# النشر على خادم افتراضي خاص (VPS)

## المقدمة

هذه هي الفصل الذي تلتقي فيه كل الأمور بالعتاد الفعلي. لديك نموذج ذهني لـ Linux (الفصل 01)،
وصور حاويات (الفصل 02)، ومكدس Compose (الفصل 03)، وباب أمامي من Nginx مُحصَّن (الفصل 04)،
وخط أنابيب CI/CD (الفصل 05) — والخادم الافتراضي الخاص (VPS) هو الحاسوب الفعلي
الذي تُشغَّل عليه هذه المكونات وتخدم المستخدمين الحقيقيين. الخادم الافتراضي الخاص هو جهاز Linux مُستأجر ذو عنوان IP عام،
وصول كامل عبر root، ولا شيء غير ذلك: لا منصة، ولا حواجز حماية، ولا أحد آخر مسؤول عنه.
هذا هو جاذبيته (رخيص، قابل للتحكم، ملكك) وهذا هو خطره أيضًا (رخيص، قابل للتحكم، *ملكك* —
بما في ذلك الأمان، والنسخ الاحتياطية، والاستدعاء في الساعة 2 صباحًا). هذا الفصل هو العملية الشاملة
لأخذ Invoicely من مستودع إلى نظام إنتاجي آمن وقابل للمراقبة يعمل على
خادم افتراضي خاص واحد: تجهيز الصندوق، وتحصين الوصول، ونشر المكدس، وتوصيل DNS وTLS،
وإعادة التشغيل دون إسقاط الطلبات، والتعافي عند تعطله.

الفكرة الأهم على الإطلاق: **الخادم الافتراضي الخاص بنية تحتية غير مُدارة، لذا فإن قابلية التكرار
والأمان وقابلية الاستعادة التي كانت ستمنحها لك منصة ما تصبح *مسؤوليتك* الصريحة —
والهدف هو خادم يمكنك إعادة بنائه من سكريبتات، وليس حيوانًا أليفًا مُعدَّلًا يدويًا
تصلي إلى الله ألا يموت أبدًا.** على منصة PaaS، تُجاب عن أسئلة "كيف أُعيد إنشاء هذا؟" و"هل SSH آمن؟" و"أين النسخ الاحتياطية؟"
بالنيابة عنك. أما على VPS، فأنت من يجيب عنها، بالكود، وإلا فلن تُجاب
حتى وقوع الكارثة. الانضباط الكامل لهذا الفصل هو تحويل آلة فريدة وهشة ومُكوَّنة يدويًا
إلى آلة *قابلة للتكرار*: تجهيز مُسَيْجَر، ونشر مؤتمت عبر خط أنابيب الفصل 05،
وإعدادات في المستودع، ومسار موثَّق *ومُختبَر* لإعادة الإعداد من الصفر.

الحكم الذي يعلِّمه هذا الفصل هو **تولَّ المسؤوليات التي كانت المنصة ستخفيها عنك.** إن نشر
VPS "يعمل" — الموقع متاح — يمكن في الوقت نفسه أن يكون مفتوحًا لـ SSH بكلمة المرور أمام العالم،
بلا جدار حماية، وبلا نسخ احتياطية، وبنشر يُسقط كل طلب قائم عند إعادة التشغيل، وبلا طريقة
للتعافي إذا مات القرص. جعله بجودة الإنتاج هو مجموعة محددة من القرارات المتراكمة على
الفصول السابقة: SSH بمفتاح فقط وجدار حماية، ومستخدم نشر غير root، ومكدس Compose
يُسحَب ويُشغَّل كقطعة أثرية (artifact)، وإعادة تشغيل بلا توقف، ونسخ احتياطية آلية لقاعدة البيانات (مع استعادة *مُختبَرة*)،
ودليل لإعادة البناء. يربط هذا الفصل المرحلة ببعضها — ويُسلِّم إلى المرحلة 9
للتحصين الأعمق، وإلى المرحلة 11 حين لا يعود خادم واحد كافيًا.

## لماذا هذا مهم

الخادم الافتراضي الخاص هو حيث تنتهي التجريدات وتصبح العواقب حقيقية:

- **"غير مُدار" تعني أن كل مسؤولية مخفية أصبحت مسؤوليتك.** تتولى المنصة بصمت تصحيح نظام التشغيل،
  وإعدادات الجدار الناري الافتراضية، وTLS، والنسخ الاحتياطية، والتعافي. يُسلمك VPS صدفة root
  وعنوان IP عام ولا يتولى شيئًا من ذلك. كل واحدة من هذه تصبح مهمة صريحة عليك تنفيذها — أو فجوة
  يجدها مهاجم أو عطل. توفير التكلفة من VPS حقيقي؛ وكذلك العبء التشغيلي.
- **الخادم المكشوف يُهاجَم خلال دقائق.** يبدأ خادم VPS جديد ذو عنوان IP عام بتلقي
  محاولات SSH brute-force وفحوصات ثغرات مؤتمتة فورًا تقريبًا. مصادقة كلمة المرور، وتسجيل دخول root،
  وعدم وجود جدار حماية ليست مخاطر نظرية — إنها إعدادات افتراضية يجري استغلالها بنشاط. أولى
  الإجراءات على أي VPS هي تحصين الوصول، قبل أن يهم التطبيق أصلًا.
- **"فقط ادخل عبر SSH واسحب" هي الطريقة التي تفسد بها عمليات النشر.** تُعيد عمليات النشر اليدوية على الخادم كل ما أزاله
  خط أنابيب CI/CD (الفصل 05): التهجير المنسي، والفرع الخاطئ، والبناء غير المُختبر،
  والطلبات المسقطة عند إعادة التشغيل العنيفة، وعدم وجود تراجع. يجب أن يكون VPS هو *هدف* النشر
  المؤتمت، لا مكانًا تسجل دخولك إليه لتشغيل أوامر من الذاكرة.
- **الخادم الأليف هو نقطة فشل واحدة غير قابلة للاستعادة.** صندوق تم تكوينه يدويًا على مدى أشهر —
  تعديلات غير موثقة، وإعدادات عُدِّلت على الهواء مباشرة، "لا تلمسه، لسنا متأكدين أنه سيعود" —
  لا يمكن إعادة بنائه عندما يتعطل القرص أو يقع حادث لدى المزود. قابلية التكرار
  (السكريبتات، والإعدادات في المستودع، ودليل إعادة البناء) هي ما يحوِّل كارثة إلى
  إزعاج بسيط.
- **لا نسخ احتياطية تعني أن لحظة سيئة واحدة تنهي العمل.** جدول محذوف، أو `down -v`، أو فشل قرص،
  أو حادثة برمجيات فدية — دون نسخ احتياطية لقاعدة البيانات *مُختبَرة*، فإن أيًا من هذه يعني
  فقدان بيانات لا يمكن استعادته. على VPS، النسخ الاحتياطية وظيفتك، والنسخة الاحتياطية التي لم تستعدها أبدًا هي
  أمل، وليست نسخة احتياطية.

أحسِنها — خادم مُسَيْجَر ومحصَّن ومؤمَّن بجدار حماية، يشغل المكدس المبني عبر CI مع
نشر بلا توقف، ونسخ احتياطية آلية مُختبرة، ودليل لإعادة البناء — ويكون VPS واحد بيئة إنتاجية
مشروعة ورخيصة وقابلة للتحكم لمنتج SaaS. أخطِئها ويصبح أصلًا مكشوفًا
غير قابل للاستعادة ويُدار يدويًا، يفشل في أسوأ وقت ممكن دون عودة.

بُعد الذكاء الاصطناعي: حين يُطلب منها "انشر هذا على خادم"، تُولِّد المساعدات المسار السعيد — تتابعًا
يجعل التطبيق يعمل — وتُغفل المسؤوليات التي كانت المنصة ستخفيها.
سيدخلون عبر `ssh` ويشغِّلون `git pull && docker compose up` (يدوي، لا تراجع، يُسقط الطلبات)،
وسيُبْقون SSH بكلمة المرور وتسجيل دخول root مُمَكَّنَيْن، وسيتجاهلون الجدار الناري، ولن يذكروا النسخ الاحتياطية أو
خطة إعادة البناء. يظهر الموقع في العرض التوضيحي؛ الخادم غير محصَّن، وغير قابل للاستعادة، ومُنشَر يدويًا.

## النموذج الذهني

VPS هو صندوق Linux خام تجعله قابلًا للتكرار ومحصَّنًا وقابلًا للاستعادة — بتراكُم المرحلة كاملة على عتاد حقيقي:

```
   A VPS = a rented Linux computer: public IP + root + NOTHING else provided.
     everything a platform hides is now YOUR job:  security · patching · backups · recovery.

   THE ORDER OF OPERATIONS (do NOT deploy before you harden)
     1. PROVISION      create the box; a non-root `deploy` user with sudo
     2. HARDEN ACCESS  key-only SSH (no passwords, no root login) · firewall (ufw: 22/80/443 only)
                          ⇒ a fresh public IP is brute-forced within MINUTES — this is step 1, not last.
     3. INSTALL RUNTIME Docker + Compose (the only real dependency; the app is images)
     4. DEPLOY         CI pushes image → server PULLS the tagged image → compose up (Ch 03/05)
     5. DNS + TLS      point the domain at the IP → Nginx + Let's Encrypt (Ch 04)
     6. BACKUPS        automated DB dumps off the box + a TESTED restore
     7. OBSERVE        logs, health, alerts (Ch 07)

   PET  vs  CATTLE   (the core VPS discipline)
     PET    = hand-configured, undocumented, edited live, irreplaceable  ← the trap
     CATTLE = provisioned by script, config in the repo, rebuildable from scratch  ← the goal
        test: "our disk died — how long to a working replacement?"  hours-from-a-runbook, not "we're doomed."

   DEPLOY = PROMOTE AN ARTIFACT, not build/pull-source on the box
     ✗ ssh in; git pull; docker build; restart      (manual, untested bytes, drops requests, no way back)
     ✓ CI built image:<sha> → server: docker compose pull → migrate → up -d → healthcheck → (rollback)

   ZERO-DOWNTIME RESTART (don't drop in-flight requests on every deploy)
     old container keeps serving → new one starts + passes healthcheck → traffic shifts → old drains → stops
        graceful SIGTERM (Ch 01/02 exec-form) lets in-flight requests finish. hard `down && up` = dropped requests.

   BACKUPS (the responsibility with no second chance)
     automated pg_dump on a schedule → stored OFF the server (object storage) → restore TESTED regularly.
        a backup you have never restored is a hope, not a backup.
```

أربعة مبادئ يحملها الفصل:

**حصِّن قبل أن تنشر.** عنوان IP عام يتعرض للهجوم فورًا؛ SSH بمفتاح فقط، وعدم تسجيل دخول root،
وجدار حماية تأتي *أولًا*، قبل التطبيق. الأمان على VPS ليس مرحلة تحصين لاحقة
(هذا العمق في المرحلة 9) — بل الحد الأدنى هو ثمن الدخول لوضع صندوق على الإنترنت.

**ابنِ ماشيةً لا حيوانات أليفة.** يجب أن يكون الخادم قابلًا للتكرار من سكريبتات وإعدادات في المستودع، حتى
يكون موت القرص أو اختراق الصندوق إعادة بناء من دليل، لا كارثة. كل تعديل يدوي على الصندوق هو
خطوة نحو حيوان أليف لا يمكن استبداله — قاومها.

**انشر بترقية قطعة أثر CI، لا بالبناء على الصندوق أبدًا.** الخادم *يسحب ويُشغِّل*
الصورة المُختبَرة المُوسومة بـ SHA (الفصل 05)، ويطبِّق التهجيرات، ويُعيد التشغيل بأناقة. إن "ادخل عبر SSH واسحب المصدر وابنِ"
يلغي كل ضمان يوفره خط الأنابيب ويُسقط الطلبات.

**اعتبر النسخ الاحتياطية والاستعادة عملًا من الدرجة الأولى.** نسخ احتياطية آلية لقاعدة البيانات خارج الصندوق و*مُختبَرة*،
ومسار إعادة بناء موثَّق، هما الفارق بين نظام قابل للاستعادة وعطل ينهي العمل.
على VPS لا أحد غيرك يقوم بهذا؛ والنسخة الاحتياطية التي لم تسترجعها لا تُحسَب.

## مثال إنتاجي

يعمل **Invoicely** في الإنتاج على VPS واحد بنظام Ubuntu 24.04 LTS (نسخة متواضعة — عدد قليل من vCPUs،
وبضعة غيغابايت من الذاكرة تكفي لـ SaaS ناشئ). المتطلب الذي يؤطِّر كل قرار: **الخادم
محصَّن قبل أن يخدم بايتًا واحدًا، وعمليات النشر تأتي فقط عبر خط أنابيب CI/CD، وكامل الصندوق — مع بياناته —
يمكن إعادة بنائه من سكريبتات ونسخ احتياطية إذا فُقد.**

دورة الحياة: سكريبت تجهيز ينشئ مستخدم `deploy` غير root، ويُثبِّت Docker وCompose،
ويُكوِّن `ufw` للسماح فقط بـ SSH/80/443، ويقفل SSH على المفتاح فقط دون تسجيل دخول root و
(من الأفضل) على منفذ غير افتراضي — كل ذلك قبل وجود التطبيق. يشير DNS لـ `app.invoicely.com` إلى
عنوان IP الخاص بالخادم؛ Nginx وCertbot (الفصل 04) يوفران TLS. النشر هو بأكمله خط أنابيب الفصل 05:
عند وسم إصدار، يدفع CI الصور الموسومة بـ SHA وتدخل خطوة النشر عبر SSH بصفة
`deploy`، ثم `docker compose pull` للصور المُختبَرة تمامًا، ويُشغِّل `alembic upgrade head` (تهجيرات التوسع/الانكماش للمرحلة 6)،
ويُشغِّل المكدس بإعادة تشغيل أنيقة، مُحقَّقة صحيتها، وقابلة للتراجع —
لا طلبات قائمة تُسقط، ولا أوامر يدوية. تُشغِّل وظيفة ليلية `pg_dump`،
وتشفِّره، وترسله إلى تخزين كائنات خارج الخادم، ووظيفة شهرية *تستعيد* أحدث
نسخة احتياطية في قاعدة بيانات تجريبية لإثبات أنها تعمل. يوثِّق `RUNBOOK.md` في المستودع إعداد
خادم بديل من الصفر. هذه آلة واحدة، تُدار كإنتاج — والمرحلة 11 هي
حيث يتخرَّج Invoicely إلى عدة عقد حين يتجاوز واحدة.

## بنية المجلدات

يُعرَّف نشر VPS عبر سكريبتات وأدلة في المستودع (ماشية، لا حيوانات أليفة) إضافة إلى التخطيط على
الخادم نفسه. أولًا، ما يوجد في نظام التحكم في الإصدارات:

```
invoicely/
├── infra/
│   ├── provision.sh            idempotent: create deploy user, install Docker, ufw, harden SSH
│   ├── backup.sh               pg_dump → encrypt → upload off-box (run by cron/timer)
│   ├── restore.sh              pull a backup → restore into a target DB (the TESTED path)
│   └── RUNBOOK.md              stand up a replacement server from zero; incident procedures
├── docker-compose.prod.yml     the stack the server runs (Chapter 03)
├── nginx/                      proxy + TLS config (Chapter 04)
├── .github/workflows/          the pipeline that deploys here (Chapter 05)
└── .env.example                the runtime config contract (real .env lives only on the server)
```

ثم التخطيط *على* الخادم — متعمَّد الحد الأدنى، لأن التطبيق صور لا ملفات:

```
/home/deploy/                   the non-root deploy user's home (never deploy as root)
├── invoicely/
│   ├── docker-compose.prod.yml  pulled from the repo at deploy time
│   ├── nginx/                   proxy config
│   └── .env                     RUNTIME secrets — chmod 600, owned by deploy (Chapter 01)
└── (Docker manages images/volumes under /var/lib/docker — the DB volume lives here)
```

لماذا هذا الشكل:

- **`infra/` تجعل الخادم قابلًا للتكرار.** سكريبتات *التجهيز* و*النسخ الاحتياطي* و*الاستعادة*
  و*الدليل* موجودة في المستودع، مُراجَعة وخاضعة للتحكم في الإصدارات — لذا يُعرَّف الخادم بالكود،
  لا بذاكرة شخص ما. هذا أهم خيار هيكلي لتجنب الحيوان الأليف.
- **`provision.sh` متطابقة (idempotent) وتجري التحصين أولًا.** إعادة تشغيلها تُقارب الصندوق إلى
  الحالة المرغوبة (آمنة للتشغيل مجددًا)، وتحصِّن الوصول قبل أي شيء آخر — الترميز يُجسِّد
  "الأمان قبل النشر."
- **`backup.sh`/`restore.sh` زوج، و`restore.sh` ليس اختياريًا.** سكريبت نسخ احتياطي وحده
  راحة زائفة؛ سكريبت الاستعادة (الذي يُشغَّل وفق جدول على قاعدة بيانات تجريبية) هو ما يثبت أن
  النسخ الاحتياطية حقيقية. يعيشون معًا حتى لا ينسى أحد النصف الثاني.
- **لا يحتفظ الخادم إلا بملف Compose وإعدادات Nginx و`.env` شبه حصرًا.** ولأن
  التطبيق يُشحن كصور (الفصل 02) والحالة تعيش في وحدة تخزين Docker، فإن نظام الملفات الخاص بالخادم
  شبه عديم الحالة — وهذا تمامًا ما يجعله قابلًا لإعادة البناء. `.env` هو السر الوحيد،
  بـ `chmod 600` ومملوك لـ `deploy`.

## التنفيذ

سكريبت التجهيز — يُشغَّل مرة واحدة على صندوق جديد، متطابق، يُحصِّن قبل أي شيء آخر:

```bash
# infra/provision.sh — run as root on a fresh Ubuntu VPS; safe to re-run.
set -euo pipefail

# 1. A non-root deploy user with sudo — the app is NEVER operated as root (Chapter 01).
id deploy &>/dev/null || adduser --disabled-password --gecos "" deploy
usermod -aG sudo deploy
install -d -m 700 -o deploy -g deploy /home/deploy/.ssh
cp /root/.ssh/authorized_keys /home/deploy/.ssh/authorized_keys   # your public key only
chown deploy:deploy /home/deploy/.ssh/authorized_keys && chmod 600 "$_"

# 2. HARDEN SSH — key-only, no root login. (A public IP is brute-forced within minutes.)
sed -i 's/^#\?PasswordAuthentication.*/PasswordAuthentication no/' /etc/ssh/sshd_config
sed -i 's/^#\?PermitRootLogin.*/PermitRootLogin no/'               /etc/ssh/sshd_config
systemctl restart ssh

# 3. FIREWALL — deny by default, allow only SSH + HTTP + HTTPS.
ufw default deny incoming && ufw default allow outgoing
ufw allow 22/tcp && ufw allow 80/tcp && ufw allow 443/tcp
ufw --force enable

# 4. Unattended security updates — the OS patching a platform would have done for you.
apt-get update && apt-get install -y unattended-upgrades && dpkg-reconfigure -f noninteractive unattended-upgrades

# 5. The only real runtime dependency: Docker + Compose. The app is images.
curl -fsSL https://get.docker.com | sh
usermod -aG docker deploy
```

النشر ذاته **ليس** سكريبتًا تشغِّله يدويًا — إنه الإجراء المركب لخط أنابيب الفصل 05 الذي يستهدف هذا الخادم.
التسلسل البعيد الذي يشغِّله، مع تفاصيل بلا توقف والتراجع التي تميِّزه عن `down && up`:

```bash
# The remote steps the CI deploy runs as `deploy@server` (from Chapter 05's composite action):
set -euo pipefail
cd /home/deploy/invoicely

PREVIOUS=$(docker compose -f docker-compose.prod.yml images -q backend || true)   # for rollback
docker compose -f docker-compose.prod.yml pull                       # the SAME tested image:<sha> (Ch 05)
docker compose -f docker-compose.prod.yml run --rm backend alembic upgrade head   # expand/contract (Stage 6)

# Rolling restart: start the new container, wait for its healthcheck, THEN retire the old one.
# `up -d` with a healthcheck + graceful SIGTERM (exec-form, Ch 02) drains in-flight requests.
docker compose -f docker-compose.prod.yml up -d --wait                # --wait blocks until healthy

curl -fsS https://app.invoicely.com/health || {                       # verify from the real edge
  echo "post-deploy health FAILED — rolling back"
  docker tag "$PREVIOUS" invoicely-backend:rollback && \
  APP_VERSION=rollback docker compose -f docker-compose.prod.yml up -d --wait
  exit 1
}
```

نسخ احتياطية آلية، خارج الصندوق، *مُختبَرة* — المسؤولية التي لا فرصة ثانية لها:

```bash
# infra/backup.sh — run nightly by a systemd timer or cron. Ships the dump OFF the server.
set -euo pipefail
STAMP=$(date +%F)
docker compose -f docker-compose.prod.yml exec -T db \
  pg_dump -U "$POSTGRES_USER" "$POSTGRES_DB" | gzip | \
  gpg --encrypt --recipient ops@invoicely.com > "/tmp/invoicely-$STAMP.sql.gz.gpg"
# Off-box storage is the point — a backup on the same disk dies with the disk.
aws s3 cp "/tmp/invoicely-$STAMP.sql.gz.gpg" "s3://invoicely-backups/db/" && rm "/tmp/invoicely-$STAMP.sql.gz.gpg"

# infra/restore.sh (run MONTHLY against a scratch DB) is what makes the above real:
#   aws s3 cp s3://invoicely-backups/db/<latest> - | gpg -d | gunzip | psql -d invoicely_restore_test
#   ...then assert row counts / a known invoice exists. An untested backup is a hope.
```

ثلاثة تفاصيل تفصل أكثر نشر VPS حقيقي عن "الموقع متاح":

- **التحصين هو الخطوات 1–3 من التجهيز، قبل التطبيق.** يُغلق SSH على المفاتيح ويُقام الجدار الناري *قبل*
  تثبيت Docker حتى. النشر أولًا والتحصين لاحقًا يعني أن الصندوق يقضي نافذة الأكثر هشاشة (الإعداد الافتراضي، عنوان IP العام) مكشوفًا للهجمات المؤتمتة التي
  تضرب خلال دقائق.
- **`up -d --wait` مع الإيقاف الأنيق هو آلية بلا توقف.** يحظر `--wait` حتى اجتياز فحص الصحة للحاوية الجديدة؛
  نقطة الدخول في صيغة exec (الفصل 02) تعني أن الحاوية القديمة تتلقى `SIGTERM` وتُصرِّف طلباتها القائمة قبل التوقف.
  إن `docker compose down && up` ساذج يقتل الحاوية القديمة فورًا — كل طلب قائم يُسقط عند كل نشر.
- **النسخة الاحتياطية حقيقية فقط لأن `restore.sh` يُشغَّل وفق جدول.** فرق لا حصر لها لديها `pg_dump` ليلي
  وتكتشف أثناء حادث أنها تفشل منذ شهور، أو أنهم لم يستعيدوا واحدة أبدًا والعملية لا تعمل.
  الاستعادة المُختبَرة هي النسخة الاحتياطية؛ التفريغ وحده ملف ذو قيمة غير مؤكدة.

## القرارات الهندسية

**حصِّن الوصول قبل نشر أي شيء.** SSH بمفتاح فقط، وعدم تسجيل دخول root، وجدار حماية رفض افتراضي
يسمح فقط بـ SSH/80/443 — كـ *أول* إجراءات على الصندوق. *السبب:* عنوان IP عام يتعرض
لهجوم مؤتمت خلال دقائق؛ الإعداد الافتراضي (مصادقة كلمة المرور، وتسجيل دخول root، وعدم وجود جدار حماية)
هو بالضبط ما تستغله تلك الهجمات. الأمان هو رسم الدخول لخادم عام، لا مرحلة لاحقة
(المرحلة 9 تتعمق أكثر؛ هذا الحد الأدنى غير قابل للتفاوض).

**شغِّل كمستخدم `deploy` غير root.** تحدث جميع عمليات النشر وتشغيل التطبيق كمستخدم غير مميز
يملك sudo مُقيَّد النطاق، لا كـ root أبدًا. *السبب:* مبدأ الامتياز الأدنى للفصل 01 على مستوى الخادم — إن
  اختُرق مسار النشر أو صدر أمر سيئ، يبقى الضرر محدودًا، لا كارثيًا. root خاص بالتجهيز لمرة واحدة،
  لا بالعمليات اليومية.

**اجعل الخادم قابلًا للتكرار: سكريبتات وإعدادات في المستودع، ودليل إعادة بناء.** التجهيز، والنسخ الاحتياطي، والاستعادة، والتعافي كلها كود ووثائق في نظام التحكم في الإصدارات. *السبب:*
الخادم المُكوَّن يدويًا حيوان أليف هو نقطة فشل واحدة غير قابلة للاستعادة؛ الخادم المُسَيْجَر والموثَّق قابل
لإعادة البناء عندما (لا لو) يموت القرص أو يُختَرق الصندوق. يجب أن يكون "ما مدى سرعة استبدالنا لهذا؟"
إجابة أفضل من "لا يمكننا."

**انشر فقط عبر خط أنابيب CI/CD، بترقية القطعة الأثرية المُختبَرة.** الخادم هو هدف
خط أنابيب الفصل 05؛ يسحب ويُشغِّل الصورة الموسومة بـ SHA، ويُهجِّر، ويُعيد التشغيل — لا `git pull` يدوي ولا بناء على الصندوق. *السبب:* عمليات النشر اليدوية على الخادم تعيد كل مخاطر أزالها CI/CD
(بايتات غير مُختبَرة، فرع خاطئ، تهجير منسي، لا تراجع). الصندوق يُشغِّل قطعًا أثرية؛ لا يبنِيها.

**أعد التشغيل بأناقة مع طرح محكوم بفحص الصحة؛ لا تُدوِّر بعنف.** `up -d --wait` مع فحوصات صحة
و`SIGTERM` أنيق، يُدحرج الحاوية الجديدة قبل تقاعد القديمة. *السبب:* إن `down && up` عنيف يُسقط
كل طلب قائم عند كل نشر — أخطاء مرئية للمستخدم كتكلفة روتينية للإصدار. عمليات الإعادة الأنيقة والمحكومة بفحص الصحة تجعل النشر غير مرئي للمستخدم وقابلة للعكس بأمان.

**أتمتة النسخ الاحتياطية خارج الصندوق واختبر الاستعادة.** `pg_dump` مشفَّر مُجدوَل إلى تخزين خارج الخادم،
إضافة إلى استعادة *مُجدوَلة* في قاعدة بيانات تجريبية. *السبب:* على VPS، النسخ الاحتياطية مسؤوليتك بالكامل،
والنسخة الاحتياطية الوحيدة التي تُحسَب هي التي استعدتها. خارج الصندوق لأن نسخة احتياطية على القرص نفسه تموت معه؛ مُختبَرة لأن نسخة تفشل بصمت أسوأ من لا شيء
(إنها ثقة زائفة).

## المفاضلات

**VPS مقابل PaaS مقابل Kubernetes المُدار.** VPS هو الأرخص والأكثر قابلية للتحكم ويعلِّمك كامل المكدس — بتكلفة
تملك الأمان والتصحيح والنسخ الاحتياطية والاستعادة. يخفي PaaS (Render, Fly, Railway) كل ذلك
مقابل سعر أعلى وتحكم أقل. Kubernetes المُدار للحالة التي تجاوزت فيها صندوقًا واحدًا وتحتاج إلى جدولة متعددة العقد — قوة هائلة وتعقيد هائل. *VPS واحد مُدار جيدًا هو
بيئة إنتاجية مشروعة لـ SaaS لفترة طويلة؛* اختر PaaS عندما تفضِّل الدفع مقابل عدم كونك مُناوَبًا على نظام التشغيل، وتوجَّه إلى التنسيق (المرحلة 11) فقط عندما
خادم واحد حقًا غير كافٍ. لا تشغِّل Kubernetes لحجم حركة تطبيق واحد.

**VPS واحد مقابل بنية تحتية متكررة.** خادم واحد بسيط ورخيص لكنه نقطة فشل واحدة —
إن مات، فأنت متوقف حتى تُعيد البناء (ولهذا النسخ الاحتياطية ودليل). التكرار
(خوادم متعددة، وموازن تحميل، وقاعدة بيانات منسوخة) يُزيل النقطة الواحدة بتكلفة
تعقيد ومال كبيرين. *ابدأ بخادم واحد إضافة إلى استعادة مُختبَرة؛* المقايضة الصادقة لـ
SaaS ناشئ هي "دقائق إلى ساعات من التوقف عند فشل نادر، وفقدان بيانات صفر" — وهو ما يوفِّره دليل ونسخ احتياطية.
Multi-node HA قرار المرحلة 11 مدفوعًا بمتطلبات توفر حقيقية، لا افتراضيًا.

**قاعدة بيانات مُدارة ذاتيًا على VPS مقابل قاعدة بيانات مُدارة.** تشغيل Postgres في مكدس Compose
مكتفٍ ذاتيًا ورخيص؛ قاعدة بيانات مُدارة (RDS, Cloud SQL, Neon) تنقل عنك النسخ الاحتياطية، وتجاوز الفشل،
والتصحيح، والاستعادة لنقطة زمنية مقابل سعر. *قاعدة بيانات Compose مقبولة للبداية — مع نسخ احتياطية حقيقية ومُختبَرة وخارج الصندوق* — لكن بيانات الإنتاج حيث تتخرَّج الفرق في الأغلب (وبحكمة)
إلى خدمة مُدارة، لأن ضمانات الاستعادة يصعب مطابقتها يدويًا. اعلم أن قاعدة البيانات المُدارة ذاتيًا هي
نقطة بداية للبيانات الحرجة، لا إجابة دائمة.

**صرامة بلا توقف مقابل بساطة النشر.** بلا توقف الحقيقي (إعادة تشغيل متدرجة محكومة بفحص الصحة،
تصريف أنيق، تهجيرات متوافقة مع الإصدارات السابقة) يحتوي أجزاء متحركة أكثر من "أوقف، اهجِّر، ابدأ."
النسخة البسيطة لها نافذة توقف قصيرة وتُخاطر بإسقاط طلبات وأخطاء متعلقة بالتهجير.
*استثمر في إعادات تشغيل أنيقة ومحكومة بفحص الصحة وفي تهجيرات التوسع/الانكماش (المرحلة 6)*
لأي شيء لديه مستخدمون حقيقيون — التعقيد متواضع والبديل هو أخطاء مرئية للمستخدم عند كل نشر. يمكن لأداة داخلية ذات مستخدم واحد
قبول النسخة البسيطة.

## الأخطاء الشائعة

**النشر قبل التحصين (أو عدم التحصين أبدًا).** يُرفع التطبيق بينما لا يزال SSH يسمح بكلمات المرور
وlogin root ولا يوجد جدار حماية، خلال النافذة الأكثر تعرضًا للصندوق. *الإصلاح:* حصِّن الوصول
(SSH بمفتاح فقط، لا root، `ufw`) كـ *أول* شيء على الخادم، قبل التطبيق.

**النشر يدويًا عبر SSH والسحب.** `git pull && docker compose up` على الخادم —
يدوي، غير مُسجَّل في الإصدارات، بايتات غير مُختبَرة، يُسقط الطلبات، لا تراجع. *الإصلاح:* انشر فقط عبر خط أنابيب CI/CD،
مع ترقية الصورة المُختبَرة.

**تشغيل كل شيء كـ root.** النشر والتشغيل كـ root، بحيث أي خطأ أو اختراق شامل. *الإصلاح:* مستخدم `deploy` غير root مع sudo مُقيَّد؛ root فقط للتجهيز لمرة واحدة.

**لا جدار حماية / كل شيء مكشوف.** منافذ الخدمات الداخلية (التطبيق، وحتى Postgres) قابلة للوصول من
الإنترنت لأنه لا يوجد جدار حماية يقيدها. *الإصلاح:* `ufw` رفض افتراضي يسمح فقط بـ SSH/80/443؛ Nginx فقط عام (الفصول 03–04).

**لا نسخ احتياطية، أو غير مُختبَرة.** إما لا يوجد `pg_dump` أصلًا، أو تفريغ ليلي لم يسترجعه أحد أبدًا
(وقد كان يفشل بصمت). *الإصلاح:* نسخ احتياطية آلية ومشفَّرة و*خارج الصندوق* **و** اختبار استعادة مُجدوَل.

**إعادة تشغيل عنيفة للمكدس عند النشر.** `docker compose down && up` يقتل الحاوية القديمة فورًا،
مُسقطًا كل طلب قائم عند كل نشر. *الإصلاح:* إعادة تشغيل متدرجة محكومة بفحص الصحة (`up -d --wait`) مع
تصريف `SIGTERM` أنيق وتهجيرات متوافقة مع الإصدارات السابقة.

**الخادم الأليف غير الموثَّق.** إعدادات عُدِّلت على الهواء، وتعديلات غير موثقة، ولا شيء مُسَيْجَر — بحيث
لا يمكن إعادة بناء الصندوق ولا يجرؤ أحد على لمسه. *الإصلاح:* سكريبتات تجهيز/نسخ احتياطي/استعادة ودليل إعادة بناء في المستودع؛ تُنشَر الإعدادات، ولا تُعدَّل يدويًا.

## أخطاء الذكاء الاصطناعي

"انشر هذا على خادم" طلب تجيب عنه المساعدات بالمسار السعيد — تشغيله — مع
إغفال المسؤوليات التي كانت المنصة ستخفيها. راجع أي نشر مُولَّد في ضوء
الأمان، وقابلية التكرار، والنسخ الاحتياطية، والاستعادة، لا "هل الموقع متاح؟"

### Claude Code: النشر اليدوي `ssh + git pull + up` دون تحصين أو تراجع

عند طلب النشر على VPS، يُنتج Claude Code عادةً تتابعًا يدخل عبر SSH، ويسحب المستودع، ويشغِّل
`docker compose up` — يجعل التطبيق يعمل، فيبدو منتهيًا، لكنه نشر يدوي لـ
مصدر غير مُختبَر دون جدار حماية، ودون تحصين SSH، ودون إعادة تشغيل أنيقة، ودون تراجع أو نسخ احتياطية.

**الكشف:** تعليمات نشر تستدعي `git clone`/`pull` و`docker build`/`up` على الخادم؛ لا
خطوة جدار حماية أو تحصين SSH؛ لا خط أنابيب CI/CD كآلية نشر؛ إعادات تشغيل `down && up`؛ لا
ذكر للنسخ الاحتياطية أو الاستعادة.

**الإصلاح:** اشترط مسؤوليات المنصة التي أصبحت الآن ملكك:

> لا تنشر عبر SSH وسحب المصدر. انشر عبر خط أنابيب CI/CD، مع سحب الصورة المُختبَرة
> الموسومة. وقبل أي من ذلك، جهِّز الخادم بالتحصين *أولًا*: SSH بمفتاح فقط، ولا
> تسجيل دخول root، وجدار حماية رفض افتراضي (SSH/80/443 فقط)، ومستخدم نشر غير root.
> أضف نسخًا احتياطية آلية لقاعدة البيانات خارج الصندوق مع استعادة مُختبَرة، وإعادة تشغيل أنيقة محكومة بفحص الصحة مع تراجع.

### GPT: منافذ مكشوفة، وتشغيل كـ root، وبلا نسخ احتياطية

عند طلب نشر كامل، غالبًا ما تكشف نماذج عائلة GPT منافذ الخدمات مباشرة (تربط التطبيق أو
حتى Postgres بـ `0.0.0.0`)، وتعمل كـ root، وتُنتج إعدادًا بلا استراتيجية نسخ احتياطي على الإطلاق —
لأن الهدف الذي تُحسِّنه هو "قابل للوصول وقيد التشغيل"، لا "آمن وقابل للاستعادة."

**الكشف:** منافذ داخلية منشورة في compose/run الإنتاجي (التطبيق/قاعدة البيانات على واجهات عامة)؛ أوامر تُشغَّل كـ root؛ لا مستخدم غير root؛ لا أتمتة `pg_dump`/نسخ احتياطي؛ لا تخزين خارج الصندوق؛ لا خطة استعادة.

**الإصلاح:** اشترط الامتياز الأدنى، والسطح المغلق، والنسخ الاحتياطية:

> Nginx فقط يجب أن يكون عامًا (80/443) — لا تكشف منافذ التطبيق أو قاعدة البيانات للإنترنت؛
> الجدار الناري يرفض افتراضيًا. شغِّل كمستخدم `deploy` غير root. وأضف قصة نسخ احتياطي حقيقية:
> `pg_dump` مؤتمت ومشفر يُشحن *خارج* الخادم، إضافة إلى اختبار استعادة مُجدوَل. النشر بلا
> نسخ احتياطية مُختبَرة ليس جاهزًا للإنتاج.

### Cursor: خادم أليف — تعديلات إعدادات على الهواء، لا شيء قابل للتكرار

عند تحرير خطوات النشر أو دليل داخل السطور، يميل Cursor إلى توجيه التحرير المباشر للإعدادات على الصندوق
(`nano /etc/nginx/...`، تعديل وإعادة تحميل) وتثبيت أشياء على نحو مخصص، مُنتجًا خادمًا فريدًا
مُكوَّنًا يدويًا بلا سكريبت وبلا مسار إعادة بناء — لأن كل تعديل محلي ولا يأخذ قابلية التكرار في الحسبان.

**الكشف:** تعليمات لتحرير إعدادات الخادم في مكانه بدلًا من نشرها من المستودع؛ `apt install` مخصص/إعداد يدوي بلا سكريبت تجهيز؛ لا دليل إعادة بناء؛ توجيهات "ادخل عبر SSH وغيِّر X"؛ إعدادات لا توجد إلا على الخادم.

**الإصلاح:** اشترط قابلية التكرار:

> اجعل الخادم قابلًا للتكرار، لا حيوانًا أليفًا. ضع التجهيز والنسخ الاحتياطي والاستعادة في سكريبتات متطابقة
> في المستودع ودليل إعادة بناء؛ انشر الإعدادات (Nginx, Compose, قالب `.env`) من نظام
> التحكم في الإصدارات بدلًا من تحريرها على الهواء على الصندوق. يجب أن أتمكن من إعداد خادم بديل من
> المستودع إضافة إلى نسخة احتياطية — اكتبه بحيث يكون ذلك صحيحًا.

## أفضل الممارسات

**حصِّن أولًا، دائمًا.** SSH بمفتاح فقط، وعدم تسجيل دخول root، وجدار حماية رفض افتراضي (SSH/80/443 فقط)،
وتحديثات أمنية تلقائية — *أول* الإجراءات على أي صندوق عام، قبل التطبيق. (تحصين أعمق: المرحلة 9.)

**شغِّل كمستخدم غير root؛ اكشف الوكيل فقط.** مستخدم `deploy` مع sudo مُقيَّد؛ الجدار الناري
وCompose يُبقيان كل شيء داخليًا عدا Nginx. امتياز أدنى وسطح عام أدنى.

**ابنِ ماشية: سكريبتات ودليل في المستودع.** تجهيز متطابق، وإعدادات تُنشَر من نظام
التحكم في الإصدارات، ودليل إعادة بناء *مُختبَر* — حتى يكون الخادم المفقود إعادة بناء، لا كارثة.

**انشر فقط عبر خط الأنابيب، بترقية القطعة الأثرية المُختبَرة.** الخادم يسحب ويُشغِّل
الصورة الموسومة بـ SHA (الفصل 05)، ويُهجِّر (المرحلة 6)، ويُعيد التشغيل بأناقة. لا `git pull`/بناء يدوي على الصندوق.

**أعد التشغيل بأناقة، وتراجع تلقائيًا.** إعادات تشغيل متدرجة محكومة بفحص الصحة (`up -d --wait`)،
و`SIGTERM` أنيق للتصريف، وتهجيرات متوافقة مع الإصدارات السابقة، وتراجع تلقائي عند فشل فحص الصحة. النشر غير مرئي للمستخدم وقابل للعكس بأمان.

**اخرج بالنسخ الاحتياطية من الصندوق واختبر الاستعادة.** `pg_dump` مشفَّر مؤتمت إلى تخزين خارج الخادم، إضافة إلى
استعادة مُجدوَلة في قاعدة بيانات تجريبية. الاستعادة المُختبَرة هي النسخة الاحتياطية. اقرنها مع المراقبة
(الفصل 07) لتعرف حالة الصندوق قبل أن يصبح حادثًا.

## الأنماط المضادة

**الصندوق غير المحصَّن.** التطبيق منشور مع SSH بكلمة مرور، وتسجيل دخول root، وبلا جدار حماية لا يزال مفتوحًا — يتعرض لـ
هجوم مؤتمت من الدقيقة الأولى. العلامة: `PasswordAuthentication yes`، `PermitRootLogin yes`، لا
`ufw`، التطبيق منشور قبل أي تحصين.

**النشر اليدوي.** الشحن عبر SSH و`git pull && up` — بايتات غير مُختبَرة، طلبات مسقطة، لا
تراجع. العلامة: "وثائق" النشر قائمة بأوامر SSH؛ "دعني فقط أسحب الأحدث على الإنتاج."

**الخادم الأليف.** صندوق فريد ومُعدَّل يدويًا وغير موثَّق لا يمكن لأحد إعادة بنائه أو يجرؤ على لمسه. العلامة:
إعدادات تُحرَّر على الهواء، ولا شيء مُسَيْجَر، "لا تُعد تشغيله، لسنا متأكدين أنه سيعود."

**سراب النسخ الاحتياطية.** تفريغ ليلي لم يُسترجع أبدًا، مخزَّن على نفس القرص الذي يُنسخ منه،
ربما يفشل بصمت. العلامة: `pg_dump` في cron بلا اختبار استعادة؛ نسخ احتياطية على الخادم
نفسه؛ "لدينا نسخ احتياطية" يُقال ولا أحد جرَّب واحدة.

**نشر بدورة عنيفة.** `docker compose down && up` عند كل إصدار، يُسقط الطلبات القائمة في كل
مرة. العلامة: لا `--wait`/بوابة فحص صحة، ولا إيقاف أنيق، والمستخدمون يرون أخطاء خلال كل نشر.

**المشغِّل root.** كل شيء يُنفَّذ كـ root — تجهيز، ونشر، وعمليات يومية — بحيث أي زلّة منهجية.
العلامة: سكريبتات نشر مليئة بأوامر root، ولا يوجد مستخدم `deploy`، والتطبيق يعمل كـ root.

## شجرة القرار

"أنا أضع تطبيقًا على VPS (أو أراجع نشر VPS) — ما الذي يجب أن يكون صحيحًا، وبأي ترتيب؟"

```
BEFORE THE APP — HARDEN (a public IP is attacked within minutes)
  Key-only SSH, no root login?                 no → fix sshd_config FIRST. this is step 1.
  Default-deny firewall, only SSH/80/443 open?  no → configure ufw before deploying.
  A non-root `deploy` user (root only for provisioning)?  no → create it.
     → do NOT proceed to deploy until these pass.

REPRODUCIBILITY — CATTLE, NOT PET
  Is provisioning an idempotent script in the repo?      no → script it.
  Is config (Nginx/Compose/.env template) deployed from version control, not edited live?  no → fix.
  Is there a tested rebuild RUNBOOK?                      no → write it. "disk died" needs an answer.

DEPLOY — PROMOTE THE ARTIFACT
  Does the server PULL the CI-built tagged image (not build/pull-source on the box)?  no → use the pipeline (Ch 05).
  Migrations run as an explicit safe step (expand/contract, Stage 6)?                 no → add it.
  Restart health-gated + graceful (up -d --wait, SIGTERM drain), with rollback?       no → make it zero-downtime.

EXPOSE — DNS + TLS
  Only Nginx public; DNS → server IP; Let's Encrypt with auto-renew (Ch 04)?  any no → fix the edge.

RECOVER — BACKUPS
  Automated DB backups, encrypted, stored OFF the server?   no → add them.
  Is the RESTORE tested on a schedule?                       no → a backup you've never restored is a hope.

OBSERVE
  Logs, health checks, alerts in place (Chapter 07)?         no → add them before you rely on the box.
  all yes → a single VPS run like production. any no → fix before real users depend on it.
```

## قائمة المراجعة

### قائمة مراجعة التنفيذ

- [ ] SSH **بمفتاح فقط** مع **تسجيل دخول root معطَّل**؛ مستخدم `deploy` غير root يُشغِّل التطبيق.
- [ ] **جدار حماية رفض افتراضي** يسمح فقط بـ SSH و80 و443؛ لا توجد منافذ خدمات داخلية مكشوفة.
- [ ] التجهيز **سكريبت متطابق في المستودع**؛ التحديثات الأمنية التلقائية مُمَكَّنة.
- [ ] الخادم **يسحب ويُشغِّل الصورة الموسومة المبنية عبر CI** (الفصل 05) — لا بناء/سحب مصدر على
      الصندوق.
- [ ] عمليات النشر تُجري التهجيرات كخطوة آمنة صريحة وتُعيد التشغيل **بأناقة** (`up -d --wait`، بوابة فحص الصحة،
      تصريف `SIGTERM`) مع **التراجع**.
- [ ] نسخ احتياطية آلية ومشفَّرة لقاعدة البيانات تُشحن **خارج الخادم**، و**الاستعادة مُختبَرة** وفق جدول.
- [ ] `.env` على الخادم بـ `chmod 600`، مملوك لـ `deploy`؛ Nginx فقط عام.

### قائمة مراجعة البنية

- [ ] الخادم **قابل للتكرار** — سكريبتات + إعدادات في نظام التحكم في الإصدارات + دليل إعادة بناء (ماشية،
      لا حيوان أليف).
- [ ] النشر يحدث **فقط عبر خط أنابيب CI/CD**، لا جلسات SSH يدوية.
- [ ] أوضاع الفشل للطبولوجيا ذات الخادم الواحد مفهومة؛ الاستعادة (نسخ احتياطية + دليل) هي
      التخفيف، مع تأجيل Multi-node HA إلى المرحلة 11.
- [ ] النسخ الاحتياطي/الاستعادة لقاعدة البيانات المُدارة ذاتيًا حقيقي، مع ملاحظة مسار انتقال إلى قاعدة بيانات مُدارة
      عندما تنمو حساسية البيانات.

### قائمة مراجعة الكود

- [ ] لا نشر قبل التحصين؛ لا SSH بكلمة مرور/root؛ لا جدار حماية مفقود (راقب عمليات النشر المولدة بالذكاء الاصطناعي).
- [ ] لا نشر `git pull`/بناء-على-الخادم يدوي؛ القطعة الأثرية المُختبَرة هي ما يعمل.
- [ ] لا تشغيل root؛ لا منافذ داخلية/قاعدة بيانات مكشوفة.
- [ ] لا إعادة تشغيل بدورة عنيفة (طلبات مسقطة)؛ أنيق، محكوم بفحص الصحة، قابل للعكس.
- [ ] النسخ الاحتياطية آلية وخارج الصندوق ومُختبَرة الاستعادة — وليست وظيفة cron لم تُجرَّب أبدًا.

### قائمة مراجعة النشر

- [ ] التحصين (SSH، جدار حماية، مستخدم غير root) مُتحقَّق منه *قبل* كشف التطبيق.
- [ ] DNS يحل إلى الخادم وTLS حي ومُجدوَل التجديد تلقائيًا (الفصل 04) قبل الإطلاق.
- [ ] **تراجع** تجريبي و**استعادة** تجريبية نُفِّذَا قبل اعتماد مستخدمين حقيقيين على
      الصندوق.
- [ ] المراقبة وفحوصات الصحة والتنبيهات (الفصل 07) نشطة؛ الدليل حيث يمكن للمُناوَب
      إيجاده.

## التمارين

**1. التجهيز والتحصين من الصفر.** على VPS جديد قابل للرمي، اكتب وشغِّل `provision.sh` متطابقًا
ينشئ مستخدم `deploy` غير root، ويقفل SSH على المفاتيح دون تسجيل دخول root، ويُكوِّن
جدار حماية رفض افتراضي — *قبل* تثبيت التطبيق. أثبت التحصين بتأكيد أن SSH بكلمة المرور
وتسجيل دخول root مرفوضان وأن 22/80/443 فقط مفتوحة. الناتج هو السكريبت ومخرجات التحقق.

**2. النشر عبر خط الأنابيب بلا توقف.** وصِّل خط أنابيب الفصل 05 لنشر مكدس Invoicely
إلى الخادم بسحب الصورة المُختبَرة، ونفِّذ إعادة تشغيل أنيقة محكومة بفحص الصحة. أثبت
بلا توقف بتشغيل مُولِّد أحمال ضد `/health` طوال عملية نشر وإظهار عدم وجود طلبات فاشلة — ثم قارن
مع `down && up` ساذج وأظهِر الطلبات المسقطة. الناتج سجلّا الطلبات.

**3. أثبت نسخك الاحتياطية باستعادتها.** نفِّذ `backup.sh` (`pg_dump` مشفر يُشحن خارج الصندوق)
و`restore.sh`، ثم حاكِ ك كارثة: دمر وحدة تخزين قاعدة البيانات، وأعد بناء البيانات من
أحدث نسخة احتياطية على خادم مُجهَّز حديثًا، مع التحقق من وجود فاتورة معروفة. زُمنها. الناتج
نسخة الاستعادة وزمن-إلى-الاستعادة.
