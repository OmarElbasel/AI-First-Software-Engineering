# CI/CD باستخدام GitHub Actions

## مقدمة

في كل مرة يُنشر فيها برنامج يدوياً بواسطة إنسان، يُنفّذ قائمة مراجعة من الذاكرة — تشغيل الاختبارات، بناء الصورة، رفعها، الاتصال عبر SSH، تطبيق الـ migrations، إعادة تشغيل الخدمات — وكل قائمة مراجعة يدوية تُنفّذ بشكل خاطئ في النهاية، في أسوأ وقت، على يد أكثر شخص مُتعب. يُستبدل هذا الطقس بأتمتة كاملة عبر CI/CD: كل دفعة (push) تُشغّل نفس الاختبارات، كل دمج (merge) يبني نفس الـ artifact، وكل إصدار (release) ينشره بالطريقة نفسها، مع نفس فحوصات الأمان، سواء كان يوم ثلاثاء هادئاً أو إصلاحاً عاجلاً ليلة الجمعة. هذا الفصل يدور حول بناء ذلك المسار (pipeline) بشكل جيد باستخدام GitHub Actions — Continuous Integration (كل تغيير يُبنى ويُختبر تلقائياً) و Continuous Delivery/Deployment (كل تغيير مقبول يُنشر تلقائياً) — لتطبيق Invoicely، محوّلين "النشر" من حدث يدوي مُقلق إلى حدث ممل، قابل للتكرار، وقابل للعكس.

الفكرة الأهم على الإطلاق: **CI/CD هو انضباط قابل للتنفيذ ومُتحكَّم في نسخته — نفس خطوات البناء والاختبار والنشر تعمل تلقائياً عند كل تغيير، لذا لا يمكن تخطي بوابات الجودة ولا يمكن ارتجال عمليات النشر.** المسار ليس ميزة توفر عليك الكتابة؛ بل هو الآلية التي تجعل "كل الاختبارات تنجح قبل الدمج" و"الإنتاج يشغل تماماً الـ artifact الذي بناه وفحصه CI" *صحيحاً بالضرورة* بدلاً من أن يكون صحيحاً عندما يتذكر الجميع. المسار الجيد يبني الـ artifact مرة واحدة ويرقّي (promote) نفس الـ artifact عبر البيئات؛ يفرض بوابات على عمليات الدمج بناءً على الاختبارات والـ lint وفحوصات الأنواع؛ يحقن الأسرار من مخزن آمن وليس من المستودع؛ و — الجزء الذي تتخطاه معظم الفرق — يمكنه *التراجع* بنفس سرعة التقدم.

الحكم الهندسي الذي يعلّمه هذا الفصل هو **أتمتة المسار كاملاً وجعله آمناً، لا سريعاً فحسب.** من السهل كتابة workflow يشغّل `pytest` ويسميه CI، أو واحد يتصل عبر SSH و `git pull` ويسميه CD. المسار الإنتاجي هو مجموعة محددة من القرارات: التخزين المؤقت (caching) ليكون المسار سريعاً بما يكفي لأن ينتظره الناس فعلياً؛ بوابة جودة حقيقية على الفروع المحمية؛ بناء الـ artifact القابل للنشر مرة واحدة ونشر *ذاك* (وليس إعادة البناء على الخادم)؛ الأسرار من المخزن المشفّر في GitHub؛ بوابة موافقة يدوية قبل الإنتاج؛ وتراجع يتم بنقرة زر وليس مشروع تنقيب أثري. يبني هذا الفصل مسار Invoicely — اختبار → بناء الصورة (الفصل 02) → رفع → نشر حزمة Compose (الفصول 03–04) إلى VPS (الفصل 06) — بكل ذلك.

## لماذا هذا مهم

المسار هو المكان الذي تُفرض فيه الجودة وحيث إما تُدار مخاطر النشر أو تُتجاهل:

- **عمليات النشر اليدوية هي العملية الأكثر تبايناً التي تقوم بها أي فريق.** إنسان يُنفّذ خطوات من الذاكرة ينسى الـ migration، ينشر الفرع الخطأ، يتخطى الاختبارات "مرة واحدة فقط"، أو يخطئ في كتابة أمر على بيئة الإنتاج. الأتمتة تجعل النشر *مطابقاً في كل مرة* — جوهر الأمر هو إزالة التباين البشري من أخطر إجراء.
- **البوابات التي تعتمد على الانضباط هي بوابات تفشل.** "نحن دائماً نُشغّل الاختبارات قبل الدمج" يظل صحيحاً حتى يحل الموعد النهائي. فحص CI إلزامي على الفرع المحمي يجعل *مستحيلاً* دمج كود فاشل — الإنفاذ ميكانيكي، لا ثقافي. بدونه، مجموعة الاختبارات مجرد ديكور.
- **البناء مرة واحدة والنشر في عدة بيئات هو ما يجعل البيئات جديرة بالثقة.** إذا بنى CI صورة، واختبر staging *تلك الصورة*، وشغّل الإنتاج *نفس الصورة عبر digest*، فإن "نجحت في staging" تعني شيئاً ذا معنى. إعادة البناء على الخادم، أو البناء بشكل منفصل لكل بيئة، تكسر السلسلة — الآن الإنتاج يشغل بايتات لم يختبرها أحد.
- **سرعة التراجع هي التحكم الحقيقي في نطاق الضرر.** كل نشر قد يكسر الإنتاج؛ ما يحدد الشدة هو مدى سرعة عودتك إلى آخر نسخة سليمة. مسار يمكنه إعادة نشر الـ artifact السابق بنقرة واحدة يحوّل الحادثة إلى انقطاع قصير لخمس دقائق؛ مسار لا يستطيع سوى التقدم للأمام يحوّلها إلى عطل وأنت تُصحح تحت الضغط.
- **الأسرار في CI هدف رئيسي للتسريب.** تحتاج المسارات إلى عناوين قواعد البيانات، وبيانات اعتماد الـ registry، ومفاتيح النشر — والطريقة الساذجة لتوفيرها (مُصمّنة بشكل ثابت في الـ workflow، ومطبوعة في السجلات) تنشرها لأي شخص يستطيع قراءة المستودع أو مخرجات البناء. تنفيذ هذا بشكل صحيح (أسرار مشفّرة، مُقنّعة في السجلات، رموز بأقل صلاحيات) جزء أمني حاسم في CI/CD.

أنجزه بشكل صحيح — مسار سريع ومُخزّن مؤقتاً يفرض بوابات على عمليات الدمج بفحوصات حقيقية، يبني artifact واحداً، يرقّيه عبر البيئات بموافقة قبل prod، ويرجع بنقرة واحدة — ويصبح النشر متكرراً ومملاً وآمناً. أفسده ويكون لديك إما مسرح اختبارات فاشلة يتجاوزه الجميع أو طقس نشر يدوي يفشل في أسوأ لحظة بلا عودة.

البُعد الخاص بالذكاء الاصطناعي: يُولّد المساعدون YAML الخاص بـ GitHub Actions بطلاقة ويُنتجون مسارات *تعمل* بينما تُفقد الأشياء التي تجعل CI/CD جديراً بالثقة — لا تخزين مؤقت (فهو بطيء جداً بحيث لا يُحترم)، أسرار ملصوقة في الـ workflow، لا حماية فرع ولا فحوصات إلزامية (فالبوابة اختيارية)، إعادة بناء عند النشر بدلاً من ترقية الـ artifact الذي تم اختباره، ولا مسار للتراجع على الإطلاق. الـ YAML صالح وعلامة الاختيار الخضراء تظهر؛ الانضباط الذي من المفترض أن يفرضه غير موجود.

## النموذج الذهني

CI/CD هو مسار آلي واحد من الـ commit إلى الإنتاج، مع بوابات وطريق للعودة:

```
   THE PIPELINE (every change takes the SAME path — no improvised deploys)

   commit / PR ──► CI ───────────────► CD ──────────────────────────► production
                   │                    │
     ┌─────────────┴───────────┐   ┌────┴──────────────────────────────────┐
     │ CONTINUOUS INTEGRATION  │   │ CONTINUOUS DELIVERY / DEPLOYMENT       │
     │ runs on every push/PR   │   │ runs on merge / tag                    │
     │  • lint + type check    │   │  • push the SAME image to the registry │
     │  • run the test suite   │   │  • [manual approval gate before PROD]  │
     │  • build the image once │   │  • deploy: pull image, migrate, restart│
     │  • scan the image       │   │  • health check → keep, else ROLL BACK │
     │  = the QUALITY GATE     │   │  = the SAFE DELIVERY of one artifact    │
     └─────────────────────────┘   └────────────────────────────────────────┘

   BUILD ONCE, PROMOTE THE SAME ARTIFACT  (never rebuild per environment)
     CI builds  invoicely-backend:<git-sha>  ──►  staging runs :<sha>  ──►  prod runs :<sha>
        the bytes tested in CI/staging are the EXACT bytes in prod. rebuilding breaks this.

   DELIVERY vs DEPLOYMENT
     continuous DELIVERY   = every accepted change is READY to deploy; a human clicks "go"
     continuous DEPLOYMENT = every accepted change deploys AUTOMATICALLY (no click)
        → most teams: auto-deploy to staging, delivery (approval) to production.

   THE FOUR THINGS A PIPELINE MUST GET RIGHT
     1. FAST (cache deps/layers) → people actually wait for it; slow CI gets bypassed
     2. GATED (required checks on protected branch) → red code CANNOT merge
     3. SECURE (secrets from the encrypted store, masked, least-privilege) → no leaks
     4. REVERSIBLE (redeploy the previous artifact in one click) → incidents stay small

   GITHUB ACTIONS VOCAB
     workflow (.github/workflows/*.yml) ─ triggered ON events (push, pull_request, tag)
        └─ job (runs on a runner; jobs parallel unless needs:)
             └─ step (an action `uses:` or a `run:` command)
        secrets → ${{ secrets.NAME }} (encrypted, masked)   environments → approval + scoped secrets
```

أربعة مبادئ يحملها الفصل:

**مسار آلي واحد، لا عمليات نشر يدوية.** كل تغيير يمر من commit → CI → CD → production عبر نفس المسار. القيمة هي *إزالة التباين البشري* من البناء والاختبار والنشر — لا ضغطات المفاتيح الموفّرة. إذا كان هناك باب خلفي يدوي، فإن ضمانات المسار لا تصمد.

**البوابة يجب أن تكون ميكانيكية.** فحوصات الجودة (الاختبارات، الـ lint، الأنواع) هي *فحوصات حالة إلزامية (required status checks)* على فرع محمي، بحيث لا يمكن دمج كود فاشل. بوابة تعتمد على تذكّر الناس لتشغيلها ليست بوابة. هكذا تصبح مجموعة الاختبارات إنفاذاً بدلاً من ديكور.

**ابنِ مرة واحدة، رقِّ نفس الـ artifact.** يبني CI صورة واحدة موسومة بـ commit SHA؛ staging والإنتاج يشغّلان *تلك الصورة بالضبط*. إعادة البناء لكل بيئة (أو على الخادم) تكسر سلسلة الحيازة التي تجعل "نجحت في staging" ذات معنى.

**النشر يجب أن يكون قابلاً للعكس وآمناً.** أي نشر قد يفشل؛ يجب أن يُرجع المسار إلى الـ artifact السابق بنقرة واحدة، ويجب أن يتعامل مع الأسرار عبر المخزن المشفّر في GitHub برموز بأقل صلاحيات — لا تكون مُصمّنة بشكل ثابت أبداً، ولا تُسجَّل في السجلات. القابلية للعكس ونظافة التعامل مع الأسرار ليسا إضافتين اختياريتين؛ بل هما ما يجعلان النشر الآلي آمناً.

## مثال إنتاجي

يُنشر **Invoicely** عبر مسار GitHub Actions يتكون من ثلاثة workflows: `ci.yml` (عند كل push و PR)، و `deploy-staging.yml` (عند الدمج إلى `main`)، و `deploy-production.yml` (عند وسم إصدار، مع بوابة موافقة). الشرط الذي يحرك التصميم: **لا يصل أي تغيير إلى `main` دون اجتياز بوابة الجودة الكاملة، والإنتاج دائماً يشغل الصورة بالضبط التي اجتازت CI و staging، وأي نشر سيئ قابل للعكس بنقرة واحدة.**

يُشغّل `ci.yml` الـ lint (`ruff`)، وفحوصات الأنواع (`mypy`)، ومجموعة الاختبارات (`pytest` ضد Postgres كـ service container)، مع تخزين مؤقت لـ pip و Docker layers ليُنهي التنفيذ في بضع دقائق؛ ثم يبني صور الـ backend والـ frontend (الفصل 02) موسومة بـ git SHA ويُشغّل فحص ثغرات. حماية الفرع على `main` تجعل هذه الفحوصات *إلزامية* — PR فاشل لا يمكن دمجه. عند الدمج، يدفع `deploy-staging.yml` الصور الموسومة بـ SHA إلى الـ registry (GitHub Container Registry) وينشر حزمة Compose (الفصول 03–04) على خادم staging، مُشغّلاً عمليات الـ migration في قاعدة البيانات (المرحلة 6) كخطوة في المسار وفاحصاً الصحة قبل إعلان النجاح. الترقية إلى الإنتاج هي إصدار موسوم يُفعّل `deploy-production.yml`، الذي يتطلب موافقة يدوية (حماية GitHub Environments) وينشر *نفس* صورة SHA التي تم التحقق منها في staging. كل سر — رمز الـ registry، مفتاح SSH للنشر، عنوان قاعدة البيانات — يأتي من أسرار GitHub المشفّرة، مفصولة حسب البيئة ومُقنّعة في السجلات. التراجع هو إعادة نشر وسم الإصدار السابق: نقرة واحدة، آخر صورة سليمة، دقائق لا ساعات. الفصل 06 هو جانب الخادم في هذا النشر.

## هيكل المجلدات

يعيش CI/CD في `.github/`، مُتحكَّماً في نسخته مثل أي شيء آخر — المسار كود، وبنيته تُبقي CI وعمليات النشر لكل بيئة والأجزاء القابلة لإعادة الاستخدام منفصلة:

```
invoicely/
├── .github/
│   ├── workflows/
│   │   ├── ci.yml                 runs on every push/PR: lint, type, test, build, scan (the GATE)
│   │   ├── deploy-staging.yml     on merge to main: push image + deploy to staging (auto)
│   │   ├── deploy-production.yml  on version tag: approval gate + deploy same image to prod
│   │   └── _build.yml             reusable workflow: build+push the image (called by the above, DRY)
│   ├── actions/
│   │   └── deploy/action.yml      composite action: the deploy+migrate+healthcheck+rollback steps
│   └── dependabot.yml             automated dependency-update PRs (they flow through the same gate)
├── docker-compose.prod.yml        what the deploy brings up on the server (Chapter 03)
├── backend/Dockerfile             the artifact CI builds (Chapter 02)
└── ...
```

لماذا هذا التخطيط:

- **workflow واحد لكل مُحفِّز/غرض، وليس workflow واحداً ضخماً.** `ci.yml` (تحقق)، نشر staging (تلقائي عند الدمج)، نشر الإنتاج (مُقيّد بوسم) اهتمامات منفصلة بمُحفِّزات وصلاحيات منفصلة. فصلها يُبقي كلاً منها مقروءاً ويسمح للإنتاج بحماية أشد من CI.
- **`_build.yml` قابل لإعادة الاستخدام و `deploy` action مُركّب يُبقيان المسار جافاً (DRY).** بناء الصورة وتسلسل النشر (رفع → migrate → إعادة تشغيل → فحص صحة → تراجع عند الفشل) متطابقان بين staging و prod؛ إخراجهما كعوامل منفصلة يعني أن منطق النشر مُعرَّف ومُثبَّت *مرة واحدة*، لا منسوخ ومنحرف بين البيئات (الدرس نفسه في DRY كما في مقتطفات Nginx في الفصل 04).
- **`dependabot.yml** يوجّه تحديثات التبعيات عبر نفس البوابة.** طلبات السحب (PRs) التلقائية للتبعيات تُشغّل مجموعة CI الكاملة قبل أن يُدمجها إنسان — إبقاء التبعيات محدّثة دون تجاوز بوابة الجودة (ممارسة أمنية تتعمق في المرحلة 9).
- **الـ workflows تشير إلى artifacts النشر، ولا تُكرّرها.** تبني الـ `Dockerfile` من الفصل 02 وتنشر `docker-compose.prod.yml` من الفصل 03 — المسار *يُنسّق* القطع التي تُعرِّفها بقية المرحلة.

## التنفيذ

workflow الـ CI — بوابة الجودة التي تعمل عند كل push و PR — مع التخزين المؤقت وقاعدة بيانات اختبار حقيقية، مشروح بـ *لماذا*:

```yaml
# .github/workflows/ci.yml
name: CI
on:
  push: { branches: [main] }
  pull_request:               # runs on every PR → this is what branch protection requires

jobs:
  test:
    runs-on: ubuntu-latest
    services:                 # a real Postgres for the tests, not mocks (Stage 8)
      postgres:
        image: postgres:16
        env: { POSTGRES_PASSWORD: postgres, POSTGRES_DB: test }
        ports: ["5432:5432"]
        options: >-
          --health-cmd pg_isready --health-interval 5s --health-timeout 3s --health-retries 5
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: "3.12"
          cache: pip          # CACHE deps → CI stays fast → people actually wait for it
      - run: pip install -r backend/requirements.txt
      - run: ruff check backend         # lint  ┐
      - run: mypy backend               # types ├ the QUALITY GATE (all required to pass)
      - run: pytest backend             # tests ┘
        env:
          DATABASE_URL: postgresql://postgres:postgres@localhost:5432/test

  build:
    needs: test                 # build ONLY if the gate passed — no artifact from red code
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }   # least-privilege token
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}     # scoped, auto-rotated — not a PAT in the YAML
      - uses: docker/build-push-action@v6
        with:
          context: ./backend
          push: true
          tags: ghcr.io/acme/invoicely-backend:${{ github.sha }}   # tag = commit SHA (promote THIS)
          cache-from: type=gha           # reuse Docker layer cache across runs (fast builds)
          cache-to: type=gha,mode=max
```

workflow نشر الإنتاج — إصدار موسوم، بوابة موافقة، نفس الـ artifact، فحص صحة مع تراجع:

```yaml
# .github/workflows/deploy-production.yml
name: Deploy Production
on:
  push: { tags: ["v*"] }        # a version tag = a release

jobs:
  deploy:
    runs-on: ubuntu-latest
    environment: production      # ← GitHub Environment: REQUIRED REVIEWER approval + scoped secrets
    steps:
      - uses: actions/checkout@v4
      - name: Deploy the SAME image that passed CI/staging
        uses: ./.github/actions/deploy       # composite action: the deploy sequence, defined once
        with:
          image_tag: ${{ github.sha }}       # promote the tested artifact, do NOT rebuild
          host: ${{ secrets.PROD_HOST }}     # all inputs from the encrypted, environment-scoped store
          ssh_key: ${{ secrets.PROD_SSH_KEY }}
```

إجراء النشر المُركّب — التسلسل الآمن، بما في ذلك التراجع الذي تُهمله معظم المسارات:

```yaml
# .github/actions/deploy/action.yml  (invoked with the image tag + host + key)
runs:
  using: composite
  steps:
    - shell: bash
      run: |
        ssh -i "$SSH_KEY" deploy@"$HOST" bash -s <<'REMOTE'
          set -euo pipefail
          PREVIOUS=$(docker inspect --format '{{.Config.Image}}' invoicely-backend || true)  # for rollback
          docker compose -f docker-compose.prod.yml pull                 # fetch the SAME tagged image
          docker compose -f docker-compose.prod.yml run --rm backend alembic upgrade head  # migrate (Stage 6)
          docker compose -f docker-compose.prod.yml up -d                # restart with the new image
          for i in {1..10}; do curl -fsS http://127.0.0.1:8000/health && exit 0; sleep 3; done
          echo "health check FAILED — rolling back to $PREVIOUS"         # ← the part teams skip
          APP_VERSION="$PREVIOUS" docker compose -f docker-compose.prod.yml up -d
          exit 1
        REMOTE
```

ثلاث تفاصيل تفصل مساراً حقيقياً عن YAML صالح لكنه أجوف:

- **`build` يحتوي على `needs: test`، لذا لا يُنتج أي artifact من كود فاشل.** ترتيب الـ jobs بحيث تُبنى الصورة فقط بعد اجتياز البوابة يعني أن الـ commit المكسور لا يمكنه حتى ترك صورة قابلة للنشر ملقاة في الأرجاء. البوابة ليست مجرد "الاختبارات اشتغلت"؛ بل "لا يحدث شيء في اتجاه التيار ما لم تنجح."
- **كل صورة موسومة بـ `github.sha`، وعمليات النشر تشير إلى ذلك الوسم — ابنِ مرة واحدة، انشر في عدة أماكن.** staging والإنتاج `pull` نفس صورة SHA؛ لا شيء يُعاد بناؤه على الخادم. هذه هي سلسلة الحيازة: البايتات التي اجتازت CI هي البايتات في الإنتاج.
- **يفحص النشر الصحة ويرجع في نفس السكريبت.** بعد إعادة التشغيل، يستطلع `/health`؛ إذا لم تظهر النسخة الجديدة، يعيد نشر الصورة السابقة ويفشل الوظيفة. نشر لا يستطيع الفشل بأمان هو نشر ينتظر أن يُسبب عطلاً — التراجع ليس اختيارياً.

## قرارات هندسية

**اجعل فحوصات الجودة إلزامية على فرع محمي.** حماية الفرع على `main` مع `test` (lint + أنواع + مجموعة) كفحص حالة إلزامي. *المبرر:* الإنفاذ يجب أن يكون ميكانيكياً — بوابة يستطيع الناس تخطيها تحت ضغط الموعد النهائي ليست بوابة. هذا ما يحول مجموعة الاختبارات من شيء ينبغي على الناس *تشغيله* إلى شيء *لا يمكن تجاوزه*.

**خزّن مؤقتاً بقوة ليكون المسار سريعاً.** خزّن تبعيات pip/npm و Docker layers (`cache: pip`، `type=gha`). *المبرر:* مسار بطيء يتم الالتفاف عليه — يدمج الناس دون انتظار، أو يُعطّلون الفحوصات "مؤقتاً". السرعة خاصية صحة في CI: كلما كان أسرع، كلما حظي باحترام أكبر. مسار مدته دقيقتان يُستخدم؛ مسار مدته 20 دقيقة يتم تجاوزه.

**ابنِ الـ artifact مرة واحدة ورقِّه عبر digest/SHA.** يبني CI صورة واحدة موسومة بـ SHA؛ كل بيئة تنشر *تلك* الصورة. *المبرر:* القيمة الكاملة للاختبار في CI/staging هي أن الإنتاج يشغل نفس البايتات. إعادة البناء لكل بيئة أو على الخادم تكسر ذلك الضمان وتُعيد "يعمل في staging، ينكسر في prod". لا تُعد البناء للنشر أبداً.

**زوّد الأسرار من المخزن المشفّر في GitHub، مفصولة ومُقنّعة.** `${{ secrets.* }}` من أسرار مفصولة حسب البيئة؛ الـ `GITHUB_TOKEN` المضمّن بأقل صلاحيات `permissions:` فوق PAT طويل العمر حيث أمكن. *المبرر:* تتعامل المسارات مع مفاتيح النشر وبيانات الاعتماد؛ تصميمها بشكل ثابت في YAML ينشرها لأي شخص لديه صلاحية قراءة المستودع، وطباعتها يُسرّبها في السجلات. الأسرار المشفّرة والمُقنّعة وبأقل صلاحيات متطلب أمني، لا تفضيل.

**احجب الإنتاج خلف موافقة يدوية؛ انشر تلقائياً إلى staging.** بيئات GitHub مع مراجع إلزامي على `production`؛ نشر تلقائي إلى staging عند الدمج. *المبرر:* يجب نشر staging باستمرار لاكتشاف المشاكل بسرعة؛ يستفيد الإنتاج من "انطلق" بشري لتوقيت الإصدار وفحص نهائي للعقلانية. هذا خيار delivery-vs-deployment يُتخذ لكل بيئة.

**اجعل التراجع مساراً من الدرجة الأولى بنقرة واحدة.** يلتقط إجراء النشر الصورة السابقة ويُعيد نشرها عند فشل فحص الصحة؛ تراجع يدوي هو إعادة نشر وسم الإصدار السابق. *المبرر:* كل نشر قد يفشل؛ تُحدد الشدة بسرعة الاستعادة. مسار لا يستطيع سوى التقدم للأمام يحوّل نشراً سيئاً إلى حادثة طويلة. صمّم طريق العودة قبل أن تحتاجه.

## المفاضلات

**النشر المستمر مقابل التسليم المستمر (نشر تلقائي مقابل بوابة موافقة).** النشر المستمر الكامل (كل دمج يصل إلى prod تلقائياً) يزيد السرعة إلى أقصى حد ويفرض تغييرات صغيرة وآمنة، لكنه يتطلب اختبارات ممتازة، ومراقبة، وتراجعاً. التسليم المستمر (موافقة قبل prod) يضيف نقطة تفتيش بشرية لتوقيت الإصدار والحكم على حساب بعض زمن الانتقال. *انشر تلقائياً إلى staging دائماً؛ أما للإنتاج، ابدأ ببوابة موافقة* وتدرّج إلى CD كامل مع نضج اختباراتك، وقابلية الملاحظة (الفصل 07)، والتراجع. لا تأتمت نشر الإنتاج بالكامل قبل أن تستطيع *كشف وعكس* نشر سيئ تلقائياً.

**السرعة مقابل الشمول في CI.** مسار سريع أدنى (lint + اختبارات وحدات) يعطي ملاحظات سريعة؛ مسار شامل (تكامل، E2E، فحوصات أمنية، إصدارات متعددة) يلتقط أكثر لكنه أبطأ وأعرَض للتذبذب. *قسّم حسب المرحلة:* شغّل البوابة السريعة (lint، أنواع، وحدات) عند كل push كفحص إلزامي، والمجموعات الأثقل (E2E — المرحلة 8، فحوصات أمنية عميقة — المرحلة 9) على جدول أو قبل النشر. أبقِ البوابة *المانعة* سريعة كي تُحترم؛ شغّل الأشياء البطيئة في مكان لن تعطّل فيه كل PR.

**البناء على الـ runner مقابل البناء على الخادم.** بناء الصور في CI (على الـ runner) يُبقي الخادم نظيفاً، ويُنتج artifact قابلاً للفحص، ويُركّز البناء؛ البناء على هدف النشر بنية تحتية أبسط لكنه يعيد البناء لكل خادم، ويربط موارد الإنتاج، ويكسر قاعدة "ابنِ مرة واحدة انشر في عدة بيئات". *ابنِ في CI دائماً*، وانشر الـ artifact الناتج — الخادم يشغل الصور، ولا يبنيها.

**GitHub Actions مقابل GitLab CI مقابل Jenkins مقابل CD المضمّن في PaaS.** GitHub Actions مدمج بشكل محكم، وله سوق ضخم، ومجاني للمستودعات العامة/الصغيرة؛ GitLab CI مشابه في بيئة GitLab؛ Jenkins أكثر مرونة وأكثر مشكلة تشغيل خاصة بك؛ PaaS (Vercel، Render) يوفر نشراً بالدفع بأقل قدر من الإعداد لكن بتحكم أقل. *Actions هو الخيار الافتراضي الصحيح عندما يكون كودك على GitHub؛* المفاهيم (البوابة، ترقية الـ artifact، الأسرار، التراجع) تنتقل إلى كل منها — لا تستثمر بشكل مفرط في لهجة YAML لأداة واحدة على حساب المبادئ.

## أخطاء شائعة

**لا فحوصات إلزامية / لا حماية فرع.** يعمل CI لكن دمج كود فاشل لا يزال ممكناً، فالبوابة استشارية ويتم تجاهلها في النهاية. *الإصلاح:* احمِ الفرع واعتبر وظيفة CI فحص حالة *إلزامياً*؛ اجعل اللون الأخضر إلزامياً للدمج.

**لا تخزين مؤقت، لذا CI بطيء ويتم تجاوزه.** كل تشغيل يُعد تثبيت التبعيات ويُعيد بناء كل Docker layers، ويستغرق أكثر من 15 دقيقة، فيتوقف الناس عن الانتظار. *الإصلاح:* خزّن التبعيات (`cache: pip`/`npm`) و Docker layers (`type=gha`) مؤقتاً؛ أبقِ البوابة المانعة لبضع دقائق.

**إعادة بناء الـ artifact عند النشر بدلاً من ترقية الذي تم اختباره.** الخادم يقوم بـ `docker build` جديدة (أو بناء CI منفصل لكل بيئة)، فالإنتاج يشغل بايتات لم تُختبر معاً أبداً. *الإصلاح:* ابنِ مرة واحدة في CI، ووسم بـ SHA، و `pull`/انشر تلك الصورة بالضبط في كل مكان.

**أسرار مُصمّمة بشكل ثابت في الـ workflow أو مطبوعة في السجلات.** بيانات اعتماد ملصوقة في YAML أو مطبوعة عبر `echo` للت debug، فتتسرّب لأي شخص لديه صلاحية الوصول إلى المستودع/السجلات. *الإصلاح:* أسرار GitHub المشفّرة، مفصولة لكل بيئة، رموز بأقل صلاحيات؛ لا `echo` للسر أبداً.

**لا مسار للتراجع.** المسار يستطيع فقط التقدم للأمام، فنشر سيئ يصبح عطلاً حتى يُعيد شخص ما بناء النسخة السابقة تحت الضغط. *الإصلاح:* تراجع بنقرة واحدة (إعادة نشر الوسم السابق) وتراجع تلقائي عند فشل فحص الصحة في خطوة النشر.

**لا بوابة موافقة على الإنتاج (أو، على العكس، بوابات على كل شيء).** إما كل دمج يصل تلقائياً إلى prod قبل أن يلتقط الفريق نشراً سيئاً، أو كل تغيير تافه يحتاج موافقة يدوية ويتوقف النشر. *الإصلاح:* انشر staging تلقائياً، واطلب موافقة للإنتاج، واضبط مع نضج الفريق.

**النشر دون تشغيل الـ migrations (أو تشغيلها بشكل غير آمن).** الكود الجديد يتوقع schema غير موجودة في قاعدة البيانات، أو تُشغَّل الـ migrations بطريقة تقفل الجدول. *الإصلاح:* شغّل الـ migrations كخطوة مسار صريحة ومرتّبة باستخدام migrations التوسيع/التقلص (expand/contract) من المرحلة 6، الفصل 06.

## أخطاء الذكاء الاصطناعي

YAML الخاص بـ CI/CD سهل على المساعدين إنتاجه وسهل إنتاجه *أجوف* — workflows صالحة تعمل لكنها لا تفرض أو تحمي شيئاً. راجع المسارات المولّدة في ضوء الأشياء الأربعة (سريع، مُقيّد ببوابة، آمن، قابل للعكس)، لا "هل الـ workflow يحمل علامة اختيار خضراء؟".

### Claude Code: مسار بلا تخزين مؤقت، ولا إنفاذ للبوابة، ولا تراجع

عند طلب "إعداد CI/CD"، يُولّد Claude Code عادةً workflows تقوم بـ lint/test/build/deploy بالتسلسل و*تعمل*، لكن بلا تخزين مؤقت للتبعيات/الـ layers (بطيئة)، ولا إشارة إلى أن الفحوصات يجب جعلها *إلزامية* عبر حماية الفرع (لذا البوابة اختيارية)، وخطوة نشر بلا فحص صحة أو تراجع — لأن "الـ workflow يعمل وينشر" يبدو مكتملاً.

**اكتشف:** لا `cache:`/`cache-from`؛ لا ذكر لحماية الفرع / الفحوصات الإلزامية؛ `run:` نشر يعيد تشغيل الخدمات بلا فحص صحة ولا تراجع؛ مسارات بطيئة؛ كود فاشل لا يزال قابلاً للدمج.

**أصلح:** اطلب الانضباط المفقود:

> اجعل هذا المسار جاهزاً للإنتاج: خزّن التبعيات و Docker layers مؤقتاً ليكون سريعاً؛ ذكّرني بجعل وظيفة CI فحص حالة *إلزامياً* مع حماية الفرع (البوابة يجب أن تكون ميكانيكية)؛ واجعل النشر يفحص الصحة بعد إعادة التشغيل ويرجع إلى الصورة السابقة عند الفشل. نشر بلا طريق للعودة ليس منجزاً.

### GPT: أسرار في الـ workflow وإعادة بناء عند النشر

نماذج عائلة GPT غالباً تلصق بيانات الاعتماد أو الإعدادات مباشرة في YAML (أو في كتلة `env:` بقيم حقيقية) وتُولّد نشراً يقوم بـ `docker build`/`git pull && build` جديدة على الخادم — وكلاهما "يعمل" بينما يُسرّب الأسرار ويكسر قاعدة "ابنِ مرة واحدة انشر في عدة بيئات".

**اكتشف:** رموز/مفاتيح/DSNs حرفية في الـ workflow أو `env:`؛ لا `${{ secrets.* }}`؛ خطوة نشر تبني الصورة (بدلاً من سحب صورة موسومة)؛ بنى مختلفة لكل بيئة؛ أسرار ظاهرة في السجلات.

**أصلح:** اطلب نظافة الأسرار وترقية الـ artifact:

> انقل كل بيانات الاعتماد إلى أسرار GitHub المشفّرة (`${{ secrets.NAME }}`)، مفصولة لكل بيئة، مع `permissions:` بأقل صلاحيات — لا شيء مُصمَّم بشكل ثابت، ولا شيء مُطبوع. ابنِ الصورة مرة واحدة في CI، ووسمها بـ commit SHA، واجعل النشر *يسحب ويشغل تلك الصورة بالضبط* — لا تُعد البناء على الخادم. البايتات المُختبرة يجب أن تكون البايتات المُنشورة.

### Cursor: تخطي بوابة الموافقة والنشر مباشرة إلى prod

عند تعديل workflow بشكل مباشر لإضافة "deploy"، يميل Cursor إلى توصيل push إلى `main` مباشرة إلى الإنتاج دون حماية بيئة أو موافقة، وغالباً دون خطوة staging — لأن التعديل المحلي هو "اجعله ينشر"، لا "اجعل النشر آمناً" — فدمج غير مراجَع يصل تلقائياً إلى prod.

**اكتشف:** وظيفة `deploy` مُحفَّزة بـ push إلى `main` تستهدف الإنتاج دون `environment:` / مراجع إلزامي؛ لا نشر staging بين الدمج و prod؛ لا ترقية/وسم يدوي للإصدارات؛ staging والإنتاج يستخدمان نفس المُحفِّز دون بوابة.

**أصلح:** اطلب بوابة البيئة ومسار staging:

> لا تنشر مباشرة إلى الإنتاج عند الدمج. انشر تلقائياً إلى *staging* عند الدمج إلى `main`، واحجب *الإنتاج* خلف بيئة GitHub مع مراجع إلزامي، مُحفَّز بوسم إصدار — ينشر نفس الصورة التي تحققت في staging. الإنتاج يحتاج "انطلق" بشري وخطوة ترقية حقيقية.

## أفضل الممارسات

**اجعل اللون الأخضر ميكانيكياً.** حماية الفرع مع وظيفة CI كفحص حالة إلزامي — الكود الفاشل لا يمكن دمجه. الإنفاذ هيكلي، لا ثقافي.

**أبقِ المسار المانع سريعاً.** خزّن التبعيات و Docker layers مؤقتاً؛ شغّل البوابة الإلزامية (lint، أنواع، وحدات) في بضع دقائق؛ ادفع المجموعات الثقيلة (E2E، فحوصات عميقة) إلى مراحل غير مانعة. CI السريع هو CI المحترم.

**ابنِ مرة واحدة، رقِّ نفس الـ artifact.** صورة واحدة موسومة بـ SHA من CI؛ staging والإنتاج تنشران *تلك* الصورة عبر الوسم/digest. لا تُعد البناء لكل بيئة أو على الخادم أبداً.

**عامل الأسرار باعتبارها حاسمة أمنياً.** أسرار مشفّرة ومفصولة حسب البيئة؛ `GITHUB_TOKEN`/`permissions:` بأقل صلاحيات؛ لا مُصمَّمة بشكل ثابت، ولا مُسجَّلة. وجّه تحديثات التبعيات (Dependabot) عبر نفس البوابة.

**انشر staging تلقائياً، احجب الإنتاج، ارجع دائماً.** نشر مستمر إلى staging؛ بوابة موافقة للإنتاج؛ تراجع بنقرة واحدة وتراجع تلقائي (مُحفَّز بفحص الصحة). شغّل الـ migrations كخطوة مسار صريحة وآمنة (المرحلة 6).

**أبقِ المسار جافاً (DRY) وفي المستودع.** workflows قابلة لإعادة الاستخدام و composite actions تُعرِّف البناء والنشر مرة واحدة؛ المسار كاملاً مُتحكَّم في نسخته ومراجَع ككود تطبيق.

## الأنماط المضادة

**البوابة الاختيارية.** يعمل CI لكن لا شيء فحص إلزامي، فيُدمج الكود الفاشل كلما كان أحدهم مستعجلاً. العلامة: لا حماية فرع؛ "الاختبارات تفشل لكنها غير ذات صلة، سأدمج على أي حال."

**المسار الجليدي.** لا تخزين مؤقت، تشغيلات مدتها 20 دقيقة، فيتوقف الناس عن الانتظار ويبدؤون التجاوز. العلامة: تبعيات تُعاد تثبيتها عند كل تشغيل؛ PRs تُدمج قبل انتهاء CI؛ "تجاوز CI هذه المرة فقط."

**إعادة البناء عند النشر.** الإنتاج يبني صورته الخاصة (أو CI يبني بشكل منفصل لكل بيئة)، فالـ prod يشغل بايتات غير مُختبرة. العلامة: `docker build`/`git pull && build` في خطوة النشر؛ "نجحت في staging لكنها انكسرت في prod."

**السر المُصمَّم بشكل ثابت.** بيانات اعتماد في YAML الخاص بالـ workflow أو كتلة `env:`، منشورة لأي شخص لديه صلاحية الوصول إلى المستودع. العلامة: رموز حرفية في `.github/workflows/`؛ أسرار في سجلات البناء.

**النشر في اتجاه واحد.** لا تراجع، فكل نشر سيئ معركة يدوية لإعادة بناء آخر حالة سليمة. العلامة: خطوة نشر تتقدم للأمام فقط؛ سؤال "كيف نعود إلى النسخة السابقة؟" يُطرح *أثناء* الحادثة.

**الدفع المباشر إلى prod.** عمليات الدمج تُنشر تلقائياً إلى الإنتاج دون staging وموافقة، فيصل خطأ فائت فوراً وبشكل واسع. العلامة: نشر الإنتاج مُحفَّز بـ push إلى `main`؛ لا حماية بيئة؛ لا خطوة إصدار/ترقية.

```
  Is that job a REQUIRED status check via branch protection? no → protect the branch. optional = ignored.
  Is `build` gated on the tests passing (needs: test)?       no → order it. no artifact from red code.

SPEED (will people actually wait for it?)
  Are dependencies AND Docker layers cached?                 no → add caching. slow CI gets bypassed.
  Is the blocking gate ~a few minutes (heavy suites deferred)? no → move E2E/deep scans off the gate.

THE ARTIFACT (build once, promote many?)
  Is ONE image built in CI, tagged by SHA?                   no → build once, tag by commit.
  Do staging AND prod deploy that SAME image (pull, not build)? no → promote the artifact; don't rebuild.

SECRETS (secure?)
  All credentials from ${{ secrets.* }}, scoped per env, masked, least-privilege token?
     no → move them to encrypted secrets; never hardcode or echo.

DELIVERY (safe promotion?)
  Auto-deploy to staging on merge; production behind an approval (environment + reviewer, tag trigger)?
     no → add the staging step and the production gate.
  Migrations run as an explicit, safe pipeline step (Stage 6)?  no → add it.

REVERSIBILITY (can you get back?)
  Does the deploy health-check and auto-roll-back on failure?   no → add it.
  Is manual rollback one click (redeploy previous tag)?         no → make it one.
  all yes → shipping is fast, gated, secure, and reversible. any no → fix before you rely on it.
```

## قائمة المراجعة

### قائمة مراجعة التنفيذ

- [ ] يُشغّل CI الـ lint وفحوصات الأنواع ومجموعة الاختبارات (ضد service DB حقيقية) عند كل push/PR.
- [ ] التبعيات **و** Docker layers مُخزَّنة مؤقتاً؛ البوابة المانعة تنتهي في ~بضع دقائق.
- [ ] تُبنى صورة واحدة في CI، موسومة بـ **commit SHA**؛ وظيفة البناء مُقيَّدة بنجاح الاختبارات.
- [ ] staging والإنتاج **تنشران نفس الصورة الموسومة** (سحب، لا إعادة بناء على الخادم).
- [ ] كل الأسرار تأتي من أسرار GitHub المشفّرة، مفصولة حسب البيئة ومُقنّعة؛ الرموز بأقل صلاحيات (`permissions:`)، لا شيء مُصمَّم بشكل ثابت أو مُطبوع.
- [ ] تُشغَّل عمليات الـ migration في قاعدة البيانات كخطوة مسار صريحة ومرتّبة وآمنة (المرحلة 6).
- [ ] يفحص النشر الصحة بعد إعادة التشغيل و**يرجع** إلى الصورة السابقة عند الفشل.

### قائمة مراجعة البنية

- [ ] للفرع المحمي وظيفة CI كفحص حالة **إلزامي** (اللون الأخضر إلزامي للدمج).
- [ ] staging تُنشر تلقائياً عند الدمج؛ الإنتاج محجوب خلف **مراجع إلزامي** (بيئة GitHub) ومُحفِّز وسم إصدار.
- [ ] منطق البناء والنشر جاف (workflows قابلة لإعادة الاستخدام / composite actions)، مُعرَّف مرة واحدة.
- [ ] تحديثات التبعيات (Dependabot) تمر عبر نفس البوابة.
- [ ] المسار كاملاً مُتحكَّم في نسخته ومراجَع ككود تطبيق.

### قائمة مراجعة الكود

- [ ] لا تخزين مؤقت مفقود (CI بطيء)؛ لا فحص إلزامي/حماية فرع مفقودة (بوابة اختيارية) — راقب workflows المولّدة بالذكاء الاصطناعي.
- [ ] لا إعادة بناء عند النشر؛ صورة SHA المُختبرة هي ما يُنشر.
- [ ] لا أسرار مُصمَّمة بشكل ثابت/مطبوعة؛ الأسرار مفصولة ومُقنّعة؛ الرموز بأقل صلاحيات.
- [ ] لا دفع مباشر إلى prod دون staging + موافقة؛ لا نشر دون مسار تراجع.
- [ ] الـ migrations خطوة متعمّدة وآمنة، لا ضمنية أو مُتخطاة.

### قائمة مراجعة النشر

- [ ] الصورة المُنشورة على الإنتاج هي SHA بالضبط التي اجتازت CI و staging (عبر digest).
- [ ] تم اختبار بوابة موافقة الإنتاج والتراجع (قم بتراجع تجريبي قبل أن تحتاجه).
- [ ] تم التحقق من أمان الـ migrations (توسيع/تقلص — المرحلة 6) وتشغيلها قبل/مع النشر كما هو مصمم.
- [ ] فحوصات الصحة بعد النشر والمراقبة (الفصل 07) تؤكد الإصدار قبل إعلانه منجزاً.

## التمارين

**1. ابنِ البوابة وأثبت أنها ميكانيكية.** أعدّ `ci.yml` (lint + أنواع + اختبارات مع Postgres كخدمة وتخزين مؤقت) لـ backend تطبيق Invoicely، ثم فعّل حماية الفرع لجعله فحصاً إلزامياً. أثبت أن البوابة تعمل بفتح PR يفشل في اختبار والتأكد من أنه *لا يمكن* دمجه، وأن PR ناجح يمكن دمجه. الـ artifact هو لقطة شاشة الـ PR الفاشل وإعدادات الحماية.

**2. رقِّ artifact واحد عبر البيئات.** وسّع المسار لبناء صورة موسومة بـ SHA مرة واحدة، ونشرها تلقائياً إلى بيئة staging عند الدمج، ونشر *نفس* الصورة إلى الإنتاج خلف بوابة موافقة على وسم إصدار. أثبت — عبر digest — أن صورة الإنتاج مطابقة بايتاً بايت لتلك التي اجتازت CI. الـ artifact هو الـ workflow والـ digests المتطابقة.

**3. اكسر نشراً وراقبه يتراجع.** نفّذ خطوة النشر بالفحص الصحي والتراجع، ثم انشر صورة مكسورة عمداً (تفشل في `/health`) وتأكد من أن المسار يكتشف الفشل ويُعيد تلقائياً نشر النسخة السابقة — مع عدم تقديم الإنتاج للبناء المكسور أبداً. الـ artifact هو سجل النشر الذي يُظهر فشل فحص الصحة والتراجع.
