# معالجة الأخطاء وإصدارات الـ API

## المقدمة

يتناول هذا الفصل موضوعين يبدوان غير مرتبطين، وفي الواقع هما نفس الموضوع:
الحفاظ على متانة عقد (contract) الـ API عبر الزمن. **معالجة الأخطاء** هي كيفية
تصرّف الـ API عند حدوث خطأ — إذ إن شكل الأكواد الخاصة بالأخطاء ورموزها جزء من
العقد الذي يعتمد عليه العملاء، تمامًا كما هي الاستجابات الناجحة. **إصدارات
الـ API** هي كيفية تطوّر العقد دون كسر العملاء المبنيين عليه أصلًا. كلاهما
يتعلّق بالوفاء بوعد للمستهلكين عبر الزمن.

ظهرت معالجة الأخطاء في جميع أنحاء الكتيب كاستثناءات (exceptions) خاصة بالنطاق
المجال تُرفع في الخدمة وتُترجَم إلى HTTP عند الحدود (Stage 2؛ Stage 3، الفصول
01–04). يجعل هذا الفصل ذلك منهجيًا: عقد خطأ واحد متّسق عبر الـ API بأكمله،
وفصل نظيف بين حالات الفشل *المتوقعة* (خطأ تحقق، عنصر غير موجود) وحالات الفشل
*غير المتوقعة* (خلل، انقطاع في خدمة أساسية)، والقاعدة الصارمة بأن التفاصيل
الداخلية لا تتسرّب أبدًا إلى العملاء. ثم تتناول الإصدارات المشكلة الأصعب
والأبطأ: الـ API عقد مع مستهلكين غالبًا لا تتحكّم فيهم، لذا فإن تغييره دون
عناية يكسرهم — وأفضل استراتيجية إصدارات هي في معظمها عن *عدم الحاجة إلى
إصدار* من خلال التطوّر بشكل تراكمي (additive).

الرؤية الجامعة هي أن الـ API الإنتاجي التزام عام طويل الأمد. يجب أن تكون
أخطاؤه قابلة للتنبؤ وآمنة، ويجب أن يكون تطوّره متوافقًا مع الإصدارات السابقة
(backward-compatible) بشكل افتراضي، لأنه على الطرف الآخر من كل endpoint يوجد
عميل — واجهة أمامية، تطبيق جوّال، تكامل خاص بالعميل — ينكسر حين ينكسر العقد.

## لماذا هذا مهم

تُعدّ معالجة الأخطاء والإصدارات المجال الذي يكشف فيه الـ API ما إذا كان قد
بُني لعرض توضيحي أم لعملاء حقيقيين على مدى سنوات.

تفشل معالجة الأخطاء السيئة بثلاث طرق مكلّفة. فهي **تسرّب التفاصيل الداخلية** —
تتبّعات المكدس (stack traces)، أخطاء SQL، نصوص الاستثناءات المُعادة إلى
العملاء — وهذا غير احترافي ويُعدّ إفصاحًا أمنيًا يمنح المهاجمين خريطة
للمكدّس والمخطّط لديك. وهي **غير متّسقة** — كل endpoint يخترع شكل خطأ خاصًا
به — لذا يجب على كل عميل معالجة كل endpoint بشكل خاص، ولا يستطيع أي عميل
معالجة الأخطاء بشكل عام. وهي **تخلط بين حالات الفشل المتوقعة وغير المتوقعة**
— خطأ العميل في المدخلات وخلل الخادم يبدوان متماثلين — فلا يستطيع العملاء
التفاعل بشكل صحيح، ولا يستطيع مراقبتك (monitoring) التمييز بين "خطأ مطبعي
من المستخدم" و"قاعدة البيانات متوقفة".

تفشل الإصدارات السيئة ببطء أكبر وبشكل أكثر علنية. الـ API عقد، وعندما تعيد
تسمية حقل، أو تحذف واحدًا، أو تغيّر نوعًا، أو تشدّد قاعدة تحقق، فإن كل
مستهلك بُني على الشكل القديم ينكسر — وبالنسبة لـ API عام فهؤلاء هم تكاملات
العملاء التي لا تستطيع إصلاحها. بدون استراتيجية إصدارات، أنت تختار بين عدم
تحسين الـ API أبدًا وكسر المستخدمين لديك؛ ومع استراتيجية، يمكنك التطوّر مع
الوفاء بما وعدت به.

يظهر بُعد الـ AI على الجانبين. يُعيد المساعدون `str(exception)` مباشرةً إلى
العميل (تسرّب داخلي)، ويُترجمون كل فشل إلى 500 عام (أو الأسوأ، 4xx مضلّل)،
ويخترعون شكل خطأ مختلفًا لكل endpoint، وبالنسبة للإصدارات — يُغيّرون عقد
endpoint موجود في مكانه دون إدراك أن التغيير كاسر. ينتج عن كل ذلك عرض
توضيحي عامل وـ API هشّ أو مُسرّب أو مكسور العقد في الإنتاج.

## النموذج الذهني

تنقسم الأخطاء إلى نوعين يجب معالجتهما بشكل مختلف، ويجب أن يكون العقد
شكلًا واحدًا متّسقًا لكليهما:

```
   AN ERROR OCCURS
        │
        ├── EXPECTED (a normal outcome the domain models)
        │     validation failed · not found · conflict · unauthorized · rate-limited
        │     └─► specific 4xx + a stable error CODE + a safe, human message
        │         (the client can react programmatically; this is part of the contract)
        │
        └── UNEXPECTED (a bug, a downstream outage — you did not model this)
              an unhandled exception, a database down, a null you didn't expect
              └─► 500, generic message to the client, FULL detail LOGGED server-side
                  with a request id, and an ALERT. Never leak the detail outward.

   ONE ERROR CONTRACT for both (RFC 9457 problem+json):
     { "type", "title", "status", "detail", "code", "request_id" }
   consistent across every endpoint — clients handle errors generically.
```

```
   API EVOLUTION OVER TIME
        │
        ├── NON-BREAKING (additive): new optional field, new endpoint, looser validation
        │     └─► just ship it. No version bump. (Tolerant clients ignore new fields.)
        │
        └── BREAKING: remove/rename a field, change a type, tighten validation,
            change semantics
              └─► for consumers you don't control: NEW VERSION + deprecate the old
                  (expand → migrate clients → contract — same pattern as DB migrations)
                  for internal-only clients: coordinate and evolve, less ceremony
```

تحمل الفصل أربعة مبادئ:

**التمييز بين حالات الفشل المتوقعة وغير المتوقعة.** حالات الفشل المتوقعة جزء
من النطاق المجال — خطأ تحقق، مورد مفقود، تعارض — وتحصل على 4xx محدد مع رمز
مستقر يستطيع العميل التفرّع عليه. حالات الفشل غير المتوقعة هي أخطاء برمجية أو
انقطاعات لم تُنمذج — تحصل على 500، ورسالة عامة للعميل، وتسجيل كامل من جانب
الخادم بالإضافة إلى تنبيه. معالجة خلل برمجي كـ 4xx تُخفيه عن المراقبة؛ معالجة
نتيجة متوقعة كـ 500 تُربك العملاء.

**لا تسرّب التفاصيل الداخلية أبدًا؛ أرفق دائمًا معرّف ارتباط (correlation ID).**
يحصل العملاء على رسالة نظيفة ورمز مستقر، وليس أبدًا تتبّع مكدس أو SQL أو نص
استثناء. تُسجَّل التفاصيل من جانب الخادم مقابل `request_id` يُعاد *أيضًا* إلى
العميل، حتى يتمكّن المستخدم من ذكر المعرّف ويمكّن الدعم من العثور على الفشل
بالضبط — تشخيص دون إفصاح.

**عقد خطأ واحد عبر الـ API بأكمله.** كل خطأ، من كل endpoint، له نفس الشكل
(معيار RFC 9457 `problem+json` هو الافتراضي المعقول)، حتى يعالج العملاء الأخطاء
بشكل عام بدلًا من معاملة كل endpoint بشكل خاص. يُفرض ذلك من خلال معالجات
استثناءات *مركزية*، وليس من خلال `try/except` لكل endpoint.

**التطوّر بشكل تراكمي؛ لا تُصدر إلا عند التغييرات الكاسرة التي لا يمكن
تجنّبها.** إضافة حقل اختياري أو endpoint جديد لا يكسر أحدًا ولا يحتاج إلى
إصدار. الإزالة أو إعادة التسمية أو إعادة الكتابة أو التشديد تكسر العملاء —
لذا تجنّبها قدر الإمكان (التطوّر التراكمي، القراء المتسامحون)، وعندما يتعذّر
ذلك، قدّم إصدارًا جديدًا وأهم الإصدار القديم بجدول زمني مُعلن. الإصدارات
هي الملاذ الأخير عندما يستحيل التطوّر التراكمي، وليست الأداة الأولى.

تعريف عملي:

> **معالجة الأخطاء تجعل الفشل جزءًا من عقد متّسق وآمن — الأخطاء المتوقعة
> كـ 4xx مرمّزة، وغير المتوقعة كـ 500 مُسجَّلة ومُنبَّه عنها، دون تسرّب داخلي.
> الإصدارات تُدير تطوّر العقد — تراكمية بشكل افتراضي، وإصدار جديد فقط عندما
> يكون التغيير الكاسر للمستهلكين الذين لا تتحكّم فيهم أمرًا لا مفرّ منه.
> كلاهما يحافظ على وعد الـ API لعملائه عبر الزمن.**

## مثال إنتاجي

تمتلك **Invoicely** دافعين قسريين لهذا الفصل. داخليًا، تُرفع endpoints الخاصة
بها أخطاء نطاق مجال (من الفصول 01–04) تحتاج إلى أن تصبح استجابات HTTP متّسقة
وآمنة بدلًا من تسرّب الاستثناءات. خارجيًا، تُبرز Invoicely **API عامًا** ليتمكّن
العملاء من بناء تكاملات (التكاملات المشابهة لـ QuickBooks من المراحل السابقة
تعني وجود مستهلكين من طرف ثالث) — ما يعني أن عقدها أصبح شيئًا تعتمد عليه
شركات أخرى ولا يمكن كسره بشكل عابر.

سنبني عقد خطأ واحدًا (`problem+json`) مع تسلسل هرمي لاستثناءات النطاق المجال
ومعالجات مركزية تُترجم الأخطاء المتوقعة إلى استجابات 4xx مرمّزة، وتمسك بكل
ما هو غير متوقع كـ 500 عام مُسجَّل ومُنبَّه عنه مع معرّف طلب. ثم سنُعدّ
إصدارات بمسار URL لـ API العام ونعمل على تطوّر حقيقي: إعادة تسمية `total` إلى
`amount_due` — تغيير كاسر — يتم بالطريقة الصحيحة من خلال التطوّر التراكمي، وعند
الضرورة القصوى، إصدار جديد مع إهمال. يُظهر المثال شقّي "الحفاظ على وعد
العقد": أخطاء يمكن للعملاء الاعتماد عليها، وتغييرات لا تكسرهم.

## بنية المجلدات

```
core/
├── errors.py             # domain-exception hierarchy (AppError + subclasses)
├── error_handlers.py     # centralized handlers → one problem+json contract
├── problem.py            # the RFC 9457 problem+json response model
└── request_id.py         # middleware: assign/propagate a correlation id
app/
├── main.py               # register exception handlers + version routers
└── api/
    ├── v1/               # version 1 routers (the public contract clients built against)
    └── v2/               # version 2 (only when a breaking change forced it)
```

سبب هذا الشكل:

- **`core/errors.py`** يحتوي على تسلسل هرمي واحد للاستثناءات حتى يرفع النطاق
  المجال أخطاء ذات معنى (`NotFoundError`، `ConflictError`) دون معرفة بـ HTTP.
- **`core/error_handlers.py`** يُركّز الترجمة إلى HTTP في مكان واحد، ليكون
  عقد الخطأ موحّدًا ولا يُنشئه أي endpoint يدويًا.
- **`core/problem.py`** يُعرّف شكل الاستجابة الوحيد الذي يستخدمه كل خطأ.
- **`api/v1/` و `api/v2/`** يجعلان الإصدارات حدًّا هيكليًا — يبقى v1 مستقرًا
  للمستهلكين الحاليين بينما يحمل v2 التغييرات الكاسرة، ولا يوجد v2 إلا عندما
  يفرض تغيير كاسر وجوده فعلًا.

## التنفيذ

**التسلسل الهرمي لاستثناءات النطاق المجال (`core/errors.py`).** قاعدة واحدة،
مع فئات فرعية تحمل حالة HTTP، رمزًا آليًا مستقرًا، ورسالة آمنة. يرفع النطاق
المجال هذه الاستثناءات؛ ولا يلمس HTTP أبدًا.

```python
class AppError(Exception):
    status_code: int = 500
    code: str = "internal_error"

    def __init__(self, message: str = "An error occurred") -> None:
        self.message = message
        super().__init__(message)


class ValidationError(AppError):
    status_code = 422
    code = "validation_error"


class NotFoundError(AppError):
    status_code = 404
    code = "not_found"


class ConflictError(AppError):
    status_code = 409
    code = "conflict"
```

**عقد الخطأ الواحد (`core/problem.py`).** RFC 9457 `problem+json` — شكل
معياري يستخدمه كل خطأ.

```python
from pydantic import BaseModel


class Problem(BaseModel):
    type: str = "about:blank"      # a URI identifying the error type
    title: str                     # short, human, stable
    status: int
    detail: str                    # safe, human message — NEVER internal detail
    code: str                      # stable machine code clients branch on
    request_id: str                # correlation id, also logged server-side
```

**معالجات مركزية — متوقع مقابل غير متوقع (`core/error_handlers.py`).**
معالجان يقومان بكل العمل: أحدهما يُترجم `AppError` المعروفة إلى استجابات 4xx
مرمّزة؛ والآخر الشامل يُحوّل *أي شيء* غير متوقع إلى 500 عام مع تسجيل
التفاصيل الكاملة ومعرّف الطلب. لا تُنشئ أي endpoint استجابات الخطأ بنفسها.

```python
import logging
from fastapi import FastAPI, Request
from fastapi.responses import JSONResponse
from app.core.errors import AppError
from app.core.problem import Problem

logger = logging.getLogger("app")


def register_error_handlers(app: FastAPI) -> None:

    @app.exception_handler(AppError)
    async def handle_app_error(request: Request, exc: AppError) -> JSONResponse:
        # EXPECTED: a modeled domain error → specific status + stable code + safe message.
        problem = Problem(
            title=exc.code.replace("_", " ").title(),
            status=exc.status_code, detail=exc.message, code=exc.code,
            request_id=request.state.request_id,
        )
        return JSONResponse(problem.model_dump(), status_code=exc.status_code,
                            media_type="application/problem+json")

    @app.exception_handler(Exception)
    async def handle_unexpected(request: Request, exc: Exception) -> JSONResponse:
        # UNEXPECTED: log the FULL detail server-side (with request id) + this fires an alert;
        # return a GENERIC message. The client learns nothing internal.
        logger.exception("unhandled error", extra={"request_id": request.state.request_id})
        problem = Problem(
            title="Internal Server Error", status=500,
            detail="An unexpected error occurred.",   # generic — no stack trace, no SQL
            code="internal_error", request_id=request.state.request_id,
        )
        return JSONResponse(problem.model_dump(), status_code=500,
                            media_type="application/problem+json")
```

**معرّف الارتباط (`core/request_id.py`).** كل طلب يحصل على معرّف، يُعاد
إلى العميل ويُرفق بكل سطر سجل — تشخيص دون إفصاح. (يُتعمّق في التتبّع
في الفصل 08.)

```python
import uuid
from starlette.middleware.base import BaseHTTPMiddleware


class RequestIdMiddleware(BaseHTTPMiddleware):
    async def dispatch(self, request, call_next):
        request.state.request_id = request.headers.get("X-Request-ID", str(uuid.uuid4()))
        response = await call_next(request)
        response.headers["X-Request-ID"] = request.state.request_id   # client can quote it
        return response
```

مع هذا في مكانه، لم تعد endpoints من الفصول 01–04 بحاجة إلى `try/except`
الخاص بها لترجمة أخطاء النطاق المجال — فما عليها سوى رفع `NotFoundError` أو
`ValidationError`، ويُنتج المعالج المركزي العقد الموحّد. يصبح أي خلل لم
تتوقّعه 500 نظيفًا (مُسجَّل، مُنبَّه عنه، مع معرّف)، لا تتبّع مكدس مُسرَّبًا.

**الإصدارات — تطوّر تراكمي، وإصدار فقط عند الإلزام.** يُعيد API Invoicely
العام `total` للفاتورة. يريد المنتج إعادة تسميته إلى `amount_due`. هذه
إعادة تسمية **كاسرة** — فكل تكامل يقرأ `total` سيُكسر. الخيارات، من الأرخص
إلى الأكلف:

```python
# ADDITIVE (preferred): add the new field, keep the old. Breaks no one.
class InvoiceReadV1(BaseModel):
    id: int
    total: Decimal                      # kept for existing clients
    amount_due: Decimal | None = None   # new; old clients ignore it (tolerant reader)

# Only if the old field MUST go, introduce v2 and deprecate v1.
# app/api/v1/invoices.py  — unchanged contract, plus a deprecation signal
@router_v1.get("/invoices/{id}")
async def get_invoice_v1(...) -> InvoiceReadV1:
    response.headers["Deprecation"] = "true"
    response.headers["Sunset"] = "Wed, 31 Dec 2025 23:59:59 GMT"  # published removal date
    response.headers["Link"] = '</v2/invoices>; rel="successor-version"'
    ...

# app/api/v2/invoices.py — the clean, breaking contract
class InvoiceReadV2(BaseModel):
    id: int
    amount_due: Decimal                 # renamed; v1 clients unaffected because v1 still exists
```

ترتيب التفضيل هو الدرس: أولًا حاول جعل التغيير **تراكميًا** (أضف
`amount_due` إلى جانب `total`)، لأن التغيير الذي لا يكسر أي عميل لا يحتاج
إلى إصدار ولا إلى مراسم. فقط عندما يجب حقًا أن يموت الشكل القديم تدفع
ثمن إصدار جديد — وحتى حينها تُبقي v1 حيًّا مع ترويسة `Deprecation`/`Sunset`
وجدول زمني مُعلن، حتى ينتقل المستهلكون بإشعار لا على صباح مكسور. الإصدارات
هي الملاذ المكلف؛ التطوّر التراكمي هو الافتراضي.

## القرارات الهندسية

تُعرِّف خمسة القرارات معالجة الأخطاء والإصدارات.

### ما صيغة استجابة الخطأ؟

**الخيارات:** (1) شكل خطأ مخصّص؛ (2) معيار RFC 9457 `problem+json`؛ (3) أي
شيء يصدف أن يُعيده كل endpoint.

**المقايضات:** الشكل المخصّص مقبول إذا كان متّسقًا لكنه يعيد اختراع مشكلة
محلولة ولا يتكامل مع شيء. `problem+json` معيار منشور يفهمه العملاء والأدوات،
بحقول للنوع، والعنوان، والحالة، والتفاصيل، ومجال لرمز. الأشكال لكل endpoint
هي الكارثة — لا يستطيع أي عميل معالجة الأخطاء بشكل عام.

**التوصية:** اعتمد `problem+json` (RFC 9457) كعقد وحيد، ممدودًا برمز آلي
مستقر `code` و`request_id`. الشكل المعياري لا يكلّف شيئًا فوق الشكل
المخصّص ويشتري قابلية التشغيل البصري والاتّساق. قبل كل شيء، يجب أن يكون
*شكلًا واحدًا*، مطبَّقًا من خلال معالجات مركزية.

### كيف تُعالَج الأخطاء المتوقعة وغير المتوقعة؟

**الخيارات:** (1) ترجمة كل شيء إلى خطأ عام؛ (2) التمييز بين أخطاء النطاق
المجال المنمذجة والاستثناءات غير المتوقعة.

**المقايضات:** معالج عام واحد بسيط ويُتلف المعلومات — لا يستطيع العميل
التفريق بين فشل تحقق وانقطاع خدمة، ويبدو الخلل الحقيقي كخطأ من العميل.
التمييز بينهما يعني تسلسلًا هرميًا لأخطاء النطاق ومعالجًا شاملًا، لكنه
يعطي سلوك عميل صحيح (تفرّع على رمز) وعمليات صحيحة (الأخطاء البرمجية تطفو
على السطح كـ 500 تُنبّه).

**التوصية:** ميّز بينهما. أخطاء النطاق المتوقعة ← 4xx محدد مع رمز مستقر
ورسالة آمنة؛ الاستثناءات غير المتوقعة ← 500، رسالة عامة، سجل كامل من جانب
الخادم، وتنبيه. هذا التقسيم هو ما يسمح للعملاء بالتفاعل بشكل صحيح
ولك بالتفريق بين "خطأ المستخدم" و"خللنا" في المراقبة.

### كم من التفاصيل يصل إلى العميل؟

**الخيارات:** (1) إعادة الاستثناء/تتبّع المكدس للمساعدة في التصحيح؛ (2)
إعادة رسالة نظيفة بالإضافة إلى معرّف ارتباط، وتسجيل التفاصيل من جانب الخادم.

**المقايضات:** إعادة التفاصيل مريحة في التطوير، وفي الإنتاج تُعدّ ثغرة إفصاح
معلومات (تتبّعات مكدس، SQL، مسارات داخلية) وغير احترافية. الرسالة النظيفة
زائد المعرّف تمنح المستخدم شيئًا قابلًا للتنفيذ يذكره، وتبقي التفاصيل
التشخيصية حيث لا يراها سواك — على حساب إعداد للتسجيل والربط.

**التوصية:** لا تسرّب التفاصيل الداخلية أبدًا إلى العملاء؛ أَعِد رسالة
آمنة، ورمزًا مستقرًا، و`request_id` مُسجَّل أيضًا. يحلّ الدعم المشاكل من
المعرّف؛ ولا يتعلّم العميل شيئًا قابلًا للاستغلال. فعِّل الأخطاء المفصّلة
في التطوير فقط، أبدًا في الإنتاج.

### أي استراتيجية إصدارات؟

**الخيارات:** (1) مسار URL (`/v1/`، `/v2/`)؛ (2) تفاوض عبر الترويسة/المحتوى
(`Accept: application/vnd.api.v2+json`)؛ (3) معامل استعلام (query parameter)؛
(4) لا إصدارات (تطوّر تراكمي فقط).

**المقايضات:** مسار URL هو الأكثر وضوحًا وقابلية للتخزين المؤقت وسهولة في
التوجيه والاختبار، على حساب عناوين URL تُرمِّز إصدارًا. التفاوض عبر الترويسة
يُبقي عناوين URL نظيفة وأكثر "RESTful" لكنه أصعب في الاختبار والتخزين
المؤقت والاكتشاف. معاملات الاستعلام بسيطة لكن محرجة. عدم الإصدارات يعمل
فقط إذا استطعت إبقاء كل تغيير تراكميًا.

**التوصية:** افتراضيًا، **تطوّر تراكمي دون رفع إصدار**، وعندما يكون التغيير
الكاسر أمرًا لا مفرّ منه، استخدم **إصدارات بمسار URL** لعمليّتها — مرئية،
قابلة للتخزين المؤقت، قابلة للاختبار بسهولة، وواضحة للمستهلكين من طرف ثالث.
احتفظ بالتفاوض عبر الترويسة للفرق التي تريد تحديدًا عناوين URL نظيفة
وقادرة على تحمّل تكلفة الأدوات.

### متى تُصدر على الإطلاق — داخلي مقابل عام؟

**الخيارات:** (1) أصدِر كل شيء بصرامة؛ (2) أصدِر APIs العامة فقط، وطوّر
الداخلية بالتنسيق.

**المقايضات:** إصدار كل شيء آمن ومكلف — فالعملاء الداخليون الذين تنشرهم
معًا نادرًا ما يحتاجون إلى هذه المراسم. عدم إصدار API عام أمر طائش — فلا
يمكنك تنسيق نشر مع تكاملات العملاء. لدى الجمهورين احتياجات مختلفة حقًا.

**التوصية:** لـ APIs **الداخلية** (العملاء الذين تتحكّم فيهم وتنشرهم معًا)،
تطوّر تراكميًا ونَسِّق التغييرات الكاسرة مع نشر متزامن — حد أدنى من مراسم
الإصدارات. لـ APIs **العامة** (مستهلكون من طرف ثالث لا يمكنك التنسيق معهم)،
أصدِر بصرامة وأَهمِل بجدول زمني مُعلن. اعرف أي نوع من المستهلكين لكل
endpoint، لأن ذلك يحدّد مقدار صرامة الإصدارات التي يحتاجها.

## المقايضات

تقايض هذه الممارسات الصلابة بالجهد، وبعضها سياقي.

**عقد خطأ متّسق يكلّف هيكلة مسبقة.** التسلسل الهرمي للاستثناءات، والمعالجات
المركزية، وسباكة معرّف الارتباط هي إعداد تدفع ثمنه مرة واحدة — مبالغة لسكريبت
يُرمى، وضروري لأي شيء لديه عملاء حقيقيون. العائد أن كل endpoint يحصل على
أخطاء آمنة وموحّدة مجانًا بعد ذلك؛ التكلفة هي السقالة الأولية التي قد لا
تحتاجها أداة داخلية صغيرة.

**إخفاء التفاصيل عن العملاء يقايض راحة التصحيح بالسلامة.** الأخطاء النظيفة
زائد معرّف الارتباط تعني أن المطوّر لا يستطيع قراءة الفشل مباشرة من الاستجابة
ويجب أن يبحث عنه بالمعرّف — أبطأ قليلًا في التصحيح، وأكثر أمانًا بشكل كبير.
هذا هو المقايض الصحيح للإنتاج دائمًا؛ التخفيف من الاحتكاك هو تسجيل جيد
مُفهرس بمعرّف الطلب (الفصل 08).

**الإصدارات تقايض الرشاقة بالاستقرار.** تتيح لك استراتيجية الإصدارات التطوّر
دون كسر العملاء، على حساب الحفاظ على إصدارات متعددة — كل إصدار مدعوم هو
كود يجب إبقاؤه يعمل، واختباره، وإهماله في النهاية. لهذا يُفضَّل التطوّر
التراكمي: يمنح الاستقرار *بدون* عبء صيانة الإصدارات المتعددة. لا تُصدر إلا
عندما لا يستطيع التطوّر التراكمي التعبير عن التغيير حقًا.

**التطوّر التراكمي فقط يكدّس الحمل.** الإضافة دائمًا والحذف أبدًا يُبقي
العملاء عاملين وينفخ الـ API ببطء بحقول مُهمَلة وتوافقية مُلحَقة. في مرحلة
ما، يصبح الإصدار الجديد *هو* القرار الصحيح للتخلّص من الوزن المتراكم —
الحكم هو أن التغييرات الكاسرة مكلّفة بما يكفي لتأجيلها، لا لتجنّبها إلى
الأبد. أَهمِل وأَنهِ الخدمة بوعي بدلًا من حمل كل شيء إلى ما لا نهاية.

## الأخطاء الشائعة

**تسرّب التفاصيل الداخلية في الأخطاء.** إعادة تتبّعات المكدس، SQL، أو نصوص
الاستثناء إلى العملاء — ثغرة إفصاح معلومات وغير احترافي. الإصلاح: رسالة
نظيفة + رمز مستقر + معرّف طلب للعميل؛ التفاصيل الكاملة مُسجَّلة من جانب
الخادم.

**أشكال أخطاء غير متّسقة.** كل endpoint يُعيد جسم خطأ مختلفًا، فلا يستطيع أي
عميل معالجة الأخطاء بشكل عام. الإصلاح: عقد `problem+json` واحد عبر معالجات
مركزية، لا إنشاء خطأ لكل endpoint.

**الخلط بين الأخطاء المتوقعة وغير المتوقعة.** ترجمة كل شيء إلى 500 عام، أو
خلل حقيقي إلى 4xx، فلا يستطيع العملاء والمراقبة التمييز بين خطأ المستخدم
وخلل الخادم. الإصلاح: أخطاء النطاق المنمذجة ← 4xx مرمّزة؛ الاستثناءات غير
المتوقعة ← 500، مُسجَّلة ومُنبَّه عنها.

**ابتلاع الاستثناءات.** التقاط واسع وإعادة نجاح (أو الاستمرار في حالة فاسدة)،
وإخفاء حالات الفشل (Stage 1، Chapter 01). الإصلاح: دع الأخطاء غير المتوقعة
تصل إلى المعالج الشامل؛ لا تُخفِها خلف `except` عارٍ.

**كسر العقد في مكانه.** إعادة تسمية، أو حذف، أو إعادة كتابة، أو تشديد حقل
على endpoint موجود دون رفع إصدار، مما يكسر العملاء الذين لا تتحكّم فيهم.
الإصلاح: تطوّر تراكمي حيثما أمكن؛ وإلا إصدار جديد مع إهمال.

**إصدار كل شيء، أو عدم إهمال شيء.** مراسم على APIs داخلية كان يمكنها التطوّر
تراكميًا، أو إصدارات عامة تُبقي حيّة إلى الأبد لأن الإهمال لا يُنفَّذ أبدًا.
الإصلاح: تراكمي افتراضيًا؛ أصدِر التغييرات الكاسرة العامة؛ أَهمِل الإصدارات
القديمة بجدول زمني مُعلن ومُنفَّذ.

## أخطاء الـ AI

تُعدّ معالجة الأخطاء والإصدارات مخاوف عقد، وعيوب العقد لا تكون مرئية حتى
يضربها عميل *مختلف* — وهو ما لا يفعله المطوّر الذي يختبر مساره السعيد
الخاص. يُحسّن المساعد الاستجابة العاملة ويترك العقد مُسرَّبًا أو غير متّسق
أو مكسورًا بهدوء. راجع الكود المُولَّد لما يختبره العملاء والمشغّلون عند
حدوث خطأ أو تغيير.

### Claude Code: تسرّب التفاصيل الداخلية إلى العميل

عند طلب معالجة الأخطاء، يُعيد Claude Code بشكل متكرر الاستثناء مباشرةً إلى
المُستدعي — `detail=str(exc)`، أو تتبّع الأخطاء (traceback) الخاص بـ FastAPI
في وضع التصحيح، أو رسالة خطأ في قاعدة البيانات — لأنه الطريقة الأكثر مباشرة
لإظهار "ماذا حدث خطأ" ويبدو مفيدًا في التطوير. في الإنتاج يُفصح عن المكدس،
والمخطّط، والمسارات الداخلية لأي شخص يُحفّز خطأ.

**اكتشاف:** `str(exception)` أو كائن استثناء في جسم الاستجابة؛ نص خام لخطأ
قاعدة البيانات أو الإطار مُعادٍ إلى العملاء؛ وضع التصحيح/تتبّعات مفصّلة يمكن
الوصول إليها في الإنتاج.

**الإصلاح:** اشترط أخطاء آمنة بالإضافة إلى تفاصيل من جانب الخادم:

> لا تُعِد أبدًا رسائل الاستثناء، أو تتبّعات المكدس، أو أخطاء قاعدة البيانات
> إلى العميل. ترجم أخطاء النطاق المتوقعة إلى رسالة آمنة ورمز مستقر؛ وللأخطاء
> غير المتوقعة أَعِد رسالة عامة و`request_id`، وسجّل التفاصيل الكاملة من
> جانب الخادم. الأخطاء المفصّلة مخصصة للتطوير فقط.

### GPT: الخلط بين حالات الفشل المتوقعة وغير المتوقعة

تميل نماذج عائلة GPT إلى توجيه كل خطأ عبر معالج عام واحد — غالبًا 400 أو
500 شامل لكل شيء — بحيث يكون خطأ تحقق العميل، والمورد المفقود، والخلل
الحقيقي في الخادم، غير قابلين للتمييز لكل من العميل والمراقبة. تختبئ
الأخطاء الحقيقية بين الأخطاء المتوقعة، ولا يستطيع العملاء التفرّع على النتيجة.

**اكتشاف:** معالج شامل واحد يُترجم كل الأخطاء إلى حالة واحدة؛ أخطاء النطاق
والاستثناءات غير المتوقعة تُعالَج بنفس الطريقة؛ الأخطاء البرمجية تظهر كـ
4xx (فلا تُنبّه)، أو النتائج المتوقعة تظهر كـ 500 (فتُنبّه جهة الاستجابة
لحادثة بسبب خطأ مطبعي من المستخدم).

**الإصلاح:** اشترط التقسيم:

> ميّز بين أخطاء النطاق المتوقعة والاستثناءات غير المتوقعة. الأخطاء المنمذجة
> (التحقق، غير موجود، تعارض) تُعيد 4xx محددًا مع رمز مستقر؛ الاستثناءات غير
> المتوقعة تُعيد 500، تُسجَّل بتفاصيل كاملة، وتُطلِق تنبيهًا. لا تُترجم خللًا
> إلى 4xx ولا نتيجة متوقعة إلى 500 أبدًا.

### Cursor: تغييرات كاسرة صامتة للعقد

عند تعديل endpoint موجود داخليًا، يُعيد Cursor تسمية حقل استجابة، أو يغيّر
نوعًا، أو يحذف حقلًا، أو يُشدد قاعدة تحقق — مما يكسر كل عميل بُني على
الشكل القديم — لأنه من موقع التعديل يبدو التغيير تحسينًا محليًا ولا توجد
إشارة مرئية إلى أن الـ endpoint عقد عام.

**اكتشاف:** إعادة تسمية/حذف/تغيير نوع/حقل مطلوب جديد على endpoint موجود؛
تشديد التحقق على نموذج طلب؛ تغيير دلالات — كل ذلك دون إصدار جديد ودون
إهمال.

**الإصلاح:** عامل العقد كقاعدة ثابتة:

> هذا الـ API عقد يعتمد عليه العملاء. فضّل التغيير التراكمي (أضف حقلًا
> اختياريًا جديدًا، وأبقِ القديم). التغيير الكاسر — إعادة تسمية، حذف، إعادة
> كتابة، أو تشديد — يتطلب إصدار API جديدًا مع إبقاء القديم وإهماله بجدول
> زمني مُعلن. لا تُغيِّر عقد endpoint موجود في مكانه.

## أفضل الممارسات

**عقد خطأ واحد، مطبَّق مركزيًا.** اعتمد `problem+json` (RFC 9457) مع
`code` و`request_id` مستقرين، تُنتجها معالجات استثناء مركزية — أبدًا
لا أجسام خطأ لكل endpoint — حتى يعالج كل عميل الأخطاء بنفس الطريقة.

**قسِّم بين المتوقع وغير المتوقع، وتَنَبَّه على غير المتوقع.** أخطاء
النطاق المنمذجة تصبح استجابات 4xx مرمّزة؛ الاستثناءات غير المتوقعة تصبح
500s تُسجَّل بالكامل ويُنبَّه عنها. هذا ما يجعل كلًا من العملاء والمراقبة
صحيحين.

**لا تسرّب التفاصيل الداخلية أبدًا؛ أَعِد دائمًا معرّف ارتباط.** يحصل
العملاء على رسالة آمنة ورمز مستقر؛ يُسجَّل تتبّع المكدس وSQL والتفاصيل
الداخلية من جانب الخادم مقابل `request_id` يُعاد إلى العميل ليتمكّن الدعم
من التتبّع (الفصل 08).

**طوِّر APIs تراكميًا.** أَضف حقولًا اختيارية وendpoints جديدة بحرية؛ عامل
الحذف وإعادة التسمية وإعادة الكتابة والتشديد باعتبارها كاسرة. التغيير
الذي لا يكسر أي عميل لا يحتاج إلى إصدار — هذا أرخص وأفضل تطوّر.

**أصدِر التغييرات الكاسرة العامة، وأَهمِل بجدول زمني.** عندما يكون التغيير
الكاسر للمستهلكين الذين لا تتحكّم فيهم أمرًا لا مفرّ منه، قدّم إصدارًا جديدًا
(مسار URL)، وأبقِ القديم حيًّا مع ترويسات `Deprecation`/`Sunset` وتواصل، وأَزله
فعليًا في التاريخ المُعلن. سجّل قرارات الإصدارات والإهمال في ADR
([`templates/adr.md`](../../templates/adr.md))؛ وثّق عقد الخطأ في `CLAUDE.md`.

## الأنماط المضادة

**الـ 500 المُسرِّب.** تتبّعات المكدس، SQL، أو نصوص استثناءات مُعادة إلى
العملاء — ثغرة إفصاح معلومات متنكّرة في زي خطأ "مفيد". العلامة: `str(exc)`
في استجابة، أو تتبّع إنتاجي يمكن الوصول إليه بتحفيز خطأ.

**خطأ ندفة الثلج.** شكل خطأ مختلف لكل endpoint، فيعامل العملاء كلًا منها
بشكل خاص ولا يمكن أن توجد معالجة أخطاء عامة. العلامة: `{"error": ...}` هنا،
`{"message": ...}` هناك، `{"detail": ...}` في موضع آخر.

**المعالج غير المميِّز.** جميع الأخطاء تُوجَّه إلى حالة واحدة، فتُخفى
الأخطاء البرمجية بين النتائج المتوقعة وتُنبَّه جهة الاستجابة لحوادث بسبب
أخطاء مطبعية من المستخدمين. العلامة: `except` وحيد يُترجم كل شيء إلى
400 أو 500.

**الإصدار عن طريق الكسر.** تغيير عقد endpoint موجود في مكانه — إعادة
تسمية، حذف، إعادة كتابة، تشديد — وكسر كل عميل بُني عليه. العلامة: حقل
أُعيدت تسميته على endpoint عام حي دون v2 ودون إهمال.

**الإهمال الأبدي.** إصدارات قديمة تُبقي حيّة إلى الأبد لأن الإنهاء لا
يُنفَّذ أبدًا، فيتراكم الـ API بالإصدارات وبحمل التوافقية إلى ما لا نهاية.
العلامة: `/v1` مُهمل منذ ثلاث سنوات ولا يزال مدعومًا بالكامل، دون تاريخ
إزالة مُحترم.

## شجرة القرار

"حدث خطأ ما، أو أُغيِّر الـ API — كيف أحافظ على العقد؟"

```
AN ERROR OCCURRED
│
├─ Is it an EXPECTED domain outcome? (validation / not found / conflict / unauthorized)
│    └─► specific 4xx + stable machine code + safe message (problem+json).
│        The client can branch on the code.
│
└─ Is it UNEXPECTED? (a bug, a downstream outage, an unhandled exception)
     └─► 500 + GENERIC message + request_id to the client.
         LOG the full detail server-side (with request_id) and ALERT.
         Never leak stack traces, SQL, or internals outward.

CHANGING THE API
│
├─ Is the change ADDITIVE? (new optional field, new endpoint, looser validation)
│    └─► ship it. No version bump. Clients tolerate new fields.
│
└─ Is it BREAKING? (remove / rename / retype / tighten / change semantics)
     ├─ Internal clients you deploy together? ─► coordinate a synchronized change.
     └─ Public / third-party consumers? ─► NEW VERSION (URL path). Keep the old,
          add Deprecation + Sunset headers, communicate, and remove on the published date.
```

## قائمة التحقّق

### قائمة التحقّق من التنفيذ

- [ ] All errors use one `problem+json` contract with a stable `code` and a `request_id`, produced by centralized handlers.
- [ ] Expected domain errors map to specific 4xx codes; unexpected exceptions map to a logged, alerted 500.
- [ ] No internal detail (stack trace, SQL, exception string) is ever returned to a client.
- [ ] Every request has a correlation ID, returned to the client and attached to logs.
- [ ] API changes are additive where possible; breaking public changes go through a new version.
- [ ] Deprecated versions carry `Deprecation`/`Sunset` headers and a published removal date.

### قائمة التحقّق من البنية

- [ ] Error handling is centralized (handlers + hierarchy), not scattered as per-endpoint `try/except`.
- [ ] The error contract is documented and consistent across every endpoint.
- [ ] Public vs internal APIs are identified, and versioning rigor matches (strict for public, additive for internal).
- [ ] A deprecation/sunset process exists and is actually executed, not perpetual.
- [ ] Error-contract and versioning conventions are in `CLAUDE.md`.

### قائمة التحقّق من مراجعة الكود

- [ ] No exception string / stack trace / SQL is returned to the client (watch AI diffs).
- [ ] Errors use the shared contract, not an ad-hoc per-endpoint shape.
- [ ] Expected and unexpected errors are distinguished (no bug mapped to 4xx, no expected outcome to 500).
- [ ] No breaking change to an existing endpoint's contract without a new version + deprecation.
- [ ] New endpoints emit the standard error contract and a correlation ID.

### قائمة التحقّق من النشر

- [ ] Verbose/debug errors are disabled in production; only safe messages leave the service (see [`checklists/production-readiness.md`](../../checklists/production-readiness.md)).
- [ ] Unexpected-error alerts are wired to on-call, keyed by request ID.
- [ ] Deprecated API versions have monitored usage, so you know when it's safe to sunset.
- [ ] Sunset dates are tracked and enforced, not indefinitely extended.

## التمارين

**1. مركزة الأخطاء.** خذ مجموعة من endpoints يبني كل منها استجابات الخطأ
الخاصة به (وأحدها يُعيد `str(exc)`)، واستبدلها بتسلسل هرمي لاستثناءات النطاق
بالإضافة إلى معالجات مركزية تُنتج عقد `problem+json` واحدًا مع معرّف طلب.
المخرج هو قبل/بعد بالإضافة إلى تحفيز 500 يُظهر رسالة عامة للعميل والتفاصيل
الكاملة في السجلات تحت نفس معرّف الطلب.

**2. إجراء تغيير كاسر بأمان.** خذ `GET /invoices/{id}` العام في Invoicely
الذي يُعيد `total` وطوّره إلى `amount_due` بطريقتين: أولًا تراكميًا (كلا
الحقلين)، ثم — بافتراض أنه يجب إزالة `total` — عبر endpoint v2 مع إبقاء v1
وإهماله. المخرج هو النهجان وتوصية من فقرة واحدة بأيهما يُستخدم ولماذا.

**3. صنف التغييرات.** بالنظر إلى قائمة تغييرات API مقترحة — إضافة حقل
`notes` اختياري، إعادة تسمية `status` إلى `state`، جعل `customer_id` مطلوبًا،
إضافة endpoint جديد، تغيير `total` من نص إلى رقم — صنف كل واحد منها باعتباره
كاسرًا أو غير كاسر وبيّن كيف ستُطلقه. المخرج هو التصنيف؛ والهدف هو استبطان
أي التغييرات يحتاج إصدارًا وأيها لا يحتاج.

## قراءات إضافية

- **RFC 9457 — Problem Details for HTTP APIs** (rfc-editor.org) — عقد الخطأ
  المعياري الذي يتبنّاه هذا الفصل، ويُعدِّل RFC 7807. قصير ويستحق القراءة
  كاملة؛ إنه الشكل الذي يجب أن تتّخذه أخطاؤك.
- **API Design Patterns** (JJ Geewax, Manning)، الفصول حول الإصدارات
  والتوافقية العكسية — معالجة صارمة لما يُعدّ تغييرًا كاسرًا وكيفية الإصدار
  والإهمال؛ أعمق مرجع لشق الإصدارات في هذا الفصل.
- **Stripe API versioning** (stripe.com/blog/api-versioning، ووثائق الـ API) —
  دراسة الحالة المعيارية في تطوّر API عام دون كسر العملاء، بما في ذلك
  الإصدارات المبنية على التواريخ وكيف يحافظون على التوافقية. اقرأها لمعرفة
  كيف يتعامل API عام جدّي مع هذا عمليًا.
- **Google API Improvement Proposals — Errors (AIP-193)** (google.aip.dev) —
  معيار عملي ذو رأي واضح لاستجابات الخطأ والرموز على نطاق واسع؛ مرجع عرضي
  مفيد لـ RFC 9457 عند تصميم عقد الخطأ لديك.
