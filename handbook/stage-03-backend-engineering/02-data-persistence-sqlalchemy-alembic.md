# Data Persistence: SQLAlchemy & Alembic

## Introduction

This chapter is about the machinery that turns a request into durable state and
back: SQLAlchemy for mapping Python objects to PostgreSQL rows and managing the
database session, and Alembic for evolving the schema over time without losing
data. These are the tools every Stage 2 chapter quietly relied on; now we build
the persistence layer deliberately and correctly.

The scope is deliberately narrow. This is *application* persistence — how to
define models, manage sessions and transactions, load data efficiently, and ship
schema changes safely. It is **not** database engineering: data modeling,
indexing strategy, transaction isolation levels, and query optimization are a
specialist discipline that belongs to Stage 6, and this chapter defers to it. The
question here is not "how should this data be modeled?" but "how do I persist and
migrate it from a FastAPI application without corrupting state, blocking the event
loop, or taking the database down during a deploy?"

Those failure modes are the point. Persistence is where a backend's most expensive
bugs live — a session shared across requests and leaking data between users, a
migration that locks a table and causes an outage, an N+1 query that works on ten
rows and melts on ten thousand. None of them show up in a happy-path test, and all
of them are things an assistant generates by default. This chapter is about
getting the mechanics right so the layers above can trust that state is durable,
isolated, and evolvable.

## Why It Matters

The database is the one part of a system where mistakes are permanent. A bug in a
request handler is fixed by a redeploy; a migration that drops a column, or a race
that double-charges a customer, is fixed by restoring backups and apologizing.
Persistence code carries the highest blast radius in the backend, which is exactly
why its correctness under real conditions — concurrency, failure, scale, and
deployment — matters more than its correctness in a demo.

Three concerns dominate:

- **Session and transaction lifecycle.** A SQLAlchemy session is a unit of work
  and is *not* safe to share across concurrent requests. Get its scope wrong — one
  global session, or a session that outlives its request — and you get data leaking
  between users, mysterious state corruption, and connection exhaustion. The
  session must be per-request, and the transaction boundary must be exactly one
  commit per use case (Stage 2, Chapter 05).
- **Query efficiency.** The ORM makes it trivially easy to write a loop that issues
  one query per iteration — the N+1 problem — which is invisible at test scale and
  catastrophic in production. Loading related data correctly is a persistence
  concern, not a database-tuning one, because the fix lives in how you write the
  query.
- **Safe schema evolution.** Every schema change on a live system is a deployment
  event that can lock tables, break the running version of the app, or lose data.
  Migrations must be reviewed, reversible where possible, backward-compatible with
  the currently-deployed code, and tested against realistic data — not
  autogenerated and applied on faith.

The AI dimension is sharp on all three. Assistants reach for a module-level global
session (the classic training-data pattern), write ORM loops that N+1 without a
second thought, and autogenerate Alembic migrations that they present as done —
including destructive or table-locking operations — because the migration "ran" on
an empty local database. Each is a production incident that a manual test cannot
surface.

## Mental Model

Persistence has three moving parts, each with a rule that prevents a class of
disaster:

```
   1. THE SESSION — a unit of work, scoped to ONE request
   ┌──────────────────────────────────────────────────────────────┐
   │  request ─► open session ─► [ do work ] ─► commit ─► close      │
   │             (one per request; NEVER shared across requests)     │
   │  the session tracks changes; ONE commit per use case at the edge│
   └──────────────────────────────────────────────────────────────┘

   2. LOADING RELATIONSHIPS — decide how related data is fetched
        lazy (default)  ─► one query per access  ─► N+1 under a loop  ✗
        eager (explicit)─► one query with a join/IN ─► loads together ✓

   3. MIGRATIONS — every schema change is a reviewed, deployable event
        model change ─► generate migration ─► REVIEW it ─► test on real-ish data
        ─► deploy (backward-compatible with the running app) ─► apply
        expand → migrate → contract  (never a breaking change in one step)
```

Three principles carry the chapter:

**The session is per-request and short-lived.** It is created when a request
starts, used to do that request's work, committed once at the edge of the use case
(Stage 2, Chapter 05), and closed when the request ends. It is never a global, never
shared between concurrent requests, and never held open across I/O it doesn't need.
Getting this wrong is the single most dangerous persistence mistake because it
corrupts data across users silently.

**Lazy loading is a loaded gun.** By default the ORM fetches a relationship the
moment you access it, so accessing `invoice.line_items` inside a loop over invoices
issues one query per invoice — the N+1 problem. It works in tests (ten rows, ten
queries, fast) and destroys production (ten thousand rows, ten thousand queries).
Loading related data efficiently is a decision you make explicitly, per query, not
a default you inherit.

**Migrations are deployments, and schema changes must be non-breaking.** A
migration runs against live data while a running version of the app still expects
the old schema. A change that renames or drops a column in one step breaks the
running app and cannot be rolled back cleanly. The safe pattern is
**expand-migrate-contract**: add the new thing, migrate to it, and only remove the
old thing in a later deploy once nothing uses it. Autogenerated migrations are a
draft, never a finished artifact.

A working definition:

> **Application persistence is managing the session as a per-request unit of work
> with one commit per use case, loading related data deliberately to avoid N+1, and
> evolving the schema through reviewed, backward-compatible migrations. Deep data
> modeling and query tuning are Stage 6.**

## Production Example

**Invoicely** needs its persistence layer built: the async database engine and
per-request session, the invoice and line-item models with their relationship, a
repository query that loads invoices *with* their line items without N+1, and the
Alembic setup that ships a real schema change — adding a `due_date` column to
invoices — the safe, backward-compatible way.

This is the layer underneath everything Stage 2 built. The models and repository
are familiar from those chapters; what this chapter adds is the parts that were
assumed: how the session is created and scoped, how the engine is configured for
production, how to load relationships without melting under scale, and how to evolve
the schema on a running system. The `due_date` migration is the worked example of
expand-migrate-contract, because "add a column" is the most common migration and the
one most often done unsafely.

## Folder Structure

```
core/
├── db.py                 # engine + async session factory + get_session dependency
└── config.py             # DATABASE_URL and pool settings
modules/invoicing/
├── _models.py            # SQLAlchemy models: Invoice, LineItem + relationship
└── _repository.py        # queries — including eager-loaded reads (no N+1)
migrations/               # Alembic
├── env.py                # wires Alembic to the models' metadata
├── alembic.ini
└── versions/
    └── 0007_add_invoice_due_date.py   # a reviewed, backward-compatible migration
```

Why this shape:

- **`core/db.py`** owns the engine and session lifecycle in one place — the engine
  is an application-lifetime singleton (Stage 2, Chapter 06), the session is
  per-request. Centralizing this is what prevents scattered, mis-scoped sessions.
- **`_models.py`** holds the ORM models and their relationships; the relationship
  configuration (and the loading strategy) lives with the model it describes.
- **`_repository.py`** is where queries — including the *eager-loading* decisions —
  live, so N+1 avoidance is encapsulated with the data access (Stage 2, Chapter 05).
- **`migrations/`** is Alembic's directory; each file in `versions/` is a reviewed
  deployment artifact, numbered and ordered, not an incidental generated file.

## Implementation

**The engine and per-request session (`core/db.py`).** The engine (and its
connection pool) is created once at startup; the session is created per request and
always closed, via a FastAPI dependency.

```python
from collections.abc import AsyncGenerator
from sqlalchemy.ext.asyncio import AsyncSession, async_sessionmaker, create_async_engine
from app.core.config import settings

# Engine + pool: created ONCE at startup (application-lifetime singleton).
engine = create_async_engine(
    settings.DATABASE_URL,
    pool_size=settings.DB_POOL_SIZE,      # bounded pool — don't exhaust Postgres
    max_overflow=settings.DB_MAX_OVERFLOW,
    pool_pre_ping=True,                    # detect dropped connections
)

_session_factory = async_sessionmaker(engine, expire_on_commit=False)


async def get_session() -> AsyncGenerator[AsyncSession]:
    # ONE session per request; always closed, even on error.
    async with _session_factory() as session:
        yield session
        # Note: this dependency does NOT commit. The use case commits once,
        # at the edge (Stage 2, Ch 05). Rollback happens automatically on exit
        # if an exception propagates.
```

**Models and their relationship (`_models.py`).** The relationship is declared once;
*how* it loads is decided per query, not hard-wired to eager on the model (which
would over-fetch everywhere).

```python
from decimal import Decimal
from datetime import date, datetime
from sqlalchemy import ForeignKey, Numeric, func
from sqlalchemy.orm import Mapped, mapped_column, relationship
from app.models.base import Base


class Invoice(Base):
    __tablename__ = "invoices"

    id: Mapped[int] = mapped_column(primary_key=True)
    account_id: Mapped[int] = mapped_column(index=True)   # tenant scoping (Ch 04)
    customer_id: Mapped[int] = mapped_column(ForeignKey("customers.id"))
    status: Mapped[str] = mapped_column(default="draft")
    total: Mapped[Decimal] = mapped_column(Numeric(12, 2), default=0)
    due_date: Mapped[date | None] = mapped_column(default=None)   # added via migration
    created_at: Mapped[datetime] = mapped_column(server_default=func.now())

    line_items: Mapped[list["LineItem"]] = relationship(
        back_populates="invoice", cascade="all, delete-orphan"
    )


class LineItem(Base):
    __tablename__ = "line_items"

    id: Mapped[int] = mapped_column(primary_key=True)
    invoice_id: Mapped[int] = mapped_column(ForeignKey("invoices.id"), index=True)
    description: Mapped[str]
    quantity: Mapped[int]
    unit_price: Mapped[Decimal] = mapped_column(Numeric(12, 2))
    invoice: Mapped["Invoice"] = relationship(back_populates="line_items")
```

**Avoiding N+1 in the repository (`_repository.py`).** The naive list query, then
the correct one. This is the difference between one query and one-thousand-and-one.

```python
from sqlalchemy import func, select
from sqlalchemy.orm import selectinload
from sqlalchemy.ext.asyncio import AsyncSession
from app.modules.invoicing._models import Invoice


class InvoiceRepository:
    def __init__(self, session: AsyncSession) -> None:
        self._session = session

    # WRONG — lazy loading inside a loop = N+1
    async def list_with_items_bad(self, account_id: int) -> list[Invoice]:
        invoices = list(await self._session.scalars(
            select(Invoice).where(Invoice.account_id == account_id)
        ))
        for inv in invoices:
            _ = inv.line_items   # each access → a separate query. 1000 invoices → 1001 queries.
        return invoices

    # RIGHT — eager-load the relationship in one extra query
    async def list_with_items(
        self, account_id: int, limit: int, offset: int
    ) -> tuple[list[Invoice], int]:
        stmt = (
            select(Invoice)
            .where(Invoice.account_id == account_id)
            .options(selectinload(Invoice.line_items))   # 2 queries total, not N+1
            .order_by(Invoice.created_at.desc())
            .limit(limit).offset(offset)
        )
        invoices = list(await self._session.scalars(stmt))
        total = await self._session.scalar(
            select(func.count()).select_from(Invoice).where(Invoice.account_id == account_id)
        )
        return invoices, total or 0
```

**A safe migration (`versions/0007_add_invoice_due_date.py`).** Adding a *nullable*
column is backward-compatible: the running app that doesn't know about `due_date`
keeps working, and the new app can start using it. This is the "expand" step;
making it non-nullable (with a backfill) would be a later, separate migration.

```python
"""add invoice due_date

Revision ID: 0007
Revises: 0006
"""
import sqlalchemy as sa
from alembic import op

revision = "0007"
down_revision = "0006"


def upgrade() -> None:
    # Nullable add: backward-compatible, no table rewrite, no lock of consequence.
    op.add_column("invoices", sa.Column("due_date", sa.Date(), nullable=True))


def downgrade() -> None:
    # Reversible: the column can be dropped to roll back.
    op.drop_column("invoices", "due_date")
```

If `due_date` eventually must be required, that is **three** deploys, not one — the
expand-migrate-contract pattern:

1. **Expand:** add the nullable column (above); deploy app code that writes it going
   forward.
2. **Migrate:** backfill existing rows in a separate migration (batched, so it does
   not lock the table for minutes), and deploy code that always sets it.
3. **Contract:** once every row has a value and all code sets it, a later migration
   makes the column `NOT NULL`.

Never do all three in one migration on a live table: it locks the table for the
backfill, and a rollback of the app leaves code that doesn't set a now-required
column.

The through-line: the session is created per request and committed once at the edge,
the list query loads its relationship in a bounded number of queries regardless of
row count, and the schema change ships in a way that never breaks the running app.
Each is a decision the ORM and Alembic will happily let you get wrong — and each
wrong version passes every test you'd write against a small local database.

## Engineering Decisions

Five decisions define a safe persistence layer.

### How is the session scoped?

**Options:** (1) a module-level global session; (2) a session per request via a
dependency; (3) a session per operation created ad hoc.

**Trade-offs:** a global session is the most common training-data pattern and a
correctness disaster — SQLAlchemy sessions are not safe to share across concurrent
requests, so a global one leaks state and data between users. Per-request (via
`get_session`) is correct and matches the request lifecycle. Per-operation sessions
fragment the transaction boundary and make one-commit-per-use-case hard.

**Recommendation:** one session per request, provided by a dependency and always
closed (Stage 2, Chapter 06 for the injection, Chapter 05 for the single commit).
This is non-negotiable; a shared session is the persistence equivalent of a shared
mutable global across threads.

### Who owns the transaction boundary?

**Options:** (1) commit inside repositories; (2) commit in the session dependency
automatically; (3) commit once at the edge of the use case.

**Trade-offs:** committing in repositories scatters boundaries and causes partial
writes (Stage 2, Chapter 05). Auto-committing in the dependency commits even when the
use case wanted to roll back, and hides the boundary. Committing at the edge makes one
use case exactly one atomic transaction.

**Recommendation:** one commit at the edge of the use case (the route or an explicit
unit of work), and a session dependency that does *not* auto-commit. Repositories
stage changes; the edge commits; an exception rolls the whole thing back. This is the
Stage 2 rule, enforced by how the session is wired.

### How are relationships loaded?

**Options:** (1) lazy loading (the default); (2) eager loading declared on the model
for all queries; (3) eager loading chosen per query.

**Trade-offs:** lazy is convenient and N+1s under any loop. Model-level eager loading
fixes N+1 but over-fetches on every query that touches the model, including ones that
don't need the relationship. Per-query eager loading (`selectinload`/`joinedload`)
loads exactly what each query needs.

**Recommendation:** choose the loading strategy per query in the repository — lazy is
fine for a single-object fetch, `selectinload` for collections you will iterate. Treat
any relationship access inside a loop as N+1 until proven otherwise. (Which eager
strategy, and index design to support it, is deepened in Stage 6.)

### Autogenerate migrations, or hand-write them?

**Options:** (1) autogenerate and apply as-is; (2) autogenerate then review and edit;
(3) hand-write every migration.

**Trade-offs:** applying autogenerated migrations blindly ships whatever Alembic
inferred — including destructive operations, missed renames (seen as drop+add, which
loses data), and table-locking changes. Hand-writing everything is safe but slow and
error-prone for routine changes. Reviewing autogenerated output combines Alembic's
speed with human judgment.

**Recommendation:** autogenerate as a *draft*, then always review and often edit
before committing — especially for renames (Alembic sees them as drop+add), data
migrations, and anything on a large table. A migration is a deployment artifact; it
gets the same review as code, and more.

### How do you make a breaking schema change safely?

**Options:** (1) change it in one migration; (2) expand-migrate-contract across
multiple deploys.

**Trade-offs:** the one-step change is simplest and breaks the running app during the
deploy window (the old code hits a schema it doesn't expect) and often can't roll back.
Expand-migrate-contract is more steps and more patience but never breaks the running
version and stays reversible at each stage.

**Recommendation:** expand-migrate-contract for any change that isn't a purely additive,
nullable column — add the new shape, migrate data, and remove the old shape only once
nothing references it, each as a separate backward-compatible deploy. Backfills on large
tables run in batches to avoid long locks. (Deployment orchestration is Stage 7; the
schema-safety pattern is here because it is a persistence concern.)

## Trade-offs

Persistence choices trade safety, performance, and effort, and a few are genuinely
contextual.

**The ORM trades control for productivity.** SQLAlchemy makes common persistence fast
to write and hides SQL — which is a benefit until a query's generated SQL matters, at
which point the abstraction can obscure performance problems (N+1, over-fetching, missing
indexes). The mitigation is to know what SQL your queries emit (echo it in development)
and to drop to explicit SQL for the rare hot query that needs it. The ORM is the right
default; blind faith in it is not.

**Eager loading trades queries for data volume.** Fixing N+1 with eager loading means
fetching related rows you might not use, and `joinedload` on a one-to-many can multiply
row counts (the "cartesian" blow-up). The right strategy depends on the shape of the data
and the access pattern — `selectinload` for collections, `joinedload` for to-one — which
is why it is a per-query decision, not a global setting.

**Migration safety trades speed for uptime.** Expand-migrate-contract turns a one-line
schema change into three deploys, which is genuinely slower. For a pre-launch app with no
users and no data, a direct change is fine and the ceremony is over-engineering; for a
live system with real data, the multi-step pattern is the difference between a deploy and
an outage. Scale the caution to whether downtime and data loss are on the table.

**When the defaults are enough.** A small app on a small database can lazy-load freely and
migrate directly — the N+1 that melts a large table is imperceptible on a small one, and a
brief lock on a tiny table is harmless. These practices earn their keep as data and traffic
grow; applying all of them to a prototype is premature (Stage 1, Chapter 07). The judgment
is knowing your scale.

## Common Mistakes

**The global session.** A module-level session shared across requests, leaking state and
data between users and exhausting connections. Fix: one session per request via a
dependency, always closed.

**Committing in the wrong place.** Commits inside repositories or auto-committing in the
session dependency, scattering the transaction boundary and causing partial writes. Fix:
one commit at the edge of the use case; repositories stage, the edge commits.

**N+1 queries.** Accessing a relationship inside a loop with lazy loading, issuing one
query per row — fine in tests, fatal at scale. Fix: eager-load the relationship in the
query (`selectinload` for collections); treat relationship-access-in-a-loop as a red flag.

**Applying autogenerated migrations blindly.** Shipping whatever Alembic inferred,
including drop+add for renames (data loss) and table-locking operations. Fix: review and
edit every generated migration; test it against production-like data.

**One-step breaking migrations.** Renaming or dropping a column, or adding a `NOT NULL`
column, in a single migration on a live table — breaking the running app and often
un-rollback-able. Fix: expand-migrate-contract across separate backward-compatible deploys,
with batched backfills.

**Unbounded connection pool or no pre-ping.** A pool that can open unlimited connections
(exhausting Postgres) or that hands out dead connections after a database restart. Fix:
bound `pool_size`/`max_overflow` and enable `pool_pre_ping`.

## AI Mistakes

Every persistence failure here shares a property: it runs perfectly against a small local
database and fails in production under concurrency, scale, or a live deploy. That is
exactly the gap an assistant cannot see, so it generates the optimistic version
confidently. Review persistence code for the conditions the local test never creates.

### Claude Code: the module-level global session

Asked to set up database access, Claude Code very often creates a single module-level
session (or `SessionLocal()` used directly in functions) shared across the app, because
that pattern saturates tutorials. It works in every single-request test and corrupts state
across concurrent requests in production.

**Detect:** a session created at module scope; functions importing and using a shared
session rather than receiving one; no per-request `get_session` dependency.

**Fix:** require per-request scoping:

> Do not create a module-level or global session. Provide one `AsyncSession` per request
> through a `get_session` dependency that always closes it, and inject it. The session is
> a per-request unit of work, never shared across requests.

### GPT: N+1 queries from lazy loading

GPT-family models write the natural ORM loop — fetch a list, then access a relationship on
each item — producing N+1 queries, because the code reads cleanly and runs fast on the few
rows a test creates. In production against thousands of rows it issues thousands of queries
and collapses.

**Detect:** a relationship accessed inside a loop (`for inv in invoices: ... inv.line_items`)
with no eager-loading option on the query; list endpoints whose query count grows with the
result set.

**Fix:** require explicit eager loading:

> This iterates a relationship and will N+1. Eager-load it in the query with
> `selectinload` (for collections) or `joinedload` (for to-one), so the number of queries is
> constant regardless of row count. Never lazy-load a relationship inside a loop.

### Cursor: unsafe autogenerated migrations

Editing models and generating a migration inline, Cursor tends to present the autogenerated
Alembic file as finished — including drop+add sequences for renames (silent data loss),
`NOT NULL` columns added in one step, and other table-locking or breaking changes — because
the migration "ran" against the empty local database.

**Detect:** a migration applied without review; a rename appearing as `drop_column` +
`add_column`; a `NOT NULL` add or type change in one step; a data-losing `downgrade` (or a
missing one) on a table that has real data.

**Fix:** treat migrations as reviewed deployment artifacts:

> Autogenerated migrations are a draft. Review this one: a rename must use `alter_column`,
> not drop+add (which loses data); a required column must go through expand-migrate-contract
> (nullable add → backfill → set NOT NULL) across separate deploys; and it must be tested
> against production-like data before it ships.

## Best Practices

**Scope the session to the request; commit once at the edge.** One `AsyncSession` per
request via a dependency that always closes it, never a global; repositories stage changes,
and exactly one commit happens at the edge of the use case (Stage 2, Chapters 05–06).

**Configure the engine for production.** The engine and its pool are a startup singleton;
bound `pool_size` and `max_overflow` so you cannot exhaust Postgres, and enable
`pool_pre_ping` so a database restart doesn't hand out dead connections.

**Load relationships deliberately.** Choose the loading strategy per query — lazy for a
single object, `selectinload`/`joinedload` for collections you iterate — and treat any
relationship access inside a loop as an N+1 to fix. Know the SQL your queries emit.

**Treat migrations as reviewed deployment artifacts.** Autogenerate as a draft, then review
and edit every one; test against production-like data; keep a working `downgrade` where the
change is reversible. Never apply a migration to a live database on faith.

**Make breaking schema changes non-breaking.** Use expand-migrate-contract — add, migrate,
then remove across separate backward-compatible deploys — with batched backfills on large
tables, so no deploy window breaks the running app. Record consequential schema decisions in
an ADR ([`templates/adr.md`](../../templates/adr.md)); deeper modeling and indexing are Stage
6.

## Anti-Patterns

**The Global Session.** One session shared across requests — state and data leaking between
users, connections exhausted. The tell: a session at module scope, or `SessionLocal()` called
inside business functions instead of injected.

**The Committing Repository.** Repositories that call `commit()`, scattering transaction
boundaries (Stage 2, Chapter 05). The tell: `commit()` anywhere but the edge of a use case.

**The N+1 Loop.** Relationship access inside a loop with lazy loading — constant on ten rows,
catastrophic on ten thousand. The tell: a query count that scales with the result set.

**The Faith-Based Migration.** Autogenerated migrations applied without review — renames as
drop+add, one-step `NOT NULL` adds, locking changes on live tables. The tell: a migration
diff nobody read, or a rename that loses data.

**The Big-Bang Schema Change.** A breaking change (rename, drop, required column) in a single
migration on a live system, breaking the running app and defying rollback. The tell: a
migration that the currently-deployed app code could not survive running against.

## Decision Tree

"I'm working on persistence — how do I keep it safe?"

```
SESSION
├─ Scope ──► one AsyncSession per request, via a dependency that always closes it.
│            NEVER a module-level/global session.
└─ Commit ─► once, at the edge of the use case. Repositories stage; they never commit.

LOADING A RELATIONSHIP
└─ Will you access it for many rows (a loop / a list)?
   ├─ YES ──► eager-load in the query (selectinload for collections, joinedload for to-one).
   └─ NO ───► lazy is fine for a single-object fetch.
   (Accessing a relationship inside a loop with lazy loading = N+1. Fix it.)

A SCHEMA CHANGE
├─ Generate the migration ──► REVIEW and edit it (never apply on faith).
│                             rename? use alter_column, not drop+add (data loss).
└─ Is it purely additive & nullable?
   ├─ YES ──► one migration is fine (backward-compatible).
   └─ NO (rename / drop / NOT NULL / type change on a live table)
        └──► expand → migrate → contract, across separate deploys.
             backfills on large tables run in batches (avoid long locks).

ENGINE
└─ Startup singleton; bound the pool (pool_size/max_overflow); enable pool_pre_ping.
```

## Checklist

### Implementation Checklist

- [ ] One `AsyncSession` per request via a dependency that always closes it; no global/shared session.
- [ ] Exactly one commit per use case, at the edge; repositories do not commit.
- [ ] Relationships accessed for collections are eager-loaded; no relationship access inside a loop lazy-loads.
- [ ] The engine/pool is a startup singleton with bounded size and `pool_pre_ping` enabled.
- [ ] Every migration has been reviewed and edited, not applied as autogenerated.
- [ ] Breaking schema changes use expand-migrate-contract with batched backfills.

### Architecture Checklist

- [ ] Session lifecycle and transaction boundary are owned in one place (`core/db.py` + the use-case edge).
- [ ] Query/loading strategy lives in the repository, not scattered across services.
- [ ] Migrations are versioned deployment artifacts, reviewed like code.
- [ ] Consequential schema decisions are recorded (ADR); deep modeling/indexing is deferred to Stage 6.
- [ ] Persistence conventions (session scope, commit boundary, N+1 policy) are in `CLAUDE.md`.

### Code Review Checklist

- [ ] No module-level/global session introduced (watch AI diffs).
- [ ] No `commit()` in a repository; one commit per use case.
- [ ] No relationship accessed in a loop without eager loading (no N+1).
- [ ] Migrations reviewed: no rename-as-drop+add, no one-step breaking change on a live table.
- [ ] Migration tested against production-like data, with a working `downgrade` where reversible.

### Deployment Checklist

- [ ] Migrations run as a deliberate deploy step, not automatically on app start with multiple instances (see [`checklists/production-readiness.md`](../../checklists/production-readiness.md)).
- [ ] Each migration is backward-compatible with the currently-running app version (expand-migrate-contract).
- [ ] Large-table backfills are batched to avoid long locks, and were timed against production-sized data.
- [ ] A tested backup/restore exists before any destructive or irreversible migration.
- [ ] Rollback is understood: the previous app version is compatible with the migrated schema.

## Exercises

**1. Kill the N+1.** Take a list endpoint that lazy-loads a relationship in a loop (write
one, or have an assistant generate "list invoices with their line items") and fix it with
eager loading. Prove it by logging the SQL (SQLAlchemy `echo=True`) before and after and
counting the queries for a 100-row result. The artifact is the before/after query counts.

**2. Ship a required column safely.** Take Invoicely's nullable `due_date` and write the full
expand-migrate-contract to make it required: the nullable add, a batched backfill migration,
and the `NOT NULL` migration — three files — plus a note on what each deploy must contain in
the app code. The artifact is the three migrations and the deploy-order note.

**3. Fix the session.** Take code with a module-level global session (write it, or find it in
AI-generated persistence code) and refactor it to a per-request `get_session` dependency,
then describe the exact production symptom the global version would cause under concurrency.
The artifact is the fix and the failure description.

## Further Reading

- **SQLAlchemy documentation — ORM Session, and Relationship Loading Techniques**
  (docs.sqlalchemy.org) — the authoritative reference for the session as a unit of work and
  for `selectinload`/`joinedload` and the N+1 problem. Read the loading-techniques page
  before writing any query that touches a relationship.
- **Alembic documentation** (alembic.sqlalchemy.org) — autogeneration, its limitations
  (renames, data migrations), and writing migrations by hand. The "auto-generating
  migrations" and "operations" sections are essential for reviewing generated migrations.
- **Online migrations / zero-downtime schema changes** — search for the
  expand-migrate-contract (a.k.a. "parallel change") pattern; write-ups from teams like
  GitHub, Stripe, and PlanetScale document how to add, backfill, and drop columns on live
  tables without downtime. The practical foundation for the deployment checklist above.
- **Architecture Patterns with Python** (cosmicpython.com), the repository and unit-of-work
  chapters — how session lifecycle and transaction boundaries fit the layered structure from
  Stage 2, in this exact stack. (Deep data modeling and query tuning are Stage 6 territory,
  covered by database-specific references there.)
