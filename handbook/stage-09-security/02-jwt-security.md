# JWT Security

## Introduction

Stage 3, Chapter 03 built Invoicely's tokens: a signed access token the client holds, a rotating
refresh token, an issue-and-verify module. It built them correctly for the happy path. This chapter
reads that same token machinery the way an attacker does — and the attacker's questions are
different from the builder's. Not "does verification succeed for a valid token?" but "what tokens
can I forge that verification will *accept*?" Not "does it expire?" but "what does the server do
with a token whose expiry I edited, whose algorithm I switched, whose signature I stripped?"

JSON Web Tokens are worth a dedicated security chapter because they concentrate an unusual amount
of trust in a single self-contained string, and because the specification is permissive in exactly
the ways that produce vulnerabilities. A JWT says "the server that holds the key vouches for these
claims" — but *which* key, verified with *which* algorithm, is information the token itself carries
in an attacker-visible, historically attacker-*controllable* header. That design choice is the
root of the two most famous JWT attacks (`alg: none` and RS256→HS256 confusion), and both are
still shipped by real systems and generated by AI assistants today.

This chapter covers the JWT-specific attack surface and the hardening that closes it: pinning the
verification algorithm so the token can't choose its own, managing signing keys (symmetric vs
asymmetric, rotation, JWKS), validating every standard claim rather than just the signature,
making stateless tokens practically revocable, and defending refresh tokens against theft and
replay with rotation and reuse detection. The boundary with Stage 3 is firm: that chapter taught
*building* token auth and the sound defaults; this one teaches the *attacks* against those tokens
and the defenses that survive a hostile, informed adversary. Where the two overlap, this chapter
goes adversarial.

## Why It Matters

A JWT is a bearer token: whoever holds a valid one *is* the user, no questions asked. That makes
every weakness in how tokens are signed, verified, or handled a direct path to account takeover —
and the weaknesses are subtle enough that a token system can be catastrophically broken while
authenticating every legitimate user perfectly.

- **Verification flaws are total, silent breaks.** If your verifier accepts `alg: none`, an
  attacker forges any identity by sending an unsigned token — no key needed. If it can be tricked
  into verifying an RS256 token with the *public* key as an HMAC secret, the attacker signs tokens
  with the key you published on purpose. Both flaws let anyone become anyone, and neither produces
  an error a user or a functional test would ever see.
- **The signing key is a skeleton key.** Whoever has the HMAC secret (or the RSA private key) mints
  valid tokens for every user forever. A weak, guessable, or leaked secret is not a degraded
  defense — it is the absence of authentication. `secret`, `changeme`, and a framework's example
  key are all crackable offline in seconds against a single captured token.
- **A signature is not the whole check.** A cryptographically valid signature on a token that
  expired last week, was issued for a different audience, or was minted by a different service is
  still *accepted* by a verifier that checks only the signature. Real JWT security is validating
  the claims — `exp`, `nbf`, `iss`, `aud` — every time, not just proving the bytes weren't altered.
- **Statelessness is a revocation problem you inherit.** The property that makes JWTs scale — no
  server-side state — is exactly what makes them impossible to revoke before expiry without adding
  state back. A stolen token with a long lifetime is a valid credential until it expires, and "just
  log them out" doesn't work the way anyone assumes.
- **Refresh tokens are long-lived and thus high-value.** They exist to avoid frequent
  re-authentication, which means they live for days or weeks. A stolen refresh token is durable
  account access unless rotation and reuse detection are in place to detect and sever it.
- **The AI dimension is acute here.** Assistants generate token code that *authenticates
  correctly* while omitting algorithm pinning, using a weak or hardcoded secret, decoding claims
  without verifying them, or skipping expiry checks — every one of which passes a login test and
  none of which a functional review catches. JWT is a domain where "it works" and "it's secure"
  are nearly uncorrelated.

## Mental Model

A JWT is three base64url segments — `header.payload.signature`. Security is decided almost entirely
by what the *verifier* does, not by what the token contains. The token is attacker-controlled input
until every check passes.

```
   THE TOKEN (all of it is attacker-VISIBLE; header+payload are attacker-EDITABLE)
     header:    { "alg": "RS256", "kid": "2026-07" }   ← the attacker can change alg and kid
     payload:   { "sub": "user_42", "aud": "invoicely-api",
                  "iss": "invoicely-auth", "exp": ..., "iat": ... }   ← readable, NOT secret
     signature: sign(header + "." + payload, KEY)      ← the only part they can't forge...
                                                          ...IF the verifier is strict

   THE VERIFIER — every one of these must hold, or the token is rejected (fail closed):
     1. ALGORITHM PINNED   verify with the algorithm the SERVER chose, not the token's `alg`.
                           reject alg:none. never let RS256 be verified as HS256.
     2. SIGNATURE VALID    against the correct key (by kid, from your own keyset — never
                           a key or URL the token points to).
     3. NOT EXPIRED        exp in the future; iat/nbf sane; small clock-skew leeway only.
     4. RIGHT AUDIENCE     aud is THIS service; iss is your issuer. a token minted for
                           service B is not valid for service A.
     5. NOT REVOKED        (for logout / compromise) — needs state: short exp + refresh store,
                           or a denylist keyed by jti.

   KEY MODEL
     HS256 (symmetric):  one secret signs AND verifies. simple; every verifier can also MINT.
                         secret must be long, random, secret. good within one trust domain.
     RS256/EdDSA (asym): private key signs, PUBLIC key verifies. verifiers can't mint.
                         right when many services or clients verify. publish public keys via JWKS.

   ACCESS vs REFRESH
     access  = short (minutes), stateless, sent on every request. theft window = its lifetime.
     refresh = long (days), stored server-side, ROTATED on use, REUSE = compromise → revoke family.
```

Three principles carry the chapter:

**The verifier is the security boundary; the token is hostile input.** Never trust anything the
token asserts about how to verify it — not the algorithm, not the key, not the issuer. The server
decides those from its own configuration and checks the token against them. A JWT library used with
its permissive defaults will happily follow the token's instructions; strictness is opt-in.

**A valid signature is necessary, not sufficient.** Signature verification proves integrity.
Authentication requires integrity *plus* an unexpired token *plus* the right audience and issuer
*plus* (for revocation) that it hasn't been killed. Checking only the signature is the most common
"but the token was valid!" breach.

**Statelessness and revocation are a trade you must make on purpose.** Pure stateless JWTs cannot
be revoked; practical systems buy revocation back with short access-token lifetimes plus a
server-side refresh store (or a `jti` denylist). Decide where you sit on this axis deliberately —
don't discover at incident time that "logout" never actually invalidated anything.

A working definition:

> **JWT security is the discipline of treating the token as hostile input and the verifier as the
> trust boundary: pinning the algorithm, verifying the signature with a strong key you control,
> validating every standard claim (expiry, audience, issuer) on every request, and buying back
> revocation with short lifetimes plus rotating, reuse-detected refresh tokens. A signature check
> alone is not authentication.**

## Production Example

**Invoicely** runs a stateless API consumed by its Next.js web app and a planned React Native app.
Stage 3 issued RS256 access tokens (5-minute lifetime) and rotating refresh tokens (14 days,
stored server-side). This chapter hardens that implementation against a real threat model and fixes
the gaps a functional build leaves open.

The concrete work: the verifier is locked to `algorithms=["RS256"]` so no token can downgrade to
`none` or trick the service into HMAC-verifying with the public key. Signing moves to an asymmetric
keypair because the mobile app and an upcoming analytics service will need to *verify* tokens
without being able to *mint* them — the private key stays in the auth service, the public keys are
published at a JWKS endpoint with a `kid` so keys can rotate without downtime. Every verification
now asserts `aud="invoicely-api"` and `iss="invoicely-auth"`, so the analytics service's tokens
can't be replayed against the main API and vice versa. Access tokens stay short precisely so that
revocation works: logging out, or an admin killing a session, revokes the *refresh* token in the
store, and the stolen access token dies on its own within five minutes.

The refresh flow gets the defense that matters most for long-lived tokens: rotation with reuse
detection. Each refresh returns a new refresh token and invalidates the old one; if an old
(already-rotated) token is ever presented again, that's the fingerprint of theft — the entire token
family is revoked and the user is forced to re-authenticate. In this chapter we take Stage 3's
working tokens and make them survive an attacker who has read the source, captured a token, and is
actively trying every JWT trick — and we contrast each hardening step with the assistant-default
version that authenticates fine and fails every one of these attacks.

## Folder Structure

```

│                         #   kid-based key selection, jti for revocation
├── _keys.py              # NEW: keypair loading, JWKS building, key rotation (kid registry)
├── _refresh.py           # NEW: refresh rotation + reuse detection (revoke-family on replay)
├── _models.py            # RefreshToken gains: family_id, rotated_at, revoked_at
└── router.py             # /refresh (rotates), /logout (revokes), /.well-known/jwks.json
core/
├── auth.py               # get_current_account — now rejects on any failed claim, fail-closed
└── config.py             # KEY IDs + TTLs from env/secret store (Ch 04); NEVER a literal secret
tests/
└── security/
    └── test_jwt_attacks.py   # alg:none, alg-confusion, expired, wrong-aud, tampered, reuse
```

Why this shape:

- **`_tokens.py`** stays the single verification chokepoint, now strict: one function every
  protected path funnels through, so algorithm pinning and claim validation cannot be
  accidentally skipped by a caller that verifies "its own way."
- **`_keys.py`** exists because asymmetric keys introduce lifecycle — loading, identifying by
  `kid`, rotating, and publishing public keys via JWKS. Isolating it keeps key management out of
  request-path code and makes rotation a config change, not a code change.
- **`_refresh.py`** isolates the rotation-and-reuse-detection logic, the highest-value defense for
  the longest-lived credential, so it's implemented once and tested hard.
- **`tests/security/test_jwt_attacks.py`** is the artifact that proves the hardening: a test *per
  attack* that must stay red-turns-green. Security you don't test against the attack is security
  you're guessing at.

## Implementation

**The hardened verifier (`_tokens.py`): pin the algorithm, validate every claim.** This one
function is the security boundary. The `algorithms` list and the claim checks are the difference
between authentication and forgery.

```python
import jwt  # PyJWT

def verify_access_token(token: str) -> TokenClaims:
    unverified_header = jwt.get_unverified_header(token)     # to pick the key by kid ONLY
    public_key = KEYSET.public_key_for(unverified_header.get("kid"))  # from OUR keyset, not the token
    try:
        claims = jwt.decode(
            token,
            key=public_key,
            algorithms=["RS256"],          # PINNED — never trust the token's alg; rejects none & HS confusion
            audience="invoicely-api",      # aud must be THIS service
            issuer="invoicely-auth",       # iss must be our issuer
            options={"require": ["exp", "iat", "aud", "iss", "sub"]},  # presence is mandatory
            leeway=30,                     # small clock-skew tolerance only
        )
    except jwt.InvalidTokenError:          # expired, wrong aud/iss, bad signature, tampered — all here
        raise AuthError("invalid token")   # fail closed; leak nothing about WHY
    if REVOCATION.is_revoked(claims["jti"]):   # optional denylist for immediate kill
        raise AuthError("invalid token")
    return TokenClaims.model_validate(claims)
```

**Why each argument matters — this is the whole chapter in one call:**

- `algorithms=["RS256"]` is the fix for both `alg: none` (rejected — not in the list) and
  RS256→HS256 confusion (the verifier will never *interpret* the key as an HMAC secret). Omitting
  this argument, or passing the token's own `alg`, is the classic catastrophic bug.
- The key is selected by `kid` **from your own keyset** — never fetched from a URL or key the token
  supplies. A token that names its own verification key verifies itself.
- `audience` and `issuer` turn "a valid signature" into "a valid signature *for this service, from
  our issuer*" — blocking cross-service replay.
- `require` makes missing claims a rejection, not a `None` that later code treats as valid.
- The bare `except InvalidTokenError` means expired, tampered, wrong-audience, and bad-signature
  all fail identically and closed — no branch accidentally lets one class through, no error message
  tells the attacker which check they tripped.

**Key management (`_keys.py`): verifiers that can't mint, and rotation without downtime.**

```python
def build_jwks() -> dict:
    # public keys ONLY — safe to serve at /.well-known/jwks.json for other services to verify with
    return {"keys": [k.public_jwk() for k in KEYSET.active_and_next()]}
```

Publishing the *public* keys lets the mobile app and analytics service verify tokens locally
without ever holding minting power. A `kid` on every token means you can introduce a new key
(sign with old, publish both), then retire the old one — rotation with zero failed verifications.

**Refresh rotation with reuse detection (`_refresh.py`): the defense for the long-lived token.**

```python
def rotate_refresh_token(presented: str) -> RefreshPair:
    record = refresh_store.get(hash_token(presented))    # store HASHES, never raw tokens
    if record is None or record.revoked_at:
        raise AuthError("invalid refresh")               # unknown or already-killed
    if record.rotated_at:                                # already used → this is a REPLAY
        refresh_store.revoke_family(record.family_id)    # theft fingerprint: burn the whole family
        raise AuthError("invalid refresh")
    refresh_store.mark_rotated(record.id)
    new_refresh = issue_refresh(family_id=record.family_id)   # same family, new token
    return RefreshPair(access=issue_access(record.sub), refresh=new_refresh)
```

The insight: a refresh token is single-use. Presenting an already-rotated one means two parties
hold it — the legitimate user and a thief. You can't tell which is which, so you revoke the family
and force re-authentication. This converts a stolen refresh token from *durable* access into
*detected-and-severed* access at the thief's next use.

**The attack tests (`tests/security/test_jwt_attacks.py`): prove the hardening.**

```python
def test_alg_none_token_is_rejected(client):

    assert client.get("/api/v1/invoices", token=forged).status_code == 401

def test_public_key_as_hmac_secret_is_rejected(client, public_key_pem):
```

## Engineering Decisions

Five decisions define a JWT implementation's security posture.

### Symmetric (HS256) or asymmetric (RS256/EdDSA) signing?

**Options:** (1) HS256 — one shared secret signs and verifies; (2) RS256/EdDSA — private key signs,
public key verifies.

**Trade-offs:** HS256 is simpler and faster and needs only one secret — but every party that can
*verify* can also *mint*, so it only suits a single trust domain where issuer and verifier are the
same service. Asymmetric keys let any number of services and clients verify with a public key while
only the auth service can sign, at the cost of key-pair management and a JWKS endpoint. The
asymmetric model also structurally prevents the RS256→HS256 confusion attack from mattering across
services.

**Recommendation:** HS256 with a long random secret is fine when a single service both issues and
verifies. The moment a second party needs to verify — a mobile app, another microservice, an API
gateway — move to RS256 or EdDSA so verification never implies minting. Invoicely uses asymmetric
because the mobile app and analytics service verify tokens.

### How is the algorithm chosen at verification time?

**Options:** (1) trust the token's `alg` header; (2) pin an explicit allowlist server-side;
(3) pin a single algorithm.

**Trade-offs:** trusting the token's `alg` is the vulnerability itself — it lets the attacker
select `none` or a confusable algorithm. An allowlist supports migration between algorithms. A
single pinned algorithm is the simplest and safest and the right default.

**Recommendation:** pin the algorithm explicitly to exactly what you issue (`algorithms=["RS256"]`),
never read it from the token. Use a two-element allowlist *only* transiently while rotating from
one algorithm to another, then collapse back to one. This single decision closes the two headline
JWT attacks; there is no reason to get it wrong.

### Where does revocation come from — or does it?

**Options:** (1) pure stateless, no revocation, rely on short expiry; (2) short access tokens +
server-side refresh store (revoke refresh); (3) a `jti` denylist checked on every request;
(4) opaque tokens with server-side session state.

**Trade-offs:** pure stateless can't revoke — a stolen token is valid until expiry, and logout is
cosmetic. Short-access-plus-refresh-store gives near-term revocation (kill the refresh; access dies
in minutes) while keeping the request path stateless. A `jti` denylist gives *immediate* revocation
but adds a state lookup to every request — reintroducing the coupling JWTs were chosen to avoid.
Opaque server-side sessions are trivially revocable and are, honestly, the simpler answer when you
don't need statelessness at all.

**Recommendation:** short access tokens (5–15 min) plus a revocable, rotating refresh store as the
default — it keeps the hot path stateless while making logout and compromise-response real. Add a
`jti` denylist only for the cases that need instant kill (a detected compromise), scoped and
short-lived so it doesn't become a session store by the back door. And genuinely consider: if you
need immediate revocation everywhere, opaque sessions may be the better engineering choice than
fighting JWT's statelessness.

### How long do access and refresh tokens live?

**Options:** (1) long access tokens for convenience; (2) short access + long refresh; (3) short
both, frequent re-auth.

**Trade-offs:** a long-lived access token means a theft is a long-lived compromise and revocation
barely matters — the worst common default. Short access plus long refresh bounds access-token theft
to minutes while keeping users logged in via rotation. Short-both hurts UX and pushes load onto the
auth service.

**Recommendation:** access tokens in minutes (5–15), refresh tokens in days-to-weeks *with rotation
and reuse detection*. The short access lifetime is what makes the stateless hot path safe; the
refresh token carries continuity and is defended by rotation rather than by being short. Tune the
refresh lifetime to your product's sensitivity — a banking app is not a photo app.

### Where does the client store the token?

**Options:** (1) `HttpOnly` `Secure` `SameSite` cookie; (2) `localStorage`/`sessionStorage`;
(3) in-memory only; (4) mobile secure storage (Keychain/Keystore).

**Trade-offs:** `localStorage` is readable by any XSS on the page — one script include and every
token leaks (Chapter 06). An `HttpOnly` cookie is unreachable by JavaScript but must then be
defended against CSRF (Chapter 07). In-memory is safe from persistence-based theft but doesn't
survive reload. On mobile, platform secure storage is the answer.

**Recommendation:** on the web, `HttpOnly` + `Secure` + `SameSite=Lax/Strict` cookies for the
tokens, paired with CSRF defenses — never `localStorage`, which trades XSS-safety for convenience.
On mobile, Keychain/Keystore, never `AsyncStorage`. This decision is where JWT security meets
Chapters 06 and 07; the token can be perfect and still be stolen from the wrong storage.

## Trade-offs

**Statelessness trades revocability for scale.** The core JWT bargain: no server-side lookup on the
hot path, but no way to invalidate a token before it expires without adding state back. You pay for
scale with a revocation gap, and you buy the gap back down with short lifetimes and a refresh store.
Pretending the gap doesn't exist is how "we logged them out" becomes "the token still worked."

**Asymmetric signing trades operational complexity for separation of powers.** RS256/EdDSA lets
verifiers exist that cannot mint — essential across services — at the cost of key generation,
rotation, a JWKS endpoint, and `kid` handling. For a single service, that machinery is overhead
HS256 avoids. Match the key model to how many parties verify.

**Short token lifetimes trade user friction and auth-service load for a smaller theft window.**
Every shortening bounds the damage of a stolen token and increases refresh traffic and the chance
of a mid-action expiry. Rotation smooths the UX; the auth service must be sized for the refresh
load. The right lifetime is a security/UX calibration, not a constant.

**Reuse detection trades occasional false logouts for theft detection.** Revoking a token family on
replay catches stolen refresh tokens — and can also trigger on benign races (a client retrying a
refresh, two tabs refreshing at once). The mitigation is careful client behavior and a short grace
window, accepting that a rare spurious re-login is a good price for detecting theft.

## Common Mistakes

**Not pinning the algorithm.** Calling `decode` without an explicit `algorithms` list, or reading
`alg` from the token — the door to `alg: none` and key-confusion. Fix: always pass an explicit,
minimal `algorithms` allowlist.

**Verifying the signature but not the claims.** Accepting a token because the signature is valid,
while `exp`, `aud`, and `iss` go unchecked — so expired and cross-service tokens sail through. Fix:
validate expiry, audience, and issuer on every request; require their presence.

**A weak or hardcoded signing secret.** `secret`, an example key from a tutorial, or a short string
committed to the repo — crackable offline from one captured token. Fix: a long, random, secret key
from the secret store (Chapter 04), never in code.

**Treating logout as if it invalidates the token.** Deleting the client-side token while the token
itself remains valid server-side until expiry — "logout" that a captured token ignores. Fix:
revoke the refresh token server-side; keep access lifetimes short; denylist `jti` where instant
kill is needed.

**Putting sensitive data in the payload.** Passwords, full PII, secrets, or fine-grained
permissions in claims — all readable by anyone holding the token, since a JWT is signed, not
encrypted. Fix: a subject ID and coarse non-sensitive claims only; look up the rest server-side.

**Long-lived access tokens.** Access tokens that live for hours or days, so theft is durable and
the short-lifetime revocation strategy doesn't apply. Fix: minutes for access, rotation for
continuity.

**Non-rotating refresh tokens.** A refresh token reusable indefinitely, so a stolen one is
permanent access with no way to detect the theft. Fix: rotate on every use and revoke the family on
replay.

## AI Mistakes

JWT is a domain where assistant output authenticates flawlessly and is wide open, because every one
of these flaws is invisible to a login test. Review token code against the attacks, not the happy
path.

### Claude Code: verifying the signature without pinning the algorithm

Asked to "verify the JWT," Claude Code frequently produces a `decode` call that checks the
signature but omits an explicit `algorithms` allowlist, or passes the algorithm pulled from the
token's own header. The code verifies real tokens correctly, so it looks done — while accepting
`alg: none` tokens (no signature at all) and, with a symmetric verifier, RS256→HS256 confusion
where the public key is used as an HMAC secret. Either one is total authentication bypass.

**Detect:** `jwt.decode`/verify calls with no `algorithms` argument or a wide one; the algorithm
read from `get_unverified_header`; verifier code that would accept a token you re-sign with `none`
or with the public key as an HMAC key (write exactly those two tests).

**Fix:** pin it, and prove the pin:

> Verification must pass an explicit `algorithms=["RS256"]` (only what we issue) and must never read
> the algorithm from the token. Add tests that a token with `alg: none` and a token signed with the
> public key as an HMAC secret are both rejected with 401.

### GPT: decoding claims without verifying, and a guessable secret

GPT-family models tend to reach for the convenient path: decoding the token to read `sub` or
`role` without a verification step (or with verification disabled "to read the claims first"), and
seeding the signing secret with a placeholder like `"secret"` or `"your-256-bit-secret"` that
survives into committed code. The result trusts attacker-editable payload data and signs with a key
an attacker cracks offline in seconds — two independent total breaks in one implementation.

**Detect:** any decode with verification off (`verify=False`, `options={"verify_signature":
False}`) whose output is then trusted; signing keys that are short literal strings, tutorial
placeholders, or committed to the repo; claims like role/tenant read from an unverified decode.

**Fix:** never trust an unverified claim; source the key from the secret store:

> Never read claims from an unverified/`verify=False` decode and then trust them — verify first,
> then use the returned claims. The signing key must be a long random value loaded from the secret
> store, never a literal in code. Add a check that the app refuses to start with a default or short
> key.

### Cursor: skipping expiry and audience validation while completing the verifier

Editing an existing verify function, Cursor completes the signature check and stops — leaving out
`exp`, `aud`, and `iss` validation because the immediate task was "make it verify," and the
surrounding code didn't already assert them. The token's signature is checked; its expiry and
audience are not. Expired tokens keep working (so revocation-by-expiry silently fails) and a token
minted for another service is accepted, because "valid signature" was mistaken for "valid token."

**Detect:** verify code that checks the signature but passes no `audience`/`issuer` and doesn't
enforce `exp`; tokens that still authenticate well past their expiry in a test; the same token
accepted by two services that should have distinct audiences.

**Fix:** make the full claim set mandatory:

> Verification must enforce `exp` (reject expired), `aud` (must equal this service), and `iss` (our
> issuer), and require those claims to be present. Add tests that an expired token and a
> wrong-audience token are both rejected, not just a tampered-signature one.

## Best Practices

**Pin the algorithm to exactly what you issue.** An explicit minimal `algorithms` allowlist on
every verification; never read `alg` from the token. This closes `none` and confusion attacks in
one line.

**Validate the whole claim set, every request.** Signature *and* `exp`/`nbf` *and* `aud` *and*
`iss`, with required-claim enforcement, failing closed and identically on every failure so nothing
leaks which check tripped.

**Use asymmetric keys once more than one party verifies.** Private key signs in the auth service;
public keys verify everywhere via a JWKS endpoint with `kid`, so verifiers can't mint and keys
rotate without downtime.

**Keep access tokens short and refresh tokens rotating.** Minutes for access so the stateless hot
path is safe to leave un-revocable; rotation with reuse detection for refresh so a stolen long-lived
token is detected and severed.

**Source signing keys from the secret store, and rotate them.** Long, random, never in code
(Chapter 04); a `kid` registry so rotation is a config change; the app refuses to boot on a default
or weak key.

**Put nothing sensitive in the payload.** Subject ID and coarse claims only; a JWT is signed, not
encrypted — treat its payload as public.

**Store tokens out of reach of scripts.** `HttpOnly`/`Secure`/`SameSite` cookies on the web (with
CSRF defense, Chapter 07), platform secure storage on mobile — never `localStorage` or
`AsyncStorage`.

**Test the attacks, not just the login.** A security test per attack — `alg:none`, key confusion,
expired, wrong audience, tampered, refresh reuse — kept green in CI. Hardening you don't test is
hardening you're guessing at.

## Anti-Patterns

**The Unpinned Verifier.** `decode` with no algorithm allowlist, or the algorithm taken from the
token. The tell: a token re-signed with `none` or with the public key as an HMAC secret still
authenticates.

**The Signature-Only Check.** Verifying integrity while ignoring `exp`/`aud`/`iss`. The tell:
expired tokens keep working; a token minted for one service is accepted by another.

**The Vanity Secret.** A signing key that's short, guessable, a tutorial placeholder, or committed
to the repo. The tell: the key appears in source or config as a literal; it would fall to an
offline crack against one captured token.

**The Cosmetic Logout.** "Logout" that only clears the client, leaving the token valid until expiry.
The tell: a captured token continues to work after the user "logged out."

**The Overstuffed Payload.** PII, permissions, or secrets packed into claims. The tell: sensitive
data visible in a base64-decoded token; the payload treated as if it were encrypted.

**The Immortal Refresh.** A refresh token reusable forever with no rotation. The tell: the same
refresh token works repeatedly across days; no family/rotation columns exist; a stolen refresh
token is undetectable.

## Decision Tree

"I'm issuing or verifying a JWT — what must be true?"

```
VERIFYING an incoming token?
├── algorithm: is it PINNED to what you issue (explicit allowlist, not from the token)?
│     └── NO ─► STOP. this accepts alg:none and key-confusion forgeries. fix first.
├── signature: valid against the correct key, selected by kid from YOUR keyset (never the token's)?
├── claims: exp not passed? aud == this service? iss == your issuer? required claims present?
│     └── any NO ─► reject, fail closed, leak nothing about which check failed.
└── revocation needed (logout / compromise)?
      ├── near-term ok ─► short access token; revoke the refresh token in the store.
      └── immediate   ─► jti denylist checked here (accept the per-request state cost),
                          or reconsider opaque server-side sessions.

ISSUING a token?
├── who verifies it?
│     ├── only this service ─► HS256 with a long random secret from the secret store is fine.
│     └── other services/clients ─► RS256/EdDSA; publish public keys via JWKS with kid.
├── lifetime: access in MINUTES; refresh in days WITH rotation + reuse detection.
├── payload: subject id + coarse non-sensitive claims only (it's readable by anyone).
└── storage told to client: HttpOnly/Secure/SameSite cookie (web) or Keychain/Keystore (mobile).
```

## Checklist

### Implementation Checklist

- [ ] Every verification passes an explicit, minimal `algorithms` allowlist; the algorithm is never read from the token.
- [ ] Verification enforces signature, `exp`/`nbf`, `aud`, and `iss`, and requires those claims to be present, failing closed.
- [ ] The signing key is long and random, loaded from the secret store, and absent from code and git; the app refuses to start on a default/weak key.
- [ ] Access tokens are short-lived (minutes); refresh tokens rotate on every use and revoke the family on reuse.
- [ ] Refresh tokens are stored hashed server-side with a family/rotation model, never in plaintext.
- [ ] The payload carries only a subject ID and coarse, non-sensitive claims.

### Architecture Checklist

- [ ] The signing model (HS256 vs RS256/EdDSA) matches how many parties verify; verifiers that shouldn't mint use public-key verification only.
- [ ] Keys carry a `kid` and are rotatable without downtime; public keys are published via a JWKS endpoint.
- [ ] The revocation strategy is explicit (short access + refresh store, or `jti` denylist) and "logout" actually invalidates something server-side.
- [ ] Token storage on each client is out of script reach (HttpOnly cookies with CSRF defense on web; secure storage on mobile).
- [ ] Distinct services use distinct audiences so tokens can't be replayed across them.

### Code Review Checklist

- [ ] No `decode`/verify call omits `algorithms` or reads the algorithm from the token.
- [ ] No claim is trusted from a `verify=False`/unverified decode.
- [ ] No signing secret, key, or JWT appears as a literal in the diff or config.
- [ ] New verification paths go through the single hardened verifier, not a hand-rolled decode.
- [ ] Security tests for `alg:none`, key confusion, expired, wrong-audience, and refresh reuse exist and pass.

### Deployment Checklist

- [ ] Signing keys come from the secret manager (Chapter 04) and are rotated on a schedule and on suspected compromise.
- [ ] The JWKS endpoint serves only public keys and is cached with a sane TTL that supports rotation.
- [ ] Token TTLs and the revocation store (refresh/denylist) are sized and monitored for the production request volume.
- [ ] Clock synchronization (NTP) is in place across services so `exp`/`nbf` with small leeway behave correctly.

## Exercises

**1. Break an unpinned verifier, then fix it.** Take a verifier that calls `decode` without an
`algorithms` allowlist. Forge two tokens — one with `alg: none`, one signed with the public key as
an HMAC secret — and show both authenticate. Then pin the algorithm and show both are rejected. The
artifact is the two forged tokens, the before/after responses, and the one-line fix that closed
them.

**2. Prove your logout is real.** Capture a valid access token, "log out" through your app, and
confirm whether the captured token still works. If it does, implement the short-access + refresh-
revocation model (or a `jti` denylist) until a logged-out token is rejected — and measure the
worst-case window between logout and the token actually dying. The artifact is the two runs and the
measured window.

**3. Implement and trip reuse detection.** Add rotation-with-reuse-detection to the refresh flow.
Rotate a refresh token once (legitimate), then present the old one (simulating theft) and show the
whole family is revoked and the user forced to re-authenticate. Then find and fix the benign race
(two tabs refreshing at once) that would false-trigger it. The artifact is the reuse test and the
race mitigation.

**4. Rotate a signing key with zero downtime.** With `kid`-tagged keys and a JWKS endpoint,
introduce a new signing key: sign new tokens with it while still verifying tokens signed by the old
key, then retire the old key. Show that no valid token was ever rejected during the rotation. The
artifact is the rotation procedure and a test that spans it.

## Further Reading

- **RFC 8725 — JSON Web Token Best Current Practices (datatracker.ietf.org/doc/rfc8725)** — the
  IETF's own catalog of JWT attacks and mitigations, including algorithm confusion and the
  `alg: none` problem; the authoritative source for this chapter's hardening.
- **RFC 7519 — JSON Web Token (JWT)** — the base specification: claim definitions (`exp`, `nbf`,
  `aud`, `iss`, `jti`) and the validation rules a strict verifier enforces.
- **OWASP JSON Web Token Cheat Sheet (cheatsheetseries.owasp.org)** — concise, practical hardening
  guidance mapped to the same attacks, useful as the review checklist while implementing.
- **Auth0 — "Critical vulnerabilities in JSON Web Token libraries" (auth0.com/blog)** — the classic
  write-up of the algorithm-confusion and `alg: none` attacks with concrete examples of vulnerable
  library usage; the case study behind why algorithm pinning is non-negotiable.
- **Stage 3, Chapter 03 — Authentication** — how Invoicely's tokens are issued and where the
  refresh store lives; the build this chapter hardens.
