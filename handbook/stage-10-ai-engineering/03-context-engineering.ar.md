# هندسة السياق

## مقدمة

مخرجات وكيل الذكاء الاصطناعي هي دالة لمُدخَلين: ما يعرفه وقت التصرف (السياق) وما تطلبه
منه (الموجِّه). يتناول هذا الفصل المُدخَل الأول — الأقوى والأقل ظهورًا من الاثنين.
**هندسة السياق هي disciplina التحكم فيما يوجد داخل نافذة سياق النموذج**: انتقاء ما
يُحمَّل في كل جلسة، وهيكلة مستودعك ووثائقك بحيث يستطيع الوكيل استرجاع المعرفة الصحيحة في
اللحظة المناسبة، وإدارة النافذة على مدى عمر الجلسة بحيث لا تُدفن المهمة الحالية تحت
بقايا المهمات السابقة.

نافذة السياق هي كل ما يراه النموذج عند تقرير إجراءه التالي: موجِّه النظام، وملفات التعليمات،
والمحادثة حتى الآن، وكل ملف قرأه، وكل مخرج أمر لاحظه. هي موارد محدودة، ومكلفة، والأهم —
أنها الشيء *الوحيد* الذي يعرفه النموذج عن نظامك. يملك النموذج معرفة تدريبية واسعة عن
FastAPI وPostgreSQL؛ لكنه لا يملك أي معرفة عن قاعدة عزل المستأجرين في Invoicely، أو وحدتك
المحاسبية المهملة، أو القرار الذي اتخذته في مارس بخصوص التعامل مع الأموال — ما لم تصل هذه
الحقائق إلى النافذة. كل فشل للوكيل يُلقى على عاتق "غباء النموذج" ينقسم، عند الفحص، إلى
كومتين: حقائق لم تصل أبدًا إلى النافذة، ونافذة ملوثة لدرجة غرقت فيها الحقائق. كلتا الكومتين
مشكلة هندسية، وهذا الفصل يبيّن كيفية حلّهما.

بَنَى الفصلان 01 و02 طبقة *المحمَّل دائمًا* (ملفات التعليمات). يَبني هذا الفصل كل ما حولها:
طبقة المعرفة عند الطلب، وبنية الاسترجاع، ونظافة الجلسة. ثم يغطي الفصل 04 المُدخَل الثاني —
الموجِّه.

## لماذا هذا مهم

- **السياق يفسر تباين المخرجات أكثر من أي شيء آخر تتحكم فيه.** نفس النموذج، ونفس الموجِّه،
  ونفس المهمة ينتج كودًا متعارفًا عليه في مستودع مُحاط بسياق جيد، وكودًا مقبولًا ظاهريًا
  لكنه غريب في مستودع مجرد. يحرق المهندسون أسابيعًا في ضبط الموجِّهات وتبديل النماذج لإصلاح
  ما هو في الحقيقة فقرة سياق مفقودة.
- **النافذة ميزانية، واليتة تتدهور قبل أن تنفد.** قبل الحدود الصلبة بوقت طويل، يحضر
  النموذج بشكل أقل موثوقية للمادة المدفونة في منتصف نافذة ضخمة؛ تتلاشى التعليمات، وتبقى
  نسخ الملفات القديمة، ويصبح السلوك فوضويًا. عبارة "النموذج يملك نافذة سياق ضخمة" حجة
  لـ*هامش*، لا للتخلي عن الانتقاء — نافذة كبيرة مليئة بالضوضاء تؤدي أداءً أسوأ من نافذة
  صغيرة تحتفظ بالحقائق ذات الصلة بالضبط، وتكلف أكثر في كل دوران.
- **السياق الخاطئ أسوأ من السياق المفقود.** المعرفة المفقرة تجعل الوكيل يسأل أو يقرأ؛ المعرفة
  المتقادمة تجعله يتصرف — بثقة، وبخطأ. وثيقة بنية قديمة، ونسخة مهملة من ملف قُرئ في وقت سابق
  من الجلسة، ووحدة مهملة تم استرجاعها كنموذج: كل منها يوجّه الوكيل بنشاط نحو الخطأ. الانتقاء
  يشمل الحذف.
- **جودة الاسترجاع خاصية في مستودعك.** يجد الوكلاء المعرفة عن طريق الإدراج والبحث وقراءة
  الملفات — بالطريقة نفسها التي يفعلها مهندس جديد. المستودع ذو التسمية المتوقعة، وخريطة
  الوثائق، والوثائق الصغيرة المركّزة هو مستودع *قابل للاستعلام*؛ أما المستودع الذي تحبست
  فيه المعرفة في سجلات المحادثات ووثيقة ARCHITECTURE.md واحدة بطول 3000 سطر فليس كذلك.
  هندسة السياق هي في معظمها هندسة المستودعات.
- **كل ما يحتاجه الوكيل هو ما يحتاجه الموظف الجديد.** خريطة الوثائق، وسجلات قرارات البنية
  (ADRs)، وملفات README للوحدات، والمسرد — كل عنصر يبنيه هذا الفصل يخدم أيضًا كوثائق
  تأهيل. الفرق التي تستثمر هنا تلاحظ الأثر نفسه مرتين: الوكلاء يتوقفون عن إعادة اختراع
  المساعدات الموجودة، والمهندسون الجدد يتوقفون عن السؤال عن مكان وجود الأشياء.
- **السياق هو أيضًا حدود أمنية.** أي شيء يدخل النافذة — سجلًا منسوخًا يحتوي رمز حامل
  (bearer token)، أو ملف `.env` قُرئ "لتصحيح الأخطاء" — قد خرج عن نطاق سيطرتك (المرحلة 9،
  الفصل 04). تقرير ما يجوز للوكيل رؤيته جزء من هذا الانضباط بقدر تقرير ما ينبغي له رؤيته.

## النموذج الذهني

عامل نافذة السياق كذاكرة عمل نادرة تُحمِّلها عن قصد، وتنفقها مهمة بمهمة، وتعيد ضبطها
قبل أن تفسد.

```
              THE CONTEXT WINDOW AS A BUDGET
  ┌──────────────────────────────────────────────────────┐
  │ ALWAYS LOADED (fixed cost — every session, every turn)│
  │   system prompt · CLAUDE.md/AGENTS.md · tool defs    │
  │   → must be small and dense; Ch 01/02's discipline   │
  ├──────────────────────────────────────────────────────┤
  │ LOADED ON DEMAND (variable cost — the working set)   │
  │   files the agent reads · docs it follows pointers to│
  │   command output · search results                    │
  │   → your repo's structure determines how cheaply the │
  │     RIGHT things load and how easily wrong ones do   │
  ├──────────────────────────────────────────────────────┤
  │ ACCUMULATED (residue — grows all session, never      │
  │   shrinks on its own)                                │
  │   conversation history · stale file versions ·       │
  │   dead-end explorations · pasted logs                │
  │   → the reason long sessions degrade; managed by     │
  │     scoping, compaction, and reset                   │
  └──────────────────────────────────────────────────────┘

  THE KNOWLEDGE SUPPLY CHAIN (where facts live → how they arrive)
    code            → agent reads it        (cheap, always current)
    docs/ + ADRs    → agent follows the map (cheap IF map exists)
    instruction file→ always present        (expensive — reserve it)
    your head       → only via prompt       (bottleneck — write it down)
    chat scrollback → NEVER arrives         (knowledge graveyard)

  SESSION LIFECYCLE
    scope task → load working set → work → verify
        └── next task? → RESET (new session / clear)
        └── long task? → COMPACT deliberately or externalize
                         state to a file the next session reads
```

ثلاثة مبادئ تنظّم الممارسة:

**الصلة أهم من الحجم.** الهدف ليس أبدًا "أعطِ النموذج كل شيء"؛ بل "أعطِ النموذج بالضبط ما
تحتاجه هذه المهمة، وطريقة رخيصة لجلب ما لم تتنبأ به." التحميل المسبق للكوني (الاتفاقيات،
والأوامر)؛ أما الاسترجاع فالخاص (الملفات الثلاثة التي تمسها هذه المهمة، وADR الواحد الذي
يقيدها).

**المستودع هو قاعدة بيانات السياق، والوكيل يستعلمها.** كل خيار هيكلي — تسمية الملفات،
وحدود الوحدات، وحجم الوثائق، وخريطة `docs/` — إما أن يسرّع استرجاع الوكيل أو يبطئه. تحسين
هذا هندسة كلاسيكية: اجعل الاستعلام الشائع سريعًا (ابحث عن الاتفاقية، وعن النموذج، وعن
القرار)، واجعل الاستعلام الخطير بطيئًا أو مستحيلًا (الأسرار، والأنماط المهملة المقدمة
كأنها حالية).

**السياق يتراكم؛ والانضباط طرح.** داخل الجلسة النافذة لا تنمو إلا — بما في ذلك كل ما تبيّن
أنه غير ذي صلة أو خاطئ. الحركات الطرحية (جلسة جديدة لكل مهمة، ضغط عند المعالم، إخراج حالة
المهام الطويلة إلى ملف خطة) ليست تحسينات؛ فبعد طول جلسة متوسط هي ما يحافظ على تماسك الوكيل.

تعريف عملي:

> **هندسة السياق هي التحكم فيما يعرفه النموذج وقت التصرف: نواة محمَّلة دائمًا بالحد الأدنى،
> ومستودع منظم بحيث تكون المعرفة الصحيحة قابلة للاسترجاع الرخيص في لحظة الحاجة (ويكون
> استرجاع المعرفة الخاطئة أو المتقادمة صعبًا)، ونظافة جلسة — تحديد نطاق، وإعادة ضبط، وضغط،
> وحالة مُخرَجة — تحافظ على صلة النافذة مع تراكم العمل. وحدتها المحاسبية هي الـtoken؛
> ومعيار جودتها هو هل يستطيع وكيل (أو مهندس) بلا تاريخ أن يصبح منتجًا من المستودع وحده.**

## مثال إنتاجي

تحتوي **Invoicely** على تسع مراحل من القرارات مغروسة فيها: لماذا الأموال سنتات صحيحة
(المرحلة 3)، ولماذا تحديد نطاق المستأجرين concern لطبقة المستودع (المرحلة 3)، ولماذا وحدة
`billing/` القديمة مجمَّدة في انتظار الانتقال إلى `credit_notes/` (إعادة هيكلة المرحلة 6)،
وأي طبقة اختبار تُشغَّل متى (المرحلة 8)، وأي endpoints حساسة لتحديد معدل الطلبات
(المرحلة 9). اليوم، يعيش معظم ذلك في ثلاثة أماكن لا يستطيع الوكلاء الوصول إليها: رؤوس
المهندسين الكبار، وخيوط PR المغلقة، وسجل المحادثات.

العرض متكرر ومكلف. الوكيل المطلوب منه إضافة ميزة خصم يعيد تنفيذ مساعد تنسيق أموال موجود
بالفعل (لأنه لم يعلم أن يبحث)، وينسخ نمط إدراج الفواتير من وحدة `billing/` المهملة (لأنه
استرجع النموذج الخاطئ)، ولا يسأل شيئًا عن تحديد نطاق المستأجرين (لأنه لم يعلم أن هناك ما
يُسأل عنه). كل إخفاق كان إخفاق سياق — النموذج كان على ما يرام.

يجعل هذا الفصل Invoicely *تصف نفسها ذاتيًا*. خريطة `docs/` قصيرة في `AGENTS.md` تخبر أي
وكيل أين توجد المعرفة. يمنح `docs/architecture.md` خريطة النظام في شاشة واحدة بمسارات
ملفات حقيقية. تسجّل ADRs القرارات التي تقيد العمل الجديد، وتقول الوحدة المهملة ذلك *في
الكود*، حيث لا يمكن أن يفوت الاسترجاع ذلك. تعطي ملفات README للوحدات القواعد المحلية
والنموذج المعياري لكل منطقة. تتغير ممارسة الجلسة أيضًا: مهمة واحدة لكل جلسة، وملف
مسودة `docs/plans/` للعمل متعدد الجلسات، وسجلات تُرشَّح قبل اللصق، وضغط عند المعالم بدلاً
من منتصف الاستدلال. اختبار القبول هو ذاك الوارد في ملف README للمرحلة: وكيل جديد، لا يمتلك
إلا المستودع وتذكرة، ينتج عملًا يقع داخل الاتفاقيات — لأن المستودع نفسه يحمل الآن المعرفة.

لماذا هذا الشكل: **ملفات صغيرة في عناوين متوقعة.** يسترجع الوكلاء بالاسم وبالبحث؛
`docs/decisions/0007-money-as-integer-cents.md` قابل للاكتشاف بثلاث طرق مختلفة (الخريطة،
واسم الملف، والمحتوى) ويكاد لا يكلف شيئًا لتحميله وحده. وثيقة مجمعة بطول 3000 سطر ستكون
قابلة للاكتشاف بطريقة واحدة بالضبط وستكلف الميزانية كاملة في كل مرة. دليل `plans/` هو أداة
نظافة الجلسة: يحتفظ العمل طويل الأمد بحالته في ملف — الهدف، والقرارات المتخذة، والخطوات
المتبقية — بحيث تستأنف جلسة جديدة من قراءة 40 سطرًا بدلاً من وراثة 100 ألف token من السجل.

## التنفيذ

### خريطة الوثائق — خمسة أسطر تغير الاسترجاع

تُلحَق بـ `AGENTS.md` (الفصل 02)، بحيث يبدأ كل وكيل ممسكًا بالخريطة:

```markdown
## Where knowledge lives (read on demand, not upfront)
- System map + file paths:      docs/architecture.md
- Why decisions were made:      docs/decisions/ (check before structural changes)
- Domain vocabulary:            docs/domain.md
- Module-local rules/exemplars: README.md inside each app/ module
- Multi-session task state:     docs/plans/<task>.md
Before writing a helper, search app/common/ and the target module —
assume it exists until proven otherwise.
```

السطران الأخيران يقابلان أكثر فشلَي استرجاع شيوعًا: عدم البحث عن كود موجود، وعدم معرفة أن
لقيد ما سببًا مسجلاً.

### architecture.md — خريطة، لا جدارية

```markdown
# Invoicely — system map

Request path: Next.js (frontend/) → FastAPI routers (backend/app/api/)
→ services (app/services/) → repositories (app/repositories/)
→ PostgreSQL. Async jobs: Celery (app/workers/) via Redis.

- AuthN/AuthZ: app/core/auth.py — JWT access + rotating refresh
  (ADR-0009). Tenant = account_id claim; scoping enforced in
  repositories (ADR-0012).
- Money: integer cents end-to-end (ADR-0007). Format only in
  frontend/lib/money.ts and app/common/money.py.
- billing/ is FROZEN (ADR-0015) — being replaced by credit_notes/.
  Never use it as a pattern.
- Test tiers: tests/README.md. Fast tier must stay under 60s.

Update this file in the same PR as any change that invalidates it.
```

شاشة واحدة، كل سطر مؤشر أو ثابت، مسارات حقيقية في كل مكان. التعليم الختامي هو آلية
الحداثة: **الوثائق التي يقرأها الوكيل وثائق حاملة للأحمال**، لذا تُحدَّث في نفس PR الذي
يحوي الكود، ويعامل المراجعون الخريطة المتقادمة كاختبار فاشل. وثيقة لا أحد ملزم بتحديثها
هي وثيقة ستكذب في النهاية — والوكلاء يصدقون الوثائق.

### ADRs — قرارات حيث يستطيع الاسترجاع إيجادها

التنسيق هو تنسيق المرحلة 1
([templates/adr.md](../../templates/adr.md))؛ الجديد هو الجمهور. `docs/decisions/0007-money-as-integer-cents.md`
موجود لكي يسترجع وكيل (أو مهندس) على وشك "تنظيف" السنتات الصحيحة إلى `Decimal`، في ملف
صغير واحد: السياق (أخطاء تقريب float في النماذج الأولية المبكرة)، والقرار، والنتائج
(التنسيق عند الأطراف فقط). الوكيل الذي يعرف *سبب* القرار يمكنه احترام القصد في حالات
لا يغطيها نص القاعدة — ويمكنه الاعتراض بشكل صحيح عندما تتعارض مهمة مع قرار مسجل بدلاً
من الامتثال بصمت.

### الإهمال عند نقطة الاسترجاع

`backend/app/billing/README.md`:

```markdown
# DEPRECATED — do not extend, do not copy patterns
Frozen since 2026-03 (ADR-0015); replaced by app/credit_notes/.
Bug fixes only, and only with a failing test demonstrating the bug.
Canonical patterns for new work: app/invoices/.
```

إعلانات المحادثات وصفحات wiki لا تصل إلى وكيل يبحث عن نموذج؛ أما README في الدليل فيصل.
بالنسبة للاسترجاع القائم على الفهرس (بنمط Cursor)، يحافظ ملف التجاهل على عدم ترشح الكود
المُولَّد والأدلة المشتراة مطلقًا. المبدأ يتعمم: **ضع المعرفة حيث سيحدث الاسترجاع الذي
يحتاجها** — حالات الإهمال في الكود المُهمل، والقواعد المحلية في الوحدة، والخطوات التشغيلية
في runbooks مسماة باسم العملية.

### نظافة الجلسة — تحديد النطاق، والإخراج، وإعادة الضبط

```markdown
# docs/plans/2026-07-discounts.md   (maintained by the agent, reviewed by you)
## Goal
Percentage + fixed discounts on invoices; applies before tax; audit-logged.
## Decisions
- Stored as basis points (int) on invoice_lines — consistent w/ ADR-0007
- New service method, no changes to totals API contract
## Done
- [x] migration 00-add-discount-columns  [x] service + unit tests
## Next
- [ ] API surface + validation  [ ] frontend form  [ ] E2E happy path
```

قواعد العمل التي تصاحبها: **مهمة واحدة لكل جلسة** — عمل الخصم وتحقيق الاختبار المتقلب لا
يتشاركان نافذة أبدًا، لأن البقايا غير المتعلقة هي الملوث الرئيسي. **اضغط عند المعالم،
ليس في منتصف الطيران** — هبوط الـmigration هو نقطة تلخيص؛ منتصف التصحيح ليس كذلك، لأن
الضغط فاقد للمعلومات ويفقد بالضبط التفاصيل الدقيقة التي يحتاجها التصحيح الحي. **رشّح قبل
اللصق** — مخرج الاختبار الفاشل، لا سجل CI بالكامل؛ إطار المكدس ذو الصلة، لا 400 سطر من
INFO (وأبدًا سجل يحتوي رموزًا أو أسرارًا). **استأنف من ملف الخطة** — جلسة الغد تقرأ 40
سطرًا من الحالة المنتقاة بدلاً من إعادة عرض سجل اليوم البالغ 100 ألف token، وهو أرخص
*وأدق*: الخطة تحتفظ بالاستنتاجات، لا المسدودات التي قادت إليها.

## القرارات الهندسية

**القرار 1: تحميل مسبق أم استرجاع؟** التكرار × الشمول. مطلوب في كل مهمة فعليًا (الأوامر،
والثوابت الأساسية، وخريطة الوثائق) ← ملف محمَّل دائمًا، يُحفظ قاسيًا في صغره. مطلوب في
بعض المهام (تفصيل البنية، ومبرر قرار ما، وقواعد الوحدة) ← وثيقة قابلة للاسترجاع في عنوان
متوقع. مطلوب مرة (قيود هذه التذكرة) ← الموجِّه. عندما يريد ملف محمَّل دائمًا أن ينمو، فإن
السؤال ليس "هل هذا صحيح؟" بل "هل يستحق إيجارًا لكل token في كل مهمة مستقبلية؟" — والجواب
عادةً مؤشر، لا فقرة.

**القرار 2: أين تعيش حقيقة معينة؟** بلحظة استرجاعها. تقيد الكود في موضع واحد ← تعليق أو
نوع في ذلك الموضع. تحكم وحدة واحدة ← README الخاص بها. تقطع النظام ← `architecture.md`.
تشرح *سبب* وجودها هكذا ← ADR. تشغيلية ← runbook. حالة مهمة عابرة ← `plans/`. القرار
المعاكس هو Slack، وخيوط PR، والرؤوس — المعرفة التي لا يصلها أي استرجاع هي المعرفة التي يدفع
الفريق ثمن إعادة اكتشافها، وكيلًا كان أم لا.

**القرار 3: كيف تبقى الوثائق صادقة؟** تحديثات نفس-PR، مفروضة في المراجعة — الخريطة، وREADME
الوحدة، وأي ADR يتأثر بالتغيير، يركبون في diff التغيير نفسه. لهذا السبب تبقى الوثائق
*صغيرة*: تكلفة صيانة الوثائق تتناسب مع حجمها، ولا تبقى جديرة بالثقة بما يكفي لتحميلها سوى
الوثائق الرخيصة بما يكفي لصيانتها. شبكة أمان مفيدة: عندما يكشف مخرج الوكيل أنه أُضلّ
بوثيقة متقادمة، يُلزَم الإصلاح بإيداع تصحيح الوثيقة إلى جانب إصلاح الكود — نفس المنعكس
كإضافة اختبار انحدار.

**القرار 4: جلسة واحدة أم كثير لمهمة كبيرة؟** أخرج وقسّم. إذا كان للعمل معالم طبيعية
(migration → service → API → frontend)، فلكل واحدة جلسة تبدأ بقراءة ملف الخطة وتنتهي
بتحديثه. تبدو الجلسات الطويلة المفردة مستمرة لكنها تتدهور بصمت؛ يحوّل ملف الخطة الاستمرارية
من خاصية داخل النافذة (هشّة، ومكلفة) إلى خاصية على القرص (متينة، وقابلة للمراجعة).
احتفظ بالضغط داخل الجلسة للعمل الذي لا يمكن حقًا عمل checkpoint له.

**القرار 5: ما الذي لا يجب أن يدخل النافذة أبدًا؟** القيم السرية، وبيانات PII الإنتاجية،
وبيانات العملاء — مفروضة هيكليًا (قواعد الرفض من الفصل 01؛ fixtures مُطهَّرة من المرحلة 8)،
لا بالنية. السياسة الأكثر دقة: *الكميات غير المفلترة* (سجلات كاملة، وتفريغات قاعدة بيانات
كاملة، وقوائم `node_modules`) محظورة ليس للسرية بل للتلوث — أي شيء يُلصَق يجب أن يمر أولًا
على "ما الجزء من هذا الدليل؟" تلك العادة، عرضيًا، هي معظم انضباط التصحيح في الفصل 07.

## الأخطاء الشائعة

- **إلقاء اللوم على النموذج عند إخفاق السياق.** "أعاد اختراع مساعدنا / استخدم الوحدة القديمة
  / تجاهل الثابت" — قبل تبديل النماذج، اسأل: *كيف كان سيعرف؟* إذا كان الجواب "سجل
  المحادثة" أو "الجميع يعرفون فقط"، فإن الفشل خطؤكم، وسينجو من أي ترقية نموذج.
- **الإغراق بدل الانتقاء.** لصق السجل كاملًا، وقراءة الدليل كاملًا، "ها هو كل شيء، استنتج."
  الحجم يبدو مساعدًا وهو العكس: يدفن الإشارة، ويُضعف الانتباه، ويُنفق ميزانية كانت المهمة
  تحتاجها. رشّح أولًا — الدليل يدخل، الضوضاء تخرج.
- **الجلسة الخالدة.** نافذة واحدة تحمل تصحيح الصباح، وإعادة هيكلة، واستكشافين، والآن تغيير
  schema. يتدهور السلوك تدريجيًا ببطء لدرجة يُقرأ على أنه "النموذج يمر بيوم سيئ." الجلسات
  مجانية؛ حدد نطاقها كمعاملات.
- **وثائق تُكتب مرة، وتُصدَّق إلى الأبد.** وثيقة بنية من إعادة هيكلتين قبل الآن خطر نشط:
  الوكلاء يولون أهمية كبيرة للوثائق وسيسيرون بها إلى الهاوية. لا توجد قاعدة تحديث في نفس
  PR ← طبقة الوثائق تصبح خصمًا على مؤقت تأخير.
- **تكرار الكود في الوثائق.** أوصاف API دالة بدالة، وschemas مُعاد سردها — متقادمة عند
  الوصول، وهدر ميزانية صرف: الوكيل يقرأ الكود بطلاقة والكود دائمًا حديث. تحمل الوثائق ما
  لا يستطيع الكود *قوله*: السبب، والحدود، وأي الأشياء الخمسة المتشابهة هو المعياري.
- **الخلط بين حجم النافذة والمناعة.** "النموذج يتعامل الآن مع مليون token" — لا يزال يحضر
  بشكل غير متساوٍ عبر نافذة محشوة، وكل دوران يعيد قراءة ما حمّلته. تغيّر الهامش؛ الاقتصاد
  والانضباط لم يتغيرا.
- **السماح لأي شيء حساس بالدخول "هذه المرة فقط".** جلسة التصحيح التي يُقرأ فيها `.env`،
  والسجل الذي يحتوي رمزًا حيًا يُلصق كاملًا. النافذة سطح تسريب؛ قواعد المرحلة 9 تنطبق
  عليها تمامًا كما تنطبق على git.

## أخطاء الذكاء الاصطناعي

يفشل المساعدون الثلاثة في السياق في اتجاهات مميزة — وكل إخفاقات الثلاثة تبدو ككفاءة للوهلة
الأولى.

### Claude Code: عمل واثق على خريطة جزئية

يقرأ Claude Code بسرعة ويتصرف بحسم — مما يعني أنه بعد قراءة ملفين أو ثلاثة غالبًا *يظن
أنه حصل على الصورة* ويشرع: يعيد تنفيذ مساعد موجود في `app/common/` ولم يفتحه، ويضيف
استعلامًا يتجاوز mixin تحديد نطاق المستأجرين المعرَّف في فئة أساسية لم يقرأها، ويمدد نمطًا
من الوحدة (المهملة) الوحيدة التي صادف النظر إليها. العمل متماسك، ومتعارف عليه، وخاطئ حول
نظامك بطرق لا يمكن أن يكتشفها إلا من يعرف *بقية* المستودع.

**الاكتشاف:** خدمات جديدة مشبوهة في شبهها بالموجودة؛ سلوك الفئة الأساسية أو الـ middleware
مُكرَّر inline؛ diffs لا يلمس أثر "Read" فيها (المرئي في الجلسة) الوحدات التي تعرّف الاتفاقيات
ذات الصلة؛ كود يُصرَّف ضد نظامك لكنه يتجاهل تعبيراته الاصطلاحية.

**الإصلاح:** اجعل الخريطة حاملة للأحمال والبحث إلزاميًا:

> Before implementing, locate the existing conventions: read docs/architecture.md, the README of
> every module you will touch, and search app/common/ and the target module for existing helpers
> that cover any part of this task. List what you found and what you'll reuse BEFORE writing new
> code. Assume a helper exists until a search proves otherwise.

### GPT: استدلال من لقطة قديمة لملفاتك الخاصة

في الجلسات الطويلة، تحفظ نماذج عائلة GPT في الاستدلال من نسخة الملف التي قرأتها سابقًا —
حتى بعد أن غيّرها (أو غيّرتها أنت). تحتوي النافذة على النسختين؛ يثبّت النموذج على الأولى.
الأعراض: تعديلات تعيد تقديم كود تمت إزالته للتو، وإشارات إلى توقيع دالة من قبل إعادة
تسميتها، و"إصلاح" مُطبَّق على سطر لم يعد موجودًا، ورقع تتعارض مع الملف كما هو قائم حاليًا.
يبدو كإهمال؛ إنه في الحقيقة سياق — النسخة المتقادمة *هناك*، تنافس الجديدة.

**الاكتشاف:** إخفاقات رقع وتعديلات قريبة من الإصابة في أواخر الجلسة؛ انحدارات لتغييرات أُجريت
في وقت سابق من الجلسة نفسها؛ النموذج يصف محتوى ملف لا يطابق `git diff`؛ ارتباك يزول فورًا في
جلسة جديدة.

**الإصلاح:** أعد القراءة قبل الكتابة، وأعد الضبط عند المعالم:

> Before editing any file, re-read its current contents — never edit from memory of an earlier
> read. After each completed step, state the current form of what changed. (And operationally:
> end the session at the milestone; start the next one from the plan file, not from history.)

### Cursor: مثال أقرب جار، إلى أينما يشير

قوة Cursor — السحب التلقائي لكود مشابه إلى السياق — هي أيضًا فشله المميز: التشابه لفظي،
لا معماري. اطلب ميزة جديدة مجاورة لميزة المدفوعات والمثال المسترجع هو وحدة `billing/`
*المهملة* (لأنها تطابق أفضل)؛ اطلب نموذجًا ويأخذ النموذج من المكون legacy الوحيد الذي يسبق
نظام التصميم الخاص بك؛ تطفو fixtures الاختبار والملفات المُولَّدة كما لو كانت كود منتج. ثم
يكتب الوكيل كودًا بطلاقة بالأسلوب الذي تحاول استئصاله — استرجاع غسل النمط legacy إلى عمل
جديد.

**الاكتشاف:** كود جديد يشبه أنماطًا مهملة أو فريدة بدلاً من أمثلتك المعيارية؛ استيراد من
وحدات مجمّدة؛ كود مُولَّد/مشترى يظهر في سياق الوكيل المُشار إليه؛ انحدارات أسلوب متجمعة في
مناطق لها توأم legacy.

**الإصلاح:** انتقي مجموعة الاسترجاع وسمِّ المعيار:

> Add the deprecated and generated paths to the ignore configuration so they never enter
> retrieval. Put a DEPRECATED README inside each frozen module. In each active module's README,
> name the canonical exemplar file for new work — and when given a task, state which exemplar
> you are following before you write.

## أفضل الممارسات

- **شغّل اختبار "الوكيل البارد" كمعيار جودتك.** جلسة جديدة، تذكرة حقيقية، بدون إحاطة لفظية:
  هل يقع المخرج داخل الاتفاقيات؟ كل إخفاق يطابق قطعة سياق مفقودة أو غير قابلة للوصول —
  أصلح المستودع، وأعد التشغيل.
- **حافظ على النواة المحمَّلة دائمًا minimal ومدفوعة إلى المؤشرات.** الخريطة تستحق الإقامة
  الدائمة؛ لا شيء آخر جديد تقريبًا يستحقها. ضغط النمو على `AGENTS.md` يعني عادة أن وجود
  وثيقة أنسب.
- **اكتب وثائق بحجم استرجاع واحد.** قرار واحد لكل ADR، وشاشة واحدة للخريطة، وREADME واحد
  لكل وحدة. الوثائق الصغيرة تُحمَّل وحدها، وتبقى قابلة للصيانة، وتبقى صادقة.
- **اجعل المعرفة متجاورة مع لحظة استرجاعها.** حالات الإهمال في الدليل المُهمل، والقواعد
  المحلية في الوحدة، وخطوات التشغيل في runbooks، والحالة العابرة في ملفات الخطة. إذا كان
  عليك أن تتذكر إخبار الوكيل، فهي في المكان الخطأ.
- **حدّث الوثائق في نفس PR مع الكود — بلا استثناءات.** الوثائق الحاملة للأحمال هي كود. تُحذف
  الإدخالات المتقادمة على الفور؛ وثيقة خاطئة أسوأ من وثيقة مفقودة.
- **حدد نطاق الجلسات كمعاملات؛ أخرج العمل الطويل.** مهمة واحدة لكل نافذة؛ ملفات خطة لأي
  شيء متعدد الجلسات؛ الضغط عند المعالم فقط.
- **رشّح كل ما تلصقه.** الاختبار الفاشل، والإطارات ذات الصلة، وكتلة التكوين المعنية. إذا
  لم تكن ستقتبسها في وصف PR كدليل، فلا تدخل النافذة.
- **راجع سطح الاسترجاع عرضيًا.** قواعد التجاهل لا تزال صحيحة؟ أي أدلة مُولَّدة/مشتراة جديدة
  تتسرب إلى البحث؟ الوحدات المهملة موسومة بشكل صحيح؟ الاسترجاع الخاطئ صامت؛ لا تكتشفه إلا
  المراجعات.
- **حافظ على عدم قابلية الوصول الهيكلي للأسرار.** قواعد الرفض + fixtures مُطهَّرة + عادة
  عدم اللصق بالجملة. "الوكيل لا يرى القيم السرية أبدًا" يجب أن تكون خاصية معمارية، لا
  أملًا.

## الأنماط المعاكسة

**نافذة حوض المطبخ.** "اقرأ قاعدة الكود كاملة أولًا، ثم نبدأ." تبدو شاملة؛ تقدم جلسة
متدهورة ومكلفة مثبّتة على مادة غير ذات صلة إلى حد كبير — والوكيل لن يكون قد قرأ ADR الواحد
الذي يهم. حمِّل الخريطة، واسترجع مجموعة العمل، واجلب الباقي عند الطلب.

**مقبرة المعرفة.** بنية الفريق الحقيقية تعيش في خيوط Slack، وتعليقات PR، ورأسي كبيرين؛
المستودع يحتوي كودًا وصمتًا. كل جلسة وكيل (وشهر أول لكل موظف جديد) تدفع ضريبة إعادة
الاكتشاف. إذا كانت تقيد العمل، تعيش في المستودع — هذه القاعدة أقدم من AI؛ الوكلاء جعلوا
انتهاكها مكلفًا لكل جلسة بدلًا من لكل موظف.

**الجدارية الوثائقية.** ARCHITECTURE.md بطول 3000 سطر، مكتملة بحب، تُحدَّث مرتين في
السنة. كبيرة جدًا لتحميلها، أحادية جدًا لئلا يُعتمد عليها، ومكلفة جدًا لصيانتها — مكافئ
الوثائق لـgod object. فككها على خطوط الاسترجاع: خريطة، وقرارات، ووحدات، وrunbooks.

**اكتناز السياق.** رفض الجلسات الجديدة بحجة "لقد فهم قاعدة الكود أخيرًا." ما لديه نافذة
ممتلئة بقراءات لمرة واحدة تتقدم نحو التقادم — فهم لا يمكن فحصه، أو مشاركته، أو الوثوق به
غدًا. أي شيء يستحق الاحتفاظ به من جلسة ينتمي إلى وثيقة أو ملف خطة؛ ثم الجلسة قابلة
للتصرف. المعرفة المتينة تعيش على القرص، أبدًا في نافذة.

**اعتماد الإحاطة اللفظية.** جودة المخرج تعتمد على تنزيل شفهي مدته خمس عشرة دقيقة ممن
يعرف المنطقة. لا تتسع لوكلاء متوازيين (الفصل 05)، ولا تنجو من إجازة ذلك الشخص، وتعفي
المستودع سرًا من أن يكون صفًا لذاته. الإحاطة *هي* الوثيقة المفقودة — اكتبها مرة.

## شجرة القرار

"قطعة من المعرفة تهم العمل — إلى أين تذهب؟"

```
Is it a secret / credential / customer data?
├── YES ──► It never enters the window. Deny rules, sanitized
│           fixtures, filtered pastes. (Stage 9 Ch 04.)
└── NO ──► Is it discoverable by reading current code?
    ├── YES ──► Leave it in code. Duplicating it into docs creates
    │           a staleness liability and spends budget twice.
    └── NO ──► Is it needed on effectively EVERY task?
        ├── YES ──► Always-loaded file (AGENTS.md) — one imperative
        │           line, or a pointer if it's bigger than a line.
        └── NO ──► Does it explain WHY something is the way it is?
            ├── YES ──► ADR in docs/decisions/, linked from the map.
            └── NO ──► Is it scoped to one module / one operation?
                ├── module ──► that module's README (+ canonical
                │              exemplar named).
                ├── operation ──► runbook.
                └── crosscutting ──► architecture.md — kept to one
                                     screen; if it won't fit, it's
                                     probably several module facts.

"Should I load X into THIS session?"
    Does the current task need it → load the relevant PART.
    Might it help → point the agent at the map and let it decide.
    It's bulk (log/dump/tree) → filter to evidence first.
    The session is old and X contradicts something read earlier →
        don't add — reset, and start clean from the plan file.
```

  في الملف نفسه
- [ ] توجد ADRs للقرارات التي ينتهكها الوكلاء في الأغلب (الأموال، والمستأجرون، والوحدات المجمّدة)
- [ ] الوحدات المهملة تحمل README DEPRECATED تسمّي البديل والمثال المعياري
- [ ] ملفات README للوحدات في المناطق النشطة: قواعد محلية + ملف مثال

- [ ] المسارات الحساسة غير قابلة للقراءة هيكليًا (تم التحقق المباشر من قواعد الرفض في الفصل 01)

**قائمة مراجعة البنية**

- [ ] النواة المحمَّلة دائمًا minimal؛ كل شيء آخر قابل للاسترجاع في عنوان متوقع
- [ ] كل حقيقة حاملة للأحمال لها بيت واحد بالضبط (كود / خريطة / ADR / README / runbook)
- [ ] الوثائق بحجم استرجاع واحد؛ لا جداريات
- [ ] سياسة جلسة مكتوبة: مهمة واحدة لكل نافذة، ضغط عند المعالم، استئناف من ملف الخطة
- [ ] اختبار الوكيل البارد يجتاز تذكرة تمثيلية — ويُعاد تشغيله بعد التغييرات الهيكلية

**قائمة مراجعة الكود**

- [ ] التغيير يُبطل وثيقة؟ تحديث الوثيقة في هذا diff — وإلا فالـ PR غير منتهي
- [ ] وحدة جديدة ← README بقواعد + مثال؛ قرار جديد ← ADR؛ إهمال جديد ← README في المكان
  + تحديثات التجاهل/الخريطة
- [ ] لا أسرار ولا رموز ولا بيانات عملاء في أي وثيقة أو ملف خطة أو مثال مُلتزَم
- [ ] ملفات الخطة تعكس الواقع وقت الدمج (الخطط المتقادمة تضلّل الجلسة التالية)
- [ ] diffs من إنتاج الوكيل: تحقق من العلامات — مساعدات مُعادة اختراعها، استيرادات أنماط
  مهملة، تعديلات تحارب حالة الملف الحالية

## التمارين

1. **شغّل تدقيق الوكيل البارد.** جلسة جديدة، تذكرة حقيقية من ركامك، سياق لفظي صفري. سجّل
   كل إخفاق (نمط خاطئ، مساعد مُعاد اختراعه، ثابت مُنتهَك، سبب مفقود) واكتب إصلاح *السياق*
   لكل واحد — سطر خريطة، أو ADR، أو README، أو قاعدة تجاهل. أعد تشغيل نفس التذكرة باردًا.
   الفرق هو عائد طبقة السياق لديك، مقاسًا.
2. **ابنِ الخريطة.** اكتب `docs/architecture.md` لمستودعك بأقل من 40 سطرًا بمسارات ملفات
   حقيقية، وخريطة الوثائق ذات الأسطر الخمسة لملف التعليمات لديك. سلّمهما لزميل لا يعرف
   المنطقة: ما يعجز عن إيجاده هو ما لا يزال مفقودًا.
3. **استخرج ثلاثة قرارات.** اعثر على ثلاثة قواعد في قاعدة كودك موجودة لأسباب غير مسجلة في
   أي مكان (اسأل الشخص الذي يعرف — هذه هي النقطة). اكتب ADRs الثلاثة. ثم اطلب من وكيل
   "تحسين" كود مقيد بكل منها لترى هل يسترجعها ويحترمها — أم يجادلها، وهي أيضًا نجاح.
4. **أصلح فخ استرجاع.** حدد أخطر مثال في مستودعك (وحدة مهملة، أو مكون legacy، أو كود
   مُولَّد) وتأكد أن وكيلًا يمكنه حاليًا استرجاعه كنمط. ثبّت الإجراءات المضادة — README في
   المكان، وقواعد تجاهل، ومعيار مسمّى في الوحدة النشطة — وتأكد أن نفس المهمة تأخذ الآن
   نمطها من الكود الصحيح.
5. **حوّل جلسة طويلة.** خذ مهمتك التالية متعددة الأيام وشغّلها كجلسات ملف خطة: أخرج الحالة
   عند كل معلم، واقتل كل نافذة، واستأنف باردًا. قارن ضد جلستك الماراثونية المعتادة في تماسك
   مخرج المرحلة المتأخرة — وفي ما أمسك به ملف الخطة عند مراجعتك لم يكن ليستكشفه أي نافذة.
