# Claude Code

## المقدمة

Claude Code هو وكيل برمجة يعمل داخل الطرفية (terminal): تعطيه مهمةً بلغة طبيعية، فيعمل
نحو إنجازها عبر قراءة الملفات، وتعديل الشيفرة، وتشغيل الأوامر في حلقة — يُلاحظ
نتيجة كل إجراء، ويُقرّر الإجراء التالي — حتى تنتهي المهمة أو يحتاج إلى مُدخلات منك.
تلك الحلقة هي الفرق بين *مساعد* و*وكيل*. المساعد يقترح الشيفرة وأنت تُدمجها؛ أما
الوكيل فيعمل مباشرة على مستودعك، ويتحقق من عمله بنفسه عبر اختباراتك، ويُسلّمك فرق
التعديلات (diff).

هذه القوة هي بالضبط سبب وجوب هندستها بدلًا من مجرد استخدامها. الوكيل الذي يُعدّل
الملفات ويُشغّل الأوامر يرث كل خاصية في البيئة التي تشغّله فيها: اتفاقياتك (أو غيابها)،
وأذوناتك (أو غيابها)، وأوامر التحقق عندك (أو غيابها). يخرج Claude Code من علبته بإعدادات
افتراضية عامة قوية وبمعرفة صفرية بنظامك. كل ما يتناوله هذا الفصل — ملف التعليمات
`CLAUDE.md`، ونموذج الإعدادات والأذونات، والـ hooks، والأوامر والمهارات المخصصة، وامتدادات
الأدوات، والتشغيل غير التفاعلي — هو سطح الإعداد الذي يستبدل تخميناته بمعاييرك الهندسية.

يتناول هذا الفصل Claude Code بعمق بصفته الوكيل المرجعي لهذه المرحلة. يتناول الفصل 02
كلًا من OpenAI Codex وCursor، والأهم من ذلك: أيٌّ من هذه المهارات تنتقل بين الأدوات —
والإجابة الصادقة هي *معظمها*، لأن الشكل الأساسي (حلقة وكيل تُهيّئها بالتعليمات والحدود
والتغذية الراجعة) مشترك. تتطور أسماء الميزات والأعلام بسرعة؛ أما الحكم الهندسي حول ما
تُهيّئه ولماذا فلا يتطور. عند تعارض تفصيل هنا مع التوثيق الرسمي الحالي، ثِق بالتوثيق.

## لماذا هذا مهم

الفجوة بين مطوّر "يستخدم Claude Code" ومطوّر هندَس مستودعه لاستخدامه أكبر من الفجوة
بين النماذج. الإعداد هو مكان الرافعة الحقيقية.

- **الوكيل غير المُعدَّل يُحسِّن للمقبول، لا لمتطلباتك.** من دون تعليمات، يخمن
  الاتفاقيات من بيانات التدريب: نمط ORM جديد هنا، وأسلوب مختلف لمعالجة الأخطاء هناك،
  واختبارات بإطار خاطئ. كل مُخرج "يعمل" بمعزل عن غيره، ويقوّض تماسك قاعدة الشيفرة
  إجمالًا. الوكيل المُعدَّل يضع الشيفرة داخل معماريتك — والفرق يتراكم مع كل مهمة.
- **حلقات التغذية الراجعة تحدد جودة المُخرج أكثر مما تحددها التعليمات (prompts).**
  الوكيل القادر على تشغيل اختباراتك والـ linter وفاحص الأنواع (type checker) بعد كل
  تغيير يلتقط أخطاءه بنفسه ويُكرّر؛ أما الذي لا يستطيع فيُسلّمك شيفرة تبدو فقط
  *منتهية*. إتاحة أوامر التحقق وتعريفُ الوكيل بها هي خطوة الإعداد الأعلى عائدًا
  على الإطلاق.
- **الأذونات هي الفرق بين أداة قوية وخطر داهم.** الوكيل القادر على تشغيل أوامر
  عشوائية قادر أيضًا على تشغيل أوامر مدمّرة — ضد قاعدة بياناتك، أو تاريخ git
  الخاص بك، أو (عبر بيانات اعتماد مُسرّبة في البيئة) بيئة الإنتاج. سياسة أذونات
  مدروسة تتيح لك منح الاستقلالية حيث تكون آمنة، وتطلب التأكيد حيث لا تكون آمنة،
  بدلًا من الاختيار بين مراقبة كل إجراء وتعطيل كل وسائل السلامة.
- **اتساق الفريق يتطلب إعدادًا مشتركًا.** إذا استدعى كل مهندس الوكيل من الصفر،
  حصلْت على خمس لهجات خاصة بقاعدة شفرتك. الإعداد المُلتزم — ملفات التعليمات،
  والإعدادات، والـ hooks، والأوامر القابلة لإعادة الاستخدام — يجعل سلوك الوكيل أصلًا
  فريقِيًّا مُراجَعًا ومُنسَّخ الإصدار، تمامًا كإعداد CI.
- **التشغيل غير التفاعلي يحوّل الوكيل إلى بنية تحتية.** يُشغَّل Claude Code
  غير تفاعليًا — في CI، وفي السكربتات، وبجدولة — مما يُتيح تمريرات المراجعة الآلية،
  وفرز المشكلات، والصيانة الدورية (الفصل 08). هذا لا يؤتي ثماره إلا إذا كان الإعداد
  التفاعلي منضبطًا أصلًا، لأنه في CI لا أحد يراقب الحلقة.
- **كلفة الخطأ هنا صامتة.** لا يحدث أي عطل عندما يكون وكيلك مُعدًّا إعدادًا خاطئًا.
  ستقضي أسابيع فقط في قبول فروق تعديلات شبه صحيحة، وتكرار التصحيحات نفسها، ومراجعة
  شيفرة تتجاهل قرارات اتخذتها قبل أشهر — تسرّب إنتاجي بلا رسالة خطأ.

## النموذج الذهني

Claude Code حلقة، وأنت تُهندس البيئة التي تعمل فيها الحلقة.

```
                        THE AGENT LOOP
             ┌────────────────────────────────────┐
             │              CONTEXT               │
             │  system prompt + CLAUDE.md +       │
             │  conversation + files it has read  │
             └──────────────────┬─────────────────┘
                                ▼
                       MODEL DECIDES ACTION
                                │
                                ▼
             ┌──────────────── TOOLS ──────────────┐
             │  read file · edit file · run command│
             │  search · git · extensions (MCP)    │
             └──────────────────┬──────────────────┘
                                ▼
                     ENVIRONMENT RESPONDS
             (file contents, test output, lint errors,
              command results — this is the feedback)
                                │
                                ▼
                    result appended to CONTEXT
                                │
                     loop until done / blocked

   YOU ENGINEER THE ENVIRONMENT, NOT THE MODEL:
     instructions  → CLAUDE.md          (what it should know every session)
     boundaries    → permissions, hooks (what it may and may not do)
     capabilities  → tools, MCP servers (what it can reach)
     feedback      → tests, lint, types (how it knows it's wrong)
     reuse         → commands, skills   (how good workflows get repeated)
```

ثلاث نتائج لشكل الحلقة تُحدّد كل شيء آخر:

**الوكيل لا يكون بجودة تغذيته الراجعة.** تُصحّح الحلقة نفسها بالضبط حيث تدفع البيئة
في الاتجاه المعاكس — اختبار فاشل، أو خطأ أنواع، أو hook يرفض تعديلًا — وتنحرف بالضبط
حيث لا يوجد دفعٌ معاكس. مستودع ذو اختبارات سريعة وقواعد lint صارمة *يُعلّم الوكيل*
في كل تكرار؛ مستودع بلا أيٍّ منهما يمرّر كل خطأ مقبول. لهذا السبب مجموعة اختبارات
المرحلة 8 هي شرط للسرعة المعتمدة على AI أولًا، وليست بديلًا عنها.

**السياق ميزانية، وكل شيء يتنافس عليها.** يُماري النموذج ضمن نافذة محدودة تحتوي
تعليماتك، والمحادثة، وكل ملف ومخرج أمر رآه. `CLAUDE.md` المُنتفخ يحرم سياق المهمة
من المساحة؛ والجلسة التي تاهت عبر عشر مهمات تدفن المهمة الحالية في بقايا التسع
السابقة. إدارة ما *يدخل* الحلقة هي موضوع الفصل 03 بأكمله؛ هنا يضع قاعدةً واحدة: كل
ما تُهيّئه ليُحمَّل في كل جلسة يجب أن يستحق رموزه (tokens).

**التعليمات احتمالية؛ الـ hooks حتمية.** `CLAUDE.md` يجعل السلوك *مرجّحًا*. الـ hooks
— السكربتات التي تُشغّلها بيئة التشغيل على أحداث مثل "على وشك تعديل ملف" أو "لتوّه
شغّل أمرًا" — تجعل السلوك *مؤكدًا*، لأنها تنفذ خارج النموذج وتستطيع حجب الإجراءات
أو تصحيحها آليًا. القواعد التي لا تتحمل كسرها تنتمي إلى التنفيذ الحتمي (hooks، CI،
أذونات)، لا إلى نثر قد يُعطي النموذج أولوية أقل له تحت سياق ممتلئ.

تعريف عملي:

> **Claude Code حلقة وكيل — نموذج، أدوات، تغذية راجعة من البيئة — تعمل مباشرة على
> مستودعك. هندستها تعني تصميم البيئة: تعليمات تُرمّز اتفاقياتك، وأذونات وhooks تحدّ
> ما يمكنه فعله، وأوامر تحقق تمكّنه من التقاط أخطائه بنفسه، وإعداد قابل لإعادة
> الاستخدام يجعل كل ذلك أصلًا فريقِيًّا مُنسَّخ الإصدار بدلًا من عادات طلب فردية.**

## مثال إنتاجي

**Invoicely** — خدمة الفوترة SaaS من المراحل 3–9 — مستودع إنتاجي حقيقي: خادم خلفي
بـ FastAPI مع طبقة خدمات، وPostgreSQL مع ترحيلات Alembic، وواجهة أمامية بـ Next.js،
ونشر قائم على Docker، ومجموعة اختبارات ذات طبقة وحدات سريعة وطبقة تكامل مُحوَّلة
بـ Docker، وتقوية أمنية يجب ألا تتراجع. يحتاج مهندس جديد أيامًا لاستيعاب كل ذلك.
الوكيل غير المُعدَّل لا يستوعبه أبدًا — يبدأ من الصفر في كل جلسة.

يُنشئ هذا الفصل طبقة Claude Code الخاصة بـ Invoicely ليكون الوكيل مُنتجًا وآمنًا
من أول مطالبة (prompt). يذكر `CLAUDE.md` الموجود في جذر المستودع المكدّس التقني،
والأوامر التي تتحقق من العمل، والقواعد البنيوية الخمس للمعمارية (المنطق في الخدمات،
والـ routers الرفيعة، والترحيل عبر Alembic فقط، والمال كعدد صحيح بالسنتات، وعزل
المستأجر في كل استعلام)، ومؤشرات إلى توثيق أعمق. يمنح `settings.json` المُلتزم
الوكيل وصولًا سلسًا إلى الأوامر قراءةً فقط وأوامر التحقق (الاختبارات، وlint، وفحص
الأنواع) مع اشتراط التأكيد على عمليات git push، ورفض الوصول إلى ملفات `.env`
والأوامر المدمّرة لقواعد البيانات. يُنفّذ hookان القواعد غير القابلة للتفاوض التي
لا يضمنها النثر: أحدهما يحجب أي تعديل مباشر على ملفات الترحيل المُنسّخة، والآخر
يُشغّل linter على كل ملف Python مُعدّل لتصحيح انتهاكات الأسلوب داخل الحلقة بدلًا
من ظهورها في المراجعة. يُغلِّف أمر مخصص سير عمل الفريق المتكرر "إضافة endpoint"
— الذي سيتناوله الفصل 04 بصفته أثر هندسة المطالبات (prompt engineering) — ويُشغّل
CI وكيل Claude Code غير تفاعليًا للمراجعة الأولية في كل طلب سحب (built in
Chapters 06 and 08).

مقياس النجاح ملموس: وكيلٌ يُطلب منه "add a `paid_at` timestamp to invoices" يُنتج
ترحيل Alembic (لا مخططًا مُعدّلًا يدويًا)، وتغييرًا في طبقة الخدمات مع عزل المستأجر،
واختبارات بالنمط القائم، وتشغيل تحقق ناجحًا — دون أن تذكر المطالبة أيًّا من هذه
المتطلبات، لأن البيئة تحملها.

## بنية المجلدات

طبقة هندسة AI في مستودع Invoicely:

```
invoicely/
├── CLAUDE.md                      # agent instructions, loaded every session — the
│                                  #   repo's standards in ~100 firm lines (see below)
├── AGENTS.md                      # cross-tool instruction file (Ch 02); CLAUDE.md can
│                                  #   simply reference it to avoid duplication
├── .claude/
│   ├── settings.json              # committed team policy: permissions, hooks — reviewed
│   │                              #   in PRs like CI config, because it IS policy
│   ├── settings.local.json        # personal overrides (gitignored) — one engineer's
│   │                              #   extra permissions never silently become team policy
│   ├── commands/
│   │   └── new-endpoint.md        # reusable slash command for a recurring workflow —
│   │                              #   captured once, versioned, improved by review
│   ├── agents/
│   │   └── test-writer.md         # subagent definition (Ch 05) — scoped role with its
│   │                              #   own instructions and reduced tool access
│   └── hooks/
│       ├── protect-migrations.sh  # deterministic guard: blocks edits to applied
│       │                          #   migrations — a rule too important for prose
│       └── lint-on-edit.sh        # feedback injector: lint output returns to the loop
│                                  #   so the agent fixes style before you see the diff
├── docs/
│   ├── architecture.md            # deep context the agent reads ON DEMAND — kept out
│   │                              #   of CLAUDE.md to protect the context budget
│   └── decisions/                 # ADRs (Stage 1) — the "why" behind rules, so the
│                                  #   agent can honor intent, not just letter
├── backend/                       # the product itself, Stages 3–9
├── frontend/
└── .github/workflows/
    └── ai-review.yml              # headless Claude Code in CI (Ch 06/08)
```

سبب هذا التقسيم: **`CLAUDE.md` هو العقد المحمَّل دائمًا** (صغير، وحازم، وعام)،
و**`docs/` عمق يُقرأ عند الطلب** (يُقرأ عند الحاجة، بلا تكلفة خلاف ذلك)، و**`.claude/`
هو السلوك** (سياسة وأتمتة، مُلتزم به ومُراجَع)، و**`settings.local.json` هو منفذك
الشخصي** الذي يُبقي التفضيلات الفردية خارج سياسة الفريق. يعكس التمييز بين الملتزم
والمحلي كل نظام إعداد آخر في هذا الدليل: السلوك المشترك مُنسَّخ الإصدار، والسلوك
الفردي ليس كذلك.

## التنفيذ

### CLAUDE.md — المعايير، لا قاعدة الشيفرة

البداية الكاملة مع تعليل لكل قسم موجودة في
[templates/claude-md-starter.md](../../templates/claude-md-starter.md). نسخة Invoicely،
مختصرة:

```markdown
# CLAUDE.md

## Project
Invoicely: multi-tenant invoicing SaaS (FastAPI + PostgreSQL backend,
Next.js frontend). In production with paying customers — prefer boring,
proven approaches; breaking changes need a migration path.

## Commands
- Test (fast tier):   cd backend && pytest -m "not integration" -q
- Test (full):        cd backend && make test-integration
- Lint + format:      cd backend && ruff check . && ruff format --check .
- Types:              cd backend && mypy app/
- Migrations:         cd backend && alembic revision --autogenerate -m "..."
- Frontend checks:    cd frontend && npm run lint && npm run typecheck

Run the fast test tier and lint after every change. A task is not done
until they pass.

## Architecture rules
- Business logic lives in app/services/, never in routers. Routers stay
  thin so logic is testable without HTTP.
- Database access goes through repositories; services never build raw SQL.
- Schema changes ONLY via new Alembic migrations. Never edit an existing
  migration file; never modify models without a migration.
- Money is integer cents everywhere. Never float, never Decimal in APIs.
- Every query on tenant-owned tables must filter by the authenticated
  account_id. No exceptions — this is our tenant isolation boundary.

## Conventions
- Follow the patterns of neighboring code over general best practices.
- New endpoints copy the structure of app/api/invoices.py.
- Tests use the factories in tests/factories.py — don't hand-build models.
- No new dependencies without asking.

## Deeper context (read when relevant)
- docs/architecture.md — system overview and layering
- docs/decisions/ — ADRs; check before proposing structural changes
```

كل سطر إمّا يُغيّر سلوك الوكيل أو يُخبره بكيفية التحقق من العمل. لا يوجد نثر يمكن
للوكيل تعلّمه بقراءة الشيفرة — فذلك سيُنفق السياق على ما هو قابل للاكتشاف. لاحظ
النبرة: أمرية ومحددة ("never edit an existing migration") لا طموحة ("we value clean
code"). ملفات التعليمات مواصفات، لا وثائق ثقافية.

### settings.json — سياسة الأذونات

```json
{
  "permissions": {
    "allow": [
      "Bash(pytest*)",
      "Bash(ruff*)",
      "Bash(mypy*)",
      "Bash(alembic revision*)",
      "Bash(npm run lint*)",
      "Bash(npm run typecheck*)",
      "Bash(git status*)",
      "Bash(git diff*)",
      "Bash(git log*)"
    ],
    "ask": [
      "Bash(git push*)",
      "Bash(alembic upgrade*)",
      "Bash(docker*)"
    ],
    "deny": [
      "Read(.env*)",
      "Read(**/secrets/**)",
      "Bash(psql*)",
      "Bash(rm -rf*)"
    ]
  },
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
                    "command": ".claude/hooks/protect-migrations.sh" }]
      }
    ],
    "PostToolUse": [
      {
        "matcher": "Edit|Write",
        "hooks": [{ "type": "command",
                    "command": ".claude/hooks/lint-on-edit.sh" }]
      }
    ]
  }
}
```

تُقرأ السياسة كثلاث درجات من الثقة. **Allow** تغطي التحقق والفحص قراءةً فقط —
الأوامر التي *تريد* أن يُشغّلها الوكيل باستمرار، بلا أي احتكاك، لأنها حلقة التغذية
الراجعة. **Ask** تغطي الإجراءات ذات الأثر المتجاوز لشجرة العمل — الدفع، وتطبيق
الترحيلات، ولمس الحاويات — حيث إن نظرة بشرية تأمين رخيص. **Deny** تغطي ما لا يحتاجه
الوكيل أبدًا: القيم السرّية (السر الملصوق في السياق مُفصح عنه — المرحلة 9، الفصل 04)
والوصول المباشر إلى قاعدة البيانات أو الوصول المدمّر لنظام الملفات. تتطور صياغة
المطابقة (matcher) وأسماء الأحداث بالضبط؛ تحقّق من التوثيق الحالي بدلًا من الوثوق
بأي مثال، بما في ذلك هذا.

### الـ Hooks — التنفيذ الحتمي

يتلقى `protect-migrations.sh` استدعاء الأداة المُعلَّق بصيغة JSON ويرفض تعديلات
الترحيلات المُطبَّقة:

```bash
#!/usr/bin/env bash
# Blocks edits to existing Alembic migration files. New migrations are
# created via `alembic revision` (allowed); history is immutable (Stage 6).
file_path=$(jq -r '.tool_input.file_path // empty')
if [[ "$file_path" == *"/alembic/versions/"* ]] && [[ -f "$file_path" ]]; then
  echo "BLOCKED: $file_path is an applied migration. Create a new migration
with 'alembic revision' instead of editing history." >&2
  exit 2   # non-zero: block the action; stderr goes back to the agent
fi
exit 0
```

يحجب كود الخروج الإجراء آليًا، وتعود رسالة stderr إلى الحلقة — فلا يفشل الوكيل
فحسب، بل يتعلّم الحركة الصحيحة *داخل المهمة نفسها*. هذا نمط كل hook: إنفاذ الحد،
ثم تعليم البديل. `lint-on-edit.sh` هو الفكرة نفسها مقلوبة — يُشغّل `ruff` على الملف
المُعدّل ويُعيد الانتهاكات كتغذية راجعة، ليُصحَّح الأسلوب داخل الحلقة بدلًا من
مراجعتك.

### أمر قابل لإعادة الاستخدام

يُغلِّف `.claude/commands/new-endpoint.md` سير عمل متكررًا في أمر slash يستدعيه
الفريق بأكمله عبر `/new-endpoint`:

```markdown
Add a new API endpoint: $ARGUMENTS

Follow this exact sequence:
1. Read app/api/invoices.py and mirror its structure precisely.
2. Schema in app/schemas/, service method in app/services/,
   route in app/api/. Router stays thin.
3. Tenant scoping: every query filters by account_id from the
   authenticated principal. Ownership returns 404, not 403.
4. Tests: happy path, validation failure, cross-tenant access
   (must 404), unauthenticated (must 401). Use tests/factories.py.
5. Run the fast test tier and lint. Fix everything before reporting.
```

هذه مطالبة (prompt) بوصفها أثرًا مُنسَّخ الإصدار — يُطوّر الفصل 04 هذا الانضباط؛
الآلية التي يجب ملاحظتها هنا أن سير العمل الجيد يُلتقط في المستودع، ويُراجَع
كالشيفرة، ويتوقف عن العيش في سجل دردشة مهندس واحد.

### غير التفاعلي — الوكيل بصفته بنية تحتية

```bash
claude -p "Run the fast test tier. If any test fails, summarize each
failure with the file and likely cause. Output markdown." \
  --allowedTools "Bash(pytest*),Read" \
  --output-format json > triage.json
```

يأخذ الوضع غير التفاعلي مطالبة، ويُشغّل الحلقة نفسها بمجموعة أدوات مضيّقة صراحة،
ويخرج — وهذا ما يجعل تمريرات مراجعة CI (الفصل 06) والصيانة المجدولة (الفصل 08)
ممكنة. أهمية تضييق `--allowedTools`: في CI لا أحد يُجيب على مطالبات الأذونات،
فتمنح بالضبط ما تحتاجه المهمة ولا أكثر.

## القرارات الهندسية

**القرار 1: ماذا يدخل في CLAUDE.md مقابل `docs/` مقابل لا شيء على الإطلاق؟**
الاختبار هو *التكرار مضروبًا في قابلية الاكتشاف*. مطلوب في كل جلسة وغير قابل
للاكتشاف من الشيفرة (الأوامر، والقواعد غير القابلة للتفاوض، والاتفاقيات التي تبدو
اختيارية) → `CLAUDE.md`. مطلوب أحيانًا وكبير جدًا ليُحمَّل دائمًا (الأساس المنطقي
للمعمارية، وADRs) → `docs/`، مع مؤشر من سطر واحد في `CLAUDE.md`. قابل للاكتشاف
بقراءة الشيفرة (ما الدوال الموجودة، وبنية المجلدات الحالية) → لا شيء؛ الوكيل يقرأ
الشيفرة بسلاسة، وتكرارها يخلق التزام تقادم. تحتفظ Invoicely بـ `CLAUDE.md` قريبًا
من 100 سطر بدفع العمق إلى `docs/` — حمايةً لميزانية السياق.

**القرار 2: تعليمات أم hooks لقاعدة ما؟** بحسب كلفة الفشل. "تفضّل دوال المصنع
الخاصة بنا في الاختبارات" — الانتهاك يكلّف تعليق مراجعة واحد، فالتعليمات
(احتمالية) تكفي. "لا تُعدّل أبدًا ترحيلًا مُطبَّقًا" — الانتهاك قد يُفسد تاريخ
المخطط عبر البيئات (المرحلة 6)، فتحصل على hook (حتمية). قاعدة الإبهام: إذا كنت
تريد لـ CI أن يرفضها، فالنثر لا يكفي؛ رمّزها آليًا. الـ hooks تحمل كلفة صيانة،
فاحتفظ بها للقواعد التي يكون انتهاكها مكلفًا — تُطلق Invoicely اثنين، لا عشرين.

**القرار 3: كم الاستقلالية؟** سياسة الأذونات قرص، لا ثنائية. تسمح Invoicely تلقائيًا
بحلقة التغذية الراجعة (الاختبارات، وlint، والأنواع — تكرار عالٍ، خطر صفر)، وتسأل
في إجراءات الأثر (push، وmigrate، وdocker — تكرار منخفض، فالمطالبات رخيصة)، وترفض
الأسرار والأوامر المدمّرة قطعيًا (لا توجد مهمة شرعية تحتاجها). توجد أوضاع ذات
صلاحيات كاملة لسبب — لكن السبب هو *بيئات معزولة وقابلة للتخلص منها* (حاوية، أو
worktree، بلا بيانات اعتماد إنتاجية)، لا محطة عمل بإمكانها الوصول إلى الإنتاج.
طابِق الاستقلالية مع نصف قطر أثر البيئة، لا مع صبرك حيال المطالبات.

**القرار 4: خطّط أولًا، أم اتركه يبني؟** للعمل متعدد الملفات، اشتراط خطة قبل
التعديلات (لـ Claude Code وضع تخطيط مخصص؛ وتعليم "اقترح نهجًا أولًا، لا تُعدّل
بعد" يعمل في أي أداة) يكلّف دقيقة مراجعة واحدة ويلتقط صنف الفشل المكلف: نهج
خاطئ نُفّذ بدقة. قاعدة Invoicely — خطّط لكل ما يلمس أكثر من ملفين أو أي مخطط؛
تنفيذ مباشر لمهام ملف واحد محددة. مراجعة الخطة دقائق؛ فك ارتباط فرق 15 ملفًا
خاطئًا بثقة يستغرق ساعات.

**القرار 5: أين يعيش إعداد الفريق؟** في المستودع، مُراجَع في طلبات السحب (PRs).
`settings.json`، والـ hooks، والأوامر سياسة — توسيع مهندسٍ لصلاحية أو إضعافه لـ hook
هو تغيير في ما يمكن للوكيل فعله بقاعدة الشيفرة، ويستحق نفس مراجعة تغيير CI.
التفضيلات الشخصية تذهب إلى `settings.local.json` المُستبعد من git. النمط المضاد هو
إعداد خفي لكل آلة: الفريق يعتقد أن لديه سياسة، وفي الحقيقة لديه خمس.

## الأخطاء الشائعة

- **قبول "تم" دون تحقق.** يُبلِّغ الوكيل النجاح بنفس الثقة سواء اجتازت الاختبارات
  أو تم تخطيها أو لم تكن موجودة أصلًا. إذا لم تكن أوامر التحقق في `CLAUDE.md`
  ومُطالَبة ("a task is not done until they pass")، فأنت تراجع تقييم الوكيل الذاتي
  لا عمله. اقرأ فرق التعديلات؛ تحقق أن الاختبارات قد نُفّذت فعلًا.
- **معاملة الأداة كنافذة محادثة لا كوكيل.** لصق أجزاء شيفرة في محادثة ونسخ
  الإجابات يضيّع الحلقة برمتها — الوصول للملفات، وتنفيذ الأوامر، والتحقق الذاتي.
  إذا وجدت نفسك تنسخ و تلصق، فأنت تستخدم 10% من الأداة.
- **جلسة أبدية واحدة.** السياق المتراكم عبر مهمات غير مترابطة يُدهور كل مهمة
  لاحقة — تتلاشى التعليمات السابقة، وتبقى نسخ ملفات قديمة، ويصبح السلوك متقلبًا.
  ابدأ من جديد (أو امسح) لكل مهمة؛ الجلسات رخيصة، السياق الملوث ليس كذلك.
- **عدم تحديث CLAUDE.md أبدًا.** تُكتب المرة الواحدة عند الإعداد، ثم يقضي الفريق
  شهورًا في إعادة كتابة نفس التصحيح في المطالبات. الانضباط: عندما تصحّح للوكيل
  مرتين على الشيء نفسه، يصبح ذلك التصحيح سطرًا في `CLAUDE.md` — إنها حلقة تغذية
  راجعة لمراجعة الشيفرة عند الوكيل، ويجب أن تتقارب.
- **إرهاق الأذونات ← yolo.** تبدو المطالبات احتكاكًا، فيُشغَّل الوكيل بكل شيء
  مسموح — على آلة بها بيانات اعتماد إنتاجية في بيئة الصدفة. الحل هو الضبط
  (السماح التلقائي بالأوامر الآمنة المتكررة لتصبح المطالبات نادرة وذات معنى)، لا
  الاستسلام.
- **مطالبات ضخمة غامضة.** "حسّن وحدة الفوترة" ينتج فرق تعديلات متشرذخًا لم
  يطلبه أحد. الوكلاء يتفوقون مع أهداف محددة الحجم قابلة للتحقق — موضوع الفصل 04،
  لكن الفشل عادةً ما يُلقى على الأداة لا على المطالبة.
- **إلقاء اللوم على النموذج لفشل بيئي.** "يستمر في كتابة SQL خام في الخدمات" ليس
  قصورًا في النموذج؛ إنه سطر مفقود في `CLAUDE.md` أو فحص مفقود في CI. قبل تبديل
  الأدوات أو النماذج، اسأل: أيّ تغذية راجعة كانت ستلتقط الخطأ داخل الحلقة —
  وأضفها.

## أخطاء AI

الحالة العودية: مساعدون يُهيّئون أدواتهم الخاصة (أو أدوات وكيل آخر). الإعداد
المُولَّد *يبدو* موثوقًا ويفشل بصمت — مفتاح إعداد غير موجود لا يُعطي خطأ، لا
يفعل شيئًا فقط — فهذه الفئة من المُخرجات تحتاج إلى تحقق أكثر من معظم الشيفرة.

### Claude Code: CLAUDE.md المُهيَّأ ذاتيًا الذي يصف بدلًا من أن يأمر

عندما يُطلب من Claude Code تهيئة أو تحسين `CLAUDE.md`، يميل إلى إنتاج *وصف
شامل للمستودع* — ملخصات مجلد بمجلد، وقوائم ميزات، ومحتوى README مُعاد صياغته —
بدلًا من التعليمات. يقرأ كنص مبهر مكتمل ولا يُغيّر السلوك تقريبًا: يمكن للوكيل
اكتشاف كل ذلك بقراءة الشيفرة، والحجم يملأ ميزانية السياق التي تحتاجها القواعد
الفعلية. يمر الملف على المراجعة البشرية بدقة لأنه يبدو شاملًا.

**الاكتشاف:** `CLAUDE.md` يتجاوز ~150 سطرًا؛ فقرات تصف ما هو موجود بدلًا من أوامر
حازمة حول ما يجب فعله؛ محتوى سيتقادم عند إعادة البناء التالية؛ لا أوامر تحقق،
أو أوامر مدفونة تحت نثر.

**الإصلاح:** أعد التوليد وفق المهمة الفعلية للملف:

> Rewrite CLAUDE.md to contain only: verification commands, non-negotiable rules stated as
> imperatives, conventions that differ from ecosystem defaults, and pointers to deeper docs. Delete
> anything discoverable by reading the code. Target under 120 lines. Every line must plausibly
> change your behavior on a future task — if it wouldn't, cut it.

### GPT: مخطط إعداد مُهلوس يفشل بصمت

عندما يُطلب من نماذج عائلة GPT كتابة `settings.json`، أو إعداد hook، أو استدعاءات
CLI، تنتج بثقة بنيةً مقبولة لكنها مُختلَقة: صياغة أذونات من إصدار قديم، وأسماء
أحداث hooks غير موجودة، وأعلام مستعارة من CLIs أخرى. لا شيء يُعطي خطأ — تُتجاهل
المفاتيح غير المعروفة عادةً — فيعتقد الفريق أن الترحيلات محمية والأسرار غير
مقروءة بينما لا يفعل الإعداد شيئًا بصمت. إنه مقابل ملف الإعداد لـ API مُهلوس،
بتأخر كشف أطول لأن الإعداد نادرًا ما يُستخدم.

**الاكتشاف:** أي إعداد مُولَّد قُبل دون اختبار حي؛ أسماء مفاتيح أو أحداث لا تجدها
في التوثيق الرسمي الحالي؛ hook لم يُلاحظ إطلاقًا أنه نُفّذ؛ أذونات لا تحجب
بشكل واضح في تشغيل تجريبي.

**الإصلاح:** تحقق من الواقع لا من المقبولية:

> Verify every key, event name, and flag in this configuration against the current official
> documentation, and state which doc section each comes from. Then give me a test procedure:
> a concrete action that should be blocked by each deny rule and each hook, so I can confirm the
> configuration actually fires before trusting it.

### Cursor: تسرّب إعداد عبر الأدوات

عندما يُطلب (داخل Cursor) إعداد تعليمات وكيل لمستودع يستخدم الفريق عليه Claude
Code أيضًا، يميل Cursor إلى كتابة كل شيء باتفاقياته الخاصة — ملفات قواعد
بـ frontmatter الخاص به ودلالات تحديد glob، أو `CLAUDE.md` مُهيكل كقاعدة Cursor —
وبيعثرة نسخ مكررة ومتباينة قليلًا عبر مواقع خاصة بكل أداة. تتبع كل أداة حينها
نسخة مختلفة من "نفس" المعايير، ولا تهبط التحديثات إلا في نسخة واحدة. الفشل
معماري: لا يوجد مصدر حقيقة وحيد لقواعد الفريق الهندسية.

**الاكتشاف:** frontmatter خاص بأداة أو صياغة تحديد نطاق داخل `CLAUDE.md`/`AGENTS.md`؛
محتوى قواعد شبه مكرر عبر `.cursor/`، و`CLAUDE.md`، و`AGENTS.md`؛ تحديث اتفاقية
غيّر ملفًا واحدًا دون إخوته.

**الإصلاح:** مصدر حقيقة واحد، ومحولات أدوات رفيعة (الفصل 02 يُطوّر هذا):

> Consolidate all engineering rules into AGENTS.md as plain markdown with no tool-specific syntax.
> Make each tool's native file (CLAUDE.md, Cursor rules) a thin pointer to it, containing only
> what that tool genuinely requires in its own format. Rules must exist in exactly one place.

## أفضل الممارسات

- **اكتب CLAUDE.md بصفته مواصفة، وصنه كالشيفرة.** أمرية، ومحددة، وقصيرة. راجع
  تغييراتها في طلبات السحب (PRs). محرّض السطر الجديد تجريبي: نفس التصحيح مرتين.
- **ابدأ بأوامر التحقق، واشترط أن تحرّس "تم".** قدرة الوكيل على التحقق من عمله
  أنفسها تساوي أكثر من أي تعليمات أسلوبية. "شغّل الاختبارات السريعة وlint بعد كل
  تغيير" ينتمي إلى كل ملف تعليمات.
- **رمّز غير القابل للتفاوض آليًا.** إذا كان انتهاك القاعدة مكلفًا، فإن hook
  أو فحص CI يُنفّذها والتعليمات يكتفي بتفسيرها. نثر للتفضيلات، وآليات
  للثوابت (invariants).
- **درّج الأذونات بحسب نصف قطر الأثر.** اسمح تلقائيًا بحلقة التغذية الراجعة،
  وأكّد الإجراءات التي تترك شجرة العمل، وارفض الأسرار والتدمير. أعد النظر في
  الدرجات عندما تصبح المطالبات متكررة بما يكفي لتولّد إرهاقًا — الإرهاق رائحة
  إعداد لا قدرًا محتومًا.
- **التزم بالإعداد المشترك؛ استبعد من git الإعداد الشخصي.** `.claude/settings.json`،
  والـ hooks، والأوامر سياسة فريق وتسافر مع المستودع. `settings.local.json` يُبقي
  الذوق الشخصي خارج السياسة.
- **التقط سير العمل المتكرر في صورة أوامر.** في المرة الثالثة التي تكتب فيها
  تقريبًا نفس المطالبة، تصبح ملف أمر مُنسَّخ الإصدار. المعرفة الفريقية تنتمي
  إلى المستودع، لا إلى سجل دردشة.
- **حدّد نطاق الجلسات بالمهام.** سياق جديد لكل مهمة؛ وضع التخطيط (أو التخطيط
  أولًا في المطالبات) لكل ما هو متعدد الملفات؛ راجع الخطة، ثم اترك الحلقة تعمل.
- **أبقِ الأسرار بعيدة هيكليًا.** قواعد deny على `.env` ومسارات الأسرار، ولا تلصق
  أبدًا قيمًا سرّية في مطالبة — سر في السياق سرٌّ مُفصح عنه (المرحلة 9، الفصل 04).
- **أعد التحقق من خصوصيات الأدوات ضد التوثيق الحالي.** تعكس أسماء الميزات
  والأعلام ومخططات الإعداد في هذا الفصل وقت الكتابة؛ المبادئ ثابتة، والصياغة لا.

## الأنماط المضادة

**yolo غير المعزول.** استقلالية كاملة — كل صلاحية ممنوحة، بلا تأكيد — على محطة
عمل بها بيانات اعتماد إنتاجية، وملفات `.env` حقيقية، ووصول push. الحُجّة هي
السرعة؛ الثمن أن *أسوأ* إجراء يمكن للوكيل اتخاذه أصبح الآن سقف يومك السيئ.
الاستقلالية تنتمي إلى بيئات مصممة لها: حاويات، وworktrees، وبيانات اعتماد محددة
النطاق.

**مكبّ CLAUDE.md.** كل حادثة تضيف فقرة، ولا يُزال شيء أبدًا، فينمو الملف إلى
تمرير 700 سطر من توثيق مُعاد صياغته، وقواعد قديمة، وسجلات أخطاء ملصوقة. بعد
حدٍّ ما، تعني التعليمات الأكثر *التزامًا أقل* — كل سطر يُضعف نصيب الآخرين من
انتباه النموذج. ملف التعليمات ميزانية منتقاة، لا سجل إلحاقي.

**الإعداد بصفته حديقة خاصة لمهندس واحد.** كل الضبط يعيش في إعدادات وذاكرة
عضلية محلية لشخص واحد؛ المستودع فارغ. تتفاوت فعالية AI في الفريق بمقدار 5×
حسب المكتب، وتُجرى التصحيحات نفسها بالتوازي، وعندما يرحل ذلك المهندس، يرحل
"سير عمل AI أولًا" معه. المعرفة المؤسسية تذهب إلى نظام التحكم بالإصدارات — لم
تتغير تلك القاعدة عندما صارت المعرفة إعداد وكيل.

**الدستور المُعدِّل لذاته.** السماح للوكيل بحرية تعديل `CLAUDE.md`، والـ hooks،
والأذونات الخاصة به كجزء من المهام العادية. سيفعل — بخدمية — ويُرخي القاعدة التي
يستمر في الاصطدام بها. تغييرات التعليمات والسياسة هي بالتحديد التغييرات التي
تتطلب مراجعة بشرية؛ يجوز للوكيل أن *يقترحها*، ولا يطبّقها بصمت أبدًا.

**التبختر في الـ Hooks.** ترميز كل تفضيل أسلوبي كـ hook حاجب حتى تُزحف الحلقة،
وينكسر نصف السكربتات على جهاز أحدهم. التنفيذ الحتمي للانتهاكات المكلفة؛ أما
الذوق فأرخص في تنفيذه من خلال إعداد lint يشغّله الوكيل على أي حال.

## شجرة القرار

"أريد من الوكيل أن يفعل (ألا يفعل أبدًا) X بثبات — أين ينتمي X؟"

```
Is X a secret, destructive, or production-touching capability?
├── YES ──► permissions: deny (or ask). Not prose — mechanism.
└── NO ──► Would X's violation be expensive (data loss, schema history,
           security regression)?
    ├── YES ──► hook (block + teach) AND a CI check as backstop.
    │           Mention it in CLAUDE.md so the agent expects the wall.
    └── NO ──► Is X needed in EVERY session (a convention, a command,
               a standing rule)?
        ├── YES ──► Is it discoverable by reading the code?
        │   ├── YES ──► leave it out — the agent reads code; duplication
        │   │           goes stale and spends context.
        │   └── NO ───► CLAUDE.md, as one imperative line.
        ├── OCCASIONALLY ──► docs/ file, one-line pointer in CLAUDE.md.
        └── ONLY FOR ONE RECURRING TASK ──► a command/skill file
            (versioned prompt), invoked when that task comes up.

Cross-check: after placing X, ask "what feedback tells the agent it
violated X?" If the answer is "nothing until human review," X is not
yet engineered — add the test, lint rule, or hook that pushes back
inside the loop.
```

## قائمة المراجعة

**قائمة مراجعة التنفيذ**

- [ ] `CLAUDE.md` موجود في جذر المستودع: أوامر التحقق أولًا، والقواعد غير القابلة
  للتفاوض كأوامر حازمة، ومؤشرات إلى توثيق أعمق — ولا شيء قابل للاكتشاف من الشيفرة
- [ ] أوامر التحقق تعمل فعلًا كما هي مكتوبة من نسخة نظيفة (الوكيل سيُشغّلها
  تمامًا كما هي مكتوبة)
- [ ] `.claude/settings.json` مُلتزم به بأذونات مُدرّجة: حلقة التغذية الراجعة
  مسموح بها، إجراءات نصف القطر مؤكدة، الأسرار والأوامر المدمّرة مرفوضة
- [ ] `settings.local.json` مُستبعد من git
- [ ] كل hook لوحظ أنه نُفّذ: تنفيذ الإجراء الممنوع، ورؤية الحجب ورسالة التعليم
- [ ] قواعد Deny مختبرة حيًا (طُلب من الوكيل قراءة `.env` ← رُفض)
- [ ] سير العمل المتكرر مُلتقط كملفات أوامر، لا عادات محادثة
- [ ] الاستدعاءات غير التفاعلية (CI) تستخدم قوائم أدوات مضيّقة صراحة

**قائمة مراجعة المعمارية**

- [ ] مصدر حقيقة واحد لقواعد الهندسة؛ ملفات خاصة بالأدوات محولات رفيعة، لا تفرعات
- [ ] القواعد موضوعة بحسب الآلية: أذونات/hooks للمكلف، و`CLAUDE.md` للدائم،
  و`docs/` للعميق، والمطالبات للمخصص بالمهمة
- [ ] كل قاعدة مهمة لها تغذية راجعة داخل الحلقة (اختبار، أو lint، أو فحص أنواع،
  أو hook) — لا مجرد مراجعة بشرية
- [ ] مستوى استقلالية الوكيل يطابق نصف قطر أثر البيئة (الاستقلالية الكاملة في
  البيئات المعزولة فقط)
- [ ] ميزانية السياق مُحترمة: المحتوى المحمَّل دائمًا صغير ويستحق مكانه

**قائمة مراجعة الكود** (لتغييرات طبقة هندسة AI)

- [ ] فرق `CLAUDE.md`: الأسطر الجديدة أوامر حازمة تُغيّر السلوك؛ لا حشو وصفي؛
  أي شيء تقادم الآن أُزيل
- [ ] فرق `settings.json`: أي صلاحية مُوسَّعة مُبرَّرة في وصف الـ PR — هذا تغيير
  سياسة، راجعه كسياسة
- [ ] تغييرات الـ Hook: السكربت مُختبر على مدخلات مطابقة وغير مطابقة؛ وضع الفشل
  (أخطاء السكربت) لا يُعطّل الجلسة
- [ ] ملفات الأوامر: المطالبة تتضمن خطوات تحقق، لا مجرد خطوات توليد
- [ ] لا قيم سرّية في أي مكان داخل ملفات التعليمات، أو الأوامر، أو سكربتات الـ hook

## التمارين

1. **جهّز مستودعًا حقيقيًا.** خذ مشروعًا تعمل عليه فعلًا واكتب `CLAUDE.md` من
   الصفر: الأوامر، وخمس قواعد معمارية، واتفاقيات تختلف عن الافتراضيات. ثم شغّل
   نفس المهمة متوسطة الحجم (تذكرة حقيقية) في جلسة جديدة مع الملف، وفي نسخة
   مؤقتة دونه. قارن بين النتيجتين — كل فارق هو مقياس لقيمة إعدادك.
2. **ابنِ حارسًا وتحقق منه.** اختر صنف الملف الوحيد في مستودعك الذي يجب ألا
   يُعدِّل فيه الوكيل أبدًا (الترحيلات، أو الشيفرة المولّدة، أو دليل مُدمَج).
   اكتب hook ما-قبل-التعديل يحجبه مع رسالة تعليم، ثم *أثبت أنه يعمل*: اطلب
   من الوكيل تعديل الملف وتأكيد الحجب، ثم تأكد أن الإجراء التالي للوكيل يتبع
   البديل المُعلَّم.
3. **صمّم سياسة الأذونات.** اكتب درجات allow/ask/deny لمستودعك، مع تبرير من سطر
   واحد لكل بند. تبادل مع زميل وهاجم كل سياسة: جد أمرًا واحدًا مسموحًا به
   يمكنه إحداث ضرر أكبر مما يوحي به درجته، وقاعدة ask متكررة بما يكفي لتوليد
   إرهاق.
4. **التقط سير عمل.** حدّد مطالبة كتبتها (في أي أداة) ثلاث مرات على الأقل.
   حوّلها إلى ملف أمر مُلتزم بخطوات تحقق، وعرّضها لمراجعة الكود. لاحظ ما
   التقطه المراجع — المطالبات تتحسن تحت المراجعة لنفس السبب الذي يجعل الشيفرة
   كذلك.
5. **شغّله غير تفاعليًا.** اكتب استدعاءً غير تفاعليًا يُشغّل مجموعة اختباراتك
   ويُنتج فرزًا بـ markdown لحالات الفشل، مع قائمة سماح أدوات تحتوي بالضبط ما
   تحتاجه تلك المهمة. شغّله من سير عمل CI أو cron. هذه بذرة أتمتة الفصل 08.
