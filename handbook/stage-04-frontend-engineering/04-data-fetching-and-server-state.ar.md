# جلب البيانات وحالة الخادم

## مقدمة

معظم البيانات التي يعرضها الـ frontend لا تنتمي إلى الـ frontend — بل تعيش على
الخادم، وتُجلب عبر الشبكة، وقد تكون قديمة بمجرد وصولها،
ويتم مشاركتها بين المكونات والمستخدمين. يتناول هذا الفصل كيفية التعامل مع تلك
**حالة الخادم** بشكل صحيح: جلبها (على الخادم افتراضياً، وعلى العميل
عند الحاجة)، وتخزينها مؤقتاً، وإبقاؤها محدّثة بعد التغييرات، وفعل ذلك دون
الشلالات (waterfalls)، وتسباقات الظروف (race conditions)، وأخطاء واجهة المستخدم القديمة التي ينتجها النهج الساذج.

الفكرة الأهم هنا هي التمييز: **حالة الخادم ليست حالة العميل.**
حالة (واجهة) العميل — هل هذه القائمة المنسدلة مفتوحة، وماذا كُتب في هذا الحقل — مملوكة
لـ frontend، ومتزامنة، ومن حقك تعديلها. حالة الخادم — قائمة الفواتير،
المستخدم الحالي — مملوكة للخادم، وغير متزامنة، ومخزّنة مؤقتاً، ومشتركة؛
يمكن أن تكون قديمة ويجب إعادة جلبها. الخلط بين الاثنين — حشو بيانات الخادم
التي تم جلبها في `useState` أو متجر عام ومزامنتها يدوياً — هو أصل نسبة كبيرة
من أخطاء بيانات الـ frontend، وهو الخيار الافتراضي الذي يلجأ إليه المساعد.

تتبع الأدوات هذا التمييز. في Next.js App Router، تُجلب البيانات الأولية
على **الخادم** (الفصلان 02–03) — بسرعة، وبأمان، وبدون شلال عميل. بالنسبة
للبيانات التي يجب جلبها أو إعادة جلبها *على العميل* — الفلاتر التفاعلية،
والاستعلام الدوري (polling)، والعمليات (mutations) التي تحدّث ذاكرة التخزين المؤقت — الأداة
المناسبة هي مكتبة حالة الخادم، **React Query (TanStack Query)**،
التي تتولى التخزين المؤقت، وإزالة الازدواجية، وإعادة الجلب في الخلفية،
وإبطال الصلاحية. ما يجب ألا تفعله تقريباً هو إدارة
حالة الخادم يدوياً باستخدام `useEffect` + `fetch` + `useState`، لأن هذا هو
مكان تواجد الأخطاء.

## لماذا هذا مهم

حالة الخادم صعبة حقاً — فهي غير متزامنة، ومشتركة، وقابلة للتخزين المؤقت، وقد
تصبح قديمة — والنهج الساذج `useEffect` + `fetch` + `useState` يخطئ في كل جزء منها تقريباً:

- **لا تخزين مؤقت ولا إزالة ازدواجية.** كل مكوّن يحتاج إلى البيانات يجلبها
  مرة أخرى؛ التنقل بعيداً ثم العودة يجلبها من الصفر؛ مكوّنان يريدان نفس
  البيانات يُصدران طلبين. تخزين حالة الخادم المؤقت يجلبها مرة واحدة ويشاركها.
- **تسباقات الظروف (Race conditions).** إطلاق جلب، ثم تغيير المُدخل، ثم إطلاق آخر
  — قد تصل الردود بترتيب خاطئ ويفوز الرد القديم، فتُعرض بيانات خاطئة. الجلب
  اليدوي القائم على الـ effect يعجّ بهذا؛ طبقة بيانات حقيقية تتعامل معه.
- **الشلالات (Waterfalls).** جلب ما يمكن جلبه بالتوازي بشكل متسلسل، أو
  الجلب على العميل لما كان يجب جلبه على الخادم، يضيف رحلات ذهاب وإياب
  ومحملات (spinners) يشعر بها المستخدم كبطء.
- **واجهة قديمة بعد العمليات (Stale UI after mutations).** عند تغيير البيانات على الخادم
  تصبح النسخة المخزنة مؤقتاً على العميل خاطئة — فلا تزال القائمة تعرض الحالة
  القديمة بعد تعديلها — ما لم تُبطل العملية (mutation) الاستعلامات المتأثرة
  (الصورة الموازية على العميل لمشكلة إبطال التخزين المؤقت من المرحلة 3، الفصل 07).

عند المعالجة الجيدة — بيانات أولية مُجلوبة من الخادم بالإضافة إلى React Query للتفاعلات
على العميل — تحصل على رسم أولي سريع، وتخزين مؤقت مشترك، وعدم وجود تسباقات، وحداثة في الخلفية،
وواجهة تعكس التغييرات فوراً. عند المعالجة الساذجة، تحصل على محملات، وشلالات،
وبيانات قديمة، ورمش (flicker).

البُعد الخاص بالذكاء الاصطناعي: يتعامل المساعدون مع حالة الخادم كأنها حالة عميل — يخزنون
البيانات المُجلوبة في `useState` أو متجر عام ويزامنونها يدوياً، وينسون
إبطال ذاكرة التخزين المؤقت بعد عملية (mutation) (واجهة قديمة)، ويبنون شلالات
بـ await متسلسل. كل واحدة منها هي المشكلة بالذات التي يسعى التمييز بين حالة الخادم وحالة العميل
و React Query إلى إزالتها.

## النموذج الذهني

نوعان من الحالة، موقعان، ومكتبة واحدة للحالة الصعبة:

```
   حالة (واجهة) العميل (CLIENT/UI)                    حالة الخادم (SERVER STATE)
   مملوكة لـ frontend                                  مملوكة للخادم
   متزامنة، ومن حقك تعديلها                            غير متزامنة، تُجلب، مُخزّنة مؤقتاً (CACHED)، قد تكون قديمة (STALE)، مشتركة
   "هل هذه النافذة المنبثقة مفتوحة؟"                   "قائمة الفواتير"، "المستخدم الحالي"
   → useState / متجر واجهة (فصل 05)                   → جلب على الخادم (SERVER)، أو React Query على العميل (CLIENT)
                                                       ليس useState + useEffect + fetch يدوياً

   أين تجلب البيانات (App Router)
     بيانات الصفحة الأولية ──► على الخادم (SERVER) (مكوّن خادم غير متزامن) — سريع، آمن، بدون شلال
     تفاعلي / إعادة جلب / عمليات / استعلام دوري ──► React Query على العميل (CLIENT)

   يعطيك React Query (مجاناً) ما لا يعطيه useEffect+fetch:
     تخزين مؤقت · إزالة ازدواجية · إعادة جلب في الخلفية · قديم-أثناء-إعادة-التحقق (SWR)
     حالة التحميل/الخطأ · معالجة تسباق الظروف · إبطال ذاكرة التخزين المؤقت بعد العمليات

   بعد عملية (mutation): أبطل الاستعلامات المتأثرة ──► تعيد الواجهة الجلب والتحديث
     (النسخة على العميل من إبطال التخزين المؤقت — المرحلة 3، فصل 07)
```

أربعة مبادئ يحملها الفصل:

**حالة الخادم وحالة العميل مشكلتان مختلفتان.** حالة العميل محلية،
ومتزامنة، ومملوكة لك؛ حالة الخادم بعيدة، وغير متزامنة، ومُخزّنة مؤقتاً، ومشتركة،
ويمكن أن تصير قديمة. أدرهما بأدوات مختلفة — `useState`/متجر واجهة لحالة
العميل (الفصل 05)، وجلب من الخادم أو React Query لحالة الخادم — ولا تضع
بيانات الخادم أبداً في `useState` أو متجر عام لمزامنتها يدوياً.

**اجلب من الخادم افتراضياً؛ واستخدم React Query لبيانات العميل.** في App
Router، اجلب البيانات الأولية للصفحة في Server Component — فهذا سريع وآمن
ويتجنب شلال العميل (الفصلان 02–03). استخدم React Query للبيانات التي تحتاج
حقاً جلباً أو إعادة جلب على العميل: الفلاتر التفاعلية، والاستعلام الدوري،
والعمليات التي تحدّث ذاكرة تخزين مؤقت على العميل.

**لا تُدِر حالة الخادم يدوياً بـ `useEffect` + `fetch`.** هذا النمط بلا
تخزين مؤقت، ولا إزالة ازدواجية، وفيه تسباقات، ولا إعادة جلب في الخلفية، ومعالجة يدوية للتحميل/الخطأ
— إنه يعيد تنفيذ (بشكل سيئ) ما تنفذه مكتبة حالة الخادم بشكل صحيح. الـ
`useEffect` + `fetch` الخام لبيانات الخادم نمط مضاد (anti-pattern)، وليس بديلاً أبسط.

**أبطل بعد العمليات (mutations).** عندما تُغيّر عملية بيانات الخادم، تصبح النسخة المخزنة مؤقتاً
على العميل قديمة حتى تُبطل (أو تحدّث) الاستعلامات المتأثرة، مما يستدعي
إعادة جلب ليعكس الواجهة التغيير. نسيان هذا هو سبب استمرار عرض القائمة للقيمة
القديمة بعد التعديل.

تعريف عملي:

> **حالة الخادم — بعيدة، ومُخزّنة مؤقتاً، وقابلة لأن تصير قديمة، ومشتركة — هي مشكلة مختلفة
> عن حالة واجهة العميل وتحتاج أدوات مختلفة: اجلبها على الخادم افتراضياً، واستخدم
> React Query للجلب على العميل والعمليات، ولا تبنِها يدوياً بـ
> `useEffect` + `fetch` + `useState`، وأبطل ذاكرة التخزين المؤقت بعد كل عملية
> (mutation) لتبقى الواجهة صحيحة.**

## مثال إنتاجي

تُظهر شاشة الفواتير في **Invoicely** هذا التوتر بشكل مثالي. يجب أن
تُعرض القائمة الأولية بسرعة وأن تكون قابلة للفهرسة، لذا تُجلب على
**الخادم** (الفصلان 02–03). لكن الشاشة تفاعلية: يُصفّي المستخدم حسب الحالة
(إعادة جلب)، ويعلّم الفواتير كمدفوعة (عملية يجب أن تحدّث كلاً من القائمة
وطريقة العرض التفصيلية). هذه التفاعلات على العميل هي بالضبط ما صُمم React Query من أجلها —
وفعلها بـ `useEffect` + `fetch` الخام سيجلب التسباقات والمحملات وأخطاء "قديم-بعد-العملية"
التي يتناولها هذا الفصل.

سنجلب القائمة الأولية على الخادم، ثم نستخدم React Query على العميل لإعادة
الجلب المُصفّى وعملية "وضع علامة مدفوع" مع إبطال ذاكرة التخزين المؤقت ليُحدّث الواجهة
فوراً. سنقابل خطأ العملية-بلا-إبطال (تعرض القائمة الحالة القديمة بعد التعديل) وشلال
`useEffect` + `fetch`. النقطة هي تقسيم العمل: الخادم للتحميل الأولي، وReact Query
للتفاعل الحي، والاثنان مُكوَّنان بنظافة.

## بنية المجلدات

```
web/src/
├── app/(app)/invoices/page.tsx      # الخادم (SERVER): يجلب القائمة الأولية على الخادم
├── lib/query.tsx                    # "use client": QueryClient + provider (مرة واحدة، في الجذر)
└── features/invoices/
    ├── api.ts                       # دوال الجلب (الخادم + العميل)
    ├── queries.ts                   # useInvoices, useInvoice — خطافات React Query
    ├── mutations.ts                 # useMarkPaid — عملية + إبطال ذاكرة التخزين المؤقت
    └── InvoicesClient.tsx           # "use client": قائمة تفاعلية (تصفية، إعادة جلب)
```

لماذا هذا الشكل: الصفحة (Server Component) تجلب العرض الأول من جانب الخادم؛
جزيرة العميل (`InvoicesClient`) تستخدم خطافات React Query من `queries.ts`/`mutations.ts`
للتفاعل. منطق الاستعلام والعملية مُقارن للميزة (feature)، ويُعدّ
موفر `QueryClient` مرة واحدة في الجذر (كمكوّن عميل).

## التنفيذ

**اجلب البيانات الأولية من الخادم (`page.tsx`).** المسار الافتراضي والأفضل في App
Router: اجلب على الخادم، واعرض فوراً، بلا محمّلة على العميل ولا شلال.

```tsx
// Server Component — initial data fetched on the server (Chapters 02-03)
import { getInvoices } from "@/features/invoices/api";
import { InvoicesClient } from "@/features/invoices/InvoicesClient";

export default async function InvoicesPage() {
  const initialInvoices = await getInvoices({ status: "all" });   // server-side fetch
  return <InvoicesClient initialInvoices={initialInvoices} />;    // handoff to the client island
}
```

**إعداد React Query (`lib/query.tsx`).** `QueryClient` واحد، يُوفَّر مرة واحدة. هذه
ذاكرة التخزين المؤقت على العميل التي تستخدمها بقية الميزة.

```tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export function QueryProvider({ children }: { children: React.ReactNode }) {
  const [client] = useState(() => new QueryClient({
    defaultOptions: { queries: { staleTime: 30_000 } },   // fresh for 30s; SWR after
  }));
  return <QueryClientProvider client={client}>{children}</QueryClientProvider>;
}
```

**استعلام لإعادة الجلب على العميل (`queries.ts` + `InvoicesClient.tsx`).** يتولى
React Query التخزين المؤقت، وإزالة الازدواجية، وإعادة الجلب في الخلفية، والتحميل/الخطأ، والتسباقات —
وأنت لا تكتب شيئاً من ذلك. البيانات المُجلوبة من الخادم تبذر ذاكرة التخزين المؤقت كـ `initialData`، فلا
يكون هناك جلب ثانٍ في العرض الأول.

```tsx
// queries.ts
import { useQuery } from "@tanstack/react-query";
import { getInvoices, type InvoiceFilter } from "./api";

export function useInvoices(filter: InvoiceFilter, initialData?: Invoice[]) {
  return useQuery({
    queryKey: ["invoices", filter],      // cache key includes the filter (like a cache key, Ch 07)
    queryFn: () => getInvoices(filter),
    initialData: filter.status === "all" ? initialData : undefined,   // seed from server
  });
}
```

```tsx
// InvoicesClient.tsx
"use client";
import { useState } from "react";
import { useInvoices } from "./queries";

export function InvoicesClient({ initialInvoices }: { initialInvoices: Invoice[] }) {
  const [status, setStatus] = useState<"all" | InvoiceStatus>("all");   // CLIENT/UI state
  const { data, isLoading } = useInvoices({ status }, initialInvoices);  // SERVER state via RQ

  // Changing `status` refetches via React Query — cached, deduped, race-safe. No useEffect.
  return (
    <>
      <StatusFilter value={status} onChange={setStatus} />
      {isLoading ? <InvoiceListSkeleton /> : <InvoiceList invoices={data!} />}
    </>
  );
}
```

**عملية (mutation) مع إبطال ذاكرة التخزين المؤقت (`mutations.ts`).** بعد تعليم
الفاتورة كمدفوعة، أبطل الاستعلامات المتأثرة لتعيد القائمة والتفاصيل الجلب وتعرض الحالة الجديدة.
هذا هو إبطال ذاكرة التخزين المؤقت على جانب العميل الذي يحافظ على صحة الواجهة.

```tsx
import { useMutation, useQueryClient } from "@tanstack/react-query";
import { markInvoicePaid } from "./api";

export function useMarkPaid() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: (invoiceId: number) => markInvoicePaid(invoiceId),
    onSuccess: (_data, invoiceId) => {
      // INVALIDATE affected queries → they refetch → UI reflects the change.
      qc.invalidateQueries({ queryKey: ["invoices"] });          // the list
      qc.invalidateQueries({ queryKey: ["invoice", invoiceId] }); // the detail
    },
  });
}
```

**الأنماط المضادة — ما يميل المساعد إلى إنتاجه.** خطآن كلاسيكيان، كلاهما
أخضر في نقرات أولية:

```tsx
// ANTI-PATTERN 1: server state by hand — no cache, races, manual loading
"use client";
function InvoicesBad() {
  const [invoices, setInvoices] = useState([]);
  const [loading, setLoading] = useState(true);
  useEffect(() => {                                   // refetch on every mount; no dedup
    fetch("/api/invoices").then(r => r.json()).then(d => { setInvoices(d); setLoading(false); });
  }, []);                                             // change a filter fast → race → wrong data
  // ...
}

// ANTI-PATTERN 2: mutation without invalidation — UI goes stale
function useMarkPaidBad() {
  return useMutation({ mutationFn: markInvoicePaid });   // no onSuccess invalidate
  // the invoice is paid on the server; the list still shows "sent" until a manual reload
}
```

الفرق هو الفصل كله: النسخة الجيدة تجلب القائمة الأولية على
الخادم (سريعة، بلا شلال)، وتدير التفاعل/إعادة الجلب/العملية على العميل بـ
React Query (مُخزّنة، ومُزالة الازدواجية، وآمنة من التسباق)، وتبطل بعد العملية (الواجهة تُحدّث
فوراً). النسخة السيئة تبنّي حالة الخادم يدوياً في `useState` (تسباقات، بلا ذاكرة) وتُجري
العملية دون إبطال (واجهة قديمة). التفاعلية متطابقة؛ الصحة والسرعة ليستا كذلك.

## قرارات هندسية

خمسة قرارات تحدد كيف تتعامل مع البيانات.

### الجلب من الخادم أم من العميل؟

**الخيارات:** (1) جلب كل شيء على العميل؛ (2) جلب البيانات الأولية على الخادم،
وجلب ما يحتاجه على العميل فقط.

**المقايضات:** الجلب الكامل من العميل عادة SPA — يُرسل هيكلاً فارغاً، ويعرض
محمّلة، ويضيف رحلة ذهاب وإياب على العميل (وشلالات) قبل أن يُعرض أي شيء.
جلب البيانات الأولية من الخادم يعرضها فوراً، ويحافظ على أمانها، ويتجنب
الشلال، لكن الأجزاء التفاعلية لا تزال بحاجة إلى حل على العميل.

**التوصية:** في App Router، اجلب البيانات الأولية للصفحة على الخادم
(الفصلان 02–03)، واستخدم React Query على العميل فقط للاحتياجات الحقيقية على جانب العميل —
إعادة الجلب التفاعلية، والاستعلام الدوري، والعمليات. الخادم للعرض الأول، وReact Query
للتفاعل الحي؛ ولا تجلب على العميل ما يستطيع الخادم عرضه.

### React Query، أم `useEffect` + `fetch`، أم متجر عام — لحالة الخادم؟

**الخيارات:** (1) `useEffect` + `fetch` + `useState`؛ (2) متجر عام (Redux/Zustand)؛
(3) React Query (مكتبة حالة الخادم).

**المقايضات:** `useEffect` + `fetch` يبنّي التخزين المؤقت، وإزالة الازدواجية، والتسباقات، وحالة التحميل يدوياً —
بشكل سيئ ومكرر. المتجر العام ليس مبنياً لبيانات الخادم غير المتزامنة ويحوّل التخزين المؤقت
والإبطال إلى عمل يدوي ستخطئ فيه. React Query مبني خصيصاً لحالة الخادم —
التخزين المؤقت، وإزالة الازدواجية، وإعادة الجلب في الخلفية، والإبطال — بتكلفة اعتماد
(dependency) ومفاهيمه.

**التوصية:** React Query (أو ما يعادلها من مكتبات حالة الخادم) لحالة الخادم
على جانب العميل، دائماً. إنها موجودة لأن `useEffect` + `fetch` والمتاجر العامة
أدوات خاطئة للمهمة. احتفظ بـ `useState`/متجر واجهة لـ *حالة* العميل (الفصل 05)؛
ولا تضع بيانات الخادم فيها.

### كيف تُبقى ذاكرة التخزين المؤقت محدّثة بعد العمليات؟

**الخيارات:** (1) لا تفعل شيئاً (اعتمد على إعادة تحميل يدوية)؛ (2) أبطل الاستعلامات المتأثرة؛
(3) تحديث متفائل (optimistic)، ثم توفيق.

**المقايضات:** عدم فعل شيء يترك الواجهة قديمة بعد كل تغيير. الإبطال بسيط
وصحيح — الاستعلامات المتأثرة تعيد الجلب وتعرض الحقيقة الجديدة — مع إعادة جلب وجيزة.
التحديثات المتفائلة تجعل الواجهة تبدو فورية بتحديث ذاكرة التخزين المؤقت قبل تأكيد الخادم،
بتكلفة التعقيد ومنطق التراجع-عند-الفشل.

**التوصية:** أبطل الاستعلامات المتأثرة بعد كل عملية (الاختيار الافتراضي والصحيح)،
وأضف تحديثات متفائلة فقط حين تستحق السرعة الإضافية التعقيد الإضافي
(مفتاح تبديل، إعجاب). نسيان الإبطال هو النسخة على العميل من خطأ ذاكرة التخزين المؤقت من
المرحلة 3، الفصل 07 — ذاكرة التخزين المؤقت لا فائدة منها ما لم تكن صحيحة.

### كيف تتجنب الشلالات؟

**الخيارات:** (1) جلب متسلسل حسب الحاجة؛ (2) جلب متوازٍ / على الخادم /
إعادة هيكلة لإزالة التبعيات.

**المقايضات:** الجلب المتسلسل هو الطريقة الطبيعية لقراءة الكود ويضيف رحلة ذهاب وإياب لكل
خطوة — سلسلة من المحمّلات يشعر بها المستخدم. الجلب المتوازٍ (بيانات مستقلة دفعة واحدة)،
والجلب من الخادم، وإعادة الهيكلة لإزالة التبعيات المصطنعة، يقلل الزمن الكلي،
بتكلفة ترتيب عمليات الجلب عمداً.

**التوصية:** اجلب البيانات المستقلة بالتوازي (`Promise.all` على الخادم، واستعلامات متوازية
على العميل)، واجلب من الخادم حيث أمكن، وأعد الهيكلة لتجنب
عمليات جلب تابعة حيث لا يوجد طلب إلا للحصول على معرف للطلب التالي. عامل سلسلة
من الـ await المتسلسلة كشلال يجب تسطيحه.

### بعدوانية إعادة جلب البيانات (ضبط حداثة البيانات)؟

**الخيارات:** (1) دائماً محدّث (إعادة جلب باستمرار)؛ (2) `staleTime`/سلوك إعادة جلب مضبوط.

**المقايضات:** دائماً-محدّث يعظّم الصحة ويُرهق الخادم بعمليات إعادة الجلب؛
أبداً-إعادة-جلب يقلل الطلبات ويقدم بيانات قديمة. يتيح لك `staleTime` في React Query، وإعادة-الجلب-عند-
التركيز/إعادة-الاتصال، والاستعلام الدوري، ضبط الحداثة لكل استعلام.

**التوصية:** اضبط `staleTime` (وسلوك إعادة الجلب) لكل استعلام بحسب سرعة
تغير البيانات وقَدْر القِدَم المقبول — يمكن أن يكون ملخص لوحة المعلومات قديماً لدقيقة
(كما في ميزانية المرحلة 3، الفصل 07)؛ قد يعيد الجلب عند التركيز لحالة حيّة. لا تقبل
الافتراضات بشكل أعمى لبيانات ذات متطلبات حداثة حقيقية، ولا تعد الجلب باستمرار لبيانات
نادراً ما تتغير.

## المقايضات

التعامل الجيد مع حالة الخادم يُقايض اعتماداً ومفاهيماً مقابل الصحة، وبعض النقاط
سياقية.

**React Query يُقايض مكتبة ومنحنى تعلّم مقابل حل مشكلة صعبة حقاً.** يضيف
اعتماداً ومفاهيمه (مفاتيح الاستعلام، وحداثة البيانات، والإبطال)، ويزيل فئة
كاملة من الأخطاء (التسباقات، والواجهة القديمة، وعمليات الجلب المكررة، وحالة التحميل اليدوية) التي
كنت ستعيد تنفيذها بشكل سيئ. لأي تطبيق فيه تفاعل حقيقي مع حالة الخادم، فإن المقايضة
تستحق بقوة؛ لتطبيق مُقدَّم من الخادم بالكامل بلا جلب على العميل، قد لا تحتاج
إليه على الإطلاق (اجلب من الخادم وتوقف).

**الجلب من الخادم يُقايض بعض مرونة العميل مقابل السرعة والأمان.** الجلب على
الخادم يعطي عرضاً أول سريعاً ويحافظ على الوصول للبيانات من جانب الخادم، ويعني أن البيانات الأولية
لا تُدار بواسطة ذاكرة التخزين المؤقت للعميل حتى تُسلّمها له. جسر `initialData` يربط
هذا؛ التكلفة هي فهم تدفق البيانات من الخادم→العميل، وهو ما يستحق العناء من أجل
الأداء.

**التخزين المؤقت يُقايض الحداثة بالسرعة — مرة أخرى.** كما في الـ backend (المرحلة 3، الفصل 07)،
تقدم ذاكرة التخزين المؤقت على العميل بيانات قديمة محتملة بسرعة، وإعدادات `staleTime`/الإبطال
هي أين تختار مقدار القِدَم. ينطبق نفس الانضباط: اضبط لكل نوع بيانات، وأبطل عند
التغيير، ولا تعامل البيانات المخزنة مؤقتاً على أنها مضمونة-الحداثة.

**التحديثات المتفائلة تُقايض التعقيد بالسرعة المُدركة.** تجعل الواجهة تبدو فورية وتتطلب
منطق تراجع وتوفيقاً دقيقاً حين يختلف الخادم. تستحق العناء
للتفاعلات عالية-التكرار، منخفضة-المخاطر؛ وتُفرط في الهندسة لعملية نادرة، مهمة،
تكون فيها محمّلة وجيزة كافية.

## أخطاء شائعة

**معاملة حالة الخادم كحالة عميل.** تخزين البيانات المُجلوبة في `useState` أو متجر
عام ومزامنتها يدوياً — بلا ذاكرة تخزين مؤقت، وإبطال يدوي، وانحراف. الإصلاح: React Query لـ
حالة الخادم؛ `useState`/متجر واجهة لحالة العميل فقط.

**`useEffect` + `fetch` لبيانات الخادم.** النمط المبني يدوياً بتسباقات، وبلا ذاكرة، وبلا
إزالة ازدواجية، وتحميل/خطأ يدوي. الإصلاح: جلب من الخادم، أو React Query — لا جلب قائم على effect خام
لحالة الخادم أبداً.

**لا إبطال لذاكرة التخزين المؤقت بعد العمليات.** إجراء عمليات على بيانات الخادم دون إبطال
الاستعلامات المتأثرة، فتعرض الواجهة بيانات قديمة حتى إعادة التحميل. الإصلاح: `invalidateQueries` (أو
تحديث متفائل) بعد كل عملية.

**الشلالات.** عمليات جلب تابعة متسلسلة، أو جلب على العميل لما كان يجب أن يكون على الخادم،
تضيف رحلات ذهاب وإياب ومحمّلات. الإصلاح: موازاة عمليات الجلب المستقلة، وجلب من الخادم،
وإزالة تبعيات الجلب المصطنعة.

**إعادة جلب خاطئة.** إعادة جلب باستمرار (إرهاق الخادم) أو أبداً (بيانات قديمة)، بدلاً من
الضبط لكل استعلام. الإصلاح: اضبط `staleTime` وسلوك إعادة الجلب من متطلبات الحداثة الحقيقية للبيانات.

**جلب مكرر.** نفس البيانات تُجلب بشكل مستقل من عدة مكونات بلا ذاكرة
تخزين مؤقت مشتركة. الإصلاح: مفتاح استعلام مشترك ليُزيل React Query الازدواجية ويشارك جلباً واحداً.

## أخطاء الذكاء الاصطناعي

حالة الخادم مشكلة صعبة ومتميزة، ويفترض المساعدون عادات حالة العميل
التي لا تناسبها — ينتجون كوداً يعمل في نقرات فردية وينكسر تحت
التفاعل، والتزامن، والعمليات. راجع كود البيانات لمعرفة ما يحدث عند إعادة الجلب، وعند تغيير
إدخال سريع، وبعد عملية.

### Claude Code: معاملة حالة الخادم كحالة عميل

عند طلب تحميل وعرض البيانات، يخزنها Claude Code في `useState` (غالباً عبر `useEffect` +
`fetch`) أو في متجر عام، ويُدير التحميل والأخطاء والتحديثات يدوياً — معاملاً
حالة الخادم البعيدة والمُخزّنة والمشتركة كأنها حالة واجهة محلية. النتيجة بلا تخزين مؤقت، ولا
إزالة ازدواجية، وتسباقات عند التغييرات السريعة، وتصير قديمة بعد العمليات.

**الكشف:** بيانات خادم مُجلوبة محفوظة في `useState`/Redux/Zustand ومُزامَنة يدوياً؛
`useEffect` + `fetch` + `setState` لبيانات الخادم؛ حالة `isLoading`/`error` يدوية حول
عمليات الجلب؛ نفس البيانات تُجلب منفصلة في عدة مكونات.

**الإصلاح:** اشترط استخدام مكتبة حالة الخادم:

> حالة الخادم (البيانات المُجلوبة من الـ backend) يجب أن تُدار بواسطة React Query، وليس `useState`،
> أو `useEffect` + `fetch`، أو متجر عام. استخدم `useQuery` (بمفتاح استعلام) للقراءة و
> `useMutation` للكتابة؛ احتفظ بـ `useState`/المتاجر لحالة العميل/الواجهة فقط. اجلب البيانات الأولية
> من الخادم حيث أمكن (App Router).

### GPT: عمليات لا تبطل ذاكرة التخزين المؤقت

تكتب نماذج عائلة GPT العملية — استدعاء الـ API، وانتهى — وتُغفل إبطال ذاكرة التخزين المؤقت، فتنجح
العملية على الخادم بينما ذاكرة التخزين المؤقت على العميل (والواجهة) لا تزال تعرض البيانات القديمة
حتى إعادة تحميل يدوية. إنه خطأ إبطال ذاكرة التخزين المؤقت على جانب العميل.

**الكشف:** `useMutation` (أو عملية يدوية) بلا إبطال/إعادة جلب في `onSuccess`؛ واجهة
تعرض بيانات قديمة بعد تغيير ناجح؛ عملية تحدّث الخادم لكن لا تحدّث الاستعلامات المتأثرة.

**الإصلاح:** اشترط الإبطال:

> بعد نجاح عملية، أبطل (أو حدّث تفاؤلياً) مفاتيح React Query المتأثرة لتعيد الواجهة الجلب وتعكس التغيير —
> `queryClient.invalidateQueries` في
> `onSuccess`. لا تُعدّ العملية منتهية حتى تُحدَّث بيانات ذاكرة التخزين المؤقت التي تؤثر فيها.

### Cursor: شلالات من عمليات جلب متسلسلة

عند كتابة تحميل البيانات ضمنياً، يميل Cursor إلى `await` لعمليات جلب واحدة تلو الأخرى — بما في
ذلك المستقلة التي يمكن أن تعمل بالتوازي — وإلى إنشاء سلاسل جلب على العميل، لأن
الكود المتسلسل يقرأ بشكل طبيعي عند موقع التعديل. كل await مُضاف هو رحلة ذهاب وإياب ينتظرها
المستخدم.

**الكشف:** عدة استدعاءات `await` متسلسلة لبيانات لا تعتمد على بعضها؛ جلبات
عميل متداخلة لا يغذي أحدها إلا الذي يليه؛ صفحة يزداد زمن تحميلها مع كل تبعية بيانات
مُضافة.

**الإصلاح:** اشترط الجلب المتوازٍ/من الخادم:

> عمليات الجلب هذه مستقلة — شغّلها بالتوازي (`Promise.all`) بدلاً من التسلسل،
> واجلب من الخادم حيث أمكن. لا تربط عمليات الجلب إلا حين يعتمد أحدها فعلاً على
> نتيجة آخر، وحتى حينئذ، فكّر في الجلب بمفتاح مشترك على الخادم لتجنب الشلال.

## أفضل الممارسات

**افصل حالة الخادم عن حالة العميل، واستخدم الأداة المناسبة لكل منهما.** حالة الخادم
(بعيدة، ومُخزّنة، ومشتركة) → جلب من الخادم أو React Query؛ حالة العميل/الواجهة (محلية، ومتزامنة)
→ `useState`/متجر واجهة (الفصل 05). لا تخزن أبداً بيانات الخادم في `useState` أو متجر عام لـ
مزامنة يدوية.

**اجلب البيانات الأولية من الخادم؛ واستخدم React Query لتفاعل العميل.** اجلب من الخادم
العرض الأول (سريع، وآمن، وبدون شلال)، وبَذّر به ذاكرة التخزين المؤقت على العميل، ودع React Query يتولى
إعادة الجلب التفاعلية، والاستعلام الدوري، والعمليات.

**أبطل ذاكرة التخزين المؤقت بعد كل عملية.** `invalidateQueries` (أو تحديث متفائل) ليعكس
الواجهة تغييرات الخادم فوراً — الانضباط على جانب العميل الذي يطابق إبطال
المرحلة 3، الفصل 07 في الـ backend.

**تجنب الشلالات؛ وازِ و اضبط الحداثة.** اجلب البيانات المستقلة بالتوازي،
وأزل تبعيات الجلب المصطنعة، واضبط `staleTime`/سلوك إعادة الجلب لكل استعلام من متطلبات
الحداثة الحقيقية للبيانات.

**دع المكتبة تتولى الأجزاء الصعبة.** لا تُعِد تنفيذ التخزين المؤقت، وإزالة الازدواجية، والتسباقات، أو
حالة التحميل بـ `useEffect` + `fetch` — هذا ما صُمم React Query من أجله. وثّق
اصطلاحات جلب البيانات (الجلب من الخادم أولياً، وReact Query للعميل، والإبطال عند التعديل)
في `CLAUDE.md`.

## أنماط مضادة

**حالة الخادم في `useState`.** بيانات مُجلوبة محفوظة ومُزامنَة يدوياً في حالة مكوّن أو متجر
عام — بلا ذاكرة، وإبطال يدوي، وانحراف. العلامة: `useState` يحتفظ باستجابة API، يبقى
مُتزامِناً عبر effects.

**جلب الـ `useEffect`.** `useEffect` + `fetch` + `setState` خام لبيانات الخادم — تسباقات، ولا
ذاكرة، ولا إزالة ازدواجية، وتحميل يدوي. العلامة: نمط `useEffect(() => { fetch().then(setData) }, [])`
لبيانات الخادم.

**العملية القديمة (Stale Mutation).** عملية بلا إبطال لذاكرة التخزين المؤقت، فتكذب الواجهة حتى إعادة التحميل. العلامة:
`useMutation` بلا إبطال في `onSuccess` وواجهة تحتاج إلى تحديث لإظهار
التغييرات.

**الشلال.** عمليات جلب متسلسلة لبيانات مستقلة، أو جلب على العميل لما كان يجب أن يكون على الخادم —
رحلات ذهاب وإياب متراكمة في زمن استجابة مرئي. العلامة: سلسلة من `await` بلا
تبعية بينها.

**الجلب المكرر.** نفس البيانات تُجلب منفصلة من عدة مكونات بلا ذاكرة/مفتاح مشترك. العلامة: طلبات
متطابقة يُطلقها مكونات شقيقة على نفس الشاشة.

## شجرة القرار

"أحتاج بعض البيانات على الشاشة — كيف أجلبها وأبقيها صحيحة؟"

```
Is this SERVER state (fetched from the backend) or CLIENT/UI state (local)?
├── CLIENT/UI state ──► useState / a UI store (Chapter 05). Not this chapter.
└── SERVER state ──►
     │
     Is it the page's INITIAL data?
     ├── YES ──► fetch on the SERVER (async Server Component). Fast, secure, no waterfall.
     │           Seed React Query with it (initialData) if the client will manage it.
     └── Interactive / refetch / poll / depends on client state?
          └──► React Query (useQuery) on the client. Query key includes all inputs (filters).
     │
     A MUTATION (write)?
     └──► useMutation, and in onSuccess INVALIDATE the affected query keys (or optimistic update).
          The mutation isn't done until the cache is refreshed.
     │
     Multiple fetches? ──► parallelize independent ones (Promise.all); don't chain unless dependent.
     Freshness? ──► set staleTime/refetch per query from how fast the data changes.

     NEVER: useState + useEffect + fetch to manage server state by hand.
```

## قائمة المراجعة

### قائمة مراجعة التنفيذ

- [ ] حالة الخادم تُدار بالجلب من الخادم أو React Query — أبداً `useState`/متجر مُزامن يدوياً.
- [ ] بيانات الصفحة الأولية تُجلب من الخادم؛ تُبذَّر ذاكرة التخزين المؤقت على العميل منها حيث ينطبق ذلك.
- [ ] القراءات على جانب العميل تستخدم `useQuery` بمفتاح استعلام يشمل كل المُدخلات (الفلاتر، المعرّفات).
- [ ] كل عملية تبطل (أو تحدّث تفاؤلياً) مفاتيح الاستعلام المتأثرة في `onSuccess`.
- [ ] عمليات الجلب المستقلة تعمل بالتوازي؛ لا توجد شلالات عرضية.
- [ ] يُضبط `staleTime`/سلوك إعادة الجلب لكل استعلام من متطلبات حداثة البيانات.

### قائمة مراجعة البنية

- [ ] حالة العميل وحالة الخادم منفصلتان بوضوح، مع الأداة المناسبة لكل منهما.
- [ ] يُعدّ React Query مرة واحدة في الجذر؛ منطق الاستعلام/العملية مُقارن للميزات.
- [ ] التسليم من الخادم→العميل للبيانات (initialData/hydration) مقصود.
- [ ] إبطال ذاكرة التخزين المؤقت في العمليات يعكس انضباط إبطال الـ backend (المرحلة 3، الفصل 07).
- [ ] اصطلاحات جلب البيانات موثّقة في `CLAUDE.md`.

### قائمة مراجعة الكود

- [ ] لا توجد بيانات خادم مخزنة في `useState`/متجر عام ومُزامنَة يدوياً (راقب فروقات الـ AI).
- [ ] لا يوجد `useEffect` + `fetch` خام يدير حالة الخادم.
- [ ] لا توجد عملية بلا إبطال/إعادة جلب لذاكرة التخزين المؤقت.
- [ ] لا يوجد شلال لعمليات جلب متسلسلة مستقلة.
- [ ] لا جلب مكرر لنفس البيانات عبر المكونات (مفتاح استعلام مشترك).

*(قائمة مراجعة النشر غير قابلة للتطبيق على هذا الفصل.)*

## تمارين

**1. استبدل الجلب المبني يدوياً.** خذ مكوّناً يحمّل بيانات الخادم بـ `useEffect` +
`fetch` + `useState` (اكتب واحداً، أو اطلب من مساعد توليد "تحميل وعرض قائمة الفواتير")
وحوّله إلى React Query، ثم بيّن فائدة كانت النسخة القديمة تفتقر إليها — لا إعادة جلب عند
إعادة التركيب، أو تغيير فلتر سريع لا يتسابق. المُنتَج هو قبل/بعد والتحسين
المُثبَت.

**2. أصلح العملية القديمة.** خذ عملية "وضع علامة مدفوع" على فاتورة بلا إبطال لذاكرة التخزين المؤقت وراقب
القائمة تعرض الحالة القديمة بعد نجاح العملية. أصلحها بـ
`invalidateQueries`، ثم أضف تحديثاً متفائلاً ولاحظ الفرق في السرعة المُدركة. المُنتَج
هما النسختان وملاحظة عن متى يستحق تعقيد التحديث المتفائل العناء.

**3. سَطِّح شلالاً.** خذ شاشة تجلب عدة قطع مستقلة من البيانات
بالتسلسل (اكتبها، أو ولّد "تحميل الفاتورة، وعميلها، ومدفوعاتها") وأعد هيكلتها
لتجلبها بالتوازي (و/أو على الخادم). المُنتَج هو قبل/بعد وفرق زمن التحميل
الذي تتوقعه.

## قراءات إضافية

- **TanStack Query (React Query) documentation** (tanstack.com/query) — الدليل المرجعي،
  خاصة "Important Defaults"، و"Query Invalidation"، و"Mutations". يتناول قسمَا الإبطال
  والعمليات خطأ الواجهة القديمة بشكل مباشر.
- **TkDodo's blog — "Practical React Query" series** (tkdodo.eu/blog) — كتبه أحد حافظي React Query؛
  أفضل إرشاد عملي حول مفاتيح الاستعلام، والتمييز بين حالة الخادم والعميل،
  وتجنب المزالق الشائعة.
- **Next.js documentation — "Data Fetching, Caching, and Revalidating"** (nextjs.org/docs) — كيف
  يعمل الجلب من جانب الخادم والتخزين المؤقت في App Router، وكيف تعيد التحقق بعد العمليات
  على جانب الخادم؛ النظير لذاكرة التخزين المؤقت على العميل في React Query.
- **"You Might Not Need an Effect" و إرشادات جلب البيانات** (react.dev) — بيان React نفسه
  بأن الجلب في `useEffect` ليس المسار الموصى به، ولماذا يجب أن يتولاه إطار عمل أو
  مكتبة بدلاً من ذلك.
