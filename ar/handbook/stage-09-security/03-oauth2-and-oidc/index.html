    OAuth 2.0 و OpenID Connect¶

    المقدمة¶

    تُبدو خاصية "تسجيل الدخول عبر Google" كميزةٍ تُضاف في فترة ما بعد الظهر: سجّل تطبيقًا، أدخل معرّف العميل، أعِد التوجيه، احصل على رمز مميز، سجّل دخول المستخدم. المكتبة تقوم بمعظم العمل. وتلك النسخة التي تُبنى في فترة ما بعد الظهر هي بالضبط المكان الذي تعيش فيه عائلة محددة وموثّقة من الثغرات الأمنية — لأن OAuth ليس بروتوكول تسجيل دخول، بل هو بروتوكول تفويض تفويضي (authorization delegation) أُعيد توظيفه على نطاق واسع لتسجيل الدخول، والفجوة بين ما يثبته فعلاً وما يفترض الناس أنه يثبته هي بالضبط مكان الهجمات.

    يدور هذا الفصل حول دمج هوية الطرف الثالث — OAuth 2.0 للتفويض التفويضي وOpenID Connect (OIDC) للمصادقة — بشكل آمن. يغطي التدفقات ولماذا يُعدّ أحدها الخيار الآمن الوحيد الافتراضي (Authorization Code مع PKCE)، والمعاملات التي تُعدّ ضوابط أمنية حاسمة وليست مجرد تفاصيل روتينية (state، redirect_uri، nonce، PKCE)، والتمييز الجوهري الذي يسهّل OAuth الخطأ فيه (رمز الوصول access token يثبت أنه يحق لك استدعاء واجهة API، وليس من أنت — هذه مهمة رمز ID token)، والهجمات الكلاسيكية: CSRF على رد النداء، التلاعب بإعادة التوجيه، استبدال الرمز المميز، وهجوم mix-up.

    ثمة حدودان يؤطّران الموضوع. أولاً، هذا فصل الدمج والهجوم: بناء المصادقة الذاتية من الطرف الأول — تجزئة كلمات المرور، إصدار الرموز المميزة الخاصة بك — كان في المرحلة 3، الفصل 03، وصلابة تلك الرموز كانت في الفصل 02 هنا؛ هذا الفصل يدور حول تفويض الهوية لمزوّد خارجي والقيام بذلك دون وراثة أخطاء OAuth. ثانيًا، رموز OIDC ID token هي رموز JWT، لذا فإن كل ما قاله الفصل 02 عن التحقق من JWT (تثبيت الخوارزمية، التحقق من المطالبات claims، جلب المفاتيح من JWKS) ينطبق مباشرة على التحقق من ID token — يبني هذا الفصل على ذلك بدلاً من تكراره. سبب فهم OAuth بعمق على الرغم من أن المكتبة تنفّذه: المكتبة تمنحك الآليات، لكن أي تدفق، وأي معاملات، وأي رمز مميز يثبت ماذا هي قرارات تتركها المكتبة لك، والخطأ فيها يشحن تسجيل دخول يعمل ويمكن اختطافه.

    لماذا هذا مهم¶

    تفشل عمليات دمج OAuth بطرق غير مرئية في العرض التجريبي ووخيمة في الإنتاج، لأن مرونة البروتوكول هي بالضبط خطره — معظم طرق استخدامه غير آمنة، ولا يوجد سوى مسار ضيق آمن.

    • رمز الوصول (access token) ليس إثباتًا للهوية — افتراض ذلك ثغرة حقيقية. يعني رمز الوصول "يجوز لحامله استدعاء واجهات API هذه نيابةً عن هذا المستخدم." إنه لا يصادق على من يقدّمه. يمكن إعادة تشغيل رمز صُكّ لتطبيق خبيث، أو سُرق، ضد خلفية تعامل "لديه رمز وصول Google صالح" على أنه "هو مستخدم Google هذا" — هجوم confused-deputy / استبدال الرمز. المصادقة هي مهمة ID token (OIDC)، وخلط الاثنتين هو الخطأ الأكثر شيوعًا في استخدام OAuth لتسجيل الدخول.
    • معامل state هو دفاع CSRF مطلوب وليس مجرد تفصيل اختياري. بدونه، يمكن للمهاجم بدء تدفق OAuth بحسابه الخاص وخداع متصفح الضحية لإكماله، مما يُسجّل دخول الضحية بصمت في حساب المهاجم (login CSRF) — أو العكس، ربط هوية المهاجم بجلسة الضحية. يربط state رد النداء بالطلب الذي بدأه؛ إغفاله يعرض رد النداء للتزوير.
    • redirect_uri هو المكان الذي تُسلَّم إليه الرموز المميزة — المطابقة الفضفاضة تسلّمها للمهاجمين. إذا كان المزوّد أو تطبيقك يقبل redirect URI حسب البادئة، أو يسمح بإعادة التوجيه المفتوحة على نطاقك، فسيسجّل المهاجم أو يصوغ redirect يُرسل رمز التفويض أو الرمز المميز إلى مُضيف يسيطر عليه. تسجيل redirect URI بالمطابقة التامة هو ما يحفظ بيانات الاعتماد من أن تُسلَّم إلى الباب الخطأ.
    • يجب التحقق من ID token كما يُتحقق من أي JWT — الملف الشخصي المُجلب ليس تحققًا. استدعاء نقطة نهاية userinfo للمزوّد أو فك ترميز ID token لقراءة البريد الإلكتروني لا يُعدّ مصادقة ما لم يُتحقق من توقيع ID token، والمُصدِر (issuer)، والجمهور (audience)، وnonce (الفصل 02). رمز ID token غير المُتحقق منه هو بيانات قابلة للتعديل من المهاجم.
    • يُغلق PKCE ثغرة اعتراض الكود، وهو الآن الخط الأساسي للجميع. كان في الأصل لتطبيقات الجوال التي لا تستطيع حفظ سر، Proof Key for Code Exchange يُوصى به الآن لـجميع العملاء، بما في ذلك الويب: يربط رمز التفويض بالعميل الذي طلبه، بحيث يكون الكود المُعترض عديم الفائدة لأي شخص آخر. تخطّيه يُبقي هجوم اعتراض الكود مفتوحًا.
    • البُعد الخاص بالذكاء الاصطناعي حاد. يُولّد المساعدون تدفقات OAuth تُكمل الدورة الكاملة وتُسجّل دخول المستخدم مع إغفال state، أو مطابقة redirect_uri بشكل فضفاض، أو التعامل مع رمز الوصول كهوية، أو تخطّي التحقق من ID token — كل واحدة منها تصادق على مستخدم happy path وتُسلّم للمهاجم إمكانية الاختطاف. إن "عمل" التدفق لا يقول شيئًا عمّا إذا كان آمنًا.

    النموذج الذهني¶

    OAuth رقصة تفويضية بين أربعة أطراف، والأمن يكمن في المعاملات التي تربط الخطوات ببعضها بحيث لا يستطيع المهاجم زرع خطوته الخاصة في تدفقك.

       THE FOUR PARTIES
     Resource Owner  = the user            Client       = your app (Invoicely)
     Authorization Server = the provider (Google)    Resource Server = the provider's API

   AUTHORIZATION CODE FLOW + PKCE (the ONE safe default for web, mobile, SPA)
     1. Client → browser → Auth Server:  /authorize?client_id&redirect_uri&scope
                                          &state=RANDOM&nonce=RANDOM&code_challenge=HASH(verifier)
     2. user authenticates & consents at the Auth Server (your app never sees their password)
     3. Auth Server → browser → redirect_uri?code=AUTHZ_CODE&state=RANDOM
                                          │
                                          ▼  CHECK state == what you sent  (CSRF defense)
     4. Client → Auth Server (back channel): exchange code + code_VERIFIER (+client secret)
                                          │
                                          ▼  PKCE: verifier must hash to the challenge from step 1
     5. Auth Server → Client:  access_token (call APIs)  +  id_token (WHO the user is, OIDC)
                                          │
                                          ▼  VERIFY id_token as a JWT (Ch 02): sig, iss, aud, exp,
                                             and nonce == what you sent
     6. Client establishes ITS OWN session for the user (your session, not the provider's token)

   WHAT EACH TOKEN PROVES  (getting this wrong is the headline bug)
     access_token → "bearer may call these APIs for this user"   NOT identity
     id_token     → "this user authenticated at this provider"   ← authentication (OIDC only)

   THE LOAD-BEARING PARAMETERS (each defends a specific attack)
     state          → CSRF on the callback (bind callback to your request)
     redirect_uri   → token delivery address; EXACT match, pre-registered (no prefix/open redirect)
     PKCE           → binds code to this client (defeats code interception) — use everywhere
     nonce          → binds id_token to this request (defeats id_token replay)

    ثلاثة مبادئ يحملها الفصل:

    المصادقة هي مهمة ID token؛ رمز الوصول مخصص لاستدعاء واجهات API. إذا كنت تُسجّل دخول مستخدم، فأنت بحاجة إلى OIDC ورمز ID token مُتحقَّق منه. لا تستنتج الهوية أبدًا من رمز الوصول، ولا تثق أبدًا في رد userinfo حصلت عليه بتقديم رمز مميز لم تتحقق من مصدره.

    كل معامل ربط هو ضابط أمني — عامل المفقود منها كأنه ثغرة. state، وredirect_uri التام، وPKCE، وnonce كل منها يدافع عن هجوم مُسمّى. ليست إعدادات اختيارية؛ تدفق ينقصه أي منها هو تدفق ينقصه دفاع، على الرغم من أنه لا يزال يُسجّل دخول الناس.

    استخدم تدفق Authorization Code مع PKCE، ولا شيء غيره. التدفق الضمني (الرموز في جزء fragment من URL) وتدفق resource-owner-password مهجوران لأسباب وجيهة — يُسرّبان الرموز ويُسلّمان تطبيقك كلمة مرور المستخدم لدى المزوّد. يوجد خيار آمن افتراضي واحد؛ استخدمه للويب وSPA والجوال على حد سواء.

    تعريف عملي:

    يُفوِّض OAuth 2.0 الصلاحيات؛ ويُضيف OpenID Connect المصادقة فوق ذلك عبر رمز ID token قابل للتحقق. يعني دمجهما بأمان استخدام تدفق Authorization Code مع PKCE، ومعاملة state، والمطابقة التامة لـredirect_uri، وPKCE، وnonce كدفاعات إلزامية خاصة بكل هجوم، واستخلاص الهوية فقط من ID token مُتحقَّق منه بالكامل (أبدًا من رمز وصول)، وإنهاء التدفق بإنشاء جلستك الخاصة بدلاً من تمرير رمز المزوّد.

    مثال إنتاجي¶

    تُضيف Invoicely خاصية "تسجيل الدخول عبر Google" ليتسنى للمستخدمين البدء دون إنشاء كلمة مرور أخرى، وتُحضّر للسماح لهم بربط Stripe بحيث تتمكّن Invoicely من إنشاء رسوم نيابةً عنهم. هذان هما وجهان للبروتوكول: تسجيل الدخول عبر Google هو مصادقة (OIDC — من هذا المستخدم)، وStripe Connect هو تفويض (OAuth — هل يجوز لـInvoicely التصرف نيابةً عن هذا الحساب). بناء الاثنين في فصل واحد يجعل التمييز ملموسًا، لأن متطلبات الأمن تتباعد تمامًا حيث تتباعد معاني الرموز.

    بالنسبة لتسجيل الدخول عبر Google، تستخدم Invoicely تدفق Authorization Code مع PKCE. تُولّد قيمتي state وnonce عشوائيتين لكل محاولة، وتخزّنهما مربوطتَين بجلسة المتصفح، وتُسجّل redirect_uri التام مسبقًا في وحدة تحكم Google — لا أحرف بدل، ولا مطابقة بالبادئة. في رد النداء، تتحقق من state قبل فعل أي شيء آخر (CSRF)، وتُبادل الكود مع مُحقّق PKCE عبر القناة الخلفية، ثم تتحقق من ID token المُعاد كرمز JWT باستخدام قواعد الفصل 02: التوقيع مقابل JWKS من Google، وiss هو مُصدِر Google، وaud هو معرّف عميل Invoicely، وexp مستقبلي، وnonce يطابق ما أرسلته. فقط بعد كل ذلك تُطابق Invoicely الـsub المُتحقَّق منه مع حساب محلي وتُنشئ جلستها الخاصة — لا تتعامل أبدًا مع رمز وصول Google كهوية، ولا تخزّنه أبدًا كجلسة للمستخدم.

    بالنسبة لـStripe Connect، تستخدم Invoicely OAuth للتفويض: تستقبل رمز وصول مُقيَّد النطاق لإنشاء رسوم على الحساب المتصل، وتخزّن هذا الرمز (مشفّرًا، الفصل 04) كـقدرة (capability) — دليل على أنه يجوز لها استدعاء Stripe لهذا الحساب — وليس كدليل على هوية أي شخص. ربط الحساب مربوط بجلسة مستخدم Invoicely الذي تمّت مصادقته بالفعل، بحيث لا يستطيع المهاجم تطعيم حساب Stripe الخاص به على تسجيل دخول الضحية.

    في هذا الفصل، نبني تدفق Google OIDC مع وجود كل معامل ربط في مكانه، ونقارنه بنسخة الإعداد الافتراضي لدى المساعد (بلا state، رمز وصول يُعامل كهوية، ID token يُفك ترميزه لا يُتحقق منه)، ونُبيّن أن كل إغفال هو الاختطاف المُحدّد الذي يُمكّنه.

    def verify_id_token(id_token: str, expected_nonce: str) -> dict:
    claims = jwt.decode(
        id_token,
        key=PROVIDER_JWKS.public_key_for(id_token),      # provider's keys, by kid, over TLS, cached
        algorithms=["RS256"],                            # PINNED (Ch 02)
        audience=CFG.client_id,                          # aud == US: this token was minted for Invoicely
        issuer=CFG.issuer,                               # iss == the provider we trust
        options={"require": ["exp", "iat", "aud", "iss", "sub", "nonce"]},
    )
    if not secrets.compare_digest(claims["nonce"], expected_nonce):
        raise AuthError("invalid nonce")                 # replay defense: this id_token is for THIS attempt
    return claims

    القرارات الهندسية¶

    خمسة قرارات تُعرّف دمج OAuth/OIDC آمنًا.

    أي تدفق؟¶

    الخيارات: (1) Authorization Code مع PKCE؛ (2) التدفق الضمني (الرموز في جزء fragment من URL)؛ (3) بيانات اعتماد مالك المورد resource-owner password؛ (4) client credentials (بلا مستخدم).

    المقايضات: يتبادل Authorization Code مع PKCE رمزًا قصير العمر عبر قناة خلفية ويربطه بالعميل — الرمز لا يركب في redirect URL، والكود المُعترض عديم الفائدة. يعرض التدفق الضمني الرموز مباشرة في الـfragment، مما يعرّضها لتاريخ المتصفح، والمُحيلين، والنصوص البرمجية — مهجور. يُسلّم resource-owner password لتطبيقك كلمة مرور المستخدم لدى المزوّد، مما يهزم نقطة التفويض بأكملها — لا تستخدمه أبدًا لتسجيل دخول الطرف الثالث. client credentials مخصص للخدمة إلى الخدمة بلا مستخدم ولا ينطبق على تسجيل الدخول.

    التوصية: Authorization Code مع PKCE لكل عميل موجّه للمستخدم — الويب وSPA والجوال على حد سواء. لم يعد PKCE مقصورًا على الجوال؛ إنه الخط الأساسي. ينبغي ألا يظهر التدفق الضمني وتدفق كلمة المرور في التعليمات البرمجية الجديدة؛ عامل وجودهما كثغرة.

    المصادقة (OIDC) أو التفويض (OAuth) — أيهما تفعل فعليًا؟¶

    الخيارات: (1) OIDC مع ID token مُتحقَّق منه لتسجيل الدخول؛ (2) رمز وصول OAuth للوصول إلى API؛ (3) معاملة رمز الوصول كتسجيل دخول (الخطأ).

    المقايضات: إذا كان الهدف "سجّل دخول هذا المستخدم"، فأنت بحاجة إلى OIDC وID token مُتحقَّق منه — الرمز الذي غرضه تأكيد المصادقة. إذا كان الهدف "استدعِ API المزوّد لهذا المستخدم"، فأنت بحاجة إلى رمز وصول، يُستخدم كقدرة. استخدام رمز الوصول لتسجيل الدخول هو ثغرة استبدال الرمز؛ واستخدام ID token لاستدعاء واجهات API ببساطة لا يعمل.

    التوصية: سمّ ما تفعله لكل دمج واستخدم الرمز المُطابق. تسجيل الدخول ← ID token مُتحقَّق منه ← جلستك الخاصة. الوصول إلى API ← رمز وصول مخزّن كقدرة مشفّرة، لا يُستشار أبدًا للهوية. عندما تحتاج إلى كليهما (تسجيل الدخول و الوصول إلى API)، تحصل على الرمزين وتبقي مهمتَيْهما منفصلتين.

    ما مدى صرامة مطابقة redirect_uri؟¶

    الخيارات: (1) مطابقة تامة لـURI مُسجّل مسبقًا؛ (2) مطابقة بالبادئة/النطاق الفرعي؛ (3) يُبنى ديناميكيًا من الطلب.

    المقايضات: تعني المطابقة التامة أنه لا يمكن تسليم رمز التفويض/الرمز المميز إلا إلى URI سجّلته — لا يستطيع المهاجم إعادة توجيهه إلى مكان آخر. تفتح المطابقة بالبادئة أو النطاق الفرعي الباب لمسار أو مُضيف يسيطر عليه المهاجم يستوفي القاعدة الفضفاضة. بناء redirect من الطلب هو الأسوأ — redirect مفتوح يُسلّم بيانات الاعتماد لأي مُضيف.

    التوصية: مطابقة تامة، redirect URI مُسجّل مسبقًا، واحد لكل بيئة، لا يُشتق أبدًا من مدخلات الطلب. تُجمع مع عدم وجود redirect مفتوح في أي مكان على نطاقك (يمكن ربط redirect مفتوح في مكان آخر لسرقة الكود). هذا قرار إعدادات ذو عواقب كلية.

    أين تعيش معاملات الأمن، وكم من الوقت؟¶

    الخيارات: (1) جلسة في جانب الخادم، تُستخدم مرة واحدة، TTL قصير؛ (2) كوكيز موقّعة؛ (3) غير مخزّنة (الثغرة).

    المقايضات: يجب أن يُولّد state، وnonce، ومُحقّق PKCE بواسطة خادمك، ومربوطًا بالمتصفح المحدد الذي بدأ التدفق، ويُستخدم مرة واحدة بالضبط، وينتهي صلاحيته بسرعة. التخزين في جانب الخادم هو النموذج النظيف. يمكن أن يعمل الكوكيز الموقّع HttpOnly إذا كان لكل محاولة ومُتحقَّقًا منه. عدم تخزينها — أو تخزينها عالميًا بدلاً من تخزينها لكل محاولة — يعني أن دفاعات CSRF وإعادة التشغيل لا تربط فعليًا أي شيء.

    التوصية: وُلّد لكل محاولة، وخزّن في جانب الخادم (أو في كوكيز موقّع لكل محاولة) مربوطًا بالجلسة، وأجبر الاستخدام مرة واحدة بحذفه عند قراءته، وأعطِه TTL قصيرًا (دقائق). تدفق OAuth يعيد استخدام هذه القيم أو يُغفلها لديه دفاعات زخرفية، لا حقيقية.

    المصادقة الذاتية، شراء مزوّد مصادقة، أو التفويض المباشر؟¶

    الخيارات: (1) ابنِ مصادقة الطرف الأول الخاصة بك (المرحلة 3)؛ (2) ادمج المزوّدين مباشرة بنفسك (هذا الفصل)؛ (3) استخدم وسيط مصادقة (Auth0، Clerk، Supabase Auth، Better Auth) يكون واجهة للمزوّدين.

    المقايضات: يمنحك الدمج المباشر سيطرة كاملة ولا تكلفة لكل مستخدم، على حساب ضرورة صحة كل معامل ربط لكل مزوّد بنفسك. يُنفّذ الوسيط التدفقات بشكل صحيح مرة واحدة، ويتعامل مع تعدد المزوّدين ودورة حياة الرمز، ويزيل فئة من المخاطر عن كاهلك — بتكلفة وبقفل. تعني الطرف الأول فقط أن المستخدمين يديرون كلمة مرور أخرى.

    التوصية: بالنسبة لمعظم الفرق، وسيط المصادقة هو الخيار الهندسي السليم تحديدًا لأن هجمات هذا الفصل يسهل أن تخطئ فيها بشكل خفي والمتخصّص يُحكمها — نفس منطق "اشترِ المصادقة" من المرحلة 1، الفصل 06 والمرحلة 3، الفصل 03. ادمج مباشرة عندما يكون لديك سبب محدد (التكلفة على نطاق واسع، السيطرة، لا طرف ثالث)، وعندما تفعل ذلك، التزم بكل قرار أعلاه.

    المقايضات¶

    تفويض الهوية يقايض تبعية بعدم حمل كلمات المرور. يعني "تسجيل الدخول عبر Google" أنك لا تخزّن كلمة مرور وترث أمان المزوّد وMFA — ويعني أن تسجيل دخولك يعتمد على توفرهم ويحتاج مستخدموك إلى حساب لديهم. بالنسبة للمنتجات الاستهلاكية، يميل التبادل لصالح التفويض؛ أما في بعض سياقات B2B فيجب أن يوجد خيار الطرف الأول إلى جانبه.

    PKCE ومعاملات الربط تقايض قليلًا من التعقيد بإغلاق هجمات مُسمّاة. يُضيف state، وnonce، وPKCE، والمطابقة التامة لـredirect تعليمات برمجية وحالة لكل محاولة. كل واحد منها يشتري إغلاق هجوم محدد ومُستغَل. ليس هذا تعقيدًا لذاته؛ إنه الحد الأدنى الذي يجعل التدفق آمنًا، وتخطّي أي قطعة يعيد فتح ثغرة معروفة.

    استخدام وسيط يقايض السيطرة والمال مقابل الصحة. يُنفّذ مزوّد المصادقة التدفقات بشكل صحيح ويُحافظ عليها مع تطور المواصفات، على حساب الرسوم، والقفل، ووجود طرف ثالث في مسار تسجيل الدخول. البديل هو امتلاك دمج حرج أمنيًا تُعدّ إخفاقاته الخفية خروقات. طابق الاختيار مع عمق الأمان في فريقك ومستوى المخاطر الكبيرة.

    الإنهاء بجلستك الخاصة (مقابل تمرير رموز المزوّد) يقايض خطوة بالسيطرة. سكّ جلستك الخاصة بعد التحقق هو قطعة إضافية يجب بناؤها، لكنه يعني أن عمر رمز المزوّد وإلغاءه ونطاقه لا يتسرّب إلى نموذج التفويض في تطبيقك — جلستك هي التي تنتهي صلاحيتها وتُلغى أنت. تمرير رمز المزوّد كبيانات اعتماد التطبيق يربطك بدلالاته ويوسّع نصف قطر انفجار سرقته.

    الأخطاء الشائعة¶

    معاملة رمز الوصول كهوية. تسجيل دخول المستخدم لأنّه قدّم رمز وصول صالحًا من المزوّد، مما يثبت تفويض الوصول إلى API وليس هويته — ثغرة استبدال الرمز. الإصلاح: صادق فقط من ID token مُتحقَّق منه (OIDC)؛ استخدم رموز الوصول كقدرات.

    إغفال state أو عدم التحقق منه. لا ربط CSRF على رد النداء، لذلك يمكن للمهاجم إكمال تدفق في متصفح الضحية (login CSRF / تطعيم الحساب). الإصلاح: وُلّد state لكل محاولة، وخزّنه في جانب الخادم، وقارنه أول شيء في رد النداء.

    مطابقة redirect_uri فضفاضة. redirect URI بالبادئة أو النطاق الفرعي أو المُشتق من الطلب تتيح للمهاجم استلام الكود/الرمز. الإصلاح: مطابقة تامة، URI مُسجّل مسبقًا؛ لا توجد redirect مفتوحة في أي مكان على النطاق.

    فك ترميز ID token بدلاً من التحقق منه. قراءة email/sub من ID token أو userinfo دون التحقق من التوقيع والمُصدِر والجمهور وnonce. الإصلاح: تحقق من ID token كرمز JWT (الفصل 02) مقابل JWKS الخاص بالمزوّد مع aud = معرّف عميلك.

    تخطّي PKCE. الاعتماد على سر العميل وحده (أو لا شيء، للعملاء العامين)، مما يعرّض رمز التفويض للاعتراض. الإصلاح: PKCE في كل تدفق، طريقة تحدّي S256.

    تمرير رمز المزوّد كجلسة التطبيق. استخدام رمز وصول Google كبيانات اعتماد جلسة Invoicely، مما يربط مصادقة التطبيق بعمر رمز المزوّد ونطاقه. الإصلاح: سكّ جلستك الخاصة بعد التحقق؛ أبقِ رموز المزوّد كقدرات منفصلة ومشفّرة.

    أخطاء الذكاء الاصطناعي¶

    OAuth بروتوكول يختلف فيه المسار السعيد (happy path) عن المسار الآمن، لذا مخرجات المساعد التي تُسجّل دخول المستخدم بشكل موثوق تُشحن دفاعات ناقصة. راجع التدفق بحثًا عن معاملات الربط ودلالات الرمز، وليس عمّا إذا كان تسجيل الدخول ينجح.

    Claude Code: التدفق بلا state (رد نداء غير محمي)¶

    عند طلب "أضف تسجيل الدخول عبر Google"، يُولّد Claude Code غالبًا تدفق Authorization Code نظيفًا يُغفل state تمامًا — بناء عنوان URL للتفويض، ومعالجة رد النداء، وتبادل الكود، وتسجيل الدخول. يعمل بشكل مثالي لمستخدم حقيقي ويترك رد النداء مفتوحًا لـCSRF: يبدأ المهاجم تدفقًا بحسابه الخاص ويُوصل الضحية إليه، أو يُطعّم هويته في جلسة الضحية. ولأن الإغفال معامل مفقود، لا يحدث خطأ ولا اختبار يتحقق فقط من "تسجيل الدخول يعمل" يكشفه.

    الاكتشاف: عنوان URL للتفويض مبني دون معامل state؛ معالج رد نداء يقرأ code لكن لا يقارن state بقيمة مخزّنة؛ لا تخزين في جانب الخادم لـstate لكل محاولة؛ رد النداء ينجح عند غياب state أو كونه عشوائيًا.

    الإصلاح: اجعل state إلزاميًا ومُفحوصًا أولًا:

    يجب على تدفق OAuth توليد state عشوائي لكل محاولة، وتخزينه في جانب الخادم مربوطًا بالجلسة، ورفض رد النداء (401) قبل أي تبادل للكود إذا لم يتطابق state المُعاد. أضف اختبارًا بأن رد النداء بـstate مفقود أو خاطئ مرفوض.

    GPT: رمز الوصول المُستخدم كإثبات هوية¶

    تُشوّش نماذج عائلة GPT كثيرًا بين المصادقة والتفويض: بعد التبادل، تقرأ ملف المستخدم بتقديم رمز الوصول (أو تفك ترميزه) وتُنشئ جلسة من ذلك — معاملة "يحمل رمز وصول صالحًا من المزوّد" على أنها "هو مستخدم المزوّد هذا". هذه هي ثغرة استبدال الرمز: رمز حصل عليه عميل خبيث، أو لتطبيق آخر، يمكن إعادة تشغيله لتسجيل الدخول كضحية، لأن الهوية لم تُربط أبدًا برمز صُكّ لتطبيقك.

    الاكتشاف: جلسات تُنشأ من رمز وصول أو استدعاء userinfo بدلاً من ID token مُتحقَّق منه؛ لا ID token مطلوب (نطاق openid مفقود) أو مطلوب لكن غير مُتحقَّق منه؛ الهوية (sub/email) مأخوذة من استدعاء API مُخوَّل برمز وصول لم يُتحقق أبدًا من جمهوره.

    الإصلاح: صادق من ID token مُتحقَّق منه فقط:

    يجب أن تستخدم المصادقة OIDC: اطلب نطاق openid، واستخلص الهوية فقط من ID token بعد التحقق من توقيعه، وiss، وaud (يجب أن يساوي معرّف عميلنا)، وexp، وnonce. لا تُنشئ أبدًا جلسة من رمز وصول أو رد userinfo غير مُتحقَّق منه. أضف اختبارًا بأن رمز وصول خام من المزوّد لا يمكنه إنشاء جلسة.

    Cursor: ID token مفكوك الترميز وغير مُتحقَّق منه (ودون nonce)¶

    عند إكمال معالج رد نداء، يقرأ Cursor غالبًا مطالبات ID token بفك ترميز غير مُتحقَّق منه — jwt.decode(id_token, options={"verify_signature": False}) أو تقسيم base64 — للحصول على البريد الإلكتروني، لأن المهمة المباشرة كانت "احصل على معلومات المستخدم"، ويُغفل جلب JWKS الخاص بالمزوّد والتحقق من aud/nonce. المطالبات قابلة للتعديل من المهاجم، وبدون nonce يكون ID token قابلًا لإعادة التشغيل. يعمل تسجيل الدخول برمز حقيقي ويقبل رمزًا مزوّرًا.

    الاكتشاف: قراءات ID token مع verify_signature: False أو فك ترميز base64 يدويًا؛ لا استدعاء لـJWKS الخاص بالمزوّد؛ aud/iss غير مُؤكَّدَين مقابل إعداداتك؛ لا nonce مولّد في البدء أو مُفحوص في الإعادة.

    الإصلاح: تحقق من ID token باستخدام مُحقّق الفصل 02 واربط nonce:

    يجب التحقق من ID token، لا فك ترميزه: التوقيع مقابل JWKS الخاص بالمزوّد (حسب kid)، وalgorithms مُثبَّتة، وiss وaud (معرّف عميلنا) مُؤكَّدَين، وexp مُفروض، وnonce مُطابق للقيمة المولّدة في بداية التدفق. أضف اختبارًا بأن ID token مُعدَّل وID token بجمهور خاطئ كلاهما مرفوض.

    أفضل الممارسات¶

    استخدم Authorization Code مع PKCE، دائمًا. تدفق واحد للويب وSPA والجوال؛ لا تدفق ضمني، لا resource-owner-password. PKCE مع S256 في كل محاولة.

    سمّ المصادقة مقابل التفويض لكل دمج. تسجيل الدخول يستخلص الهوية من ID token مُتحقَّق منه وينتهي بجلستك الخاصة؛ الوصول إلى API يستخدم رمز وصول كقدرة مشفّرة. لا تخلط الأسلاك أبدًا.

    عامل state، وredirect_uri التام، وPKCE، وnonce كإلزامية. كل واحد يدافع عن هجوم مُسمّى (CSRF، سوء تسليم الرمز، اعتراض الكود، إعادة التشغيل). وُلّد العشوائية لكل محاولة، وخزّن في جانب الخادم، تُستخدم مرة واحدة، TTL قصير؛ تحقّق من state قبل أي شيء آخر.

    تحقق من ID token كما يُتحقق من أي JWT. JWKS الخاص بالمزوّد حسب kid، خوارزمية مُثبّتة، iss/aud/exp مُؤكَّدة، nonce مُطابق — قواعد الفصل 02 ضد رمز سكّه شخص آخر.

    أنهِ التدفق بجلستك الخاصة. سكّ جلسة Invoicely بعد التحقق؛ أبقِ رموز المزوّد منفصلة ومشفّرة. جلستك هي ما تتحكم فيه، وتنتهي صلاحيتها، وتُلغيها.

    فضّل وسيط مصادقة مُختبَرًا ما لم يكن لديك سببًا لعدم ذلك. من السهل أن تخطئ في هذه الهجمات بشكل خفي؛ يُنفّذ المتخصّص التدفقات بشكل صحيح ويُحافظ عليها. الدمج المباشر خيار متعمّد بتكلفة أمنية تقبلها عن دراية.

    اختبر الهجمات. state مفقود/مزوّر، ID token مُعدَّل وبجمهور خاطئ، تبادل كود دون PKCE، رمز وصول كهوية — اختبار أمني لكل هجوم، أخضر في CI.

    الأنماط المضادة¶

    التدفق الضمني. الرموز المُعادة في جزء fragment من URL، مكشوفة للتاريخ والمُحيلين والنصوص البرمجية. العلامة: response_type=token؛ رموز وصول تظهر في redirect URL؛ لا تبادل كود عبر قناة خلفية.

    الهوية من رمز الوصول. تسجيل دخول المستخدمين من رمز وصول أو userinfo غير مُتحقَّق منه. العلامة: جلسات تُنشأ دون ID token؛ نطاق openid مفقود؛ رمز خام من المزوّد مقبول كبيانات اعتماد تسجيل دخول.

    رد النداء عديم الحالة. رد نداء OAuth بلا فحص state. العلامة: لا تخزين لـstate لكل محاولة؛ رد النداء ينجح بـstate عشوائي أو غائب.

    redirect الفضفاض. redirect_uri بالبادئة/النطاق الفرعي/المُشتق من الطلب. العلامة: تسجيل redirect بأحرف بدل أو بادئة؛ تجميع redirect URI من مدخلات الطلب؛ redirect مفتوحة في مكان آخر على النطاق.

    ID token مفكوك الترميز. مطالبات ID token تُقرأ دون تحقق. العلامة: verify_signature: False أو تقسيم base64 لقراءة المطالبات؛ لا جلب JWKS من المزوّد؛ aud/nonce لا تُفحص أبدًا.

    رمز المزوّد كجلسة. رمز وصول المزوّد يُستخدم كبيانات اعتماد جلسة التطبيق. العلامة: لا سكّ لجلسة الطرف الأول؛ تفويض التطبيق مرتبط بعمر رمز المزوّد ونطاقه.

    شجرة القرار¶

    "أنا أدمج مزوّد هوية/تفويض طرف ثالث — كيف أفعل ذلك بأمان؟"

    What am I actually doing?
├── LOGGING A USER IN ──► OIDC.
│     ├── flow: Authorization Code + PKCE (never implicit/password)
│     ├── request `openid` scope; generate state + nonce + PKCE verifier per attempt (server-side)
│     ├── callback: check `state` FIRST → exchange code with PKCE verifier
│     ├── verify the ID TOKEN as a JWT (Ch 02): sig via provider JWKS, iss, aud==our client id,
│     │     exp, nonce==ours
│     └── establish YOUR OWN session from the verified `sub` — never the provider's token
│
├── CALLING A PROVIDER API FOR THE USER ──► OAuth authorization.
│     ├── flow: Authorization Code + PKCE; request the specific scopes only
│     ├── store the access token as an ENCRYPTED capability (Ch 04), bound to the app user
│     └── never consult it for identity; refresh/rotate per the provider's rules
│
└── UNSURE WHETHER TO BUILD IT AT ALL ──►
      strongly consider a vetted auth broker — these attacks are subtle and specialists get
      them right. integrate directly only for a concrete reason, holding every decision above.

Any flow: redirect_uri EXACT-match pre-registered · no open redirects on the domain ·
          state/nonce/verifier single-use + short TTL · fail closed on any mismatch.

    قائمة التحقق¶

    قائمة التحقق من التنفيذ¶

    • [ ] جميع التدفقات الموجّهة للمستخدم تستخدم Authorization Code مع PKCE (S256)؛ لا تدفق ضمني ولا تدفق resource-owner-password موجود.
    • [ ] يُولّد state، وnonce، ومُحقّق PKCE لكل محاولة، ويُخزّن في جانب الخادم مربوطًا بالجلسة، ويُستخدم مرة واحدة، وTTL قصير.
    • [ ] يقارن رد النداء state قبل أي تبادل كود ويفشل بشكل مغلق عند عدم التطابق.
    • [ ] تُستخلص الهوية فقط من ID token مُتحقَّق منه كرمز JWT (التوقيع عبر JWKS الخاص بالمزوّد، الخوارزمية مُثبّتة، iss، aud = معرّف العميل، exp، nonce).
    • [ ] ينتهي التدفق بإنشاء جلسة من الطرف الأول؛ تُخزّن رموز وصول المزوّد على حدة كقدرات مشفّرة، أبدًا كهوية.
    • [ ] redirect_uri مطابقة تامة ومُسجّل مسبقًا لكل بيئة؛ لا توجد redirect مفتوحة على النطاق.

    قائمة التحقق المعمارية¶

    • [ ] يُصنَّف كل دمج صراحةً كمصادقة (OIDC/ID token) أو تفويض (OAuth/access token)، مع استخدام الرمز المُطابق.
    • [ ] إعدادات المزوّد (المُصدِر، JWKS URL، معرّف العميل، redirect URI، النطاقات) مركزية وغير مُشتقّة من مدخلات الطلب.
    • [ ] ربط الحساب (ربط مزوّد بمستخدم موجود) مربوط بجلسة التطبيق المصادق عليها بحيث لا يمكن تطعيم الهويات.
    • [ ] يُتخذ قرار البناء مقابل الوسيط بشكل متعمّد؛ إذا كان الدمج مباشرًا، تُسنَد مسؤولية صحة معاملات الربط.
    • [ ] التحقق من ID token يعيد استخدام مُحقّق JWT المُصلَّب (الفصل 02)، لا فحصًا ثانيًا أضعف.

    قائمة التحقق من مراجعة الكود¶

    • [ ] لا يُبنى عنوان URL للتفويض دون state، وnonce، وتحدّي PKCE.
    • [ ] لا جلسة تُنشأ من رمز وصول أو فك ترميز userinfo/ID token غير مُتحقَّق منه.
    • [ ] لا redirect_uri يُطابق بالبادئة، أو بأحرف بدل، أو يُبنى من الطلب.
    • [ ] معاملات الأمن تُستخدم مرة واحدة (تُحذف عند القراءة) ولا تكون عالمية أو مُعاد استخدامها عبر المحاولات.
    • [ ] اختبارات الهجوم موجودة لـstate مفقود/مزوّر، وID token مُعدَّل وبجمهور خاطئ، ورمز وصول كهوية.

    قائمة التحقق من النشر¶

    • [ ] تأتي أسرار العميل وبيانات اعتماد المزوّد من مدير الأسرار (الفصل 04)، لا من الكود أو ملفات الإعداد.
    • [ ] تُسجّل redirect URI تامة لكل بيئة (dev/staging/prod) دون أحرف بدل مشتركة.
    • [ ] يُجلب JWKS الخاص بالمزوّد عبر TLS ويُخزّن مؤقتًا بـTTL يدعم تدوير مفاتيح المزوّد.
    • [ ] يمنح إعداد نطاق openid/النطاقات الحد الأدنى من الامتياز؛ تُزال النطاقات غير المستخدمة (وموافقتها).

    التمارين¶

    1. اختطف رد نداء عديم الحالة، ثم أغلقه. ابنِ (أو خذ) تسجيل دخول OAuth يُغفل state. أثبت login-CSRF: أكمل تدفقًا بحيث ينتهي متصفح الضحية في جلسة بدأها المهاجم. ثم أضف state لكل محاولة وأثبت أن الهجوم نفسه يفشل الآن عند رد النداء. المُخرج هو شرح الهجوم والحالة قبل/بعد.

    2. أثبت أن رمز الوصول ليس هوية. في دمجك، حاول إنشاء جلسة باستخدام رمز وصول من المزوّد (رمز صُكّ لعميل مختلف، إن أمكنك الحصول عليه). أثبت ما إذا كان مقبولًا. إذا كان كذلك، انقل الهوية إلى ID token مُتحقَّق منه مع فحص aud، وأثبت أن الاستبدال مرفوض الآن. المُخرج هو التشغيلتان وفحص الجمهور الذي أغلقه.

    3. تحقق من ID token من الصفر. بالنظر إلى ID token حقيقي وJWKS URL الخاص بالمزوّد، اكتب التحقق الكامل: جلب المفاتيح حسب kid، تثبيت الخوارزمية، تأكيد iss/aud/exp، ومطابقة nonce ولّدته أنت. ثم أطعمه رمزًا مُعدّلًا، ورمزًا بجمهور خاطئ، ورمزًا منتهي الصلاحية وأثبت أن كل واحد مرفوض. المُخرج هو المُحقّق وحالات الاختبار الأربع.

    4. قارن الدمج المباشر بالوسيط. نفّذ تسجيل الدخول عبر Google مرتين: مرة مباشرة مع كل معامل ربط، ومرة عبر وسيط مصادقة (Auth0/Clerk/Supabase/Better Auth). قارن الكود الذي كان عليك كتابته وتأمينه، واذكر أي هجمات من هذا الفصل يجعلك كل نهج مسؤولاً عنها. قرر أيهما ستشحن وبرّر ذلك. المُخرج هو المقارنة والقرار.

    قراءات إضافية¶

    • OAuth 2.0 Security Best Current Practice (RFC 9700 / datatracker.ietf.org) — إرشادات الأمان المُحدَّثة والمُوحَّدة من IETF: لماذا يُعدّ Authorization Code + PKCE الخيار الافتراضي، ومعالجة redirect URI، والهجمات (CSRF، mix-up، حقن الكود) التي يدافع عنها هذا الفصل.
    • OpenID Connect Core 1.0 (openid.net/specs) — مواصفات طبقة المصادقة: ID token، وnonce، وقواعد التحقق التي تفصل مصادقة OIDC عن OAuth المجرد.
    • RFC 7636 — Proof Key for Code Exchange (PKCE) — الآلية ونموذج التهديد لربط رمز التفويض بالعميل؛ يُوصى به الآن لجميع أنواع العملاء.
    • OAuth 2.0 Simplified بقلم Aaron Parecki (oauth.com) — شرح واضح وعملي للتدفقات والمعاملات، جيد لبناء النموذج الذهني قبل قراءة RFCs.
    • الفصل 02 — أمان JWT والمرحلة 3، الفصل 03 — المصادقة — قواعد التحقق من ID token التي يعيد هذا الفصل استخدامها، ومصادقة الطرف الأول التي يُعدّ هذا التدفق المفوَّض بديلاً عنها.
