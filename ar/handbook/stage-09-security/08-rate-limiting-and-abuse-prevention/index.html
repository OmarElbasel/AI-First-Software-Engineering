    تحديد معدّل الطلبات ومنع إساءة الاستخدام¶

    المقدمة¶

    كل فصل سابق في هذه المرحلة دافع ضد ثغرة — عيب في الكود يستغلّه مهاجم. هذا الفصل الأخير يدافع ضد هجوم لا يحتاج إلى أي ثغرة على الإطلاق، بل إلى الحجم فقط: إرسال طلب صحيح مكوّن بشكل سليم مراتٍ كثيرة حتى يصبح هجومًا. Credential stuffing يجرّب مليون زوج مسروق من كلمات المرور ضد تسجيل الدخول لديك. Account enumeration يتحسّس أي الإيميلات مسجّلة. أداة سحب بيانات (scraper) تُفرغ قاعدة بياناتك بالكامل صفحة بترقيم تلو الأخرى. نقطة نهاية مكلّفة (عرض PDF، تقرير، بحث) تُضرب حتى يسقط الخادم. لا شيء من هذا يستغلّ خطأ برمجيًا. كل واحد منها يسيء استخدام ميزة شرعية باستعمالها بمقياس لم تُصمَّم له — والسبيل الوحيد للدفاع هو تحديد معدّل الطلبات.

    تحديد معدّل الطلبات بسيط بشكل مُخادع في المفهوم (عدّ الطلبات، ارفض فوق عتبة) ومليء بالأحكام في الممارسة: ماذا تعدّها ضد (IP؟ مستخدم؟ API key؟)، أيّ خوارزمية (fixed window، sliding window، token bucket — لكل منها سلوك اندفاع مختلف)، أين يعيش العدّاد (in-memory يفشل لحظة تشغيل أكثر من نسخة)، أين في الطبقات تفرضه (الحافة، البوابة، التطبيق)، وكيف تحدّ الإساءة دون معاقبة المستخدمين الشرعيين الذين يصادف وجودهم خلف IP مشترك أو يومًا مزدحمًا. إن أخطأت في هذه، فإما أن يكون لديك محدّد لا يعمل (يُعاد ضبطه عند إعادة التشغيل، يُتجنّب بتدوير IP، مفتاحه على ترويسة قابلة للتزوير)، أو يَحجب عملاءك الحقيقيين.

    يتناول هذا الفصل فئات الإساءة (brute force، credential stuffing، enumeration، السحب، استنزاف الموارد)، آليات تحديد المعدّل (الخوارزميات، المفاتيح، التخزين، الموضع)، ومجموعة أدوات منع الإساءة الأوسع التي يرسيها تحديد المعدّل: backoff تصاعدي، CAPTCHAs، قفل حساب آمن، وترويسات الاستجابة التي تتيح للعملاء الجيدين التعاون. الحدود: طبقة البنية التحتية للدفاع ضد DoS — حماية الفيضانات على مستوى الشبكة، الـ CDN/WAF، توزيع الأحمال — هي المرحلة 7 والمرحلة 11 (تصميم الأنظمة)؛ هذا الفصل هو دفاع طبقة التطبيق. نقاط نهاية المصادقة التي يجري حمايتها بُنيت في المرحلة 3، الفصل 03؛ هذا الفصل يُغلّفها بحدود إساءة الاستخدام. والتخزين المؤقت (caching) الذي يمتصّ الحمل أيضًا (المرحلة 3، الفصل 07) مُكمِّل يُشار إليه هنا، ولا يُعاد تعليمه.

    لماذا هذا مهم¶

    الإساءة المعتمدة على الحجم هي فئة الهجوم الأكثر احتمالًا لأن تضرب فعلًا نظام SaaS في الإنتاج، لأنها لا تتطلب مهارة، ولا ثغرة، ولا بحثًا خاصًا بالهدف — الأدوات مؤتمتة وتعمل بنقطة والتصويب.

    • تسجيل الدخول تحت هجوم مؤتمت باستمرار. بوتات Credential stuffing تعيد تشغيل مليارات أزواج اسم المستخدم/كلمة المرور المسرّبة من خروقات أخرى ضد كل نموذج تسجيل دخول على الإنترنت، باستمرار. بدون تحديد معدّل وكشف إساءة، يكون تسجيل الدخول لديك عرّافًا مفتوحًا لاختبار بيانات الاعتماد المسروقة — وكل ضربة ناجحة هي اختراق حساب لم تفعل شيئًا لمنعه. هذا ليس افتراضيًا؛ هذا هو الإشعاع الخلفي لتشغيل تطبيق ويب.
    • يكشفEnumeration من هم مستخدموك. نقطة نهاية تستجيب بشكل مختلف لإيميل مسجّل مقابل غير مسجّل (تسجيل الدخول، إعادة تعيين كلمة المرور، التسجيل) تتيح للمهاجم حصاد قائمة المستخدمين عبر التحسّس — قيّمة للتصيد المستهدف وcredential stuffing أعلاه. يبطئها تحديد المعدّل؛ الاستجابات الموحّدة (الفصل 01) تُغلقها؛ معًا يصنعان الفرق.
    • النقاط المكلّفة طريقة رخيصة لإسقاطك. أي نقطة نهاية تكلّف الخادم عملًا حقيقيًا — توليد PDF، تشغيل تقرير، بحث معقّد، إرسال بريد — يمكن تحويلها إلى denial-of-service باستدعائها في حلقة. المهاجم ينفق طلبًا واحدًا؛ أنت تنفق ثوانٍ من CPU أو استعلامًا ساحقًا لقاعدة البيانات. بدون حدود لكل نقطة نهاية، أكثر ميزاتك نفعًا هي أكثرها قابلية للاستغلال.
    • السحب يستنزف البيانات والموارد. يستطيع منافس أو وسيط بيانات اجتياز كامل مجموعة بياناتك عبر نقاط نهاية مرقّمة مشروعة، مستهلكًا عرض النطاق وحمل قاعدة البيانات ومُخرجًا البيانات التي تبيع الوصول إليها. حدود المعدّل (بالإضافة إلى التفويض) هي ما يجعل الاستخراج بالجملة مكلفًا بما يكفي لردعه.
    • الإجراءات غير المقنّنة تكلّف مالًا حقيقيًا. نقاط النهاية التي تُشغّل SMS أو بريدًا أو استدعاءات API لطرف ثالث تحوّل حجم الطلبات مباشرة إلى فاتورتك. حلقة إساءة على زر "إعادة إرسال SMS التحقق" هي هجوم مالي، لا مجرد هجوم تقني.
    • البُعد الخاص بالذكاء الاصطناعي يظهر في محدّدات لا تعمل فعلًا. Assistants تولّد محدّدات معدّل تكون in-memory (تُعاد ضبتها عند كل إعادة تشغيل، ومستقلة لكل نسخة خلف load balancer، فالحد الحقيقي هو N× ما ضبطت)، أو مفتاحها فقط على IP (يُتجاوز بالتدوير، أو يحجب مكتبًا كاملًا خلف NAT واحد)، أو مفتاحها على ترويسة قابلة للتزوير من العميل (تُهزم بسهولة). كل واحدة تبدو تحديد معدّل ولا تفرض تقريبًا شيئًا.

    النموذج الذهني¶

    تحديد معدّل الطلبات يعدّ الطلبات ضد مفتاح خلال نافذة باستخدام خوارزمية، مدعومة بـتخزين مشترك، مطبَّقة في الطبقة الصحيحة — وكل واحد من هذه الخيارات الخمسة له طريقة للخطأ تُعطّل الحماية بصمت.

       THE FIVE CHOICES (get any wrong and the limit is weaker than it looks)

   1. KEY — what do you count against?
        per IP        → bypassed by rotation; blocks whole NATs (offices, mobile carriers)
        per user/account → the right key for authenticated abuse (login: also key on the account tried)
        per API key   → the right key for API clients
        best: LAYERED — e.g. login limited per-IP AND per-account-being-tried
        ✗ never key on a client-controllable header (X-Forwarded-For) unless the edge sets it trusted

   2. ALGORITHM — how do you count?
        fixed window   simple; allows 2x burst at the window edge (100 at 0:59, 100 at 1:00)
        sliding window smooths the edge; a bit more state
        token bucket   allows controlled bursts, refills at a steady rate — good for APIs
        leaky bucket   smooths output to a constant rate

   3. STORAGE — where does the counter live?
        in-memory   ✗ resets on restart; SEPARATE per instance → behind a load balancer the real
                       limit is (instances x your limit). useless for multi-instance.
        shared store (Redis) ✓ one counter all instances share; survives restarts. the correct default.

   4. LAYER — where is it enforced?
        edge / CDN / WAF   absorbs volumetric floods before they reach you (Stage 7 / 11)
        gateway            coarse per-client limits
        application        per-user, per-endpoint, business-aware limits ← this chapter
        (defense in depth: multiple layers)

   5. RESPONSE — how do you reject?
        429 Too Many Requests + Retry-After  → lets good clients back off cooperatively
        progressive: slow down, then challenge (CAPTCHA), then block — not instant hard-block

   THE ABUSE CLASSES THIS DEFENDS
     brute force / credential stuffing (login) · enumeration (login/reset/signup) ·
     scraping (list/search) · resource exhaustion (expensive endpoints) · cost abuse (SMS/email)

    ثلاثة مبادئ ترافق الفصل:

    حدّد ضد المفتاح الصحيح، عادةً أكثر من واحد. المفتاح يقرّر ما يقيده الحد فعلًا. IP وحده خشن جدًّا (يحجب NATs مشتركة) وسائب جدًّا (يُهزم بالتدوير). إساءة المصادقة تُحد لكل حساب؛ تسجيل الدخول يُحد لكل IP و لكل حساب-يُجرَّب؛ عملاء API لكل مفتاح. ضع طبقات من المفاتيح بحيث لا ينفذ أي تجاوز وحيد (تدوير IP، أو ضرب حساب واحد من IPs كثيرة).

    يجب أن يكون العدّاد مشتركًا ودائمًا، وإلا فالحد وهم. العدّاد in-memory يُعاد ضبطه عند إعادة التشغيل وهو لكل نسخة — خلف أي load balancer، يتضاعف الحد الفعلي بعدد نسخك، وأي نشر يمسحه. تحديد المعدّل يتطلب تخزينًا مشتركًا (Redis) بحيث تفرض كل النسخ حدًا واحدًا يصمد أمام إعادة التشغيل.

    تحديد المعدّل أداة واحدة في نظام منع إساءة؛ أفسِد تدريجيًا، لا تحجب فقط. اقرن الحدود باستجابات موحّدة (مضادة لـ enumeration)، احتكاك تصاعدي (بطء → تحدٍّ → حجب)، قفل حساب آمن، وترويسات متعاونة (429 + Retry-After). الهدف جعل الإساءة مكلّفة مع إبقاء الاستخدام الشرع سلسًا — حجب فوري قاسٍ على IP مشترك يعاقب مستخدمين حقيقيين.

    تعريف عملي:

    تحديد معدّل الطلبات يعدّ الطلبات ضد مفتاح مختار جيدًا (غالبًا متعدد الطبقات: لكل IP ولكل حساب) خلال نافذة، باستخدام خوارزمية تفهم سلوك اندفاعها، مدعومة بتخزين مشترك دائم لتظل صامدة عبر النسخ وإعادة التشغيل، مطبَّقة في طبقة التطبيق كجزء من نظام منع إساءة أوسع — احتكاك تصاعدي، استجابات موحّدة مضادة لـ enumeration، قفل آمن، وترويسات 429/Retry-After متعاونة — تجعل إساءة الحجم مكلّفة دون معاقبة المستخدمين الشرعيين.

    مثال إنتاجي¶

    Invoicely لديها بالضبط الأسطح التي تستهدفها إساءة الحجم: نقطة نهاية تسجيل الدخول (credential stuffing)، إعادة تعيين كلمة المرور والتسجيل (enumeration)، نقاط نهاية عرض PDF للفواتير وإصدار التقارير (استنزاف موارد)، قائمة الفواتير المرقّمة (السحب)، وإجراء "إعادة إرسال بريد التحقق" (إساءة تكلفة). يحمي هذا الفصل كلًّا منها، مدعومًا بالـ Redis الذي تشغّله فعلًا (المرحلة 3، الفصل 07)، ويُظهر محدّد المعدّل الافتراضي لدى المساعد الذي لا يحمي تقريبًا شيئًا.

    تسجيل الدخول هو الأولوية. يُحد على مفتاحين في آنٍ واحد: لكل IP المصدر (لإبطاء مهاجم واحد) و لكل حساب يجري استهدافه (لإيقاف هجوم موزّع يدوّر الـ IPs ضد إيميل ضحية واحدة). الحدود صارمة — حفنة محاولات في الدقيقة — وتجاوزها يستثير احتكاكًا تصاعديًا: أولًا تأخير قصير، ثم تحدٍّ CAPTCHA، وبعد ذلك فقط حجب مؤقت، بحيث مستخدم شرعي أخطأ في كتابة كلمة المرور ثلاث مرات لا يُقفل بقسوة. إعادة تعيين كلمة المرور والتسجيل يعيدان استجابات موحّدة سواء كان الإيميل موجودًا أم لا (الفصل 01) ويجري تحديد معدّلهما، فلا يمكن استخدامهما لـ enumeration قاعدة المستخدمين. العدّادات تعيش في Redis، فكل نسخة من التطبيق تشارك حدًّا واحدًا ولا يُعيد النشر ضبط الحماية — نمط الفشل للافتراضي in-memory.

    النقاط المكلّفة (PDF، التقارير) تحصل على حدود token-bucket لكل مستخدم بحجم الاستخدام الحقيقي، فيستطيع المستخدم الاندفاع قليلًا لكن لا يستطيع حلقة العرض في denial-of-service؛ الحد نفسه يحمي إجراء "إعادة إرسال البريد" الذي يحمل تكلفة. نقاط نهاية القوائم المرقّمة لديها حدود لكل مستخدم تجعل اجتياز كامل مجموعة البيانات بطيئًا بما يكفي لردع السحب مع عدم التأثير على التصفح العادي. كل رفض هو 429 مع ترويسة Retry-After فيرجع العملاء ذوو السلوك الحسن (والواجهة الأمامية لـ Invoicely نفسها) تعاونيًا. والمنظومة كاملة متعددة الطبقات: الـ CDN/WAF (المرحلة 7) يمتصّ الفيضانات الحجمية قبل أن تصل إلى التطبيق، بينما حدود التطبيق هذه تتعامل مع الإساءة الواعية بالأعمال التي لا تفهمها الحافة. في هذا الفصل نبني هذه ونقابل كلًّا منها بالافتراضي لدى المساعد: المحدّد in-memory، المفتاح IP فقط، المفتاح القابل للتزوير عبر الترويسة.

    التنفيذ¶

    المحدّد المدعوم بـ Redis (rate_limit.py): مشترك، دائم، مفاتيح متعدد الطبقات. in-memory هو الفشل؛ المخزن المشترك هو المقصود كله.

    async def check_rate_limit(key: str, limit: int, window_seconds: int) -> RateLimitResult:
    # one counter in Redis that ALL instances share and that survives restarts
    count = await redis.incr(key)
    if count == 1:
        await redis.expire(key, window_seconds)
    if count > limit:
        ttl = await redis.ttl(key)
        raise RateLimitExceeded(retry_after=ttl)   # -> 429 + Retry-After
    return RateLimitResult(remaining=limit - count)

    # login: LAYERED keys — per source IP AND per account being tried
async def rate_limit_login(ip: str, email: str):
    await check_rate_limit(f"login:ip:{ip}", limit=10, window_seconds=60)        # slows one attacker
    await check_rate_limit(f"login:acct:{email}", limit=5, window_seconds=300)   # catches IP rotation

    المفتاحان يُغلقان فجوة كلٍّ منهما: مفتاح IP يُبطئ آلة واحدة، مفتاح الحساب يوقف هجومًا موزّعًا يدوّر IPs ضد إيميل ضحية واحدة. لا يكفي أيٌّ منهما وحده؛ معًا يغطّيان كلا التجاوزين.

    حل موثوق لـ IP العميل (client_ip.py): المفتاح بجودة IP.

    def client_ip(request) -> str:
    # trust X-Forwarded-For ONLY as set by our own edge/proxy — never the raw client value
    xff = request.headers.get("X-Forwarded-For")
    if TRUSTED_PROXY and xff:
        return xff.split(",")[0].strip()   # the edge appends the real client; take the left-most it set
    return request.client.host            # otherwise the direct socket peer — unspoofable

    إن مفاتيح تحديد المعدّل على ترويسة يتحكم بها العميل، فالعميل يضبط قيمة عشوائية لكل طلب والحد لا ينطلق أبدًا. الـ IP موثوق فقط عندما تضبطه بنيتك التحتية؛ هذه الوحدة ترمّز حد الثقة هذا.

    احتكاك تصاعدي وقفل آمن (abuse.py): أفسِد تدريجيًا، لا تحجب بقسوة.

    async def on_failed_login(ip: str, email: str):
    failures = await bump_failure_counter(email)
    if failures >= 10:
        raise AccountChallenge("captcha")     # challenge, not permanent lock
    elif failures >= 5:
        await asyncio.sleep(backoff(failures))  # progressive delay slows automation, barely felt by humans
    # NOTE: lockout is TIME-BASED (auto-expires), never permanent — a permanent lock is a
    #       denial-of-service an attacker can trigger against any known email.

    قرارات هندسية¶

    خمسة قرارات تُعرّف منظومة منع إساءة.

    ما المفتاح (المفاتيح) الذي تحدّ ضده؟¶

    الخيارات: (1) لكل IP؛ (2) لكل مستخدم/حساب مصادق عليه؛ (3) لكل API key؛ (4) توليفات متعددة الطبقات؛ (5) معرّف يقدّمه العميل.

    المقايضات: IP متاح قبل المصادقة (ضروري لتسجيل الدخول) لكنه خشن جدًّا (NAT مؤسسي أو شركة اتصالات mobile يشارك IP واحدًا بين الآلاف) وسائب جدًّا (المهاجمون يدوّرون IPs رخيصًا). لكل مستخدم دقيق للإساءة المصادق عليها لكن عديم النفع قبل تسجيل الدخول. لكل API key يناسب العملاء البرمجيين. معرّف يقدّمه العميل لا قيمة له — المهاجم فقط يغيّره. التعدد الطبقي (IP و حساب لتسجيل الدخول) يغطّي الفجوات التي لكلٍّ منها وحده.

    التوصية: اختر المفتاح بحسب ما تدافع عنه واطبُقه على طبقات. تسجيل الدخول: لكل IP و لكل حساب-يُجرَّب (يلتقط الهجمات أحادية المصدر والموزّعة-التدوير على حدٍّ سواء). نقاط النهاية المصادق عليها: لكل مستخدم (بالإضافة إلى لكل IP كحاجز خلفي خشن). APIs: لكل مفتاح. لا تحد أبدًا على قيمة يمكن للعميل التحكم بها. المفتاح المتعدد الطبقات هو ما يجعل التجاوز صعبًا.

    أيّ خوارزمية؟¶

    الخيارات: (1) fixed window؛ (2) sliding window؛ (3) token bucket؛ (4) leaky bucket.

    المقايضات: fixed window الأبسط (عدّاد بـ TTL) لكنه يسمح باندفاع 2× عبر حدود النافذة — 100 طلب عند 0:59 و100 عند 1:00. sliding window يملّس ذلك على حساب قليل من الحالة الإضافية. token bucket يسمح باندفاع محكوم ثم إعادة تعبئة ثابتة — طبيعي لـ APIs حيث الاندفاع العرضي مقبول. leaky bucket يفرض معدّل إخراج ثابتًا.

    التوصية: fixed أو sliding window لحدود وقائية بسيطة (تسجيل الدخول، إعادة التعيين) حيث تحتاج فقط سقفًا؛ token bucket لحدود API والنقاط المكلّفة حيث الاندفاع الصغير مشروع لكن يجب تحديد المعدّل المستدام. طابق سلوك اندفاع الخوارزمية مع نقطة النهاية: سقف صارم للمصادقة، قابل للانفجار لـ APIs المواجهة للمستخدم. لا تُفرط في الهندسة — عدّاد بنوافذ في Redis يغطّي معظم الاحتياجات.

    أين يعيش العدّاد؟¶

    الخيارات: (1) in-memory (لكل عملية)؛ (2) مخزن مشترك (Redis)؛ (3) عند الحافة/البوابة؛ (4) قاعدة البيانات.

    المقايضات: in-memory تافه ومكسور لأي نشر متعدد النسخ — كل نسخة تعدّ على حدة، فالحد الحقيقي هو عدد النسخ × حدك، وكل إعادة تشغيل/نشر تعيد ضبطه. Redis يعطي عدّادًا واحدًا مشتركًا وسريعًا ودائمًا عبر كل النسخ — الجواب القياسي. تحديد المعدّل عند الحافة/البوابة خشن لكن يمتصّ الحجم مبكرًا. قاعدة البيانات تعمل لكنها تضيف حملًا لما تحميه.

    التوصية: مخزن مشترك (Redis) لحدود معدّل التطبيق — الخيار الوحيد الذي يفرض حدًّا حقيقيًا واحدًا عبر النسخ ويصمد أمام إعادة التشغيل. احفظ in-memory فقط لخدمة أحادية النسخة بحق (نادرة في الإنتاج). أضف حدود الحافة/البوابة كطبقة خارجية خشنة. المحدّد in-memory هو الطريقة الأكثر شيوعًا لأن يتبين أن حد المعدّل وهم.

    أين في الـ stack تُعالَج الإساءة؟¶

    الخيارات: (1) التطبيق فقط؛ (2) الحافة/CDN/WAF فقط؛ (3) متعدد الطبقات (حافة + بوابة + تطبيق).

    المقايضات: حدود التطبيق فقط واعية بالأعمال (لكل مستخدم، لكل نقطة نهاية) لكن تجلس وراء كل شيء، فالفيضانات الحجمية لا تزال تستهلك عرض النطاق وسعة الاتصال للوصول إليها. حدود الحافة فقط تمتصّ الفيضانات مبكرًا لكن لا تستطيع فهم "هذا المستخدم يسحب فواتير" — تفتقر إلى السياق التجاري. التعدد الطبقي يأخذ الاثنين: الحافة تُسقط الحجم، التطبيق يفرض حدودًا دلالية.

    التوصية: اطبُق طبقات. الفيضانات الحجمية وعلى مستوى الشبكة تُسقط عند الحافة/CDN/WAF (المرحلة 7، المرحلة 11)؛ الإساءة الواعية بالأعمال (credential stuffing، السحب، حلقات النقاط المكلّفة) تُعالَج في التطبيق حيث يعيش السياق. هذا الفصل يبني طبقة التطبيق؛ إنها حلقة واحدة من دفاع يبدأ عند حافة الشبكة.

    كيف تستجيب للإساءة — تحجب، أم تُفسد؟¶

    الخيارات: (1) حجب قاسٍ (رفض، أو قفل دائم)؛ (2) احتكاك تصاعدي (بطء → تحدٍّ → حجب مؤقت)؛ (3) 429 + Retry-After للتعاون؛ (4) حدود ظل/ناعمة.

    المقايضات: الحجب الفوري القاسي بسيط لكنه يعاقب الإيجابيات الكاذبة بقسوة — NAT مشترك أو مستخدم يمر بيوم سيّئ يُرفض، والقفل الدائم للحساب هو بحدّ ذاته denial-of-service يمكن للمهاجم استثارته بالاسم. الاحتكاك التصاعدي يُبطئ الأتمتة بينما بالكاد يؤثر على البشر. 429/Retry-After يتيح للعملاء ذوي السلوك الحسن (بما في ذلك واجهتك الأمامية) الرجوع بشكل صحيح. الحدود الناعمة (سجّل دون حجب) تساعد في ضبط العتبات قبل الفرض.

    التوصية: الاحتكاك التصاعدي كوقف افتراضي — تأخير، ثم تحدٍّ (CAPTCHA)، ثم حجب محدود بوقت — مع 429 + Retry-After على كل رفض لِيُتعاون العملاء ذوو السلوك الحسن. لا تستخدم أبدًا قفل حساب دائم (DoS قابل للتسليح)؛ اجعل الأقفال تنتهي تلقائيًا. فكّر في مرحلة ظل/إبلاغ-فقط لمعايرة العتبات ضد الحركة الحقيقية قبل الفرض، حتى لا تحجب مستخدمين شرعيين في اليوم الأول.

    المقايضات¶

    تحديد معدّل الطلبات يتبادل سقفًا صارمًا على المستخدمين الشرعيين الثقيلين مقابل الحماية من الإساءة. أي حد صارم بما يكفي لإيقاف الإساءة سيقيّد أحيانًا مستخدمًا ثقيلًا حقيقيًا أو IP مشترك. التخفيف هو مفاتيح متعددة الطبقات (بحيث يكون الحد دقيقًا)، حدود سخية على الإجراءات العادية مع حدود صارمة فقط حيث تتركّز الإساءة (تسجيل الدخول، النقاط المكلّفة)، واحتكاك تصاعدي بدل الحجب القاسي. التكلفة حقيقية لكن يمكن إدارتها؛ لا حد ليس خيارًا.

    العدّ في مخزن مشترك يتبادل اعتماد Redis مقابل حد يعمل فعلًا. التحديد المدعوم بـ Redis يضيف اعتمادًا في مسار الطلب وتكلفة زمن انتقال صغيرة لكل فحص. ما يشتريه هو الفرق بين حد حقيقي ووهمي (in-memory يعطي عدد النسخ × الحد). لأي تطبيق متعدد النسخ، هذا الاعتماد غير قابل للتفاوض — ومن المرجح أنك تشغّل Redis فعلًا (المرحلة 3، الفصل 07).

    الاحتكاك التصاعدي يتبادل تعقيد التنفيذ مقابل عدم معاقبة المستخدمين الحقيقيين. تصعيد التأخير → CAPTCHA → قفل مؤقت أكثر شيفرة من حجب مسطّح، والـ CAPTCHAs تضيف احتكاكًا للمستخدم. العائد هو إيقاف الإساءة المؤتمتة بينما كلمة مرور مكتوبة خطأ من إنسان تكلّفه لا شيء تقريبًا — أفضل بكثير من حمل الدعم والتسرب الناتج عن حجب قاسٍ للمستخدمين الشرعيين. احفظ الاحتكاك لأينما تتركّز الإساءة فعلًا.

    العتبات الأشد تتبادل إيجابيات كاذبة أكثر مقابل التقاط إساءة أكثر. الحدود المنخفضة تلتقط مهاجمين أكثر وتحجب أبرياء أكثر؛ الحدود المرتفعة تفعل العكس. لا يوجد رقم صحيح عالميًا — يعتمد على نقطة النهاية وحركتك. السبيل عبر هو القياس: شغّل الحدود في وضع إبلاغ-فقط، راقب الحركة الحقيقية، واضبط العتبات بالبيانات بدل التخمين، ثم راقب معدّل 429.

    الأخطاء الشائعة¶

    محدّد in-memory خلف load balancer. كل نسخة تعدّ على حدة وإعادة التشغيل تعيد ضبطه، فالحد الفعلي هو عدد النسخ × رقمك والنشر يمسحه. الإصلاح: مخزن مشترك (Redis) بحيث تفرض كل النسخ حدًّا واحدًا دائمًا.

    المفتاح فقط على IP. يُتجاوز بتدوير IP ويحجب NATs مشتركة بالكامل (مكاتب، شركات اتصالات). الإصلاح: اطبُق طبقات من المفاتيح — لكل حساب للإساءة المصادق عليها، لكل IP و لكل حساب لتسجيل الدخول.

    المفتاح على ترويسة قابلة للتزوير. تحديد معدّل على X-Forwarded-For/X-Real-IP الخام يتحكم بها العميل، فيغيّرها المهاجم ولا ينطلق الحد أبدًا. الإصلاح: حلّ IP العميل فقط من ترويسة تضبطها حافّتك، أو استخدم نظير المقبس.

    لا حد على النقاط المكلّفة أو ذات التكلفة. عروض PDF، تقارير، عمليات بحث، ومحفّزات SMS/بريد تُترك بلا حد، فالحلقة denial-of-service أو فاتورة. الإصلاح: حدود لكل مستخدم بحجم الاستخدام الحقيقي على كل إجراء مكلّف.

    قفل حساب دائم. قفل حساب إلى الأبد بعد N فشل، يسلّحه المهاجم لقفل أي مستخدم بإيميله. الإصلاح: قفل محدود بوقت، ينتهي تلقائيًا، مع احتكاك تصاعدي.

    نقاط نهاية مصادقة قابلة للـ enumeration. تسجيل الدخول/إعادة التعيين/التسجيل يستجيب بشكل مختلف للإيميلات المعروفة مقابل المجهولة، مسرّبًا قائمة المستخدمين. الإصلاح: استجابات موحّدة (الفصل 01) بالإضافة إلى تحديد معدّل على تلك النقاط.

    أخطاء الذكاء الاصطناعي¶

    تحديد معدّل الطلبات مجال يبدو فيه خرج المساعد كحماية ولا يفرض تقريبًا شيئًا، لأن المحدّد يعمل ويُعيد 429 في اختبار أحادي النسخة ويفشل في كل ظرف إنتاجي. راجع تخزين المحدّد، مفتاحه، وثقة IP — لا ما إذا كان يُعيد 429 محليًا.

    Claude Code: المحدّد in-memory الذي لا يصمد في الإنتاج¶

    عند طلب إضافة تحديد معدّل، Claude Code كثيرًا ما يُنتج محدّد in-memory — dict على مستوى الوحدة أو عدّاد يتتبع الطلبات لكل مفتاح في ذاكرة العملية. يعمل بشكل مثالي في الاختبار المحلي وفي عرض أحادي النسخة، مُعيدًا 429 في حينه. في الإنتاج يفشل بطريقتين في آنٍ واحد: خلف load balancer لكل نسخة عدّادها، فالحد الحقيقي هو (عدد النسخ × الحد المضبوط)، وكل إعادة تشغيل أو نشر يمسح العدّادات بالكامل. تبدو الحماية موجودة في المراجعة وتغيب إلى حدٍّ بعيد في الإنتاج.

    اكتشف: حالة تحديد معدّل في dict على مستوى الوحدة، defaultdict، lru_cache، أو بنية داخل العملية؛ لا Redis/مخزن مشترك في المحدّد؛ حدود تُعاد ضبتها عند إعادة تشغيل التطبيق؛ محدّد لا تصمد أرقامه عند تشغيل أكثر من عامل/نسخة.

    أصلح: ادعمه بمخزن مشترك دائم:

    يجب أن يستخدم تحديد معدّل الطلبات مخزنًا مشتركًا (Redis) بحيث تفرض كل النسخ حدًّا واحدًا ويصمد أمام إعادة التشغيل — أبدًا dict in-memory، الذي خلف load balancer يسمح بـ (عدد النسخ × الحد) ويُعاد ضبطه عند النشر. أضف اختبارًا أن الحد لا يزال صامدًا حين تتشارك نسختان المخزن.

    GPT: المفتاح فقط على IP (قابل للتجاوز ويحجب زائدًا في آنٍ واحد)¶

    نماذج عائلة GPT تفتقر افتراضيًا إلى مفتاح IP لكل شيء، بما في ذلك تسجيل الدخول. يفشل في الاتجاهين: مهاجم يدوّر عبر بروكسيات رخيصة أو botnet يتجاوز بسهولة حد تسجيل الدخول لكل IP (كل IP يحصل على كامل الحصة)، بينما IP لـ NAT مؤسسي مشترك أو شركة mobile يعني العديد من المستخدمين الشرعيين يتشاركون دلوًا واحدًا ويُحجبون بنشاط بعضهم البعض. بالنسبة لـ credential stuffing — الموزّع بطبيعته — حد IP فقط بالكاد يساعد، ويحجب تبعيًا مستخدمين حقيقيين خلف IPs مشتركة.

    اكتشف: مفتاح تحديد المعدّل الوحيد هو IP؛ تسجيل الدخول محدود لكل IP دون مفتاح لكل حساب؛ تقارير دعم عن مكاتب كاملة محجوبة؛ محاولات credential stuffing تنجح رغم "حد IP" لأن كل IP يبقى تحته.

    أصلح: اطبُق طبقات من المفاتيح، خصوصًا لكل حساب لتسجيل الدخول:

    لا تجعل تحديد معدّل الطلبات على IP وحده. لتسجيل الدخول، حدّد لكل IP و لكل حساب-يُجرَّب بحيث يُلتقط هجوم موزّع يدوّر IPs ضد إيميل واحد، ولا يُحجب المستخدمون العاديون خلف NAT مشترك من قِبل بعضهم. لنقاط النهاية المصادق عليها، امفتاحح لكل مستخدم. أضف اختبارًا أن تدوير IP ضد حساب واحد لا يزال يُطلق الحد.

    Cursor: الثقة في ترويسة يتحكم بها العميل كمفتاح تحديد معدّل¶

    عند إكمال محدّد يحتاج IP العميل، Cursor غالبًا يقرأه مباشرة من X-Forwarded-For أو X-Real-IP ويمفتاحح عليه الحد — متبعًا النمط الشائع دون التحقق ما إذا كانت هذه الترويسة موثوقة. لأن هذه الترويسات يضبطها العميل ما لم تُعد كتابتها حافة، يرسل المهاجم ببساطة X-Forwarded-For مختلفًا في كل طلب وكل واحد يحصل على دلو جديد. يعمل المحدّد، يُعيد 429 في اختبار لا يزوّر الترويسة، ويُتجاوز بسطر واحد من أيّ شخص يفعل.

    اكتشف: مفتاح تحديد المعدّل مستمد من X-Forwarded-For/X-Real-IP الخام دون فحص بروكسي موثوق؛ IP العميل مأخوذ من ترويسة طلب بدلًا من نظير المقبس أو قيمة تضبطها الحافة؛ الحد لا ينطلق حين يغيّر نفس العميل الترويسة المُعادة.

    أصلح: حلّ IP فقط من مصدر موثوق:

    ثق بـ X-Forwarded-For/X-Real-IP فقط إذا ضبطتها حافتنا/بروكسينا الخاص؛ وإلا استخدم عنوان نظير المقبس. لا تجعل أبدًا مفتاح حد معدّل على ترويسة يمكن للعميل ضبطها بحرية. أضف اختبارًا أن تغيير الترويسة المُعادة لا يُنشئ دلاء تحديد معدّل جديدة.

    أفضل الممارسات¶

    حدّد ضد مفاتيح متعددة الطبقات. لكل حساب للإساءة المصادق عليها؛ لكل IP و لكل حساب لتسجيل الدخول (لالتقاط الهجمات أحادية المصدر والموزّعة على حدٍّ سواء)؛ لكل مفتاح لـ APIs. أبدًا قيمة يمكن للعميل التحكم بها.

    ادعم العدّاد بمخزن مشترك دائم. Redis (أو ما يعادله) بحيث تفرض كل النسخ حدًّا واحدًا يصمد أمام إعادة التشغيل — أبدًا in-memory لتطبيق متعدد النسخ.

    حلّ IP العميل فقط من مصدر موثوق. نظير المقبس، أو X-Forwarded-For فقط كما تضبطها حافّتك — أبدًا الترويسة الخام من العميل.

    حدّد كل نقطة نهاية مكلّفة وذات تكلفة. حدود لكل مستخدم بحجم الاستخدام الحقيقي على عروض PDF/تقرير/بحث وعلى إجراءات SMS/بريد/محفّز طرف ثالث — أسطح denial-of-service والتكلفة.

    أفسِد تدريجيًا؛ لا تحجب بقسوة أو تقفل دائمًا. تأخير → CAPTCHA → حجب محدود بوقت؛ 429 + Retry-After على كل رفض؛ أقفال تنتهي تلقائيًا (القفل الدائم DoS قابل للتسليح).

    اقرن الحدود باستجابات موحّدة مضادة لـ enumeration. تسجيل الدخول/إعادة التعيين/التسجيل يستجيب بنفس الشكل للحسابات المعروفة والمجهولة (الفصل 01) ويُحدّد معدّلها — الاثنان معًا يُغلقان enumeration.

    اطبُق طبقات مع الحافة. الفيضانات الحجمية تُسقط عند CDN/WAF (المرحلة 7/11)؛ الإساءة الواعية بالأعمال تُعالَج في التطبيق. لا طبقة وحدها كافية.

    عاير بالبيانات وراقب. شغّل حدودًا جديدة في وضع إبلاغ-فقط لضبط العتبات ضد الحركة الحقيقية، ثم افرض؛ نبّه على معدّلات 429 وعلى أنماط الإساءة (ارتفاع فشل تسجيل الدخول).

    أنماط مضادة¶

    المحدّد In-Memory. حالة المعدّل في ذاكرة العملية خلف load balancer. العلامة: عدّاد/dict على مستوى الوحدة؛ لا مخزن مشترك؛ حدود تُعاد ضبتها عند النشر وتُضرب بعدد النسخ.

    مفتاح IP فقط. كل حد مفتاحه فقط على IP. العلامة: لا مفتاح لكل حساب على تسجيل الدخول؛ credential stuffing ينجح تحت "حد IP"؛ مكاتب كاملة محجوبة.

    مفتاح قابل للتزوير. حد مفتاحه على ترويسة عميل خام. العلامة: X-Forwarded-For يُقرأ دون فحص بروكسي موثوق؛ الحد لا ينطلق حين تتغير الترويسة.

    نقطة نهاية مكلّفة غير مقنّنة. إجراءات مكلّفة أو قابلة للفوترة بلا حد. العلامة: نقاط نهاية PDF/تقرير/بحث أو SMS/بريد بلا فحص معدّل؛ حلقة ترفع CPU أو الفاتورة.

    القفل الدائم. حسابات تُقفل إلى الأبد بعد N فشل. العلامة: لا انتهاء صلاحية على القفل؛ مهاجم قادر على قفل أي مستخدم بتقديم كلمات مرور خاطئة لإيميله.

    عرّاف enumeration. استجابات مصادقة محدّدة المعدّل لكن قابلة للتمييز. العلامة: حالة/ رسالة/توقيت مختلف للإيميلات المعروفة مقابل المجهولة؛ يُعامل التحديد على أنه كل دفاع مضاد لـ enumeration.

    شجرة القرار¶

    "أحتاج حماية نقطة نهاية من إساءة الحجم — كيف؟"

    What's being abused?
├── LOGIN (brute force / credential stuffing)
│     └─► layered keys: per-IP AND per-account-being-tried, in Redis. strict limits.
│         progressive friction: delay -> CAPTCHA -> TIME-LIMITED lock (never permanent).
│         uniform responses so it's not also an enumeration oracle (Ch 01).
├── PASSWORD RESET / SIGNUP (enumeration)
│     └─► rate-limit + UNIFORM responses (same reply whether the email exists or not).
├── EXPENSIVE endpoint (PDF, report, search) — resource exhaustion
│     └─► per-user token-bucket limit sized to real usage. burstable, bounded.
├── COST-bearing action (SMS, email, 3rd-party call)
│     └─► strict per-user limit — this is a financial control, not just technical.
└── LIST / SEARCH (scraping)
      └─► per-user limit that makes full-dataset walking slow; + authorization (Stage 3 Ch 04).

For ANY of them:
   KEY      → the right key(s), layered; never a client-controllable value.
   STORAGE  → shared + durable (Redis). never in-memory for multi-instance.
   IP       → from the socket peer or an edge-set header only — never raw XFF.
   RESPONSE → 429 + Retry-After; degrade progressively, don't hard-block.
   LAYER    → edge/WAF sheds floods (Stage 7/11); app enforces business-aware limits (here).

    قائمة مراجعة التنفيذ¶

    • [ ] عدّادات تحديد معدّل مدعومة بمخزن مشترك دائم (Redis)، أبدًا in-memory، بحيث تصمد الحدود عبر النسخ وإعادة التشغيل.
    • [ ] المفاتيح مختارة لكل تهديد ومطبّقة على طبقات (تسجيل الدخول: لكل IP و لكل حساب؛ المصادق عليها: لكل مستخدم؛ APIs: لكل مفتاح)؛ لا مفتاح يمكن للعميل التحكم به.
    • [ ] IP العميل مُحَلَّل من نظير المقبس أو ترويسة تضبطها الحافة فقط، أبدًا X-Forwarded-For الخام.
    • [ ] كل نقطة نهاية مكلّفة وذات تكلفة (عروض، تقارير، بحث، SMS/بريد) لها حد لكل مستخدم بحجم الاستخدام الحقيقي.
    • [ ] الرفض يُعيد 429 مع Retry-After؛ الإساءة تُقابل باحتكاك تصاعدي وقفل محدود بوقت (أبدًا دائم).
    • [ ] نقاط نهاية المصادقة تجمع بين تحديد المعدّل والاستجابات الموحّدة حتى لا يمكن استخدامها لـ enumeration.

    قائمة مراجعة البنية¶

    • [ ] تحديد معدّل مركزي (تنفيذ واحد) ومطبّق باتّساق عبر نقاط النهاية، لا يُخترع من جديد لكل route.
    • [ ] سياسة الإساءة (احتكاك، قفل، عتبات) مفصولة عن آلية العدّ.
    • [ ] حدود التطبيق مطبّقة على طبقات مع حماية الحافة/WAF (المرحلة 7/11)، كلٌّ يتعامل مع ما يناسبه.
    • [ ] منظومة منع الإساءة تستخدم الـ Redis الذي يشغّله التطبيق فعلًا (المرحلة 3، الفصل 07) بدلًا من إضافة بنية تحتية جديدة بلا داعٍ.
    • [ ] العتبات مضبوطة من حركة مقاسة (معايرة إبلاغ-فقط) بدلًا من التخمين.

    قائمة مراجعة مراجعة الشيفرة¶

    • [ ] لا توجد حالة تحديد معدّل تعيش في ذاكرة العملية لخدمة متعددة النسخ.
    • [ ] لا حد مفتاحه على IP وحده حيث يلزم مفتاح لكل حساب، ولا حد مفتاحه على ترويسة عميل خام.
    • [ ] لا نقطة نهاية مكلّفة أو ذات تكلفة تُدمج دون حد معدّل.
    • [ ] لا يُدخل قفل حساب دائم؛ الأقفال تنتهي تلقائيًا.
    • [ ] نقاط نهاية المصادقة الجديدة تجمع بين تحديد المعدّل والاستجابات الموحّدة؛ الرفوض تحمل 429/Retry-After.

    قائمة مراجعة النشر¶

    • [ ] الـ Redis (أو المخزن المشترك) الداعم لحدود المعدّل مراقب ومُحجَّم لحجم الطلب؛ نمط فشله مقرّر (fail-open مقابل fail-closed) بقصد.
    • [ ] حماية الحجم عند الحافة/CDN/WAF في مكانها قبل حدود التطبيق (المرحلة 7/11).
    • [ ] معدّلات 429 وارتفاع فشل تسجيل الدخول مُنبَّه عليها، بحيث تكون كل من الإساءة والحدود المضبوطة بشكل خاطئ مرئية.
    • [ ] إعداد البروكسي الموثوق صحيح في كل بيئة بحيث لا يكون حلّ IP العميل قابلاً للتزوير في الإنتاج.

    التمارين¶

    1. أثبت أن محدّدك حقيقي عبر النسخ. خذ محدّد معدّل in-memory وشغّل نسختين خلف load balancer (أو عاملين). بيّن أن الحد الفعلي يتضاعف وأن إعادة التشغيل تعيد ضبطه. ثم انقل العدّاد إلى Redis وبيّن حدًّا مشتركًا واحدًا يصمد أمام إعادة التشغيل. المُسلَّمة هما التشغيلان وإصلاح المخزن المشترك.

    2. اهزم، ثم دافع، ضد credential stuffing موزّع. هاجم حد تسجيل دخول لكل IP فقط بتدوير IPs ضد حساب واحد وبيّن أن المحاولات تنجح. أضف مفتاحًا لكل حساب وبيّن أن الهجوم الموزّع نفسه محجوب الآن، بينما سيناريو NAT مشترك (مستخدمون كثيرون، IP واحد) لا يُحجب تبعيًا. المُسلَّمة هما الهجومان وإصلاح المفتاح المتعدد الطبقات.

    3. تجاوز محدّد ترويسة قابل للتزوير. ابنِ محدّدًا مفتاحه على X-Forwarded-For الخام، ثم تجاوزه بتغيير الترويسة لكل طلب. أصلحه بحلّ IP بروكسي موثوق وبيّن أن التزوير لم يعد يُنشئ دلاء جديدة. المُسلَّمة التجاوز وإصلاح IP الموثوق.

    4. أضف احتكاكًا تصاعديًا آمنًا لتسجيل الدخول. نفّذ تأخير → CAPTCHA → قفل محدود بوقت على فشل تسجيل الدخول المتكرر. بيّن أن الإساءة المؤتمتة تُبطّأ بشدة بينما إنسان يخطئ في كتابة كلمة المرور ثلاث مرات بالكاد يتأثر، وأنه لا يمكن إثارة قفل دائم ضد إيميل معروف. المُسلَّمة منطق التصعيد والسيناريوهان (بوت مقابل إنسان أخطأ في الكتابة).

    قراءة إضافية¶

    • OWASP — "Blocking Brute Force Attacks" ودليل المصادقة (cheatsheetseries.owasp.org) — إرشاد عملي حول إساءة تسجيل الدخول، التأخيرات التصاعدية، القفل الآمن، ولماذا القفل الدائم بحدّ ذاته مخاطرة.
    • OWASP Automated Threats to Web Applications (owasp.org/www-project-automated-threats) — تصنيف إساءة الحجم (credential stuffing، السحب، enumeration، carding) التي يدافع عنها هذا الفصل، مع أنماط الكشف والتخفيف.
    • كتابات هندسية "كيف بنينا تحديد المعدّل" (مدونات Stripe، Cloudflare، GitHub) — معالجات إنتاجية حقيقية لـ token buckets، عدّادات مدعومة بـ Redis، وتحديد متعدد الطبقات عند الحافة+التطبيق بمقياس واسع.
    • توثيق Redis — أنماط INCR/EXPIRE وتحديد المعدّل (redis.io) — الأوليات الدقيقة وراء محدّد العدّاد المشترك، بما في ذلك أنماط النوافذ وtoken bucket.
    • المرحلة 3، الفصل 03 (المصادقة) والفصل 01 (استجابات موحّدة مضادة لـ enumeration) — نقاط نهاية تسجيل الدخول التي يحميها هذا الفصل واتّساق الاستجابة الذي يقترن بتحديد المعدّل؛ دفاع DoS على مستوى الشبكة هو المرحلة 7 والمرحلة 11.
