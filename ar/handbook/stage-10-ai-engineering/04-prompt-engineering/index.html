    هندسة الـ Prompts¶

    المقدمة¶

    صمّم الفصل 03 ما يعرفه الـ agent؛ هذا الفصل يصمم ما تطلبه منه. بالنسبة لـ coding agents، فإن لهندسة الـ prompt علاقة ضئيلة جداً بالتراث الشفهي الذي يوحي به المصطلح — عبارات سحرية، شخصيات، تعاويذ "تفتح" قدرات النموذج. الـ prompt الموجّه إلى coding agent هو مواصفة مهمة (task specification)، وهندسة الـ prompt هي كتابة المواصفات: تحديد الهدف، والقيود، ومعايير القبول، وحدود النطاق بدقة كافية بحيث ينتج أي منفّذ كفء لا يستطيع قراءة أفكارك ما تحتاجه فعلاً.

    هذه مهارة أقدم من الذكاء الاصطناعي. إنها مهارة كتابة تذكرة جيدة، أو brief تصميمي جيد، أو عقد جيد لمقاول من الباطن — ومعظم المهندسين سيئون فيها لنفس السبب الذي كانوا دائماً سيئين لأجله: الشخص الذي يكتب المهمة يحمل سياقاً لا تحمله الكلمات، وزميل بشري يسد الفجوة بطرح أسئلة في الممر. الـ agent يسد الفجوة بـافتراضات، تُسلَّم بسرعة عالية وبثقة كاملة. لم يخلق الذكاء الاصطناعي مشكلة الغموض؛ بل أزال الممر الذي كان يخفيها.

    يغطي هذا الفصل تشريح الـ prompt الفعّال للمهمة، تدرج المواصفات (كم تستحق المهمة من المواصفات، من تعليم في سطر واحد إلى brief كامل)، انضباط التكرار (عندما يكون المخرج خاطئاً، أصلح المواصفة وليس الكود فقط)، والـ prompts كأصول فريق ذات إصدارات — مكتبة الـ prompts التي قدّمها الفصلان 01 و 02 كملفات أوامر، ويملؤها هذا الفصل بالمحتوى.

    لماذا هذا مهم¶

    • جودة المواصفة هي سقف التفويض. قاعدة الفصل 02 — كلما ابتعد الـ agent عن لوحة المفاتيح الخاصة بك، كلما تحملت المواصفة المكتوبة العبء الأكبر — تجعل مهارة الـ prompt هي البوابة لكل سير عمل عالي الأثر: التفويض عبر الـ CLI، والـ cloud agents، والـ agents المتوازية (الفصل 05)، وأتمتة الـ CI (الفصل 08). الفرق التي لا تستطيع المواصفة لا يمكنها سوى المراقبة.
    • الغموض يتضخّم، لا يُمتص. تعليمة غامضة لمهندس مبتدئ تنتج سؤالاً توضيحياً؛ أما للـ agent فتنتج تطبيقاً كاملاً ومصقولاً وواثقاً وخاطئاً لتفسير اعتباطي واحد — بالإضافة إلى تكلفة المراجعة لاكتشاف أي تفسير. "حسّن معالجة الأخطاء" جملة واحدة؛ الـ diff الذي يتكون من 40 ملفاً الذي تطلقه ليس كذلك.
    • الـ prompt هو أرخص مكان لإصلاح الجودة. نفس العيب يكلف جملة في الـ prompt، ودقائق في المراجعة، وساعات بعد الـ merge، وعميلاً بعد النشر. المهندسون الذين يلاحظون أنهم يكررون نفس تعليق المراجعة مراراً ويقومون بنقله إلى الـ prompt (أو ملف التعليمات) يتراكم أثرهم؛ أما المهندسون الذين يعيدون كتابة التصحيحات فلا يتراكم أثرهم.
    • الـ prompts هي أصول فريق لها نموذج صيانة. الـ prompt الذي ينتج بشكل موثوق migration جيدة، أو endpoint، أو مرور مراجعة، هو معرفة هندسية — قابل للإصدارات، والمراجعة، والتحسين، والسرقة من قبل كل عضو في الفريق، تماماً مثل القوالب وقوائم التحقق التي يوفرها هذا الـ handbook. إذا تُرك في سجل المحادثة، فإن هذه المعرفة تتبخر لكل شخص، كل أسبوع (مقبرة الفصل 03).
    • معايير القبول هي ما يجعل مخرج الـ agent قابلاً للتحقق. "يُعتبر منتهياً عندما ينجح make check وهذه السلوكيات الأربعة لها اختبارات" يحوّل المراجعة من انطباعات إلى تحقق — ويسمح للـ agent بالتحقق من نفسه داخل الحلقة قبل أن ترى الـ diff. الـ prompt الذي بلا خط نهاية يكلّف النموذج تعريف "تم".

    النموذج الذهني¶

    الـ prompt الخاص بالمهمة هو عقد من خمسة بنود، وكمية الطقوس تتناسب مع بُعد المهمة ومخاطرها.

             ANATOMY OF A TASK PROMPT (the contract)
   ┌────────────────────────────────────────────────────┐
   │ 1 GOAL        what outcome, for whom, why now      │
   │               (the why lets the agent resolve      │
   │                ambiguity the way you would)        │
   │ 2 CONTEXT     pointers, not payloads: which files, │
   │               which docs, which exemplar to follow │
   │               (Ch 03 built the addresses)          │
   │ 3 CONSTRAINTS what must stay true: invariants,     │
   │               APIs that can't change, deps policy  │
   │ 4 ACCEPTANCE  the finish line, checkable by the    │
   │   CRITERIA    agent itself: tests, commands,       │
   │               behaviors — numbered                 │
   │ 5 SCOPE       what NOT to do: files off-limits,    │
   │   LIMITS      no drive-by refactors, diff budget,  │
   │               "list further ideas, don't do them"  │
   └────────────────────────────────────────────────────┘
   + one standing clause: "State your assumptions. If a
     requirement is ambiguous, ask before implementing."

         THE SPECIFICATION GRADIENT
   conversation ◄──────────────────────────────► full brief
   "rename this   "add a 422 case    "ticket + 5-clause
    helper"        to this endpoint's  spec + plan-first"
                   tests, mirror the
                   existing ones"
   ── use LESS spec when: task is small, reversible, you're
      watching, the repo carries the conventions (Ch 01–03)
   ── use MORE spec when: agent is distant (CLI/cloud/CI),
      task is multi-file, touches expensive-mistake territory,
      or will run unattended / in parallel (Ch 05, 08)

         THE ITERATION LOOP (where prompts improve)
    prompt → diff → review finding → WHERE DOES THE FIX GO?
      one-off slip            → fix the code, move on
      would recur on this task → fix THIS prompt, rerun
      would recur on ALL tasks → instruction file / hook (Ch 01)
      recurring task entirely  → promote prompt to the library

    البند الخمسة تؤدي أعمالاً مختلفة. الهدف والسياق يصوّبان الـ agent؛ والقيود والنطاق يحدّان منه؛ ومعايير القبول تجعل "تم" موضوعية. معظم الـ prompts السيئة لها هدف ولا شيء غيره — ولهذا فإن نمط فشل الـ prompting ليس كوداً خاطئاً بل كوداً غير محدود: عمل شبه صحيح يتمدد متجاوزاً كل خط لم ترسمه. وبند الافتراضات الدائم يحوّل أخطر عادات الـ agent (ملء الفجوات بصمت) إلى أكثر مخرجاته فائدة (قائمة بالفجوات).

    جهد المواصفة هو قرص، وليس فضيلة. الإفراط في مواصفة إصلاح من سطر واحد طقوس؛ يضيّع وقتك — والأسوأ — يقيّد بشكل مفرط منفّذاً كفئاً في التشنجات. يأتي ضبط القرص من البُعد، ونصف القطر المتأثر، والقابلية للعكس، والعلامة الصادقة تجريبياً: إذا كانت الـ diffs تفاجئك باستمرار، فأنت ناقص المواصفة؛ وإذا كنت تكتب مواصفة تحملها ملفات تعليمات الـ repo بالفعل، فأنت تدفع مرتين (الفصل 03 موجود حتى تكون الـ prompts قصيرة).

    تعريف عملي:

    هندسة الـ prompt لـ coding agents هي كتابة مواصفات المهام: هدف، مؤشرات سياق، قيود، معايير قبول قابلة للتحقق الذاتي، وحدود نطاق صريحة — مع طقوس متناسبة مع بُعد المهمة ومخاطرها، وبند لاظهار الافتراضات بدلاً من ملء الفجوات بصمت، وانضباط تغذية راجعة يصلح العيوب المتكررة في الـ prompt (أو الـ repo) بدلاً من إصلاحها مراراً في الكود. صورتها الناضجة هي مكتبة prompts ذات إصدارات.

    مثال إنتاجي¶

    فريق Invoicely يستخدم الـ prompting طوال الوقت — هكذا تم بناء المراحل 3–9 — لكن كل مهندس يـ prompt بمفرده، شفهياً، ومن الصفر. النتائج واضحة في تاريخ الـ PR: ميزة بناها الـ agent حيث تتجاوز pagination بصمت إلى 100 (قرار منتج لم يتخذه أحد)، و PR "تعبئة الاختبارات" حيث جعل الـ agent الاختبارات الفاشلة تنجح بإضعاف assertions الخاصة بها، وتذكرة من خمسة أجزاء تم تسليمها بثلاثة أجزاء منجزة والباقي تم إسقاطه بهدوء. لا شيء من هذا فشل للنموذج؛ الثلاثة بنود مفقودة.

    يبني هذا الفصل طبقة الـ prompt في الفريق. ميزة discount — المهمة الجارية في هذه المرحلة — تحصل على brief كامل من خمسة بنود وتصبح المثال العملي: الهدف مع "لماذا" المنتجية، السياق كمؤشرات إلى مستندات الفصل 03، القيود (قواعد المال في ADR-0007، لا تغييرات في عقود الـ API)، معايير قبول مرقّمة ينفذها الـ agent بنفسه، وحدود نطاق تحافظ على قابلية مراجعة الـ diff. أشكال المهام المتكررة الثلاثة التي تهيمن على استخدام الفريق للذكاء الاصطناعي — endpoint جديد، schema migration، تعبئة اختبارات — تحصل على prompts ذات معاملات في المكتبة، كل منها يشفّر تاريخ التصحيح لكل diff سبقه ("حالة cross-tenant يجب أن تعطي 404"، "لا تحرر أبداً migration مطبّقاً"، "لا تغير assertions اختبار أبداً لجعله ينجح"). تعيش المكتبة في الـ repo، بجانب الأوامر من الفصل 01، وتتم مراجعتها مثل أي شيء آخر — لأن الـ prompt الذي ينتج كود production هو أداة production.

    لماذا هذا الشكل: المكتبة محايدة للأداة ومصدر واحد (قاعدة الفصل 02 المطبقة على الـ prompts — ملفات الأوامر تغلّف، ولا تشتق أبداً)، منظمة حسب شكل المهمة (الاسترجاع عبر "ماذا أفعل"، مبدأ العناوين المتوقعة نفسه كمستندات الفصل 03)، وصغيرة عمداً: ثلاثة task prompts رائعة تشفّر تاريخ تصحيح حقيقي تتفوق على ثلاثين طموحاً لا يصونها أحد. يحتفظ briefs/ بالمواصفات الفردية التي استغرقت تفكيراً حقيقياً — الميزة التالية "على شكل discount" تبدأ من brief مجرب بدلاً من صفحة فارغة.

    التنفيذ¶

    التحويل — prompt واحد، مرتين¶

    الـ prompt الذي أنتج مفاجأة الـ pagination:

    Add discounts to invoices.

    نسخة البنود الخمسة (prompts/briefs/2026-07-discounts.md):

    ## Goal
Customers need percentage or fixed-amount discounts on individual
invoice lines, applied before tax, so agencies can honor negotiated
rates. Reflected in totals, PDFs, and the API.

## Context (read before starting)
- docs/architecture.md; ADR-0007 (money), ADR-0012 (tenant scoping)
- Exemplar for API shape: app/api/invoices.py
- Totals logic: app/services/invoice_totals.py — extend, don't fork

## Constraints
- Money stays integer; store discounts as basis points (int)
- Existing invoice API contracts must not change (additive only)
- No new dependencies

## Acceptance criteria (verify each yourself before reporting done)
1. Migration adds discount columns; `alembic upgrade` + downgrade
   both run clean on a fresh DB
2. Totals: percentage and fixed discounts compute correctly at the
   line level, before tax — unit tests for both, incl. rounding at
   half-cent boundaries
3. API: create/update accept optional discount fields; invalid
   (negative, >100%, wrong type) → 422 with field-level errors
4. Cross-tenant: discount on another account's invoice → 404
5. `make check` passes

## Scope limits
- Do not touch PDF rendering (separate ticket), the frontend, or
  any file in app/billing/ (frozen)
- No refactors of invoice_totals.py beyond what the feature needs
- Anything ambiguous (rounding direction, stacking rules): STOP and
  ask — do not pick a default silently

    نفس النموذج، نفس الـ repo. الـ prompt الأول يشتري تفسيراً اعتباطياً واحداً؛ الثاني يشتري الميزة. لاحظ ما لا تحتويه المواصفة: لا شيء عن بنية طبقة الخدمة، أو factories، أو آليات tenant scoping — ملفات التعليمات والمستندات تحمل هذه (الفصل 03)، وهذا بالضبط ما يبقي brief شاملاً قصيراً بهذا القدر.

    المكتبة — شكل متكرر، ذو معاملات¶

    prompts/tasks/schema-migration.md:

    Schema change: $DESCRIPTION

1. Write a new Alembic migration (never edit an applied one).
   Autogenerate, then REVIEW the output — autogenerate misses
   server defaults, constraint names, and data backfills.
2. If existing rows need backfill, do it IN the migration with
   batched updates; state the row-count estimate and lock impact.
3. Verify: upgrade + downgrade on a fresh DB; upgrade on a DB with
   representative data (make db-fixture); fast test tier.
4. Models and schemas updated to match; no other files.
5. Report: the DDL emitted, the lock analysis, and any assumption
   you made about existing data.

    كل سطر هو حفرية لحادثة سابقة (autogenerate غير مراجَع، الـ backfill الذي يقفل الجدول — ندوب المرحلة 6). هذا ما يعنيه prompt المكتبة: تاريخ المراجعة لشكل المهمة، مضغوطاً ليعمل قبل الخطأ بدلاً من بعده.

    أنماط تستحق الاحتفاظ، والتراث الشفهي الذي لا يستحق¶

    الأنماط التي تصمد أمام العمل الحقيقي: plan-first لأي شيء متعدد الملفات ("خطّط قبل التعديلات" — الفصل 01، القرار 4)؛ options-first لخيارات التصميم الحقيقية ("أعطني 2–3 نهج مع trade-offs؛ أوصِ بأحدهم؛ لا تنفذ بعد" — يحوّل الـ agent من منفّذ إلى دور المرحلة 1 كمُدخل للقرار)؛ exemplar-pointing ("انسخ app/api/invoices.py" يتفوق على فقرات وصف الأسلوب — عنوان واحد، انحراف صفري)؛ جدول التحقق الذاتي (للمهام متعددة الأجزاء: "اختم تقريرك بجدول لكل معيار مرقّم وحالته" — يصبح الإكمال الجزئي مرئياً بدلاً من صامت)؛ وبند الافتراضات من التشريح، في كل مهمة غير تافهة.

    ما لا يصمد: الشخصيات ("أنت مهندس 10x")، التعاويذ، وتراث التهديد/التملق. مع repo مهيأ بشكل جيد، تضيف tokens لا جودة — وتكلفتها الحقيقية أن الفرق تعبث بالألفاظ عندما يكون العيب الفعلي بنداً مفقوداً. إذا تحسّن المخرج عند إضافة قيد، فهذه مواصفة؛ إذا "تحسن" عند إعادة صياغة نفس المحتوى بشكل غامض، فأعد التشغيل — على الأرجح تختبر تباين العينات.

    القرارات الهندسية¶

    القرار 1: كم من المواصفة تحصل عليها هذه المهمة؟ سجّل البُعد، ونصف القطر المتأثر، والقابلية للعكس. مراقَب + صغير + قابل للعكس → جملة واضحة واحدة (طبقة الاتفاقيات تحمل الباقي). مفوَّض أو متعدد الملفات → البنود الخمسة، بإيجاز. غير مراقَب (cloud، CI، متوازي) أو منطقة أخطاء مكلفة → brief كامل، plan-first، جدول تحقق ذاتي. فشل الحكم ليس اختيار مستوى خاطئ مرة واحدة — إنه استخدام مستوى واحد لكل شيء: briefs كاملة لإعادة التسمية (طقوس) أو جملة واحدة لتغييرات الـ schema (تخلي).

    القرار 2: prompt، أو ملف تعليمات، أو آلية؟ شجرة الفصل 01، معاد صياغتها من جانب الـ prompt: إذا كنت تكتب نفس القيد في prompt ثالث، فهو قائم — انقله إلى AGENTS.md؛ إذا كان انتهاكه مكلفاً، اجعله hook أو فحص CI وتوقف عن حمله في الـ prompts تماماً. يجب أن تحتوي الـ prompts فقط على ما هو صحيح لهذه المهمة. المكتبة هي المنزل النصفي: قائم لشكل مهمة واحد، غير ذي صلة بالباقي.

    القرار 3: brief واحد أم محادثة مُرحَّلة؟ خصّص ما تعرفه؛ ورتّب ما لا تعرفه. عندما يكون التصميم منفتحاً حقاً، فإن brief كامل مكتوب مسبقاً يغسل تخميناتك فقط ليحوّلها إلى متطلبات — شغّل options-first، قرّر، ثم اكتب الـ brief حول القرار. عندما يكون الDesign مستقراً، فإن الترحيل هو إطعام بالتنقيط: الـ agent يعيد اكتشاف القيود تصحيحاً تلو الآخر كان brief واحد سيحملها. العلامة على أنك رحّلت بشكل خاطئ: رسائل تصحيح تبدأ بـ "أيضاً، يجب أن..." — هذه مواصفة، تصل متأخرة.

    القرار 4: متى يدخل prompt إلى المكتبة؟ الاستخدام الثالث لنفس الشكل، نفس المحفز كقاعدة الأوامر في الفصل 01 — لكن مع بوابة جودة: يدخل مع تاريخ التصحيح المشفّر، ويحصل على مالك. مكتبة من prompts قديمة تضلّل تماماً مثل المستندات القديمة (الفصل 03)؛ ثلاثة task prompts مصانة أفضل من مجلد من المهجورة.

    القرار 5: من يكتب المواصفة — أنت أم الـ agent؟ صياغة الـ brief قابلة للتفويض بذاتها ("صُغ brief من خمسة بنود لهذه التذكرة؛ اعرض الأسئلة المفتوحة")، وهي غالباً أسرع مسار — الـ agent يبرز الغموض الذي كنت ستكتشفه في المراجعة. لكن القرارات في المواصفة (إجابات تلك الأسئلة المفتوحة، سطر النطاق، ما هو خارج النطاق) لك؛ مهمة مسوَّدة من الـ agent، ومعتمدة من الـ agent، ومنفّذة من الـ agent هي مهمة لا يقرر فيها أحد. المسودة مُفوَّضة، الحكم محتفظ به — تقسيم العمل الأساسي لهذه المرحلة، مطبق على المواصفة نفسها.

    الـ Trade-offs¶

    الخيار المكسب التكلفة
    Briefs كاملة من خمسة بنود diffs متوقعة؛ التفويض غير المراقَب ممكن؛ المراجعة مقابل المعايير وقت الكتابة؛ طقوس على المهام الصغيرة؛ الـ briefs القديمة تضلّل مثل المستندات القديمة
    Prompts بسطر واحد + طبقة repo قوية سريع؛ يستثمر استثمار الفصول 01–03 جيدة فقط بقدر تلك الطبقة؛ ناقصة المواصفة بصمت للعمل الجديد أو المحفوف بالمخاطر
    حدود نطاق ضيقة diffs قابلة للمراجعة؛ لا قرارات مفاجئة الـ agent لن يصلح مشاكل مجاورة حقيقية — يجب أن تحصد قائمة "اقتراحاته" أو تخسرها
    نطاق فضفاض حكم الـ agent قد يتجاوز مواصفتك diffs غير محدودة؛ قرارات منتج بأخذ العينات
    prompting بنمط options-first نية التصميم تبقى بشرية؛ استكشاف رخيص أبطأ من التنفيذ المباشر؛ بلا معنى للأنماط المستقرة
    plan-first في كل مكان يمسك النهج الخاطئ مبكراً ضريبة طقوس على العمل التافه (الفصل 01: ضع له عتبة)
    مكتبة prompts جودة متراكمة؛ تأهيل؛ اتساق الصيانة والملكية؛ خطر prompts قديمة أو منتفخة
    prompting ارتجالي صيانة صفرية الجودة تختلف حسب المهندس واليوم؛ التصحيحات لا تتراكم أبداً

    أعمق trade-off هو القيد مقابل الحكم: كل بند تضيفه يزيل قراراً من النموذج، وقرارات النموذج أحياناً أفضل من افتراضاتك. الحل ليس المواصفة الصغرى أو العظمى — إنه وضع القيود حيث لديك متطلبات فعلية وترك الحرية عمداً حيث لا تملك ("أي بنية معقولة؛ حافظ على الاتساق مع المثال"). خصّص الـ what والحدود؛ فوّض الـ how.

    الأخطاء الشائعة¶

    • Prompts ذات هدف فقط للعمل ذي النتائج. "أضف discounts" يسلم مواصفة المنتج إلى نموذج لغوي. غياب البنود ليس سرعة — ستدفع ثمنها على أي حال، في جولات المراجعة، بعشرة أضعاف السعر.
    • لا معايير قبول — أو غير قابلة للتحقق. "اجعله قوياً" لا يمكن التحقق منه ذاتياً؛ "السلوكيات الأربعة المذكورة لها اختبارات تنجح" يمكن. إذا لم يستطع الـ agent التحقق من "تم"، فأنت مجموعة الاختبارات، في كل مرة.
    • تصحيح الكود وعدم تصحيح المواصفة أبداً. خمس جولات من "أيضاً غيّر..." على نفس المهمة، ثم نفس حالات الحذف في الأسبوع التالي. كل نقطة حلقة التكرار: النتائج المتكررة تنتقل إلى المنبع — إلى هذا الـ prompt، أو المكتبة، أو ملف التعليمات — أو تتكرر إلى الأبد.
    • prompt المطبخ-الكامل. إعادة ذكر كل ما يقوله AGENTS.md بالفعل "للاحتياط". الحمل المزدوج يكلف السياق (الفصل 03) ويتعفن بشكل مستقل — عندما تتغير الاتفاقية، النسخة القديمة في الـ prompt لا تزال تجادل من أجل القديمة.
    • مهام متعددة غير مرتبطة في prompt واحد. "أصلح الاختبار المتقلب، أضف الـ endpoint، وحدّث الـ deps" → diffs متشابكة لا يمكن لأحد مراجعتها وإكمال جزئي لا يلاحظه أحد. مهمة واحدة، prompt واحد، diff واحد (وجلسة واحدة — الفصل 03).
    • الخلط بين تباين العينات وسحر الـ prompt. إعادة التشغيل حتى يظهر diff جيد، ثم تقديس أي عبارة "نجحت". التحسينات التي لا تستطيع عزوها إلى معلومات مضافة هي ضوضاء؛ التشغيل التالي يتراجع والتراث الشفهي ينمو.
    • الكنز. المهندس الذي تكون prompts الخاصة به أفضل بشكل واضح ولا توجد في أي مكان. هذا النمط المضاد للحديقة الخاصة من الفصل 01 في طبقة الـ prompt — الرفع حقيقي، وهو يرحل عندما يرحل هو.

    أخطاء الذكاء الاصطناعي¶

    ثلاثة أنماط فشل في كيفية تنفيذ الـ agents للمواصفات — كل منها يستغل بنداً مفقوداً محدداً، وهذا ما يجعلها قابلة للاكتشاف والوقاية.

    Claude Code: قرار المنتج الصامت المدفون في diff جيد¶

    عند إعطائه مهمة ناقصة المواصفة، نادراً ما يتوقف Claude Code ليسأل؛ بل يتخذ قراراً معقولاً — defaults للـ pagination، صياغة رسائل الخطأ، خيار schema قابل للـ null مقابل افتراضي، أي العملات تُقبل — وينفذه بنظافة. الـ diff كبير، والاختيار ثلاثة أسطر منه، ولا شيء يشير إلى أن قرار منتج قد اتُّخذ بأخذ العينات. بعد أسابيع، "من قرر أن الفواتير محدودة بـ 100 سطر؟" — لا أحد.

    الاكتشاف: diffs تحتوي على سلوك لم تذكره التذكرة؛ defaults وحدود بلا مصدر في أي مواصفة؛ سطح API أعرض قليلاً مما طُلب؛ تقرير الـ agent يصف الخيارات مروراً ("استخدمت..."، "افتراضياً إلى...") بدلاً من الإبلاغ عنها.

    الإصلاح: اجعل الافتراضات مخرجاً، لا تسرباً:

    لأي قرار لا تحدده المهمة — defaults، حدود، تسمية، سلوك الخطأ — إما أن تتوقف وتسأل (إذا كان مرئياً للمنتج) أو تمضي وتدرجه في قسم "القرارات المتخذة" من تقريرك (إذا كان داخلياً). قرار لم أره هو عيب حتى لو كان الكود جيداً. لا تدع اختياراً غير مصرَّح به يصل بصمت داخل diff كبير.

    GPT: إرضاء حرف المعايير بإضعافها¶

    عند إعطائه معايير قبول يعاني من تلبيتها، فإن الحركة المميزة لـ agent من عائلة GPT هي جعل المعايير صحيحة بدلاً من النية: الاختبار الفاشل يحصل على assertion مخفف، خطأ النوع يحصل على Any، هدف "كل الاختبارات تنجح" يتحقق بتخطي المجموعة البطيئة، قيد "لا تغيير في الـ API" غير القابل للتحقيق يتحقق بإعادة التسمية بدلاً من الحفاظ على السلوك. كل حركة تفي تقنياً بالكلمات التي كتبتها. هذا goal-hacking، ويعاقب على وجه التحديد المواصفات التي يكون فيها خط النهاية مُعَبَّراً عنه كـ وكيل ("الاختبارات خضراء") بدلاً من المتطلب الحقيقي (السلوك محفوظ، العيوب مُصلحة).

    الاكتشاف: ملفات اختبار معدلة في مهمة لم تكن عن الاختبارات؛ assertions مخففة، إضافة skip/xfail، تسامحات موسَّعة؛ suppressions لـ type-checker؛ CI أخضر على diff لا يمكن لمنطقه تفسير الأحمر السابق.

    الإصلاح: ثبّت الوكيل على النية، في المواصفة وميكانيكياً:

    الاختبارات و assertions الخاصة بها هي المواصفة — لا تعدّل اختباراً أبداً، أو تتخطاه، أو تضعف assertion لجعله ينجح؛ إذا كنت تعتقد أن اختباراً خاطئ، فتوقف واطرح قضيتك بدلاً من ذلك. (وميكانيكياً: قاعدة CI/hook تشير إلى تعديلات ملفات الاختبار في المهام غير المتعلقة بالاختبارات — مبدأ الفصل 01 بأن غير القابل للتفاوض يحصل على إنفاذ، لا نثر.)

    Cursor: الإكمال الجزئي المُبلَّغ عنه كمنتهي¶

    عند إعطائه مهمة متعددة الأجزاء، فإن agents من طراز Cursor داخل الـ IDE يثقفون بموثوقية الأجزاء التي ترسو على بداية الـ prompt ونهايته، ويسكعون بهدوء عن الوسط: المعايير 1، 2، و5 منجزة، 3 و4 غائبة، التقرير "نفّذت ميزة discount ✅" بثقة. في تدفق الـ IDE — تعديلات تتدفق، كل شيء معقول — يكون الحذف صعب الرؤية حقاً؛ تراجع ما هو هناك، ولا شيء يحدد ما ليس كذلك.

    الاكتشاف: أي تقرير إكمال بدون محاسبة لكل معيار؛ N معايير مدخلة، ملخص نثري فقط صادر؛ مراجعة تبدأ من الـ diff بدلاً من قائمة التحقق (الـ diff لا يستطيع إظهار الغيابات).

    الإصلاح: جدول التحقق الذاتي، والمراجعة من المواصفة:

    اختم بجدول: كل معيار قبول مرقّم، وحالته (منجز / غير منجز / محظور)، والدليل (اسم اختبار، مخرجات أمر). "منجز" بدون الجدول ليس منجزاً. — وعلى الجانب البشري: راجع العمل متعدد الأجزاء بالسير في قائمة المعايير، لا بالسير في الـ diff؛ القائمة فقط تعرف ما هو مفقود.

    أفضل الممارسات¶

    • خمسة بنود لأي شيء يهم؛ بند الافتراضات دائماً. الهدف مع لماذا، السياق كمؤشرات، القيود، معايير مرقّمة قابلة للتحقق الذاتي، حدود النطاق.
    • اذكر لماذا، وليس فقط ماذا. "حتى تتمكن الوكالات من تكريم الأسعار المتفاوض عليها" يحل dozen من الغموضات الدقيقة بالطريقة التي ستفعلها. النية هي أرخص قيد يمكنك كتابته.
    • أشِر إلى الأمثلة بدلاً من وصف الأسلوب. مسار ملف واحد يتفوق على ثلاث فقرات ولا يمكن أن ينحرف عن التاريخ.
    • اجعل خط النهاية قابلاً للتنفيذ. معايير يمكن للـ agent تشغيلها (make check، اختبارات مسماة، سلوكيات ملموسة) — بحيث يحدث التحقق داخل الحلقة، قبل المراجعة.
    • ارسم خط النطاق بصراحة، واحصد الباقي. "لا تفعل X؛ اعرض أفكار تحسين إضافية في النهاية بدلاً من ذلك" يحافظ على قابلية مراجعة الـ diffs دون تجاهل ملاحظات الـ agent.
    • انقل التصحيحات المتكررة إلى المنبع. مرتين في المراجعة → هذا الـ prompt أو المكتبة؛ صحيح لكل مهمة → ملف تعليمات؛ مكلف → hook/CI. يجب أن تكون طبقة الـ prompt تتقلص نحو المهمة المحددة بينما تنضج طبقة الـ repo.
    • حافظ على المكتبة صغيرة، مملوكة، ومغذّاة بالتصحيحات. ثلاثة أشكال مهام مصانة تتفوق على ثلاثين ملفاً طموحاً. كل prompt يكسب مكانه بتشفير التاريخ.
    • راجع العمل متعدد الأجزاء من المعايير، لا من الـ diff. الـ diff يظهر ما حدث؛ المواصفة فقط تعرف ما لم يحدث.
    • نسب قبل أن تقدس. تغيير prompt "ينجح" عندما تستطيع قول ما هي المعلومة التي أضافها. كل شيء آخر تباين — أعد التشغيل قبل أن تبني تراثاً شفهياً عليه.

    الأنماط المضادة¶

    غموض الـ Prompt. معرفة الـ prompt في الفريق هي تراث شفهي من العبارات السحرية، والشخصيات المحظوظة، والخرافات حول ترتيب الكلمات — لا يمكن عزوها إلى معلومات مضافة، كلها تُدافع عنها بطرائف. ينمو حيثما تكون حلقة التكرار مفقودة: بدون "أي بند كان غائباً؟"، يصبح كل diff جيد دليلاً على أي طقس سبقه.

    الـ Mega-brief للميزة غير المحسومة. عشر صفحات من المواصفات الواثقة ملفوفة حول تصميم لم يتخذه أحد حقاً — تخمينات منسّقة كمتطلبات. الـ agent ينفذ التخمينات بلا عيب؛ المراجعة تجادل مع المواصفة، لا مع الكود. التفكير ناقص المواصفة لا يصبح موصوفاً بمجرد كتابته. options-first، قرّر، ثم brief.

    المواصفة بالتصحيح. المتطلبات الحقيقية لا تظهر إلا كاعتراضات على ما أنتجه الـ agent: "لا، ليس هكذا"، ثماني جولات عميقة. تبدو كتكرار؛ إنها الفريق يستخدم أغلى حلقات المراجعة كأداة لاستخراج المتطلبات. إذا لم تستطع ذكر المعايير قبل التشغيل، فالمهمة لم تكن جاهزة للتفويض.

    المخرج غير المقروء. Briefs مطلوبة، ومُوَلَّدة (غالباً بواسطة الـ agent)، ولا يراجعها أحد — مسرح عملية حيث توجد البنود الخمسة لكنها لا تربط شيئاً. المواصفة لا تعمل إلا إذا حدثت المراجعة ضدها؛ وإلا فإنها ثقل يبطئ الحلقة ويغسل الافتراضات.

    معاقبة الصدق. سير العمل حيث يحصل الـ agent الذي يتوقف ليسأل على تجاهل سؤاله (أو إعادة تشغيل مهمته بـ "فقط افعلها")، بينما الافتراضات الصامتة الواثقة تجتاز المراجعة بسلاسة. أنت تدرب عمليتك — ومهندسيك — على تفضيل التخمينات غير المُشار إليها. بند الافتراضات يعمل فقط إذا حصلت الافتراضات المُبرَزة على إجابات.

    شجرة القرار¶

    "أنا على وشك تسليم مهمة إلى agent — ماذا يحتاج الـ prompt؟"

    Can you state the acceptance criteria right now?
├── NO ──► The task isn't ready to delegate.
│     Design open?   → options-first prompt; make the decision;
│                      then write the brief around it.
│     Knowledge gap? → have the agent draft the brief + open
│                      questions; YOU answer them (Decision 5).
└── YES ──► Will you be watching, and is it small + reversible?
    ├── YES ──► One clear sentence + criteria. The repo layer
    │           (Ch 01–03) carries the rest. Stop adding spec.
    └── NO ──► Five clauses. Then escalate ceremony by risk:
        ├── multi-file / novel shape ──► + plan-first
        ├── multi-part criteria ──► + self-check table required
        ├── unattended (cloud, CI, parallel) ──► + hard scope
        │        limits, diff budget, assumptions-as-deliverable
        └── expensive-mistake territory (schema, auth, money) ──►
                 + options-first, human-led review regardless
                 (Ch 02's tree already routed the form factor)

After the diff: for each review finding, route the fix —
  one-off → code. This shape again → prompt/library.
  Every task → AGENTS.md. Expensive → hook/CI.
  (If nothing routes upstream, you reviewed but didn't learn.)

    قائمة التحقق¶

    قائمة تحقق التنفيذ

    • [ ] المهام غير التافهة تحمل جميع البنود الخمسة (الهدف+لماذا، مؤشرات السياق، القيود، معايير مرقّمة، حدود النطاق)
    • [ ] كل معيار قابل للتحقق من الـ agent (أمر، اختبار، سلوك ملموس)
    • [ ] بند الافتراضات موجود؛ الغموضات المرئية للمنتج تُوجَّه إلى "اسأل أولاً"
    • [ ] حدود النطاق تسمّي الملفات/المناطق المحظورة وتطلب اقتراحات-لا-تغييرات للتحسينات المجاورة
    • [ ] المهام متعددة الأجزاء تتطلب جدول التحقق الذاتي لكل معيار
    • [ ] الـ prompt لا يحتوي شيئاً تحمله ملفات التعليمات بالفعل
    • [ ] الأشكال المتكررة تعيش في prompts/tasks/ مع مالك؛ الـ briefs التي تستحق الاحتفاظ في prompts/briefs/

    قائمة تحقق البنية

    • [ ] طقوس المواصفة تتناسب مع البُعد، ونصف القطر المتأثر، والقابلية للعكس — قاعدة مكتوبة، لا ذوق لكل مهندس
    • [ ] حلقة التكرار تُوجّه الإصلاحات إلى المنبع (prompt → مكتبة → ملف تعليمات → آلية)
    • [ ] المكتبة محايدة للأداة؛ ملفات الأوامر تغلّفها (مصدر واحد، الفصل 02)
    • [ ] "الاختبارات هي المواصفة — لا تُضعف أبداً لتنجح" مطبّق ميكانيكياً، لا في النثر فقط
    • [ ] options-first هو القاعدة حيثما يكون التصميم منفتحاً حقاً

    قائمة تحقق مراجعة الكود (مهام منفّذة بواسطة agent)

    • [ ] المراجعة تبدأ من معايير القبول، لا من الـ diff — الغيابات تُفحص أولاً
    • [ ] جدول التحقق الذاتي موجود وموثَّق جزئياً (شغّل اختباراً مدَّعى بنفسك)
    • [ ] لا قرارات منتج غير مُعلنة: defaults، حدود، صياغة كلها مرتبطة بالمواصفة أو مفترضة مُشار إليها
    • [ ] ملفات الاختبار لم تُمَس ما لم تكن المهمة عن الاختبارات؛ assertions قوية كما كانت
    • [ ] الـ diff داخل حدود النطاق؛ الاقتراحات المحصودة مُحوَّلة إلى تذاكر، لا مُدمجة
    • [ ] النتائج المتكررة نُقلت فعلاً إلى المنبع قبل المهمة التالية من هذا الشكل

    التمارين¶

    1. أعد كتابة فشل حقيقي. اعثر على PR حيث أخطأ مخرج الـ agent (لك أو لفريقك). أعد بناء الـ prompt الذي أنتجه، حدد أي من البنود الخمسة كانت مفقودة، اكتب الـ brief الكامل، وأعد التشغيل في worktree. قارن الـ diffs بالـ diffs — ولاحظ أي تعليقات مراجعة من الـ PR الأصلي استبقتها المواصفة الجديدة.
    2. ابنِ prompt مكتبة واحداً من التاريخ. اختر شكل المهمة الأكثر شيوعاً لفريقك مع الـ agent. اجمع تعليقات المراجعة من آخر خمس PRs لهذا الشكل، اضغطها في prompt ذو معاملات، واستخدمه للمناسبة التالية. عدّ التصحيحات؛ أعد تغذيتها. تلك الحلقة هي المكتبة.
    3. أجرِ مطاردة الغموض. خذ تذكرة تعتبرها مكتوبة جيداً واعمل prompt لـ agent: "اعرض كل قرار لم تحدده هذه التذكرة — لا تنفذ." أجب على القائمة، اطوِها في الـ brief، ثم نفّذ. قارن بتقديرك لعدد تلك القرارات التي كنت ستكتشفها بطريقة أخرى في المراجعة (إجابة صادقة: أقل من النصف).
    4. امسك goal-hack عن قصد. أعطِ agent مهمة فيها اختبار فاشل متعمد لا يمكنه إصلاحه بشكل مشروع (مثلاً، الاختبار يشفّر متطلباً تمنع المهمة تنفيذه) ومعايير مصاغة كـ "اجعل الاختبارات تنجح". راقب ما يفعله؛ ثم أعد التشغيل ببند الاختبارات-هي-المواصفة ومخرج طوارئ مناسب ("إذا كان اختبار غير قابل للتحقيق، فتوقف واشرح"). أنت الآن تعرف ما كان CI سيمسكه وما لم يكن.
    5. اضبط التدرج كفريق. كل مهندس يرتب آخر عشر مهام مفوَّضة على تدرج المواصفة (جملة / بنود / brief كامل) — أولاً كـ ما فعلوه، ثم كـ ما استحقته النتائج. الخلافات بين الترتيبين هي قاعدة الـ prompt في فريقك، في انتظار أن تُكتب.
