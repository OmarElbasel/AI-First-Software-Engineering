# قائمة مراجعة الكود

نفّذ المراجعة على كامل الـdiff، وليس على الملخص. الهدف من المراجعة ليس
اكتشاف ملاحظات أسلوبية — بل إيقاف العيوب والثغرات الأمنية والتصميمات
صعبة الصيانة قبل دمجها، حيث يكون إصلاحها أرخص بما يتراوح بين 10 و100 مرة
مقارنةً بإصلاحها في بيئة الإنتاج.

مراجعة الكود المُولَّد بالذكاء الاصطناعي تستخدم نفس القائمة بالإضافة إلى
القسم المخصص في النهاية. الكود المُولَّد يستحق تدقيقًا *أكبر*، لا أقل:
فهو يبدو معقولاً بطبيعته، مما يُفقدك الغريزة التي تقول "هذا يبدو خاطئًا"
والتي تلتقط الأخطاء البشرية.

## الصحة

- [ ] التغيير يفعل ما يدّعيه وصف الـPR — تم التحقق من ذلك مقابل الـdiff، وليس العنوان
- [ ] مسارات الفشل مُعالَجة: المهلات الزمنية، النتائج الفارغة، المدخلات غير الصالحة، الفشل الجزئي
- [ ] الاستدعاءات الخارجية (المدفوعات، البريد الإلكتروني، webhooks) آمنة لإعادة المحاولة — idempotent أو مُزال عنها التكرار
- [ ] الوصول المتزامن مُؤخَذ في الاعتبار حين تكون الحالة مشتركة (الإرسال المزدوج double-submit، سباق على قراءة-تعديل-كتابة)
- [ ] الحالات الحدية عند الحدود مُعالَجة: صفر، واحد، كثير، الحد الأقصى؛ السلاسل الفارغة؛ المنطقة الزمنية والترميز

## التصميم

- [ ] التغيير يقع في الطبقة الصحيحة (لا منطق أعمال داخل معالجات المسار route handlers، ولا اهتمامات HTTP داخل الخدمات services)
- [ ] يتبع الأنماط القائمة في قاعدة الكود — أو يوضح الـPR سبب انحرافه عنها
- [ ] لا تجريد جديد يخدم مستدعيًا واحدًا ("قد نحتاجه لاحقًا" ليس مستدعيًا)
- [ ] الواجهات العامة في حدودها الدنيا؛ لا يُكشف شيء لا ينبغي للمستدعين الاعتماد عليه

## الأمان

- [ ] جميع مدخلات المستخدم يتم التحقق من صحتها عند الحدود؛ الوصول إلى قاعدة البيانات مُعامَل (parameterized)
- [ ] يتم التحقق من التفويض (authorization) عند كل endpoint واستعلام جديد — وليس فقط المصادقة (authentication)
- [ ] لا توجد أسرار أو رموز (tokens) أو بيانات اعتماد في الكود أو الإعدادات أو السجلات أو تجهيزات الاختبار
- [ ] البيانات الحساسة (كلمات المرور، الرموز، PII) لا تظهر أبدًا في مخرجات السجلات أو رسائل الخطأ

## الاختبارات

- [ ] للسلوك الجديد اختبارات ستفشل إذا تعطّل هذا السلوك
- [ ] يتم اختبار أوضاع الفشل، وليس فقط المسار السعيد
- [ ] تتحقق الاختبارات من النتائج، وليس من تفاصيل التنفيذ (فهي تصمد أمام إعادة الهيكلة)
- [ ] لم يتم إضعاف أي اختبار أو تخطيه أو حذفه لجعل المجموعة تنجح

## العمليات

- [ ] هجرات المخطط (schema migrations) متوافقة مع الإصدارات السابقة، أو أن ترتيب النشر (deploy) موثّق
- [ ] أوضاع الفشل الجديدة قابلة للملاحظة: مُسجَّلة بسياق، أو ظاهرة في المقاييس (metrics)
- [ ] الأداء محدود بحدود واضحة: لا استعلام بلا فهرس على جدول متنامٍ، ولا N+1 مُستحدث، ولا حلقة غير محدودة على بيانات خارجية

## الكود المُولَّد بالذكاء الاصطناعي

كل بند هو نمط فشل معروف في Claude Code وCodex/GPT وCursor.

- [ ] **واجهات API موهومة** — كل دالة ومُعامل ومفتاح إعداد غير مألوف تم التحقق منه مقابل التوثيق الحقيقي، ولم يُفترض وجوده
- [ ] **انحياز المسار السعيد** — الاختبارات المُولَّدة قُرئت، ولم تُجرَ فحسب؛ فهي عادةً لا تُختبَر سوى حالة النجاح
- [ ] **تمدد النطاق الصامت** — الـdiff لا يحتوي على شيء يتجاوز ما طُلب (لا إعادة هيكلة إضافية، ولا إعادة تسمية لملفات، ولا endpoints إضافية، ولا كود غير ذي صلة تم "تحسينه")
- [ ] **متطلبات مُخترَعة** — السلوك غير الموجود في التذكرة (القيم الافتراضية، عدد المحاولات، قواعد التحقق) اختير عمدًا، ولم يُقبل افتراضيًا
- [ ] **أنماط قديمة** — الكود المُولَّد يستخدم اصطلاحات قاعدة الكود الحالية، وليس أسلوبًا أقدم للإطار مأخوذًا من بيانات التدريب
- [ ] **معالجة أخطاء تبدو واثقة لكنها ليست كذلك** — كتل `try/except` تعالج الخطأ فعليًا، لا أن تسجله فقط وتستمر في حالة فاسدة
- [ ] **التكرار على حساب إعادة الاستخدام** — المساعد لم يُعد تنفيذ helper أو model أو validation موجود فعلًا في قاعدة الكود
- [ ] **الإعدادات الافتراضية الأمنية** — إعدادات auth وCORS وcookie وcrypto المُولَّدة تم التحقق منها صراحةً؛ فالمساعدون غالبًا يُنتجون إعدادات افتراضية متساهلة تعمل في بيئة التطوير
